## 核心概念

### 1. TLC 的內部運作

TLC（TLA+ model checker）是 **explicit-state**：
1. 從 `Init` 算出所有 initial states
2. 對每個 state $s$，apply `Next` 算出所有 successor states
3. BFS 整個 reachable graph
4. 每訪問新 state 檢查 INVARIANTS
5. 對 liveness 用 **tableau-based 演算法** 找 fairness-violating cycles

實際工程考量：
- **State storage**：TLC 用 fingerprint hash（64-bit）省記憶體；fingerprint collision rate 對 ~$10^9$ state 仍可接受
- **Multi-core**：TLC 支援 multi-worker BFS（disk-based queue）
- **Distributed TLC**：跨多機器（適合 ~$10^{12}$ state，但 setup 複雜）
- **State constraint**：用 `CONSTRAINT MyBound` 在 model config 限制 state space 不展開

實務上 TLC 對 ~$10^7$-$10^8$ state 在 desktop 跑得動，超過要 cluster。

### 2. State space explosion 的工程緩解

#### 緩解 1: Symmetry reduction

若 spec 對 process identity 對稱（典型 N 個對等 client），TLC 在 model config 寫:
```
SYMMETRY Permutations
```

對 Set `Processes = {p1, p2, p3}` 自動套用 symmetry，將 N! 倍 reduction。

#### 緩解 2: View abstraction

```
VIEW MyView
```
`MyView` 定義一個 function 把 concrete state 映到 abstract value。TLC 在 abstract state space 上 explore。

#### 緩解 3: State constraint

```
CONSTRAINT NotTooMany
```
`NotTooMany == counter < 10`。TLC 不展開 counter >= 10 的 state。但**這只能 sound bug-finding，不能 sound full verification**——constraint 外的 state 可能有 bug 沒被檢測。

#### 緩解 4: Action constraint
```
ACTION_CONSTRAINT NoNewConnection
```
`NoNewConnection == \neg (action_label = "NewConnection")`。限制 transition 不展開特定 action。

### 3. Apalache：symbolic model checker

**Apalache** (https://apalache-mc.org/) 是 ETH / Informal Systems 開發，TLA+ 語法 + symbolic backend。

關鍵差異:
- TLC：explicit-state，跑 finite model
- Apalache：用 SMT solver (Z3) 對 **bounded-depth path** 做 symbolic execution

優勢:
- **無限 data type**：unbounded integers、unbounded sequences
- 處理 spec 含有 `\E x \in Int: ...` 之類無限量化
- 找 inductive invariant 更快（不需 enumerate）

劣勢:
- 只能 **bounded-depth**：通常 check 「length-k path 內無 violation」，非完整 reachability
- TLA+ 語法 subset：某些 operators 不支援
- 對 enum-rich state（如 message-type tag）速度可能不如 TLC

**對我們協議：Apalache 適合證 unbounded integer arithmetic invariant**（如 「packet number ≤ MAX_VARINT」、「flow control window 任何時候 ≥ 0」）。

### 4. Inductive invariants

寫好 invariant 後，TLC enumerate reachable state 確認。但對**極大 state space**，更好的方式是寫 **inductive invariant** + 對應 proof。

**Inductive invariant** 定義：

$$\text{IInv} \text{ is inductive iff } \text{Init} \Rightarrow \text{IInv} \text{ AND } \text{IInv} \land \text{Next} \Rightarrow \text{IInv}'$$

如果你能 prove IInv inductive，再 prove `IInv => Inv` (你想要的 invariant)，就完整 verify 了——**不需 explicit state enumeration**。

對 unbounded systems (任意數量 client / unlimited packet number)，inductive invariant 是唯一可行解。

工具:
- **TLAPS**: TLA+ proof system, 手動寫 proof 證 `Init => IInv` + `IInv /\ Next => IInv'`
- **Apalache**: 自動或半自動 inductive invariant generation（仍 experimental）

### 5. Refinement: 抽象 ⊑ 具體

**Concept**: high-level spec $H$ 跟 low-level spec $L$，若 $L \Rightarrow H$（在 temporal logic 意義下），稱 $L$ refines $H$。

Refinement 用途:
- **Spec layered design**：先寫 abstract spec 證 high-level property（如 「consensus 達成」），再寫 detailed spec 證 implementation steps 滿足 abstract
- **TLA+ refinement mapping**: 用 `INSTANCE` keyword 寫對應關係

例子（QUIC stream layered design）:

```tla
\* High-level: stream delivers bytes in order
HighSpec ==
    /\ HighInit
    /\ [][HighNext]_HighVars

\* Low-level: stream frame with offset, sorted by frameSorter
LowSpec ==
    /\ LowInit
    /\ [][LowNext]_LowVars

\* Refinement mapping: 把 LowSpec 的 frame queue + sorted output 映到 HighSpec 的 delivered bytes
HighStreamBytes(low) == [delivered |-> SortByOffsetThenConcat(low.frames)]

\* Prove:
THEOREM LowSpec => HighSpec
```

對我們協議：
- Abstract spec: 「stream 是 reliable ordered byte channel」
- Concrete spec: 「stream 由 STREAM frame (with offset) 組成，frame sorter 重組 」
- Refinement：證 concrete 滿足 abstract

### 6. Liveness 與 fairness 的細節

5.2 提過 `WF_vars(A)` 是 weak fairness。完整 definition:

- **Weak fairness on action $A$** (WF): If $A$ is **continuously enabled** from some point onwards, then $A$ must eventually be executed
- **Strong fairness on action $A$** (SF): If $A$ is **enabled infinitely often**, then $A$ must be executed infinitely often

對 protocol design:
- **WF on Send**: 「sender 永遠願意送，最終 send 動作會發生」
- **SF on Receive**: 「receiver 即使偶爾忙也最終 process incoming message」

**沒有 fairness 假設, liveness 通常 unprovable**——system 可以「永遠不做任何事」。

對 adversarial setting (security protocol):
- **不對 attacker 加 fairness**：attacker 可以拒絕 forward message
- **對 honest party 加 fairness**：honest party 最終會 send/receive

### 7. 實戰 1: QUIC packet number space invariant

對應 RFC 9000 §12.3，三個 packet number space：

```tla
---------------------------- MODULE QUICPnSpace ----------------------------
EXTENDS Naturals, Sequences

CONSTANT MaxPn  \* upper bound for model checking (real spec uses 2^62)

VARIABLES
    sent_initial,      \* sequence of sent packet numbers in Initial space
    sent_handshake,    \* sequence of sent packet numbers in Handshake space
    sent_application,  \* sequence of sent packet numbers in Application space
    last_received_initial,   \* highest received pn in Initial
    last_received_handshake,
    last_received_application

Init ==
    /\ sent_initial = << >>
    /\ sent_handshake = << >>
    /\ sent_application = << >>
    /\ last_received_initial = -1
    /\ last_received_handshake = -1
    /\ last_received_application = -1

\* Sender: each space's packet number strictly monotonic
SendPacket(space) ==
    LET seq == CASE space = "initial" -> sent_initial
                 [] space = "handshake" -> sent_handshake
                 [] space = "application" -> sent_application
        next_pn == IF Len(seq) = 0 THEN 0 ELSE seq[Len(seq)] + 1
    IN
    /\ next_pn <= MaxPn
    /\ CASE space = "initial" ->
         /\ sent_initial' = Append(sent_initial, next_pn)
         /\ UNCHANGED << sent_handshake, sent_application,
                          last_received_initial, last_received_handshake,
                          last_received_application >>
        [] space = "handshake" -> ...

\* Receiver: track highest seen pn per space
ReceivePacket(space, pn) ==
    /\ CASE space = "initial" ->
         /\ pn > last_received_initial
         /\ last_received_initial' = pn
         /\ UNCHANGED << sent_initial, sent_handshake, sent_application,
                          last_received_handshake, last_received_application >>
        [] ...

Next ==
    \/ \E s \in {"initial", "handshake", "application"}: SendPacket(s)
    \/ \E s \in {"initial", "handshake", "application"}, p \in 0..MaxPn:
         ReceivePacket(s, p)

vars == << sent_initial, sent_handshake, sent_application,
            last_received_initial, last_received_handshake,
            last_received_application >>

Spec == Init /\ [][Next]_vars

\* Invariants
PnMonotonic ==
    /\ \A i \in 1..(Len(sent_initial)-1): sent_initial[i] < sent_initial[i+1]
    /\ \A i \in 1..(Len(sent_handshake)-1): sent_handshake[i] < sent_handshake[i+1]
    /\ \A i \in 1..(Len(sent_application)-1): sent_application[i] < sent_application[i+1]

PnDistinctSpaces ==
    \* No packet number 同時 in two spaces — vacuously hold by construction
    TRUE  \* explicit by structure; verify via model

PnReceivedMonotonic ==
    last_received_initial >= -1
    /\ last_received_handshake >= -1
    /\ last_received_application >= -1
=======================================================================
```

跑 TLC 設 `MaxPn = 5`：能很快驗證 `PnMonotonic`。

對 real-world `MaxPn = 2^62`，要用 Apalache 證 inductive invariant 不依賴具體 bound。

### 8. 實戰 2: QUIC connection migration safety

對 RFC 9000 §9 path validation:

```tla
VARIABLES
    server_path,        \* current path (peer addr) server thinks is active
    pending_challenge,  \* outstanding PATH_CHALLENGE on new path
    paths_validated     \* set of paths that passed validation

PathValidationStart(new_path) ==
    /\ new_path \notin paths_validated
    /\ pending_challenge' = [path |-> new_path,
                              data |-> RandomBytes()]
    /\ UNCHANGED << server_path, paths_validated >>

PathValidationSuccess(response) ==
    /\ pending_challenge # NULL
    /\ response.data = pending_challenge.data
    /\ paths_validated' = paths_validated \union {pending_challenge.path}
    /\ pending_challenge' = NULL
    /\ UNCHANGED << server_path >>

\* Safety: server 不對 new path 解除 anti-amplification 直到 validation 完成
NeverSendUnvalidated(amplification_limit_state) ==
    LET amp_active == amplification_limit_state.path \notin paths_validated
    IN \neg (amp_active /\ amplification_limit_state.exceeded)

\* Invariant: 同一個 path 不會 simultaneously pending + validated
NotBothPendingAndValidated ==
    pending_challenge # NULL =>
        pending_challenge.path \notin paths_validated
```

對應 RFC 9000 §8.2.4「Validation Failures」、§9.3「Use of Tokens for Address Validation」。

### 9. 實戰 3: Anti-replay window invariant

對 TLS 1.3 0-RTT anti-replay：

```tla
VARIABLES
    accepted_ticket_age,  \* set of (ticket_id, observed_age) pairs
    window_low, window_high  \* time window edges

Init ==
    /\ accepted_ticket_age = {}
    /\ window_low = 0
    /\ window_high = 30  \* RFC 8446 §8 example window

AcceptTicket(ticket_id, claimed_age) ==
    /\ claimed_age >= window_low
    /\ claimed_age <= window_high
    /\ <<ticket_id, claimed_age>> \notin accepted_ticket_age  \* 不重複
    /\ accepted_ticket_age' = accepted_ticket_age \union {<<ticket_id, claimed_age>>}
    /\ UNCHANGED << window_low, window_high >>

\* Invariant: 任何 ticket 在 window 內 accept 唯一次
UniqueAcceptance ==
    \A t1, t2 \in accepted_ticket_age:
        (t1[1] = t2[1] /\ t1 # t2) => FALSE   \* 即 ticket_id same → must be same pair
```

對應 RFC 8446 §8.2 Mechanism #3 (freshness)。

### 10. AWS 使用案例

Newcombe et al. CACM 2015 報告：AWS 對 **S3、DynamoDB、EBS** 三個核心系統做 TLA+ verification。發現 bug 列表（公開）：
- DynamoDB 找到 1 個「在 replica failure 期間 consistency violation」 → 改 protocol 防止
- EBS 找到「concurrent network partition + replica recovery 可能 stale data」 → patch
- S3 找到 lifecycle automation 的 invariant violation → 改設計

**經驗教訓**:
1. **TLA+ 學習曲線 ~ 2 weeks 對 senior engineer**
2. **70% 的價值在「寫 spec 過程強迫想清楚」**，不是 TLC 找到的具體 bug
3. **invariant 寫得對比 spec 寫得對更難**

Bornholt et al. SOSP 2021 是後續 update：S3 用 TLA+ + Stateright (Rust executable model) 做 hybrid verification — 跨 spec 跟 implementation。

### 11. 寫 spec 的 best practice

從多個 industrial project 抽出：

1. **從 trace 開始**：先想 protocol 的 happy path message 序列，再 generalize 到 spec
2. **State 越少越好**：每加一個 variable 增加 state space 指數爆炸
3. **顯式 stuttering**：每個 action 寫清楚 `UNCHANGED << ... >>`
4. **Type invariant 第一**：先 verify `TypeOK` 才驗其他 invariant
5. **Inductive invariant 想清楚**：寫 invariant 時自問「這條 inductive 嗎？如果不是，需要什麼 stronger invariant 才 inductive？」
6. **TLC config 小規模先**：constants 設小（N=2 process, MaxPn=3）跑通再 scale
7. **Liveness 最後寫**：safety 是基礎，liveness 對 fairness 假設極度敏感

### 12. 何時放棄 model checking 改 hand proof

state space 太大 + Apalache 也撐不住時:
- 用 **TLAPS** 寫 inductive invariant proof
- 用 **paper proof** 對 protocol 用 informal mathematical reasoning
- 用 Coq / Lean / Isabelle 對 critical part 做 formal proof（不通過 TLA+）

實際上 protocol verification 是 mixed：transport state machine 用 TLA+；cryptographic security 用 ProVerif/Tamarin；implementation 用 F\*。Part 11.10 + Part 12 用 hybrid。

## 與我們協議設計的關聯

Part 11.10 用 TLA+ 證明的具體 invariants 清單：

1. **Packet number space monotonic per epoch**（本堂例 1）
2. **Migration safety**（本堂例 2）
3. **Anti-replay window**（本堂例 3）
4. **Stream ID allocation**: client-initiated IDs from 0,4,8,... never repeat
5. **Flow control: `0 <= bytes_in_flight <= max_data`**
6. **Connection close finality**: graceful close 後 no new frames accepted
7. **Inner protocol multiplex**: tunnel stream IDs 不衝突
8. **Key epoch ordering**: Initial → Handshake → 0-RTT → 1-RTT key install 順序

每條都用 TLA+ module 寫，跑 TLC 對 small constants verify；對 unbounded versions 用 Apalache 或 TLAPS。

### 2. 對手分類學

TLA+ 不天然 model attacker。實踐上把 attacker 當 process:
```tla
VARIABLES attacker_knowledge
AttackerLearn(msg) ==
    /\ msg \in wire
    /\ attacker_knowledge' = attacker_knowledge \union {msg}
AttackerInject(msg) ==
    /\ msg \in DerivableFrom(attacker_knowledge)
    /\ wire' = wire \union {msg}
```

但 `DerivableFrom` 對 crypto 推導很 awkward — 所以 crypto 用 ProVerif/Tamarin。

### 3. 形式化定義

**Inductive invariant**:

$$I \text{ inductive iff } \text{Init} \Rightarrow I \land (I \land \text{Next}) \Rightarrow I'$$

**Refinement**: $L \sqsubseteq H$ iff $L \Rightarrow H$（temporal implication）

**Stuttering equivalence** (Lamport 1983): two behaviors equivalent if differ only in stuttering steps
