## 核心概念

### 1. Noise framework 整體

**Noise** (Trevor Perrin, 2015–) 是一個 protocol **family**——不是一個 protocol，而是一組 building block 讓你**組合** handshake pattern。

核心元素：
- **DH function** (`s` for static, `e` for ephemeral)
- **Symmetric state** (chaining key `ck`, hash `h`)
- **Patterns** (specify 哪些 token 在哪些 message 中)

**Noise pattern 命名**：

```
Pattern XX_25519_ChaChaPoly_BLAKE2s
        ^^   ^^^^^   ^^^^^^^^^^   ^^^^^^^
        |    |       |            |
        |    DH      AEAD         hash
        pattern name
```

Pattern 名後幾個字母標記 initiator/responder key 是 known 還是 in-handshake 提供:
- **N**: no static key for initiator
- **K**: static key for initiator **known** to responder beforehand
- **X**: static key for initiator transmitted **in handshake**
- **I**: static key for initiator **immediately** sent in 1st message

幾個主要 pattern（initiator → responder）:

| Pattern | Initiator 條件 | Responder 條件 | 應用 |
|---|---|---|---|
| **NN** | none | none | anon-anon, weak |
| **NK** | none | known to initiator | TLS-like, server-authenticated only |
| **NX** | none | sent in handshake | TLS-like, server-authenticated only |
| **KK** | known | known | mutual auth, pre-shared keys |
| **XX** | sent | sent | mutual auth, ID 都在 wire 上 (WireGuard 替代) |
| **IK** | immediate | known | mutual auth, **WireGuard 用** |

### 2. Noise IK 的 message flow

Noise IK：
- Initiator 知道 responder's static public key beforehand
- Initiator 在 message 1 立刻送自己的 static public key
- Responder 在 message 2 完成 mutual auth

Message tokens:
```
IK:
  -> s
  <- s
  ...
  -> e, es, s, ss
  <- e, ee, se
```

`-> e, es, s, ss` 意思（message 1, initiator → responder）:
- `e`: initiator 產生 ephemeral keypair, 把 ephemeral public 寫進 message
- `es`: DH(initiator's ephemeral, responder's static) → 混進 chaining key
- `s`: initiator 把自己 static public 寫進 message (**encrypted** by current chaining key)
- `ss`: DH(initiator's static, responder's static) → 混進 chaining key

`<- e, ee, se` (message 2, responder → initiator):
- `e`: responder 產生 ephemeral, 寫進 message
- `ee`: DH(responder's ephemeral, initiator's ephemeral) → mix
- `se`: DH(responder's static, initiator's ephemeral) → mix

完成後：
- 雙方都 derive 相同 chaining key
- 從 chaining key derive 兩個 traffic keys (`send`, `recv`)
- mutual authentication: initiator 知道 responder static (assumption); responder 透過 `ss` 與 `se` 知道 initiator static

### 3. WireGuard 對 Noise IK 的具象

WireGuard 把 Noise IK_25519_ChaChaPoly_BLAKE2s instantiate，再加：
- **Cookie reply** for DoS 保護（類比 QUIC Retry）
- **Timestamp (TAI64N)** 防 replay
- **Pre-shared key (PSK)** optional：把 PSK 混進 chaining key 提供 post-quantum 部分保護
- **Roaming**: peer endpoint 可變但 connection state 不破

完整 handshake (從 WireGuard NDSS 2017 paper Fig. 2):

```
Initiator → Responder: handshake_initiation message {
    type, sender_index, ephemeral_pub_i,
    encrypted{static_pub_i, mac1, mac2},
    encrypted{timestamp},
    mac1, mac2
}

Responder → Initiator: handshake_response message {
    type, sender_index, receiver_index, ephemeral_pub_r,
    encrypted{empty, mac1, mac2},
    mac1, mac2
}
```

加密 `static_pub_i` 用 `ss` 推出的 key — 隱藏 initiator identity (responder 才能解)。

加密 `timestamp` 用後續 chaining key — anti-replay。

`mac1`, `mac2`: HMAC over message 之前部分, 用 responder static key — DoS 保護 (mac1 用 responder pub, mac2 用 cookie when under attack)。

### 4. WireGuard 安全屬性

Donenfeld 2017 paper 與 Donenfeld-Milner 2018 formal verification 列出 11 條：

1. **Confidentiality of transmitted data**
2. **Authentication** (mutual)
3. **Forward secrecy of session keys**
4. **KCI resistance** (即使 attacker 取得 long-term key, 過去 session 仍保密)
5. **Identity hiding** of responder (但不是 initiator's, 因 known beforehand 假設)
6. **PSK 模式下的 post-quantum partial protection**
7. **Anti-replay** via timestamp
8. **DoS resistance** via cookie reply
9. **Resistance to unknown key-share**
10. **Resistance to channel binding violation**
11. **Resistance to replay across sessions**

每條都對應一個 ProVerif query 或 Tamarin lemma。

### 5. Lipp-Blanchet-Bhargavan 2019 ProVerif spec 結構

WireGuard.pv 大致結構（簡化版）:

```proverif
(* === Channels === *)
free internet: channel.

(* === Types === *)
type point.       (* Curve25519 point *)
type scalar.      (* Curve25519 scalar / private key *)
type key.         (* symmetric key *)
type nonce.

(* === Equational theory: Curve25519 === *)
const g: point [data].
fun pmul(point, scalar): point.
equation forall x: scalar, y: scalar;
    pmul(pmul(g, x), y) = pmul(pmul(g, y), x).

(* === Hash (BLAKE2s) === *)
fun hash(bitstring): bitstring.

(* === HKDF === *)
fun kdf1(bitstring, bitstring): key.
fun kdf2(bitstring, bitstring): bitstring.
... etc.

(* === AEAD (ChaCha20Poly1305) === *)
fun aead(key, nonce, bitstring, bitstring): bitstring.
reduc forall k: key, n: nonce, m: bitstring, ad: bitstring;
    adec(k, n, aead(k, n, m, ad), ad) = m.

(* === Initiator process === *)
let initiator(sk_i: scalar, pk_r: point) =
    (* generate ephemeral *)
    new ek_i: scalar;
    let epk_i = pmul(g, ek_i) in
    (* compute ss, es DH values *)
    let ss = pmul(pk_r, sk_i) in
    let es = pmul(pk_r, ek_i) in
    (* chaining key + hash *)
    let c1 = hash(...) in
    let c2 = kdf2(c1, es) in
    ...
    (* build message *)
    let pk_i = pmul(g, sk_i) in
    let enc_pk_i = aead(key_from(c2), zero_nonce, pk_i, h) in
    ...
    out(internet, (epk_i, enc_pk_i, enc_timestamp, mac1, mac2));
    ...
    (* receive responder message, compute final keys *)
    in(internet, (epk_r, enc_empty, mac1_r, mac2_r));
    ...
    event InitiatorFinished(pk_i, pk_r, k_send, k_recv).

(* === Responder process === *)
let responder(sk_r: scalar, allowed_initiators: ...) =
    ...
    event ResponderAccepted(pk_i, pk_r, k_send, k_recv).

(* === Adversary === *)
(* free init keys to attacker for KCI tests *)

(* === Queries === *)
(* secrecy *)
query attacker(transmitted_message).

(* mutual authentication *)
query pi: point, pr: point, ks: key, kr: key;
    inj-event(ResponderAccepted(pi, pr, ks, kr)) ==>
        inj-event(InitiatorFinished(pi, pr, ks, kr)).

(* forward secrecy *)
(* model: 在 long-term keys 洩漏之後仍 query secrecy of past session data *)
phase 1.
out(internet, sk_i).
out(internet, sk_r).

(* === Process === *)
process
    new sk_i: scalar;
    new sk_r: scalar;
    out(internet, pmul(g, sk_r));  (* responder pub key public *)
    ( !initiator(sk_i, pmul(g, sk_r)) | !responder(sk_r, ...) )
```

跑 ProVerif 對每個 query 解：
- secrecy: 大部分 PASS
- authentication: PASS（with inj-event）
- forward secrecy: PASS（用 `phase` mechanism）
- KCI resistance: PASS
- identity hiding of responder: PASS (但 initiator identity 由前提，不評估)

### 6. WireGuard 形式化發現的 issue

Donenfeld-Milner 2018 + Lipp-Blanchet-Bhargavan 2019 verify 過程發現幾個 spec ambiguity:

1. **MAC1 / MAC2 cookie 場景下 timestamp 沒覆蓋**：spec 已 fix
2. **Roaming 的 endpoint update 行為** verification 內 over-approximate (允許任意 endpoint 變動)；實際 Linux impl 有 rate-limit
3. **PSK 模式下 PSK 不變 = lifetime key**: 若 PSK 洩漏，所有 session FS 失效（這是設計上 trade-off）

沒找到 fundamental attack——WireGuard 安全屬性 robust。**這是「設計時用 Noise pattern 自帶可 verify-able structure」的結果**。

### 7. Noise Explorer

Kobeissi-Nicolas-Bhargavan 2019 *Noise Explorer*: 把 Noise pattern notation 當輸入，**自動生成** ProVerif / Tamarin spec 並 verify。

100+ Noise variants 全 verify：
- IK: WireGuard 用
- XK: like IK but initiator key 也 in-handshake
- NK: server-auth only (TLS-like)
- KK: pre-shared keys

對每個 variant 自動產生 security analysis：authentication / secrecy / FS / KCI / identity hiding 在 N 種 compromise scenarios 下的 status。

**意義**：protocol designer 可以快速比較不同 Noise variant；不用每個都手寫 ProVerif spec。

→ **對我們協議**: Part 11 可以用 Noise Explorer 對候選 handshake pattern 自動比較。

### 8. 與 TLS 1.3 handshake 對比

| 屬性 | TLS 1.3 | WireGuard (Noise IK) |
|---|---|---|
| Mutual authentication | Optional (server-side default) | 強制 |
| Identity hiding (initiator) | Yes (after ServerHello encryption) | Yes (encrypted with ss-derived key) |
| Identity hiding (responder) | No (cert leaks; ECH 解) | Yes (responder pub key 已知前提) |
| RTT | 1-RTT, 0-RTT optional | 1.5-RTT (initiator 等 response 才能 send data) |
| Cipher agility | Yes (cipher suites) | No (固定 ChaCha20-Poly1305 + BLAKE2s + Curve25519) |
| Anti-replay | Mechanism #1/2/3 (RFC 8446 §8) | Timestamp + per-peer counter |
| DoS resistance | Stateless retry-token (HRR) | mac1/mac2 + cookie reply |
| Formal verification | TLS 1.3: extensive (Cremers, Bhargavan, etc.) | WireGuard: completely (Lipp et al.) |
| Code size | Reference impl: 10K+ LOC | 4K LOC kernel |
| Anti-fingerprint | None native (need ECH/REALITY) | None native (need wrapper) |

### 9. WireGuard 的弱點：anti-fingerprint

WireGuard handshake message size + UDP signature 對 GFW 是 obvious 指紋。

GFW 對 WireGuard 反應 (2024-2025 measurement):
- 主動探測 + UDP port block
- Initial handshake packet pattern matching
- 中國境內 user 普遍 WireGuard 連不上 default port，需要 obfuscation wrapper (WireProxy, Cloak, AmneziaWG)

**這就是 WireGuard 對 anti-censorship 用途的最大限制**——formal verify 完美的協議，wire format 太特殊。

→ **對我們協議的設計教訓**: handshake 設計**必須**從 day 0 考慮 anti-fingerprint，不能事後 wrap。Part 11.4 + Part 11.9 詳。

### 10. AmneziaWG: WireGuard + obfuscation

AmneziaWG (https://github.com/amnezia-vpn/amneziawg-go) 是 WireGuard fork 加 obfuscation layer：
- Initial handshake packet 大小隨機
- 加 junk bytes / dummy packets
- Header pattern obfuscation

**Trade-off**: 失去 formal verification 的乾淨性，因 obfuscation layer 沒 formal model。但 deployment 對中國 user 顯著有效。

我們協議要不要走這條 path? Part 11.4 評估。

## 與我們協議設計的關聯

從本堂結論：

1. **Noise IK 是 mutual authentication + identity hiding 的 best-in-class baseline**
2. **WireGuard 是 4000-LOC formally verified VPN 的 existence proof**
3. **我們協議若採 Noise pattern, 自動繼承 verification scaffolding**
4. **anti-fingerprint 必須 baked in handshake design，不能事後 wrap**

具體設計取捨（Part 11.4）：
- 候選 pattern：IK (WireGuard-like) / XK (initiator pub in-handshake but encrypted) / 自定 pattern
- 候選 cipher: ChaCha20-Poly1305 + BLAKE2s vs AES-GCM-SIV + SHA-256
- 必加 ECH-style outer/inner SNI 分離（WireGuard 沒這層）
- 必加 traffic shaping (Wu-FEP-resistance, Part 10)

→ 結論：我們協議**不**直接是 WireGuard——但它是設計 starting point 跟對比基線。

### 練習 C：自己寫 Noise IK simplified spec

不用 WireGuard 完整 pv（太大），自己寫一個 simplified Noise IK:

```proverif
type scalar.
type point.

const g: point [data].
fun pmul(point, scalar): point.
equation forall x: scalar, y: scalar;
    pmul(pmul(g, x), y) = pmul(pmul(g, y), x).

free internet: channel.
free secret_data: bitstring [private].

fun hash(bitstring): bitstring.
fun kdf(bitstring, bitstring): bitstring.

type key.
fun key_of(bitstring): key.
fun aead(key, bitstring): bitstring.
reduc forall k: key, m: bitstring;
    adec(k, aead(k, m)) = m.

event InitFinished(point, point, key, key).
event RespAccepted(point, point, key, key).

let initiator(sk_i: scalar, pk_r: point) =
    new ek_i: scalar;
    let epk_i = pmul(g, ek_i) in
    let es = pmul(pk_r, ek_i) in
    let c1 = hash(es) in
    let pk_i = pmul(g, sk_i) in
    let enc_pk_i = aead(key_of(c1), pk_i) in
    out(internet, (epk_i, enc_pk_i));

    in(internet, (epk_r: point, ack: bitstring));
    let ee = pmul(epk_r, ek_i) in
    let se = pmul(epk_r, sk_i) in
    let c2 = kdf(c1, kdf(ee, se)) in
    let k_send = key_of(c2) in
    let k_recv = key_of(c2) in
    event InitFinished(pk_i, pk_r, k_send, k_recv);
    out(internet, aead(k_send, secret_data)).

let responder(sk_r: scalar) =
    in(internet, (epk_i: point, enc_pk_i: bitstring));
    let es = pmul(epk_i, sk_r) in
    let c1 = hash(es) in
    let pk_i = adec(key_of(c1), enc_pk_i) in
    new ek_r: scalar;
    let epk_r = pmul(g, ek_r) in
    let ee = pmul(epk_i, ek_r) in
    let se = pmul(pk_i, ek_r) in
    let c2 = kdf(c1, kdf(ee, se)) in
    let k_send = key_of(c2) in
    let k_recv = key_of(c2) in
    out(internet, (epk_r, aead(k_send, hash(c2))));
    event RespAccepted(pk_i, pmul(g, sk_r), k_send, k_recv);
    in(internet, ct: bitstring);
    let _ = adec(k_recv, ct) in
    0.

query attacker(secret_data).
query pi: point, pr: point, ks: key, kr: key;
    inj-event(RespAccepted(pi, pr, ks, kr)) ==>
        inj-event(InitFinished(pi, pr, ks, kr)).

process
    new sk_i: scalar;
    new sk_r: scalar;
    let pk_r = pmul(g, sk_r) in
    out(internet, pk_r);
    ( !initiator(sk_i, pk_r) | !responder(sk_r) )
```

跑 ProVerif。期望兩個 query 都 PASS。

## 自我檢查

1. **Noise IK 在 message 1 就送 initiator static public key（encrypted）**。如果 attacker 截獲 message 1, 能否 derive initiator identity？描述為什麼/為什麼不。
2. **WireGuard 的 `ss` DH value (initiator-static × responder-static)** 是 long-term 不變的（只要兩端 key 不變）。這對 forward secrecy 有什麼影響？提示：考慮 ephemeral `ee` 的角色。
3. **WireGuard 的 timestamp anti-replay** 假設 client 跟 server 的 clock skew 在容忍範圍內。如果 attacker 控制 client 的 clock，能否 replay 舊 handshake？
4. **Noise IK 不對 initiator 提供 identity hiding to passive observer**（initiator 已知 responder pub key, but observer 仍可推斷 pattern）。如何 fix? 提示：考慮 ECH-style outer/inner。
5. **AmneziaWG 加 obfuscation 後失去 formal verification cleanness**。對 anti-censorship deployment 是值得 trade-off 嗎? 列出兩種設計策略對比。

### 2. 對手分類學

對 Noise IK 的 adversary:

| 等級 | 能力 |
|---|---|
| W1 | passive observer |
| W2 | active MITM on wire |
| W3 | initiator long-term key compromised (post-handshake) |
| W4 | responder long-term key compromised (post-handshake) |
| W5 | both compromised |
| W6 | adaptive RNG manipulation |
| W7 | global key escrow scenario |

WireGuard 對 W1-W4 安全（FS + KCI resistance）。W5 在 PSK 模式下對過去 sessions 仍部分保護。W6/W7 不在 spec scope。

### 3. 形式化定義

**Noise IK security properties** (per Perrin spec §7):

1. **Message integrity** (every message): destination 收到的 message 內容 attacker 無法 modify undetected
2. **Sender authentication** (after handshake): destination 知道 source 是 unique party
3. **Forward secrecy**: long-term key compromise 後 past sessions 仍 secret
4. **KCI resistance**: 即使 attacker 拿 initiator long-term key, 不能 impersonate responder to initiator
5. **Identity hiding**: 對 responder identity (公開前提) 與 initiator identity (encrypted under ss) 分別處理
