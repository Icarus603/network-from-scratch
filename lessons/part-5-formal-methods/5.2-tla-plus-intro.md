## 動機

Part 5.1 結束時你接受了「**spec-first + mechanical verification 是 PhD-level 設計協議的方法論**」。本堂課介紹這條 pipeline 的第一個工具：**TLA+**。

**TLA+ 的角色**：
- 不證密碼學安全（那是 ProVerif/Tamarin/CryptoVerif 的活）
- **證 protocol 的 state machine 不變量**：例如「flow control window 永遠 ≥ 0」、「stream 不會 deliver 同樣 byte 兩次」、「migration 不會讓兩個 endpoint 算出不同 session keys」
- TLA+ 用於 **distributed systems correctness**：concurrent state、message ordering、liveness、fairness

**為什麼 TLA+ 對我們協議重要**：
- QUIC connection migration 跨 path 的 state consistency
- Stream multiplexing 的 frame ordering invariant
- Anti-replay window 的 monotonic update
- 0-RTT acceptance 的 timing constraint
- Inner protocol multiplexing 的 mutual exclusion

讀完應該能：
1. 看懂 TLA+ 基本語法（VARIABLES, Init, Next, INVARIANTS）
2. 用 PlusCal（TLA+ 的 procedural sugar）寫 simple protocol
3. 用 TLC 跑 model checker 找 invariant violation
4. 對 SOCKS5 握手寫一個 minimal TLA+ spec

## 核心概念

### 1. 為什麼是 TLA+，不是 Coq / Isabelle / Lean

| 工具 | 強項 | 弱項 |
|---|---|---|
| **TLA+** | spec language for concurrent systems, 自動 model checking (TLC) | 不是定理證明（雖然有 TLAPS）；不擅長深度 math |
| **Coq / Lean** | 完整定理證明，dependent types | learning curve 陡，spec 跟 implementation 混在一起 |
| **Isabelle** | 大 lib（含 crypto）| 同上 |
| **F\* / EasyCrypt** | computational crypto proof | 不擅長 distributed state |
| **Spin / Promela** | model checking | 語法老，less expressive |

**TLA+ 的甜蜜點**：
- Math-flavor spec（first-order logic + ZF set theory + temporal logic）
- 自動 model check finite-state subset (TLC)
- 業界友善（AWS、Microsoft、Mongo 大規模採用）

對我們協議 Part 11 目的，TLA+ 主要用來證 **transport 層 state machine invariants**（packet number monotonic, ACK consistency, migration safety）。

### 2. TLA+ 的核心觀念：spec 是個 state machine

一個 TLA+ spec 描述：
- **State**：一組 variables 的 valuation
- **Initial state**：satisfying predicate `Init`
- **Transition**：predicate `Next(s, s')` 對 current state $s$ 跟 next state $s'$
- **Invariant**：predicate that holds in all reachable states
- **Liveness**：「最終某事會發生」（temporal property）

整個 spec 就是 temporal logic 公式：

$$\text{Spec} = \text{Init} \land \square[\text{Next}]_v \land \text{Fairness}$$

- $\square$ = "always" temporal operator
- $[\text{Next}]_v$ = "either Next holds, or all variables in $v$ unchanged" (stuttering)
- Fairness = weak/strong fairness condition

### 3. 一個極簡 TLA+ spec

寫一個 simplest possible spec：**counter 從 0 到 N**。

```tla
---------------------------- MODULE Counter ----------------------------
EXTENDS Naturals
CONSTANT N
ASSUME N \in Nat

VARIABLES counter

Init == counter = 0

Increment ==
    /\ counter < N
    /\ counter' = counter + 1

Next == Increment

vars == << counter >>

Spec == Init /\ [][Next]_vars /\ WF_vars(Next)

\* Invariant
TypeOK == counter \in 0..N
NeverExceed == counter <= N

\* Liveness: eventually counter reaches N
EventuallyN == <>(counter = N)
=======================================================================
```

逐行解：
- `EXTENDS Naturals`：import natural number theory
- `CONSTANT N`：parameter to spec
- `ASSUME N \in Nat`：assumption（spec 有效前提）
- `VARIABLES counter`：state variable
- `Init == counter = 0`：初始 state predicate
- `Increment` action: `counter < N` (precondition) AND `counter' = counter + 1` (primed = next-state value)
- `Next == Increment`：transition relation
- `vars == << counter >>`：tuple of all state vars (used in stuttering)
- `Spec == Init /\ [][Next]_vars /\ WF_vars(Next)`：完整 spec
  - `[][Next]_vars` = always (Next or all vars unchanged)
  - `WF_vars(Next)` = weak fairness — 如果 Next 一直 enabled，最終會發生

最後三個是要 verify 的 properties：
- `TypeOK`：type invariant
- `NeverExceed`：safety property
- `EventuallyN`：liveness property

### 4. 用 TLC 跑 model checking

把這個 spec 存成 `Counter.tla`，在 TLC 設定:
```
SPECIFICATION Spec
INVARIANT TypeOK
INVARIANT NeverExceed
PROPERTY EventuallyN
CONSTANT N = 5
```

TLC 會 **enumerate all reachable states**：
- s0: counter=0
- s1: counter=1
- ...
- s5: counter=5

對每個 state 檢查 invariants；對整體檢查 liveness。

如果你改 `Increment` 把 `counter < N` 條件去掉:
```
Increment == counter' = counter + 1
```

TLC 跑 NeverExceed 就會找到 counter=6 的 reachable state，輸出 counter-example trace。

### 5. PlusCal：給 TLA+ 加 procedural 糖

TLA+ 純 declarative，初學者覺得 reverse engineer 困難。**PlusCal** 是個 procedural language 編譯到 TLA+。同一個 counter:

```pluscal
--algorithm Counter
variables counter = 0;
begin
    while counter < N do
        counter := counter + 1;
    end while;
end algorithm;
```

TLA+ Toolbox 把這段編譯成 TLA+ Spec section（含 `pc` program counter variable）。**好處**：寫起來像普通 imperative code；**壞處**：state space 通常比 hand-written TLA+ 大（多個 pc variable）。

對 protocol modeling，**hand-written TLA+ 通常更乾淨**——protocol 本來就是 message-passing concurrent system，不是 imperative procedure。

### 6. 多 process: 用 `\E p \in Processes` 量化

Protocol 場景常有「N 個 client + 1 個 server」。TLA+ 用 set 與 existential quantifier 表達 concurrency。

```tla
---------------------------- MODULE TwoCounters ----------------------------
EXTENDS Naturals
CONSTANT N
VARIABLES counters \* function: Process -> Nat

Processes == {"P1", "P2"}

Init == counters = [p \in Processes |-> 0]

Incr(p) ==
    /\ counters[p] < N
    /\ counters' = [counters EXCEPT ![p] = @ + 1]

Next == \E p \in Processes: Incr(p)

TypeOK == \A p \in Processes: counters[p] \in 0..N
=======================================================================
```

- `counters` 是 **function** from Process to Nat（TLA+ 的 record/map 同一個概念）
- `Init` 用 `[p \in Processes |-> 0]` 建 function (uniform 0)
- `Incr(p)` action 帶 parameter `p`
- `Next == \E p: Incr(p)` 表示「**某個 process** 做 Incr」— TLA+ semantics：每個 step 任意一個 process 動

這個 pattern 是 protocol modeling 核心。

### 7. Message-passing: queue + atomic send

模型化 wire 上的 message：用 `Bag`（multiset）或 set 表示 "in-flight messages"。

```tla
VARIABLES network \* set of messages on the wire

Init == network = {}

Send(msg) ==
    network' = network \union {msg}

Recv(msg) ==
    /\ msg \in network
    /\ network' = network \ {msg}
```

對 protocol 中每個 message type 寫一個 action。

### 8. 一個小型 SOCKS5 握手 spec

SOCKS5 handshake (RFC 1928):
1. Client → Server: VER + NMETHODS + METHODS[]
2. Server → Client: VER + METHOD
3. Client → Server: CONNECT request (VER + CMD + RSV + ATYP + DST.ADDR + DST.PORT)
4. Server → Client: REPLY (VER + REP + RSV + ATYP + BND.ADDR + BND.PORT)

寫成 TLA+:

```tla
---------------------------- MODULE Socks5 ----------------------------
EXTENDS Naturals, FiniteSets

CONSTANTS Methods, Targets, Replies

VARIABLES clientState, serverState, wire

Init ==
    /\ clientState = "init"
    /\ serverState = "wait_methods"
    /\ wire = {}

\* Step 1: client sends method offer
ClientOfferMethods ==
    /\ clientState = "init"
    /\ clientState' = "wait_method_choice"
    /\ wire' = wire \union {[ type |-> "MethodOffer",
                              methods |-> Methods ]}
    /\ UNCHANGED << serverState >>

\* Step 2: server selects method
ServerSelectMethod ==
    /\ serverState = "wait_methods"
    /\ \E m \in wire :
         /\ m.type = "MethodOffer"
         /\ \E chosen \in m.methods :
              /\ serverState' = "wait_request"
              /\ wire' = (wire \ {m}) \union {[ type |-> "MethodChosen",
                                                method |-> chosen ]}
    /\ UNCHANGED << clientState >>

\* Step 3: client sends CONNECT request
ClientConnect ==
    /\ clientState = "wait_method_choice"
    /\ \E m \in wire :
         /\ m.type = "MethodChosen"
         /\ clientState' = "wait_reply"
         /\ \E t \in Targets :
              wire' = (wire \ {m}) \union {[ type |-> "Connect",
                                             target |-> t ]}
    /\ UNCHANGED << serverState >>

\* Step 4: server replies
ServerReply ==
    /\ serverState = "wait_request"
    /\ \E m \in wire :
         /\ m.type = "Connect"
         /\ serverState' = "established"
         /\ \E r \in Replies :
              wire' = (wire \ {m}) \union {[ type |-> "Reply",
                                             reply |-> r ]}
    /\ UNCHANGED << clientState >>

\* Step 5: client transitions to established or aborts
ClientFinish ==
    /\ clientState = "wait_reply"
    /\ \E m \in wire :
         /\ m.type = "Reply"
         /\ IF m.reply = "succeeded" THEN
              clientState' = "established"
           ELSE
              clientState' = "aborted"
    /\ UNCHANGED << serverState, wire >>

Next ==
    \/ ClientOfferMethods
    \/ ServerSelectMethod
    \/ ClientConnect
    \/ ServerReply
    \/ ClientFinish

vars == << clientState, serverState, wire >>

Spec == Init /\ [][Next]_vars /\ WF_vars(Next)

\* Invariants
TypeOK ==
    /\ clientState \in {"init", "wait_method_choice", "wait_reply",
                        "established", "aborted"}
    /\ serverState \in {"wait_methods", "wait_request", "established"}

ServerEstablishedImpliesClientResponded ==
    serverState = "established" =>
        \/ clientState = "established"
        \/ clientState = "aborted"
        \/ clientState = "wait_reply"  \* client 還沒收到 reply

\* Liveness
EventuallyEstablished ==
    <> (clientState = "established" /\ serverState = "established")
=======================================================================
```

跑 TLC 設:
```
CONSTANTS
    Methods = {0, 2}      \* NO_AUTH, USER_PASS
    Targets = {"google", "vps"}
    Replies = {"succeeded", "refused"}
```

TLC 會 enumerate 所有 reachable state，驗 invariants 是否 hold。

### 9. 找 bug 的真實場景

對上面 spec 你應該注意一個 implicit 假設：**wire 是一個 set，不是 ordered queue**。這意味:
- 如果 ClientOfferMethods + ClientConnect 兩條 message 都在 wire 上，Server 可以**先處理 Connect 再處理 MethodOffer**
- 這在真實 TCP wire 不會發生

要 fix，把 wire 改為 ordered:
```tla
VARIABLES wire  \* sequence (ordered)
network' = Append(network, msg)
\* Recv only the head:
head_msg = Head(network)
```

**這就是 TLA+ 找出 protocol design bug 的方式**：你自己以為 spec 沒問題，TLC 跑 model check 找到 reachable state 違反 invariant，trace 顯示「server 在 client 還沒發 ClientOffer 前就 reply Connect」——這在 wire-as-set 模型成立，wire-as-sequence 不成立。**Spec 模型化決策影響 verification 結果**。

### 10. State space explosion 與 abstraction

TLC 是 explicit-state model checker——enumerate **每個 reachable state**。Protocol 加 N processes、無限 message space → state space 爆炸。

緩解策略：
1. **限制 constant**：N=3 processes 而非 N=∞
2. **State constraint**：`StateConstraint = clientState # "established"` 讓 TLC 不展開 established 之後的 state space
3. **Symmetry reduction**：對 processes 對稱的 spec，TLC 自動只看一個 representative
4. **View abstraction**：自定 abstract function 把多個 concrete state 合併
5. **Apalache (symbolic)**: 5.3 講

### 11. TLA+ 的 LIMITATION

**TLA+ 不能：**
- 證 cryptographic security（沒 Dolev-Yao，沒 unification）
- 證 timing-related side channel（spec level 看不到 nanoseconds）
- 證 implementation correctness（spec ≠ Go/C code，需另外 prove refinement）
- 證 statistical / probabilistic property（沒概率 reasoning）

**TLA+ 適合：**
- Concurrent state machine correctness
- Distributed protocol agreement (consensus, leader election, replication)
- Resource management invariants (flow control, queue bounds)
- Liveness with fairness assumption

對我們協議：
- **transport state machine** invariants（QUIC packet number space, stream ID allocation, flow control）→ TLA+ ✅
- **handshake authentication / secrecy** → ProVerif / Tamarin (5.4-5.6) ✅
- **cryptographic primitive bound** → CryptoVerif (5.7) ✅
- **implementation matches spec** → F* / Coq (Part 12 可選)

## 與我們協議設計的關聯

我們協議用 TLA+ 證明的 invariants（Part 11.10 預告）：

1. **Stream ID monotonicity**: client-initiated bidi stream ID 嚴格遞增
2. **Packet number space isolation**: Initial / Handshake / Application 三個 space 互不污染
3. **Flow control window invariant**: `0 <= bytes_in_flight <= max_data`
4. **Migration safety**: PATH_CHALLENGE 完成前不對新 path 解除 anti-amplification
5. **Anti-replay window monotonic**: window 內已用 nonce 不被 accept 第二次
6. **Connection close finality**: graceful close 後不接受新 frame

每條都會在 Part 11.10 寫成 TLA+ module 並用 TLC verify。

### 安裝環境

```bash
mise use java@temurin-25-LTS
# Download TLA+ Toolbox or use VS Code extension
# Option A: TLA+ Toolbox - https://lamport.azurewebsites.net/tla/toolbox.html
# Option B: VS Code - search "TLA+" by Andrew Helwer (官方推薦)
```

VS Code 安裝後，新增 `Counter.tla` 跟上節範例。

### 練習 A：建 Counter spec + 跑 TLC

跟著上節 Counter 範例：
1. 寫 `Counter.tla`
2. 寫 model config (`Counter.cfg`):
   ```
   SPECIFICATION Spec
   INVARIANT NeverExceed
   PROPERTY EventuallyN
   CONSTANT N = 5
   ```
3. 跑 TLC，看是否找到 violation

故意把 `Increment` 條件改錯（去掉 `counter < N`），看 TLC 印 counter-example。

### 練習 B：兩個 process 的 mutex

寫 TLA+ spec for **simple mutual exclusion**:
- 兩個 process A, B
- 一個共享 lock 變數
- 各 process 狀態 ∈ {try, in_critical, done}
- Invariant: 不能兩個 process 同時 in_critical

故意把 lock check 弄錯（如 Peterson's algorithm 不對版本），看 TLC 找 violation。

### 練習 C：寫上節的 SOCKS5 spec

照抄上節 SOCKS5 spec，跑 TLC：
- 確認 `TypeOK` 不會 fail
- 確認 `ServerEstablishedImpliesClientResponded` 在 wire=set 模型下會 fail（為什麼？）
- 改 wire 為 ordered sequence，重跑

### 練習 D：QUIC packet number space invariant

對 QUIC 簡化版（只有 Initial 跟 Handshake space）：
```tla
VARIABLES pn_initial, pn_handshake

\* Invariant: 兩個 space 的 packet number 各自 monotonic
\* Invariant: 不會在 same connection 內混淆
```

寫一個 minimal spec：每個 step 任意 send Initial 或 Handshake packet，verify packet number monotonic per space。

## 自我檢查

1. **TLA+ 的 `[][Next]_vars` 中 `[Next]_vars` 包含 "stuttering"**。為什麼 stuttering 對 concurrent system spec 必要？

### 2. 對手分類學

TLA+ 對 attacker modeling 直接：把 attacker 當成 process。但 TLA+ 不擅長 modeling **cryptographic adversary**（無 unification, 無 implicit knowledge）。

### 3. 形式化定義

**TLA+ spec** 是 temporal logic formula:
$$\text{Spec} \equiv \text{Init} \land \square[\text{Next}]_v \land L$$

其中 $L$ 是 fairness conjunction。

**Refinement** 在 TLA+ 是 implication:
$$\text{Spec}_{low} \Rightarrow \text{Spec}_{high}$$

意味 low-level spec implements high-level spec。

**Hyper-property**（如 non-interference）TLA+ 直接表達困難——需 TLA+ extensions 如 HyperTLA。
