## 動機

TLA+ 證 state machine correctness 但不證 cryptographic security。**ProVerif** 是密碼學協議自動 verifier，吃 applied pi-calculus 寫的 spec，輸出 secrecy / authenticity 是否成立。

讀完應該能：
1. 看懂 applied pi-calculus 基本 process operators (`!`, `|`, `new x`, `out`, `in`)
2. 寫 simple protocol 的 ProVerif spec
3. 用 ProVerif `query` 證 secrecy / authentication / observational equivalence
4. 親手驗一個 Diffie-Hellman protocol

ProVerif 是 Part 5.5 (Noise IK) 和 Part 11.10 (我們協議) 的工作引擎。

## 核心概念

### 1. Pi-calculus 與 applied pi-calculus

**Pi-calculus** (Milner 1989) 是 process algebra，描述 concurrent processes via channel-based message passing。

核心 operators:

| Operator | 意義 |
|---|---|
| `0` | nil process (done) |
| `P | Q` | parallel composition: P 跟 Q 同時跑 |
| `!P` | replication: 無限多個 P copies |
| `new x; P` | new name: 創建 fresh name x, P 內可用 |
| `out(c, m); P` | 在 channel c 上送 m，然後跑 P |
| `in(c, x); P` | 在 c 上收，bind 進 x，然後跑 P |
| `if M = N then P else Q` | branch |

**Applied pi-calculus** (Abadi & Fournet POPL 2001) 加了：
- **Function symbols** with equational theory (對 cryptographic primitives)
- **Names vs variables** distinction
- **Active substitution** for value flow

讓你 model `enc(m, k)` (encryption), `dec(enc(m, k), k) = m` (decryption equation), 等密碼學 primitives。

### 2. ProVerif 內部架構

**ProVerif** 把 applied pi-calculus spec **abstract 成 Horn clauses**:
- `att(M)` predicate: attacker may know M
- Protocol step `in(c, x). out(c, f(x))` → Horn clause `att(x) ⇒ att(f(x))`
- Cryptographic equation `dec(enc(m, k), k) = m` → clause `att(enc(m, k)) /\ att(k) ⇒ att(m)`

然後對 `att(secret)` query 用 resolution algorithm 找 derivation。若找不到 → secret 確實 secret。若找到 → 試 reconstruct attack trace.

**優勢**:
- **Unbounded sessions**: 不限制 protocol run 數量（不像 TLC）
- **Unbounded data**: 不限制 message size
- **Fast**: 對中等 protocol 秒級

**劣勢**:
- **Horn clause abstraction** = over-approximation: 可能有 false positive（ProVerif 說有 attack，實際無）
- **No global mutable state**：對 stateful protocol (如 anti-replay window) 處理 awkward
- **Inequality 困難**：「x ≠ y」condition 不易表達

### 3. ProVerif 基本語法

```proverif
(* Free names: 公開可用 *)
free c: channel.

(* Types *)
type key.
type host.

(* Functions *)
fun enc(bitstring, key): bitstring.
reduc forall m: bitstring, k: key; dec(enc(m, k), k) = m.

(* Constants and events *)
free secret_data: bitstring [private].

(* Queries *)
query attacker(secret_data).

(* Process *)
let alice =
    new k: key;
    out(c, enc(secret_data, k)).

process
    alice
```

跑 ProVerif:
```bash
proverif simple.pv
```

輸出大致:
```
RESULT not attacker(secret_data[]) is true.
```

意思：attacker 無法 derive secret_data。

如果你把 `let alice` 改成 `out(c, k); out(c, enc(secret_data, k));`（先送 key 再送 ciphertext），ProVerif 跑出:
```
RESULT not attacker(secret_data[]) is false (attack found).
```
並顯示 attack trace。

### 4. ProVerif 核心 query 種類

#### Secrecy
```proverif
query attacker(secret).
```
證 secret 不會 leak 給 attacker.

#### Strong secrecy (observational equivalence)
```proverif
query secret M.
```
證 attacker 無法 distinguish M 是 secret value 還是別的——比 secrecy 強。

#### Authentication via events
```proverif
event AliceFinished(host, host, bitstring).
event BobStarted(host, host, bitstring).

query x: host, y: host, k: bitstring;
    inj-event(AliceFinished(x, y, k)) ==> inj-event(BobStarted(x, y, k)).
```

意義：Alice claim 完成 protocol with y, sharing k → 必有對應的 Bob start with x, sharing k。**`inj-` 表示 injective**: 每次 Alice finish 對應**一次** Bob start (防 replay)。

對應 Lowe 1997 hierarchy:
- `event A ==> event B` = non-injective agreement
- `inj-event A ==> inj-event B` = injective agreement

#### Observational equivalence (privacy)
```proverif
process
    !( ... ) | !( ... )
```

ProVerif `--observational-eq` 模式驗 「兩個 process distribution 對 attacker indistinguishable」。

用於 privacy properties（ECH server identity privacy 之類）。

### 5. 親手驗 Diffie-Hellman

DH 是密碼學最基本協議，也是 ProVerif tutorial 的標準起步:

```proverif
(* Diffie-Hellman over a finite group *)
type G.
type exponent.

const g: G [data].

fun exp(G, exponent): G.

equation forall x: exponent, y: exponent;
    exp(exp(g, x), y) = exp(exp(g, y), x).

(* Channel *)
free c: channel.

(* Secret shared *)
free shared: bitstring [private].

(* Symmetric encryption *)
fun senc(bitstring, G): bitstring.
reduc forall m: bitstring, k: G;
    sdec(senc(m, k), k) = m.

(* Events for authentication queries *)
event AliceCommit(G, G).
event BobAccept(G, G).

(* Queries *)
query attacker(shared).
query A: G, B: G;
    event(BobAccept(A, B)) ==> event(AliceCommit(A, B)).

(* Processes *)
let alice =
    new x: exponent;
    let ga = exp(g, x) in
    out(c, ga);
    in(c, gb: G);
    let k = exp(gb, x) in
    event AliceCommit(ga, gb);
    out(c, senc(shared, k)).

let bob =
    in(c, ga: G);
    new y: exponent;
    let gb = exp(g, y) in
    out(c, gb);
    in(c, ct: bitstring);
    let k = exp(ga, y) in
    let m = sdec(ct, k) in
    event BobAccept(ga, gb).

process
    !alice | !bob
```

跑 ProVerif：
```
RESULT not attacker(shared[]) is true.
RESULT event(BobAccept(A,B)) ==> event(AliceCommit(A,B)) is false.
```

第二條 fail！為什麼？因為 **textbook DH 沒 authentication**——MITM 可以攔截 Alice 的 ga，自己跟 Bob 跑 DH，再對 Alice 跑另一個 DH。Bob 「accept」的不是 Alice。

修補：加 signature:
```proverif
fun pk(skey): pkey.
fun sign(bitstring, skey): bitstring.
reduc forall m: bitstring, k: skey;
    verify(sign(m, k), pk(k)) = m.

let alice(skA: skey) =
    new x: exponent;
    let ga = exp(g, x) in
    out(c, (ga, sign(ga, skA)));
    ...
```

跑 ProVerif 再次，secrecy + authentication 都成立。

**這個練習就是 5 分鐘讓你體會 ProVerif 的快**——手動證 textbook DH 不安全要寫滿幾頁 paper，ProVerif 一個 query 解決。

### 6. Equational theory

ProVerif 對 cryptographic primitives 透過 **equational theory** 建模：

```proverif
(* Symmetric encryption (perfect) *)
fun senc(bitstring, key): bitstring.
reduc forall m: bitstring, k: key;
    sdec(senc(m, k), k) = m.

(* Public-key encryption *)
fun pk(skey): pkey.
fun aenc(bitstring, pkey): bitstring.
reduc forall m: bitstring, k: skey;
    adec(aenc(m, pk(k)), k) = m.

(* Hash function (one-way) *)
fun hash(bitstring): bitstring.

(* Signature *)
fun sign(bitstring, skey): bitstring.
reduc forall m: bitstring, k: skey;
    verify(sign(m, k), pk(k)) = m.
reduc forall m: bitstring, k: skey;
    msg(sign(m, k)) = m.

(* MAC *)
fun mac(bitstring, key): bitstring.

(* Diffie-Hellman *)
equation forall x: exponent, y: exponent;
    exp(exp(g, x), y) = exp(exp(g, y), x).
```

注意 **Dolev-Yao 假設**:
- `senc` 是 perfect: 沒 key 不能 decrypt, 沒 collision
- `hash` 是 one-way: 不能 invert
- `sign / verify` 是 perfect: 沒 sk 不能 forge

**這個 idealization 是 symbolic model 的核心**。Realistic crypto attack（如 oracle attack）在 symbolic 看不到——要用 computational model (CryptoVerif, Part 5.7)。

### 7. 處理 stateful protocol

ProVerif 對 global mutable state 不友善。例：anti-replay window 是 stateful。

緩解方案：
1. **GSVerif** (Cheval-Cortier-Turuani 2018): ProVerif 的 stateful extension
2. **Table** (ProVerif `table` 機制): 半 stateful
3. 把 state 改 model 成 message passing on private channel

```proverif
(* Use private channel to model state *)
new state_chan: channel.

let state_manager =
    out(state_chan, init_state);
    !(
        in(state_chan, s: state_type);
        in(c, request: bitstring);
        let new_s = update(s, request) in
        out(state_chan, new_s)
    ).
```

但這 idiom 有 known 限制：對 unbounded sessions ProVerif 可能 over-approximate。

### 8. ProVerif 的局限

1. **False positive**: Horn clause abstraction 太寬，可能說有 attack 但實際無
2. **Termination not guaranteed**: 對 complex protocol ProVerif 可能跑不完
3. **No XOR support**: XOR equation `x XOR x = 0` 對 resolution 不友善（後續改 GS Verif 支援）
4. **No timing**: 不能 model timing-based property
5. **Probabilistic 不支援**

但對 majority of authenticated key exchange protocols (TLS, Noise, IKE, X3DH, MLS) ProVerif 工作良好——這也是它能輻射到 Signal, WireGuard, TLS 1.3 等 production protocol 的原因。

### 9. ProVerif 的 industry track record

| Protocol | ProVerif 用途 | 結果 |
|---|---|---|
| **TLS 1.3** | Bhargavan-Blanchet-Kobeissi S&P 2017 verify | 大部分屬性 OK；找出 0-RTT subtle issue |
| **Signal** | Kobeissi-Bhargavan-Blanchet 2017 | X3DH + Double Ratchet 主要屬性 OK |
| **WireGuard** | Lipp-Blanchet-Bhargavan EuroS&P 2019 | 用 ProVerif + CryptoVerif 全 verify |
| **Noise framework** | Kobeissi-Bhargavan-Blanchet 2019 *Noise Explorer* | 100+ Noise variant 自動 verify |
| **MLS** | Cremers-Hale 對 group 通訊 | 持續分析 |
| **ECH** | Bhargavan-Cheval-Wood CCS 2022 | privacy properties |

**Test of Time Award** at CSF 2023 給 Blanchet CSFW 2001 paper — ProVerif 真的成為密碼學社群 backbone。

## 與我們協議設計的關聯

我們協議 Part 11.10 用 ProVerif 證明的 properties:

1. **Inner session secrecy**: inner application data 對 outer attacker 保密
2. **Mutual authentication**: client 跟 server 完成 handshake 後相互 inj-event authentic
3. **Forward secrecy**: long-term key 洩漏不影響 past sessions
4. **Anti-replay**: 0-RTT data 不被 server accept 兩次（with GSVerif stateful extension）
5. **Outer SNI privacy**: outer ClientHello observation 不洩漏 inner SNI（observational equivalence）

ProVerif 是 Part 11.10 主要 verification tool；Tamarin (5.6) 用於 deeper Diffie-Hellman / equation property；TLA+ 用於 transport state。

## 自我檢查

1. **Pi-calculus 跟 process algebra**（CSP, CCS）的差別是什麼？為何 protocol verification 偏好 pi-calculus？
2. **ProVerif 的 Horn clause abstraction 是 over-approximation**。給一個 false positive 的具體例子。
3. **`inj-event(A) ==> inj-event(B)` 跟 `event(A) ==> event(B)`** 的差別是什麼？對應 Lowe 1997 hierarchy 哪兩層？
4. **Equational theory 對 perfect crypto**。如果 attacker 有 oracle attack 能力（如 timing oracle 或 ASN.1 parsing oracle），ProVerif 看得到嗎？
5. **ProVerif 對 stateful protocol 表現差**。為什麼 anti-replay window 必須用 GSVerif extension or private channel idiom?

### 2. 對手分類學

ProVerif 預設 Dolev-Yao:
- 攻擊者完全控制 wire
- 看得到所有 public message
- 可以 inject / drop / replay / reorder
- 知道所有 public keys 與 user identity
- **無法**：break crypto primitive (assumption); side-channel; timing
- **可選**：long-term key compromise（perfect/imperfect FS query）

### 3. 形式化定義

**Attacker knowledge** 由 Horn clause derivation 定義：

$$\text{att}(M) \iff \exists \text{ derivation tree from public knowledge + protocol clauses to } \text{att}(M)$$

**Secrecy**:
$$\text{Protocol } P \text{ provides secrecy of } s \iff \neg \exists \text{ derivation to } \text{att}(s)$$

**Authentication via correspondence**:
$$P \text{ authenticates } A \text{ to } B \iff \forall \text{ trace}: B \text{ executes event}_B \Rightarrow A \text{ has executed event}_A$$
