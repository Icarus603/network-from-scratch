## 動機

reliability 是 TCP 的核心承諾，也是**過去 40 年最被優化的單一 protocol property**。對 G6：

1. **G6 baseline 走 QUIC，但 QUIC 的 reliable delivery 是 TCP 的進化版**：QUIC RFC 9002（loss detection 與 congestion control）直接 reference RFC 8985 RACK-TLP——**理解 TCP reliable delivery = 理解 QUIC reliable delivery 的 70%**
2. **「可靠的封包流」是任何代理協議的 minimum bar**：G6 不僅要把 packet 送到，還要**正確順序、無 silent loss、控速合理**。TCP 為此演化的所有 mechanism（cumulative ACK / SACK / DSACK / RTO / fast retransmit / TLP / RACK / F-RTO / ER / PRR）都直接適用到 G6 設計決策
3. **錯誤重傳（spurious retx）是 throughput 殺手**：RTO 估算錯導致重傳已收到的 packet → 觸發 congestion control 縮窗 → throughput 降。Karn 1987 + Jacobson 1988 + F-RTO + RACK 是 30 年解這問題的 progression
4. **GFW / 中間設備對 reliable delivery 有干擾能力**：selective drop、reorder、delay——理解 TCP 對這些反應才知道 G6 在抗審查場景該如何處理
5. **Jacobson 1988 是 distributed system 經典之一**：「conservation of packets」原則 + 動態 RTT estimator + AIMD（下 lesson 詳）——三個 invariant **支撐了整個 internet 過去 40 年的 stability**。**這篇論文跟 Perlman 1985 / Saltzer 1984 / Clark 1988 並列「網路 architecture 必讀」清單**

教科書講 TCP reliable delivery 的問題：cumulative ACK + fast retransmit 講一遍就完，**完全跳過 SACK / RACK / TLP**——但這三者是 modern internet **跑 99% 流量** 的真正機制。本堂從 2026 SOTA（RACK-TLP）切入，反向理解整套演化。

## 核心概念

### 1. Sequence number 與 ACK 的真實語義

TCP header 兩個 32-bit field：
- **Seq num**：「**我這個 segment 的第一個 byte 在整個 stream 中的位置**」
- **ACK num**：「**我已連續收到截至此 byte（exclusive）為止的所有 data，下次想要的 byte 從這開始**」

```
sender: 寫 "Hello, World!" (13 byte) at offset 1000
        → TCP segment: seq=1000, data=...
receiver: ACK 1013（不是 1012，因 ACK 是 next expected byte）
```

**關鍵性質**：
- ACK 是 **cumulative**——回 ACK=1013 表示「**1000-1012 全到了**」，不只「**1012 到了**」
- ACK 與 data 都用同一個 32-bit space——overlap：seq 也是 ACK 計數的對象
- **SYN 與 FIN 各佔 1 個 seq number**——這就是為什麼 ACK = ISN+1 in handshake

#### 32-bit seq number 在 1 Gbps+ 速度下會 wraparound

1 Gbps full duplex → 每秒 ~125 MB → 32-bit seq wrap 在 ~34 秒。**MSL（120 sec default）內**可能 wrap → **舊 segment 跑出去再回來時 seq 已被重用 → 嚴重 bug**。

**Mitigation**：**PAWS (Protection Against Wrapped Sequences, RFC 7323)**——用 TCP Timestamp option 加 64-bit logical timestamp 區分 wraparound。**Linux/macOS/Windows 全預設 enabled**。

### 2. Cumulative ACK 的兩個致命 bug

#### 2.1 Hole 之後的 packet 「**被白送**」

```
sender: send seg1(seq=1000), seg2(seq=2000), seg3(seq=3000), seg4(seq=4000)
                    ↓             ↓            ↓             ↓
                  delivered    LOST!       delivered      delivered

receiver:
  receives seg1 → ACK 2000
  seg2 lost → no ACK update
  receives seg3 → still ACK 2000 (cumulative不認 hole 後 data)
  receives seg4 → still ACK 2000
```

⇒ sender 看到 3 個 duplicate ACK（"**still want 2000**"）後**才**重傳 seg2（fast retransmit, ≥ 3 dupACK）。但 seg3 與 seg4 sender 不知道 receiver 已經收到——**可能誤重傳全部**。

#### 2.2 Tail loss 無 dupACK 觸發

若**最後 N 個 segment 全丟**，receiver 沒有後續 packet 可以觸發 dupACK → sender 等 **RTO（典型 200ms ~ 1s+）** 才重傳。**對 short-lived flow（HTTP request）災難**——RTO 比正常 RTT 大 1-10×。

⇒ 1996 後 30 年所有 loss detection 改進都圍繞解決這兩個 bug。

### 3. SACK（RFC 2018, 1996）⭐ — 第一個關鍵改進

#### 3.1 SACK 怎麼解 cumulative ACK 的洞

SACK 是 TCP option（kind=5），可帶 **最多 4 對 (left edge, right edge)**——告訴 sender「**我除了 ACK 那個 cumulative 邊界，還收到了這幾個不連續 block**」。

```
sender: send seg1-4
seg2 lost
receiver: ACK=2000, SACK=[3000-5000]
  // 意思：「I want byte 2000 next, but I have 3000-4999 already」
sender: 看 SACK → 立即知道 seg3, seg4 已到 → 只需重傳 seg2
```

#### 3.2 DSACK（Duplicate-SACK, RFC 2883）

SACK 擴展：用 SACK block 告訴 sender「**這個 block 我已經收到 N 次了**」——其中 N 表示 spurious retransmission。

```
receiver: 不小心收到 seg1 兩次（sender 誤重傳）
receiver → sender: ACK=2000, DSACK=[1000-1500]
  // 「我重複收到 1000-1499，所以你的重傳是 spurious」
```

sender 根據 DSACK 推斷「**之前的 RTO 太短，估錯**」→ **增大 RTO**，避免再 spurious。

### 4. RTO 計算的演化：Karn 1987 + Jacobson 1988

#### 4.1 為什麼 RTO 難算

- **太短**：spurious retransmit 浪費頻寬、誤觸發 congestion control
- **太長**：真正 loss 後拖延，throughput 慘
- RTT 在 wide-area network 從 ~10ms (datacenter) 到 ~500ms (satellite)、抖動可達 ±200ms

#### 4.2 Karn's algorithm（Karn & Partridge 1987）⭐

**問題**：若一個 segment 重傳了，**怎麼算 RTT**？sender 看到 ACK 不知道對應原 packet 還是重傳。

**Karn's rule**：**重傳的 segment 不算 RTT sample**。直到收到 **新 segment 的 ACK** 才更新 RTT estimate。

**Clamped retransmit backoff**：重傳後 RTO 必須 exponential backoff（×2）且**忽略後續 ACK timing**——直到收到一個 unambiguous ACK 為止。

**意義**：保證 RTT 估算永不被 retx ambiguity 污染。**這個演算法 1987 後 40 年沒變，所有 modern TCP 都用**。

#### 4.3 Jacobson RTT estimator（1988）

定義：
- `SRTT` (Smoothed RTT) = 平滑後的 RTT 估計
- `RTTVAR` (RTT Variance) = RTT 變動量估計

每收到 unambiguous RTT sample `R`：
```
RTTVAR ← (1 - β) × RTTVAR + β × |SRTT - R|       (β = 1/4)
SRTT   ← (1 - α) × SRTT   + α × R                (α = 1/8)
RTO    ← SRTT + max(G, 4 × RTTVAR)                (G = clock granularity)
```

**為什麼 4 倍 RTTVAR**：原 1988 paper 用 2×；**revised version（後加，因為觀察到 slow SLIP link 的 spurious retx）改為 4×**——這是 Van Jacobson 自己加在 paper 的 footnote。

**Initial RTO**：RFC 6298 規定 **1 second**——若 path RTT 遠小（DC 內 1ms），第一個 missing packet 等 1 sec 才重傳——不可接受。**所以 fast retransmit / TLP 必須補洞**。

#### 4.4 Linux 進階：MIN_RTO clamp

Linux 設 `TCP_RTO_MIN = 200ms`——RTO 不能小於 200ms。原因：避免 RTO 過短造成 spurious retx。**對 DC 場景太大**——所以 Google 等大用戶 patch kernel 改 50ms。

### 5. Fast retransmit + Fast recovery（RFC 5681）

#### 5.1 Fast retransmit（≥ 3 dupACK）

```
sender: send seg 1..5
seg 2 lost
receiver: ACK=2000 (× 3 due to seg 3, 4, 5 arrival)
sender 看到 3 dupACK → 立即重傳 seg 2，不等 RTO
```

**為什麼 3**：歷史經驗值，平衡 reorder false-positive vs detection latency。1 個 dupACK 太敏感（reorder 也會），> 3 太晚。

#### 5.2 Fast recovery

重傳後**不**立即回 slow start——把 cwnd 縮一半繼續用「**inflated** cwnd」推送，每個 dupACK 加 1 MSS（因為 1 packet 已離開網路）。Reno 1990 起這個 logic。

**問題**：3 dupACK 是 heuristic，**對 reorder-heavy 環境**（如 multipath、wifi、衛星）誤判率高。

### 6. RACK-TLP（RFC 8985, 2021）⭐ 2026 SOTA

#### 6.1 RACK：基於時間而非 ACK 計數

**核心想法**：不數 dupACK，**用 timestamp**：

```
For each segment S:
    record S.xmit_ts (transmit time)

When receiving ACK that covers some segments:
    let T = max{xmit_ts of newly-ACKed segments}
    For every unacked segment U with U.xmit_ts < T - reo_wnd:
        mark U as LOST
```

**意思**：「**如果一個 segment xmit 時間明顯比已被 ACK 的 segment 早，且超過一定 reorder window，就視為 LOST**」

**好處**：
- **同時偵測 multiple loss**——一次 ACK 可標多個 segment LOST
- **無需 3 dupACK heuristic**
- **對 reorder 友善**——`reo_wnd` 自適應（DSACK 觸發增大）
- **對 application-limited flow 友善**——少量 packet 也能 detect tail loss

#### 6.2 TLP（Tail Loss Probe）

當 sender 寫完最後 packet，**過 PTO（Probe Timeout, typical 2 RTT）**還沒 ACK：
- 送一個 **probe segment**（new data 或 retx 最末 packet）
- Probe 的 ACK 會觸發 RACK 偵測 tail loss
- 避免等到 RTO（典型 200ms ~ 1s）

#### 6.3 RACK-TLP 對 short-lived flow 的革命性改善

**RFC 8985 開頭範例**：100 個 segment 送出，最後 3 個丟。
- **無 RACK-TLP**：RTO 等，重傳第一個 unacked，slow start 重建 cwnd → ~3 RTT + 1 RTO 完成
- **有 RACK-TLP**：TLP 觸發 ACK → RACK 標 98、99 lost → fast recovery → ~4 RTT 完成
- **延遲改善** ~10×，cwnd 不必重建

#### 6.4 對 G6 / QUIC 的影響

**QUIC RFC 9002（loss detection）幾乎是 RACK-TLP 的 datagram 版**：
- 每 packet 帶 xmit_ts
- 用 packet number（而非 byte seq）但概念對應
- TLP probe 變 PING frame
- DSACK 對應 QUIC 的 ACK frame 含「**newest larger packet 的 ack delay**」

⇒ **G6 baseline = QUIC = inherits RACK-TLP 的 loss detection paradigm**。

### 7. F-RTO（Forward RTO recovery, RFC 5682）

#### 7.1 問題

突然網路改變（如 WiFi → cellular 切換），路徑 RTT 從 10ms 跳到 200ms。RTO 來不及更新 → spurious RTO → sender 重傳 + 縮窗 → 但 path 沒事，純粹是 RTO 估錯。

#### 7.2 F-RTO 演算法

RTO 觸發後**不立即進 slow start**：
- 重傳第一個 unacked（傳統行為）
- **送 2 個 new segment**（不在原本應重傳的範圍）
- 等下一個 ACK：
  - 若 ACK 對應 new segment → **沒丟，只是 RTT 變化**——退出 RTO recovery、繼續正常
  - 若 ACK 仍對應已有 segment → 真丟了——進 standard recovery

⇒ F-RTO 把「spurious RTO」與「true loss」區分開——避免錯誤縮窗。

#### 7.3 對 G6 在 mobile 場景的意義

mobile 用戶 WiFi↔cellular 切換頻繁——**spurious RTO 是常見**。QUIC 內建 connection migration（[1.7 lesson](./1.7-nat-taxonomy.md)）+ F-RTO-equivalent recovery 是 mobile 場景 G6 比 TCP-based VPN 順暢的主因。

### 8. Early Retransmit (ER, RFC 5827)

對 **small flow**（< 4 segment）無法觸發 3 dupACK 的 fast retransmit。**ER**：當 sender 知道 in-flight < 4 packet 時，**1 個 dupACK 就視為 fast retransmit trigger**——降低 RTO 等待。

**RACK-TLP 把 ER 場景吸收**——ER 在 RFC 8985 deployment 後 effectively obsolete。

### 9. PRR（Proportional Rate Reduction, RFC 6937）

#### 9.1 為什麼需要

Reno fast recovery 把 cwnd 縮半然後**累加 dupACK**——但 burst-like 釋放 packet 可能造成新 burst loss。
Linux 引入 **PRR**：在 recovery 期間，**精確平衡 「網路內 packet 數」與 「ACK 觸發釋放 rate」**——確保 recovery 過程平滑、不造成 secondary loss。

#### 9.2 PRR 演算法（簡化）

```
target_cwnd = ssthresh + (max_acked - prr_delivered) × (target / unacked_when_loss_detected)
sndcnt = max(min(target_cwnd - in_flight, segs_since_last_send), 1)
```

意思：根據已 deliver / 已 send / loss detection 時的 unacked，**動態調節**每次能送的 segment 數，使 recovery 過程平滑。

#### 9.3 Linux 預設 enable PRR——這是現代 TCP recovery 的隱性主力

### G6 loss detection 設計選擇

```yaml
loss_detection:
  base_algorithm: RACK-TLP-derived  # QUIC RFC 9002 spec
  reordering_window: dynamic        # 由 DSACK / out-of-order ACK 自適應
  tail_loss_probe:
    enabled: true
    pto: max(2 * SRTT, 200ms)
  initial_rto: 333ms                # 比 Linux default 1s 短，但保守於 100ms
  min_rto: 100ms                    # 比 Linux 200ms 短
  spurious_rto_detection: F-RTO-style
  recovery_pacing: PRR-style smooth
  packet_number_encryption: mandatory  # 阻止 middlebox 看 packet seq
  ack_frequency: adaptive            # 不固定每 packet 一 ACK
  ack_delay: ≤25ms                   # 啟發自 QUIC RFC 9000
```

## 研究級補遺

### 1. 學界詞彙

- **SRTT (Smoothed RTT) / RTTVAR (RTT variance)**

- **Reordering window (reo_wnd)** in RACK
- **Spurious retransmission / spurious RTO**
- **F-RTO (Forward RTO, RFC 5682)**
- **ER (Early Retransmit, RFC 5827)**
- **TLP (Tail Loss Probe, RFC 8985)**
- **RACK (Recent ACK, RFC 8985)**
- **PRR (Proportional Rate Reduction, RFC 6937)**

- **SCTP / DCCP** — 替代 TCP 的 reliable / partial-reliable 協議
- **HoL (Head-of-Line) blocking**：TCP 對 application-multiplexed stream 的痛點 → QUIC stream multiplexing 解

- **Bandwidth-delay product (BDP) = bandwidth × RTT**：window size 上界
- **MSS (Maximum Segment Size)**：與 path MTU 互動
- **Karn's algorithm** (Karn & Partridge 1987)
- **Jacobson algorithm** (Jacobson 1988)

### 2. 對手分類學

| 對手能力 | 對 TCP 可靠傳輸的影響 |
|---|---|
| **passive observer** | 可推斷 throughput、RTT、loss rate；流量分析素材 |
| **selective drop** (GFW selective DPI) | 故意丟特定 segment 觸發異常 recovery；可被識別為「GFW intervention」 |
| **reorder** (rare 但 ECMP 場景常見) | 觸發 false fast retransmit；RACK 對此友善 |
| **delay injection** | 故意延遲 RTT estimator → 強制 spurious RTO → throughput 崩 |
| **RST injection** (already in 1.8) | 直接斷連線 |
| **ACK splitting / ACK division attack** | client side 偽造 ACK 騙 server 認為已收到 → 加速 cwnd 增加 → DoS |
| **Optimistic ACK attack** | 受害者 server 因 attacker 偽 ACK 加速送 data → 反向放大 |

### 3. 形式化定義

#### 3.1 TCP reliable delivery 形式化

設 sender 送 data sequence D = b_1, b_2, ..., b_n。

**Safety property**：receiver 給 application 的 stream 是 D 的 prefix（無亂序、無缺漏）。
**Liveness property**：在 finite loss rate p < 1 下，sender 在 finite expected time 內把 D 全部送達。
**No spurious delivery**：receiver 不傳給 application 任何不在 D 中的 byte。

**Cumulative ACK 滿足這些 property**——只要 RTO 不爆炸（Karn 演算法保證）+ retransmit 邏輯正確。

#### 3.2 RTO 的安全性

**RFC 6298 RTO**：
```
RTO_n = SRTT_n + max(G, 4 × RTTVAR_n)
SRTT_n ← (1-α) × SRTT_{n-1} + α × R_n   (α = 1/8)
RTTVAR_n ← (1-β) × RTTVAR_{n-1} + β × |SRTT_{n-1} - R_n|   (β = 1/4)
```

**性質**：對 stationary RTT distribution，RTO 收斂到 (mean RTT) + 4 × (mean abs deviation)。對 ~ Gaussian RTT，4σ 涵蓋 99.99% sample → spurious RTO probability ~10^-4。

**現實 RTT 非 Gaussian**——常有 heavy-tail（cellular handoff、CGN）。⇒ 4σ 不夠保守——所以 Linux 加 `MIN_RTO=200ms` clamp。

#### 3.3 BDP 與 Mathis equation

**Mathis et al. 1997**：在 random loss rate p、constant RTT、AIMD congestion control 下：

```
Throughput ≈ MSS × C / (RTT × √p)
```

其中 C ≈ 1.22。對 G6 設計：
- p = 1% loss → throughput proportional to 1/0.1 = 10× 比 p = 0.01% 場景
- **高丟包鏈路上 TCP 表現極差**——這推動 BBR、Hysteria 等 non-AIMD 演算法（下 lesson 詳）

#### 3.4 RACK reordering window 形式化

```
reo_wnd_init = min(SRTT/4, 100ms)
on DSACK observed (suggesting prior retx was spurious):
    reo_wnd ← min(2 × reo_wnd, SRTT)
```

自適應放大——對高 reorder path 自動容忍更大 reorder。

### 5. 我們協議的座標 / 設計取捨

| 設計面 | TCP reliability 知識影響 |
|---|---|
| **packet number 而非 byte seq** | 學 QUIC RFC 9000：simpler、可加密保護 |
| **RACK-TLP-derived loss detection** | mandatory；對 mobile 必要 |
| **PRR-style recovery pacing** | 避免 secondary loss |
| **F-RTO equivalent for spurious RTO** | 對 mobile handoff 必要 |
| **per-packet xmit timestamp** | enable RACK；不暴露 in plain |
| **ack frequency adaptive** | 不每 packet ACK；節省頻寬 |
| **ack range encoding** | 學 QUIC ACK frame：可表達多 range，比 SACK 多 |
| **packet number encryption** | 阻止 middlebox 推斷 loss rate（GFW 可能用此做 detection） |

### 7. 開放問題

- **Post-quantum TCP options** (TCP-AO 升級到 PQ MAC)
- **TCP over 5G mmWave / LEO satellite** — 高變動 BDP、handoff 頻繁——RACK + F-RTO 是否足夠？
- **AI/ML-tuned loss detection**：用 RL 動態調 reo_wnd / RTO——學術已有 prototype（Indigo, Pantheon）但 production 部署慢
- **Multipath TCP (MPTCP) 與 reliable delivery 互動**：多 subflow 同時 active 時 loss attribution 困難——MPTCP RFC 8684 有規範但實際細節 trial-and-error
- **形式化驗證 TCP loss detection state machine**：part of Linux TCP code 已被 Verifast / SeaHorn 工具掃過，**完整 mechanized proof of RACK-TLP** 尚未存在
- **量子網路下的可靠傳輸**：quantum channel 的 loss model 與 classical 截然不同——quantum-aware reliable delivery 是 open research
- **adversarial environment 下 TCP loss detection 的可破壞性**：GFW selective drop 對 RACK 觸發 recovery 的 pattern 是否成為 fingerprint？open
