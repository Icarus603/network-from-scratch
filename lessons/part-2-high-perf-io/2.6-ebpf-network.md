## 動機

> eBPF 是 G6 在 Linux 上的「擴展執行平台」

[2.5](./2.5-ebpf-intro.md) 把 eBPF 當 framework 講。本堂只看**對網路有用的 hook**——這是 G6 直接會用上的部分。

本堂講 6 個 hook 點，每個對 G6 有具體用法：

| Hook | G6 用法 |
|---|---|
| **TC ingress/egress eBPF** | server 端 packet 抗指紋自動檢驗 / DDoS 第二層防禦 |
| **Socket filter** | 細粒度 socket 級 filter（少用，被 TC/XDP 取代） |
| **Sockmap / sk_msg / sk_skb redirect** | plaintext proxy fast-path（baseline mode） |
| **SO_ATTACH_REUSEPORT_EBPF** | 多 worker 自訂 connection 分配策略 |
| **Cgroup-bpf / sockops** | 客戶端透明代理 / per-cgroup 流量隔離 |
| **SK_LOOKUP** | server 動態 socket 分派（多協議共用 port） |

XDP（driver-level）是另一個 networking eBPF hook 但**夠複雜要單獨一堂**——[2.7 XDP](./2.7-xdp.md) 講。本堂集中在 stack 內 hook。

## 核心概念

### 1. TC eBPF（traffic control classifier）

#### 1.1 TC 是什麼

TC (Traffic Control) 是 Linux qdisc / classifier / action subsystem。歷史上靠 user-space `tc` 命令配置 token bucket、HTB、netem 等 qdisc。

整個 Linux network stack 的 eBPF hook points（從低層往高層）：

```mermaid
flowchart LR
    NIC[NIC RX] --> XDP{{"XDP<br/>(driver/HW)"}}
    XDP -- XDP_PASS --> tci{{"TC ingress<br/>(cls_bpf / tcx)"}}
    tci --> NF1[netfilter prerouting] --> RT[routing]
    RT --> NF2[netfilter forward/input]
    NF2 --> SOCK[socket layer]
    SOCK --> SKM{{"sk_msg / sk_skb<br/>(sockmap, sockhash)"}}
    SKM --> APP[user-space app]
    APP --> SOP{{"sockops<br/>(cgroup/sock)"}}
    SOP --> NF3[netfilter postrouting]
    NF3 --> tce{{"TC egress<br/>(cls_bpf / tcx)"}}
    tce --> NICTX[NIC TX]
    classDef hook fill:#fde,stroke:#c39
    class XDP,tci,tce,SKM,SOP hook
```

每個粉色節點都是一個可以掛 eBPF 程式的 hook。本堂涵蓋除 XDP（[2.7](./2.7-xdp.md)）以外的所有 hook。

`cls_bpf` classifier 讓你**用 eBPF 寫 classifier**：每個 packet 進 TC ingress/egress hook，BPF 程式決定 verdict。

[2.13 tc / netem](./2.13-tc-netem.md) 會詳講 qdisc。本堂只看 BPF 的整合面。

#### 1.2 attach 一段 TC eBPF

```bash
# 編譯 BPF
clang -target bpf -O2 -c tc_classifier.bpf.c -o tc_classifier.bpf.o

# load + attach to eth0 ingress
sudo tc qdisc add dev eth0 clsact   # 一次性建立 clsact pseudo-qdisc
sudo tc filter add dev eth0 ingress bpf da obj tc_classifier.bpf.o sec ingress
# da = "direct action": BPF 直接返回 verdict，不用接 tc action

# 觀察
sudo tc filter show dev eth0 ingress
```

新 API（5.13+）用 `bpf_link + BPF_TCX_INGRESS`，比 `tc` 命令乾淨：

```c
// libbpf
struct bpf_tcx_opts opts = { .handle = 1, .priority = 1 };
struct bpf_link *link = bpf_program__attach_tcx(prog, ifindex, &opts);
```

#### 1.3 TC eBPF 程式長什麼樣

```c
SEC("classifier/ingress")
int ingress(struct __sk_buff *skb) {
    void *data = (void *)(long)skb->data;
    void *data_end = (void *)(long)skb->data_end;

    struct ethhdr *eth = data;
    if ((void *)(eth + 1) > data_end) return TC_ACT_OK;

    if (eth->h_proto == bpf_htons(ETH_P_IP)) {
        struct iphdr *ip = (void *)(eth + 1);
        if ((void *)(ip + 1) > data_end) return TC_ACT_OK;

        if (ip->protocol == IPPROTO_TCP) {
            // 量 TCP packet size 分布 (for G6 抗指紋驗證)
            u32 key = ip->daddr;
            u64 *cnt = bpf_map_lookup_elem(&size_per_dst, &key);
            // ...
        }
    }

    return TC_ACT_OK;   // 放行
    // TC_ACT_SHOT  = drop
    // TC_ACT_REDIRECT = ...
    // TC_ACT_PIPE = 繼續下一個 filter
}
```

verdict 常數見 `<linux/pkt_cls.h>`。

#### 1.4 TC vs XDP 的差別

| 維度 | TC eBPF | XDP |
|---|---|---|
| 位置 | 進 stack 後（skb 已建立） | driver hook（skb 還沒建） |
| 可改包 | 是（含 metadata） | 是 |
| 可 drop | 是（TC_ACT_SHOT） | 是（XDP_DROP） |
| egress 支援 | 是 | 否（只 ingress） |
| 性能 | ~5-10 Mpps | 24+ Mpps |
| 通用性 | 任何 NIC | 需 driver 支援（多數新 NIC） |
| 對 G6 | server 出口量測 / egress shaping | server 入口 DDoS / fast-drop |

⭐ **G6 server 設計**：XDP 在最前線（DDoS / 早期 filter），TC 在第二線（細粒度 shaping、出口量測）。

### 2. Socket filter (古典 BPF + eBPF)

`SO_ATTACH_FILTER` 是最古老的 BPF 接入點（cBPF 時代），eBPF 版本叫 `SO_ATTACH_BPF`：

```c
struct sock_fprog fp = { ... };
setsockopt(sock, SOL_SOCKET, SO_ATTACH_FILTER, &fp, sizeof(fp));
```

語意：每個 packet 進此 socket recv buffer 前，BPF program 決定 drop/keep。

**現在很少用**：TC ingress 比 socket filter 更早（packet 還沒分到 socket）、更通用。socket filter 主要還是 tcpdump 的傳統。

G6 不用 socket filter——TC 與 XDP 已涵蓋。

### 3. Sockmap / sk_msg / sk_skb：socket-to-socket 直連

這是**對 proxy 最直接 relevant** 的 eBPF feature。

#### 3.1 概念

**問題**：傳統 user-space proxy 流程：

```
client → socket A → recv() → user buffer → send() → socket B → upstream
                    1 syscall              1 syscall
```

如果 packet **不需要 user 處理**（如 plaintext pass-through），這 2 syscall 是浪費。

**Sockmap 解法**：在 kernel 內 redirect。

```
client → socket A → ★ sk_skb / sk_msg program decides ★ → directly forward to socket B
```

完整 kernel-side，**user 不參與 data path**。

#### 3.2 三個關鍵元件

1. **Sockmap / sockhash**：map type 存「(key) → socket pointer」
2. **sk_skb program**：附在 socket，packet 進 receive queue 時觸發；decide 是否 `bpf_sk_redirect_map(skmap, key, flags)`
3. **sk_msg program**：附在 socket，sendmsg() 時觸發；decide 是否 redirect

#### 3.3 範例：plaintext proxy in kernel

```c
struct {
    __uint(type, BPF_MAP_TYPE_SOCKHASH);
    __uint(max_entries, 65535);
    __type(key, u32);
    __type(value, int);   // socket fd
} sock_map SEC(".maps");

SEC("sk_skb/stream_verdict")
int verdict(struct __sk_buff *skb) {
    // 從 skb 取 client connection id（user-space 預先存）
    u32 key = ...;   // e.g. hash of 4-tuple
    return bpf_sk_redirect_hash(skb, &sock_map, &key, 0);
}
```

user-space：

```c
// 把 client socket A 跟 upstream socket B 都加進 sockmap
bpf_map_update_elem(map_fd, &keyA, &sockA, BPF_ANY);
bpf_map_update_elem(map_fd, &keyB, &sockB, BPF_ANY);
// 把 sk_skb program attach 到 A 跟 B
bpf_prog_attach(prog_fd, sockA, BPF_SK_SKB_STREAM_VERDICT, 0);
```

之後 A 收到 packet → BPF 決定送 B → kernel 內直接 enqueue 到 B 的 send buffer → 出去。**0 user-space 參與**。

#### 3.4 對 G6 的應用：plaintext fallback / TLS-in-tunnel

**G6 主流量必須加密**，不能直接 sockmap forward。但有兩個情境可用：

1. **TLS-in-tunnel 模式**：如果 G6 用「外層 TLS + 內層加密」雙層，且**外層 TLS 由 kTLS 接管**——剩下 ciphertext byte stream 可以 sockmap forward 到 backend service（如 nginx fronting）
2. **Plaintext debug / benchmark mode**：開發過程量 baseline 用

效能：sockmap 在 4KB packet 場景下 vs user-space splice，提速 ~30%（Cilium 數據）。

#### 3.5 限制

- 只支援 TCP / UDP
- 必須 user-space 在 connection 建立時把 socket 加進 map（add 不是 BPF 端做）
- packet boundary 不一定保留（TCP byte stream 特性）
- 6.x 起穩定，但 5.x 有不少 bug

### 4. SO_ATTACH_REUSEPORT_EBPF

[2.1 §7](./2.1-select-poll-epoll.md#7-驚群問題thundering-herd) 提過 `SO_REUSEPORT` 是 N worker 共用 listen port 的 hack。`SO_ATTACH_REUSEPORT_EBPF` 讓你**用 eBPF 自訂分配**。

#### 4.1 預設 hash 策略

```
socket_id = hash(src_ip, src_port, dst_ip, dst_port) mod N
```

問題：

- 不能保證「同 client」永遠 hash 到同 worker（4-tuple 變了）
- 不 NUMA-aware
- worker 動態 add/remove 時，現有 connection rebalance 不平滑

#### 4.2 用 eBPF 自訂

```c
struct {
    __uint(type, BPF_MAP_TYPE_REUSEPORT_SOCKARRAY);
    __uint(max_entries, 32);
    __type(key, u32);
    __type(value, u64);
} sk_array SEC(".maps");

SEC("sk_reuseport/select")
int select_sock(struct sk_reuseport_md *ctx) {
    // 自訂邏輯：例如根據 client IP 親和度
    u32 key = ((u32)(uintptr_t)ctx->ip) % 32;
    bpf_sk_select_reuseport(ctx, &sk_array, &key, 0);
    return SK_PASS;
}
```

user-space 把每個 worker 的 listen socket 放 array key 0..N-1。BPF 程式決定每條 incoming connection 給哪個 worker。

#### 4.3 對 G6 的用法

1. **Per-CPU pinning**：用 client IP hash mod 8 → 8 個 worker，每個 pin 在固定 CPU。L1/L2 cache locality 最大化
2. **Sticky session**：對「同 client」持續走同 worker（連線狀態 cache 在那個 worker 內）
3. **Graceful worker rollout**：滾動更新時逐個替換 worker，不像 default hash 那樣斷既有連線
4. **Adversarial defense**：若某條 client IP 行為可疑，引導到「sandbox worker」做 deep inspection

⭐ G6 server 預期默認用 `client-IP-affinity-then-CPU-load-balance` eBPF program。**比 nginx 預設的 round-robin 細很多**。

### 5. Cgroup-bpf：per-cgroup network policy

cgroup-bpf 把 BPF program attach 到 cgroup，影響該 cgroup 內所有 process 的網路行為。

#### 5.1 attach 類型

| Attach type | 觸發時機 | 用途 |
|---|---|---|
| `BPF_CGROUP_INET_INGRESS` | packet 進 cgroup 內 socket | per-cgroup firewall |
| `BPF_CGROUP_INET_EGRESS` | packet 出 cgroup | egress filter |
| `BPF_CGROUP_INET_SOCK_CREATE` | `socket()` syscall | 限制 socket 型態 |
| `BPF_CGROUP_INET4_CONNECT` / `INET6_CONNECT` | `connect()` syscall | **重寫目標位址（透明代理！）** |
| `BPF_CGROUP_UDP4_SENDMSG` / `UDP6_SENDMSG` | UDP sendmsg | per-message 控制 |
| `BPF_CGROUP_SOCK_OPS` | TCP state change | tune TCP option |
| `BPF_CGROUP_SETSOCKOPT` / `GETSOCKOPT` | setsockopt | 攔截 / 修改 socket option |

#### 5.2 範例：透明代理（connect rewrite）

對 cgroup 內所有 process 的 `connect()` syscall 攔截，把 `dst = upstream_real_addr` 改寫成 `dst = local_proxy`，**user 程式完全不知情**：

```c
SEC("cgroup/connect4")
int redirect(struct bpf_sock_addr *ctx) {
    // 原本要 connect 1.2.3.4:443
    // 我們改成 connect 127.0.0.1:8080
    ctx->user_ip4 = bpf_htonl(0x7f000001);   // 127.0.0.1
    ctx->user_port = bpf_htons(8080);
    return 1;
}
```

attach：

```c
bpf_prog_attach(prog_fd, cgroup_fd, BPF_CGROUP_INET4_CONNECT, 0);
```

#### 5.3 對 G6 客戶端的應用

**這就是 G6 client 透明代理的最乾淨機制**：

1. 用戶啟動 G6 client（自己進入一個 cgroup，例如 `/sys/fs/cgroup/g6/`）
2. cgroup connect4 BPF program 把所有 outbound connect 改寫成 `127.0.0.1:G6_PORT`
3. G6 client 在 `127.0.0.1:G6_PORT` listen，從 socket 拿原始 dst（透過 `SO_ORIGINAL_DST` 或 BPF 設定的 metadata）
4. G6 client 用我們協議 encrypted forward 到 upstream

**比 iptables redirect / tun2socks 強的點**：

- 不需要 root（只需要 cgroup 寫權限）
- 不修改 routing table
- 跨 process 透明（不像 SOCKS5 要 application 支援）
- 比 iptables OUTPUT REDIRECT 性能好（不過 conntrack）

#### 5.4 限制

- 只支援 Linux。macOS / Windows 沒有
- 需要 cgroup v2 (cgroupv2 unified hierarchy)
- 必須 root 或有 CAP_NET_ADMIN load BPF
- 對 IPv6 / DNS / non-TCP/UDP 流量需另外處理

### 6. SK_LOOKUP：動態 socket 派發

5.9+ 引入。允許用 BPF program 自訂「哪個 socket 接收這條新連線」。

#### 6.1 應用：多協議共用 port

```
:443 → BPF SK_LOOKUP →
       SNI = "g6.example.com" → G6 server
       SNI = "site.example.com" → nginx
       no SNI / fallback → nginx default
```

不用 SO_REUSEPORT trick、不用 nginx stream proxy——**kernel 內直接派發**。

#### 6.2 範例

```c
struct {
    __uint(type, BPF_MAP_TYPE_SOCKMAP);
    __uint(max_entries, 16);
    __type(key, u32);
    __type(value, u64);
} backends SEC(".maps");

SEC("sk_lookup/lookup")
int lookup(struct bpf_sk_lookup *ctx) {
    if (ctx->protocol != IPPROTO_TCP) return SK_PASS;
    if (ctx->local_port != 443) return SK_PASS;

    // peek 第一個 byte 判斷 TLS ClientHello / 其他
    // (但 SK_LOOKUP 沒法 read packet payload，這是 limit)

    u32 key = 0;
    struct bpf_sock *sk = bpf_map_lookup_elem(&backends, &key);
    if (!sk) return SK_PASS;
    return bpf_sk_assign(ctx, sk, 0);
}
```

#### 6.3 對 G6 的用法

1. **單 port 多服務**：G6 server 跟 nginx 共用 443，BPF 根據 connection metadata 派發
2. **動態 backend swap**：rolling update 時切換 backend pool 不用斷連
3. **REALITY 整合**：REALITY 設計核心是「**G6 server 跟 nginx 共用 443，由 G6 探測 TLS ClientHello 決定是真 nginx 還是 G6 流量**」——這個邏輯**可以用 SK_LOOKUP 實作**（雖然目前 REALITY 用 user-space 切換）

⭐ Part 7.7 講 REALITY 時會回頭討論「能否用 BPF SK_LOOKUP 寫一個更乾淨的 REALITY」。

### 7. sockops + setsockopt 動態 tune

`BPF_CGROUP_SOCK_OPS` 在 TCP state change 時被呼叫，能 inline tune TCP option：

```c
SEC("sockops")
int tune(struct bpf_sock_ops *skops) {
    if (skops->op == BPF_SOCK_OPS_ACTIVE_ESTABLISHED_CB) {
        // connection established，立刻調 congestion control
        bpf_setsockopt(skops, SOL_TCP, TCP_CONGESTION, "bbr", 3);
        // 調 TCP_NOTSENT_LOWAT
        int v = 16384;
        bpf_setsockopt(skops, SOL_TCP, TCP_NOTSENT_LOWAT, &v, sizeof(v));
    }
    return 1;
}
```

對 G6 用法：

- 動態切換 congestion control 算法（檢測到 lossy 路徑切 BBR、低延遲切 CUBIC）
- 動態調整 `TCP_USER_TIMEOUT` 等參數
- 觀察 RTT 分布做 telemetry

### 8. SK_REUSEPORT / SK_LOOKUP / sockmap 對比

| 機制 | 何時觸發 | 對 G6 用法 |
|---|---|---|
| **SO_REUSEPORT default hash** | 新 connection | 啥都不做的 default |
| **SO_ATTACH_REUSEPORT_EBPF** | 新 connection | per-client-IP 親和分配 |
| **SK_LOOKUP** | 新 connection（更早） | 多服務共 port（REALITY-style） |
| **sockmap + sk_skb redirect** | packet 進 socket recv queue | proxy fast-path（plaintext only） |
| **cgroup connect4 redirect** | client 端 `connect()` syscall | 透明代理 |

### 10. 工程實踐：bpftool / tcx / production deployment

#### 10.1 bpftool 命令速查

```bash
bpftool prog list                  # 列出所有 loaded program
bpftool map list                   # 列出 map
bpftool prog dump xlated id NN     # 反組譯 BPF bytecode
bpftool prog dump jited id NN      # 反組譯 native code
bpftool net attach xdp prog id NN dev eth0
bpftool net attach tcx_ingress prog id NN dev eth0
bpftool feature probe              # 看 kernel 支援哪些 BPF feature
```

#### 10.2 用 systemd-managed BPF

systemd 247+ 支援 BPF program lifecycle management：

```ini
[Service]
BPFProgram=ingress:bpffs/my-prog
IPAddressDeny=any
IPAddressAllow=10.0.0.0/8
```

對 G6 server 部署：用 systemd unit 管理 BPF program load + attach，cleanup 自動。

#### 10.3 多 program tail call / freplace

複雜邏輯拆多個 BPF program，用 **tail call** chain：

```c
SEC("tc/dispatcher")
int dispatch(struct __sk_buff *skb) {
    bpf_tail_call(skb, &prog_array, 0);
    return TC_ACT_OK;
}
```

或 **freplace**（5.10+）：在 production runtime 替換已 loaded 的 BPF program，**不斷服務**。Cilium 重用。

## 與我們協議設計的關聯

1. **客戶端透明代理**：cgroup connect4 redirect 是 Linux 透明代理首選。比 iptables 強很多
2. **Server worker 分配**：SO_ATTACH_REUSEPORT_EBPF + per-client-IP affinity
3. **單 port 多服務**：SK_LOOKUP 可實作 REALITY-style port 共用
4. **觀測**：TC eBPF + ring buffer 量出口 packet 特徵，自我驗證抗指紋
5. **不用 sockmap 主流量**：因加密；但 plaintext baseline / TLS-in-tunnel 場景可考慮
6. **TCP tuning**：sockops + bpf_setsockopt 動態調 congestion control（packet loss 高自動切 BBR）
7. **macOS client 沒有對等品**：cgroup / sockmap 都是 Linux 專屬。macOS 走 Network Extension（[2.10 講](./2.10-macos.md)）

## 動手

### 實驗 A：寫一個 TC eBPF 量自家出口流量 size 分布

```c
// size_hist.bpf.c
SEC("tc/egress")
int egress(struct __sk_buff *skb) {
    u32 bucket = skb->len / 100;     // 100B per bucket
    if (bucket >= 64) bucket = 63;
    u32 zero = 0;
    u64 *cnt = bpf_map_lookup_elem(&hist, &bucket);
    if (cnt) __sync_fetch_and_add(cnt, 1);
    return TC_ACT_OK;
}
```

attach 到自己服務的 interface，跑你的 prototype 測試流量，用 `bpftool map dump` 看分布。**這就是 G6 抗指紋驗證的最小 lab**。

### 實驗 B：寫一個 SO_ATTACH_REUSEPORT_EBPF program

簡單版：永遠選 worker 0；之後改成「依 client IP hash mod N」。
測試 N=4 worker 是否真的按你寫的策略分配。用 `ss -tnl --bpf` 看附加狀態。

### 實驗 C：cgroup connect4 redirect 做 minimal 透明代理

```bash
mkdir -p /sys/fs/cgroup/g6test
echo $$ > /sys/fs/cgroup/g6test/cgroup.procs   # 把當前 shell 加進去
bpftool prog load redirect.bpf.o /sys/fs/bpf/redirect type cgroup/connect4
bpftool cgroup attach /sys/fs/cgroup/g6test connect4 pinned /sys/fs/bpf/redirect

# 現在這個 shell 啟動的 process 所有 IPv4 connect 都被重定向
curl https://example.com   # 實際上連到 127.0.0.1:8080
```

debug 用 `tc exec bpf debug` 或 `bpftool prog tracelog`。

### 實驗 D：sockmap plaintext echo

寫一個 minimal echo server：accept client，把 socket 加進 sockmap，attach sk_skb program 把收到的 packet 直接 redirect 回 client socket。

用 `bpf_printk` 在 BPF 內 log。用 `cat /sys/kernel/debug/tracing/trace_pipe` 看。

### 2. 對手分類學：BPF 對 GFW 升級的影響

| 對抗階段 | GFW 工具 | G6 對策 |
|---|---|---|
| TCP/IP fingerprint | conntrack + iptables | TCP_CONGESTION control via sockops |
| TLS SNI inspection | DPI parser | encrypted SNI / 不暴露 SNI |
| Packet size pattern | flow analysis | TC eBPF self-monitoring + 隨機化 |
| Timing pattern | inter-packet delay analysis | jitter injection + measurement |

⭐ **重點**：BPF 既是 GFW 升級的潛在工具，也是 G6 自我驗證的工具。**You can't defend what you can't measure**——eBPF 給你「能 measure」的能力。

### 3. 形式化定義：proxy data path latency budget

對一條 client→G6→upstream 的 packet，定義端到端延遲：

$$
L_{\text{total}} = L_{\text{net,c→s}} + L_{\text{queue,c}} + L_{\text{stack,c}} + L_{\text{proxy,user}} + L_{\text{stack,s}} + L_{\text{queue,s}} + L_{\text{net,s→u}}
$$

每段優化：

- $L_{\text{queue}}$：sockmap redirect 把 user-space 段壓到 0（但加密不適用）
- $L_{\text{stack}}$：XDP 略過 stack（但加密不適用）
- $L_{\text{proxy,user}}$：io_uring + register_buf_ring 壓最低
- $L_{\text{net}}$：路徑問題，無解

對 G6 加密協議，$L_{\text{proxy,user}}$ 是優化最大空間，本堂 eBPF tooling 是「**監控** $L$ 各段的工具」而非優化機制本身。

1. **加密協議 + sockmap 整合**：能否設計一個「sk_msg program 內呼叫 AEAD」的擴展？需要 BPF verifier 支援 SIMD-like crypto ops。對 G6 是 future win
