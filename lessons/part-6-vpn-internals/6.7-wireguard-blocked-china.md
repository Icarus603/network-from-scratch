## 動機

我們花了 4 堂課讓你看到 WireGuard 的 spec / 原始碼，**這是 Phase II 抗審查分析的第一個實戰**。Donenfeld 2017 寫 whitepaper 時，明確說「stealth」目標只覆蓋 passive observer。GFW 是 active adversary，於是 WireGuard 在中國境內**早晚會被打**——而我們現在看到的，正是這個「早晚」的真實時間線。

對研究 G6 的目標，這堂課是**最關鍵的「不要怎麼做」清單來源**：
- WireGuard 已經是 "best practice spec"——可被機器驗證、被密碼學家 audit、被部署 8 年。
- **它仍然被打**。為什麼？因為「passive stealth ≠ probe resistance ≠ traffic-analysis resistance」三件事完全不同。
- AmneziaWG 是社群對「怎麼補救已部署 WireGuard」的工程嘗試。它不完美，但它的 mod 思路是 G6 設計時必須學的範本。

## 核心概念

### 1. WireGuard 的 fingerprint 矩陣

把 Xue 2022 framework（[6.2 §3](6.2-openvpn-anatomy.md)）套到 WireGuard：

| Fingerprint 類別 | WireGuard 在 spec 層的洩漏 | 對應 source code |
|---|---|---|
| **#1 byte pattern** | `msg_type ∈ {0x01, 0x02, 0x03, 0x04}` 後接 3 bytes 0（即 type 在 LE uint32 的低位） | noise-protocol.go:58-62 |
| **#2 packet size #1** | handshake init = 148 bytes，handshake response = 92 bytes，cookie reply = 64 bytes | noise-protocol.go:65-67 |
| **#3 packet size #2** | transport packet 大小 = 16-byte header + (16-byte multiple inner) + 16-byte tag | send.go:439 (RoutineEncryption) |
| **#4 timing pattern** | handshake init → response RTT 與後續 transport burst 模式 | timer state machine |
| **#5 active probe response** | 對 valid-pk MAC1 的 init → response；invalid MAC1 silent drop | RoutineHandshake (receive.go:271+) |
| **#6 IP/port pattern** | UDP，server 通常固定 port（例如 51820，但不強制） | conn/bind_std.go |

**這構成完整的 fingerprint 集合**。對手只需要把 **#1 + #2 + #5** 串起來——跟 Xue 2022 對 OpenVPN 的 framework 結構完全一樣。

### 2. byte-pattern fingerprint：第一封 packet 第一 byte

`MessageInitiationType = 1`。整個 4-byte type field 在 little-endian 上長這樣：

```
offset 0: 0x01 0x00 0x00 0x00
```

**第一 byte = 0x01**。對 DPI 而言，UDP packet 第一 byte 為 0x01 已是極強訊號。再加上其他 type（0x02/0x03/0x04），總 entropy 不到 2 bits。

對比一個「真實 entropy uniform first byte」protocol：第一 byte 應該是 `[0, 256)` 接近 uniform，且**對外無 protocol identity 資訊**。

> **研究級觀察**：Wu et al. 2023 對 Shadowsocks 的偵測精度核心就是「fully-encrypted protocol 應該前若干 byte 接近 uniform」——而 GFW 用 entropy test 把 SS 流量挑出來。WireGuard 第一 byte 0x01 連這個 entropy test 都過不了——**比 SS 還容易被識別**。SS 至少前面是真的加密過的 random bytes；WireGuard 是明文常數。

### 3. packet-size fingerprint：固定 148/92/64

```
ClientHandshakeInit:    148 bytes
ServerHandshakeResponse: 92 bytes
CookieReply:             64 bytes
```

這三個常數**寫在 spec 裡**（noise-protocol.go:65-67），任何 RFC-compliant 實作都這個 size。

組合 #1 + #2 = **deterministic** detection：
```
UDP packet
  AND first byte ∈ {0x01, 0x02, 0x03, 0x04}
  AND length ∈ {148, 92, 64}
  → 100% confident WireGuard handshake
```

DPI box 用 4 個 if 即可篩出 99.9% WireGuard handshake。**False positive 來源**只有真的剛好同 size 的 UDP——實務上幾乎不存在。

### 4. transport packet size：16-byte multiple 洩漏 inner traffic

[6.5 §1.3](6.5-wireguard-source-datapath.md) 看到：`calculatePaddingSize` 把 inner packet padding 到 16-byte multiple，再加 16-byte header + 16-byte AEAD tag。

實際 outer UDP packet size = `16 + ((inner_size + 15) & ~15) + 16`。

**這意味著什麼**：對手看到 outer packet size 即知道 inner packet size 在 ±16 bytes 內。inner traffic 的 size distribution **完全反映**在 outer。

對流量分析（Beauty, FlowPrint, k-fingerprinting）：
- 看到 outer size 序列 → 知道 inner packet size 序列。
- inner size 序列 → 識別 inner protocol（HTTP / Video / SSH）。
- 識別出 inner → 即可推斷 user behavior。

[Part 10 流量分析](../part-10-traffic-analysis/) 完整展開。**WireGuard 對流量分析的防禦近乎零**——這是 spec 設計者明確說「不做」的事。

### 5. active probing 對 WireGuard 的攻擊

MAC1 對 random probe 有效——但對「**已知 server static pk** 的對手」無防護。

**對手如何取得 server static pk**？
- 任何 legit user 拿到 server config 都有 server pk。
- 商業 VPN 提供商把 server pk 寫在 official config 裡分發給數百萬用戶。
- **GFW 只需要購買一份商業 VPN 服務**，取得 server pk，即可對該 IP 構造 valid MAC1 探測。

```python
# 假設 GFW 已知 server static pk
import nacl.bindings as nacl_b
mac1_key = blake2s(b"mac1----" + server_static_pk)
init_msg = bytes(148)
init_msg[0:1] = b"\x01"
init_msg[1:4] = b"\x00\x00\x00"
init_msg[4:8] = random_session_id
init_msg[8:40] = random_ephemeral_pk  # 隨機 X25519 公鑰
# 跳過 Static + Timestamp 都填 random（無法解密但 server 會驗證 MAC1 後才嘗試解密）
init_msg[116:132] = blake2s_mac(mac1_key, init_msg[0:116])
init_msg[132:148] = bytes(16)  # MAC2 全零，server 不在 UnderLoad 時接受

# 送到 suspected WG server
sock.sendto(init_msg, (suspected_ip, 51820))
reply, _ = sock.recvfrom(1500)
if len(reply) == 92 and reply[0] == 0x02:
    print("Confirmed WireGuard server!")
```

**這就是 Xue 2022 第三條 fingerprint 在 WireGuard 上的等價**。**MAC1 阻止不了知道 server pk 的對手**。

> **研究級觀察**：MAC1 設計時 Donenfeld 的假設是「server pk 是 secret-ish」。但**任何 deployed 商業 VPN 都會洩漏 server pk**。GFW 對中國境內活躍 server IP 做 enumerate + probe，極低成本識別 WG。**這是 protocol 設計 vs deployment reality 的 gap**——G6 必須假設 server identity **必然洩漏**，做 day-1 防禦。

### 6. 2022-2023 中國對 WireGuard 的封鎖時間線

| 時期 | 觀察 | 來源 |
|---|---|---|
| 2018-2021 | WireGuard 在中國少量使用，主要靠 niche，GFW 興趣不大 | 站內回報 |
| 2022 H1 | Xue 2022 USENIX Sec 公布 OpenVPN framework；研究界開始把 framework 套到 WG | Xue 2022 |
| 2022 Q4 | GFW.report 與用戶觀察：對若干 WG endpoint 出現 selective interference | GFW.report |
| 2023 H1 | Wu 2023 USENIX Sec 公布 fully encrypted detection；framework 適用於 WG | Wu 2023 |
| 2023 H2 | 主流商業 VPN 提供商的 WG endpoint 在中國境內**大規模被打**：握手成功率 < 30% | 多家 provider blog |
| 2024 | AmneziaWG 2.0 發佈，加入 ranged headers + I1-I5 signature CPS | AmneziaWG release notes |
| 2025 | AmneziaWG 變成中國 / 俄羅斯境內主流「WG-derived」協議 | 社群觀察 |

**GFW 對 WG 的打法（推測；未公開源頭）**：
1. **passive DPI**：filter UDP packet with first byte 0x01 + length 148 → 標記 suspect。
2. **timer-based correlation**：suspect 出現後 < 5 秒看到 first-byte 0x02 + length 92 → confirmed handshake。
3. **drop or RST**：對已 confirmed flow drop 後續 UDP，或用 ICMP unreachable 觸發 client 重連風暴。

實證：在中國境內，**裸 WireGuard 對某些 IP/port 的握手成功率 < 50%**（GFW.report 數據）。

### 7. AmneziaWG 的 mod 範本

[AmneziaWG 文件](https://docs.amnezia.org/documentation/amnezia-wg/) 把 WG 的 fingerprint 一條條補。它是**最完整的 WG 抗審查 fork**。

#### 參數總覽：

| 參數 | 範圍 | 作用 | 對應 fingerprint |
|---|---|---|---|
| **Jc** (Junk count) | 0~10 | 握手前送 Jc 封純隨機 UDP packet | #4 timing + 提早噪音 |
| **Jmin / Jmax** | 0~MTU | junk packet size 範圍 | #2 |
| **S1** | 15~150 | handshake init 前 prepend S1 bytes random | #2 → init size 變 148+S1 |
| **S2** | 15~150（S1+56 ≠ S2 hard rule） | handshake response 前 prepend S2 bytes random | #2 → response size 變 92+S2 |
| **S3 / S4**（AWG 2.0） | 0~32/64 | cookie / data packet padding | #2 → variable size |
| **H1 / H2 / H3 / H4** | uint32 | 把 message type 0x01/0x02/0x03/0x04 改成任意 4-byte value | #1 → first byte 不再是 0x01-0x04 |
| **I1~I5**（AWG 2.0） | byte spec | handshake 前送 N 封偽 protocol（如 mimic QUIC）packet | #1+#2+#4 偽裝成別的 protocol |

#### S1+56 ≠ S2 的硬限制

```
148 + S1 == 92 + S2
=>  S2 - S1 == 56
```

若 S2 - S1 = 56，padded init 與 padded response 同 size——對手用「兩封 same-size UDP packet 短 timing」即可重新 fingerprint。AmneziaWG 在配置時 enforce S1+56 ≠ S2。

**這是極好的設計細節**：**fix 一個 fingerprint 不能引入另一個**。對 G6 同樣警示：所有 obfuscation 參數要做 conflict analysis。

#### H1-H4 的意義

```
H1: init magic header (取代 0x01_00_00_00)
H2: response magic header (取代 0x02_00_00_00)
H3: transport magic header (取代 0x04_00_00_00)
H4: under-load (cookie) magic header (取代 0x03_00_00_00)
```

四個 32-bit 值，**per-deployment 自定義**。client 與 server 必須一致。AmneziaWG 2.0 進一步把它擴成 **ranged headers**（每次握手從 range 內挑），降低 H1~H4 本身的可 fingerprint 性。

**對 G6 的對照**：完全動態 ranged header 是 G6 的目標——但 G6 應該更進一步：**第一 byte 必須 entropy-uniform**，連 H1~H4 都不該有 stable 集合。

#### I1-I5 signature CPS（chained protocol simulation）packets

「**握手前送 5 封偽協議 packet**」。每封 `I_n` 用 tag 語言定義內容：
- `<r 128>` = 隨機 128 bytes
- `<b 0xc000000001>` = 固定 prefix bytes（如 QUIC Initial header magic）
- `<t>` = current timestamp

例：`<b 0xc000000001><r 64><t>` 看起來像「**QUIC Initial packet 開頭**」——DPI 可能把 flow 分類成 QUIC 而非 WG。

**這是「protocol mimicry」**：偽裝成主流允許的 protocol。[Part 7] 詳述歷史上 mimicry 的成敗（FTE, Marionette 等）。

### 8. AmneziaWG vs G6：mod 的局限

AmneziaWG 是**最好的「現有 WG 救火」方案**，但**還不是 G6 級的設計**。差距：

| 議題 | AmneziaWG 做法 | G6 要做 |
|---|---|---|
| First byte | ranged H1-H4，仍可能 cluster | **完全 entropy-uniform 第一 byte** |
| Padding | S1-S4 隨機 prepend | **MTU-fill padding policy**（每封都 MTU）|
| Mimicry | I1-I5 偽 QUIC header | **真實 QUIC handshake outer envelope**（不是模仿） |
| Server probe-resistance | 仍依賴 MAC1 + server pk secrecy | **REALITY-style: 借用 real website TLS**（[Part 7.10](../part-7-proxy-protocols/)）|
| Transport size hiding | S4 padding | **per-packet cover traffic + size sampling** |
| PQ 安全 | 無 | **X25519 + ML-KEM-768 hybrid** |
| Formal verification | 無 | **CryptoVerif 機器驗證 spec** |

**關鍵差異**：AmneziaWG 是 **patch on top of WG**——根本架構仍是 WG。G6 是**重新設計**——day-1 把所有 anti-censorship 與 PQ 寫進 spec。

### 9. WireGuard "fundamental" 缺陷：probe-resistance vs. server identity 洩漏

**根本問題**：WireGuard 把 server identity（static pk）作為「**OOB-distributed secret-ish**」處理。MAC1 設計假設只有合法 client 知道 server pk。**這在小規模 self-hosted 成立，在大規模商業 deployment 完全失效**。

任何 protocol 只要把 server identity 視為「broadcast secret」(分發給 N 用戶 = 給 N 個 prober 機會)，必然遇到同問題。

**解法**（Part 11 G6 設計）：
1. **server identity 為公開公鑰，但握手前不洩漏被連的是誰**：用 PSK 或 KEM-encapsulation 把 server identity 包進加密層。
2. **借殼 real server**：REALITY 模式——讓 prober 看到的是真實第三方 server 的 TLS。
3. **CDN / Domain Fronting**：用 fronted server 隱藏真實 endpoint。

每條路都是 [Part 7] 翻牆協議演化的主題。

### 10. 對研究路線的綜合啟示

**研究級觀察**：WireGuard 在 2017 是 best-of-class spec design；2025 仍是最簡潔可審計的 VPN protocol；但**在 anti-censorship 維度它從 day 1 就不夠**。這不是 Donenfeld 的錯——他清楚說了 "stealth" 限於 passive。**錯的是用戶把它當作翻牆協議**。

G6 的設計起點正是「**承認 WireGuard 級的 protocol simplicity 不能犧牲，但 anti-censorship 必須 day-1 加入**」。具體做法在 [Part 11] 展開，但本堂結束你應該已能列出 G6 必須回答的 8 個問題：
1. 第一 byte 怎麼 uniform？
2. handshake size 怎麼 hide？
3. transport size 怎麼 hide？
4. Active probing 怎麼防（不依賴 server pk secrecy）？
5. PQ migration 怎麼做？
6. Roaming 怎麼避免 path spoof？
7. Cover traffic 何時送、多少送？
8. spec 怎麼形式化驗證？

## 與我們協議設計的關聯

### G6 day-1 設計守則（直接抄）

從本堂與 AmneziaWG 對比直接落地的 G6 守則：

1. **type byte must be encrypted-or-uniform**：第一 byte 永遠不能透露 protocol identity。
2. **handshake messages must be size-uniform-padded or padded to MTU**：消除 #2 fingerprint。
3. **probe response must be indistinguishable from generic non-protocol server**（REALITY-style）：消除 #5 fingerprint。
4. **server static pk must not gate probe resistance**：假設 server pk 公開。
5. **constraints on obfuscation params must be enforced at config time**：學 AmneziaWG `S1+56 ≠ S2` 教訓——所有參數組合必須做 conflict analysis。
6. **AmneziaWG-style 配置複雜性是技術債**——G6 設計應**內建** entropy-uniform，不是 user 設定 H1~H4。
7. **mimicry 是脆弱的**：用 real-protocol-shape 而非「mimicked-shape」（REALITY > FTE）。

## 動手（可選）

### 實驗 6.7.A：寫 Python script 對 self-hosted WG server 做 active probe

```python
# probe.py
import socket, os, hashlib

server_ip = "10.0.0.1"
server_port = 51820
# Assume you know server static pk (from your own config)
server_static_pk = bytes.fromhex("...")

# Build minimal handshake init that passes MAC1
def build_init(server_pk):
    import hashlib, secrets
    # mac1 key = blake2s(LABEL_MAC1 || server_pk)
    mac1_key = hashlib.blake2s(b"mac1----" + server_pk, digest_size=32).digest()
    msg = bytearray(148)
    msg[0] = 0x01
    msg[4:8] = secrets.token_bytes(4)
    msg[8:40] = secrets.token_bytes(32)   # ephemeral
    # Static (encrypted) + Timestamp (encrypted): random bytes - 我們不關心
    msg[40:88] = secrets.token_bytes(48)
    msg[88:116] = secrets.token_bytes(28)
    # MAC1
    h = hashlib.blake2s(bytes(msg[:116]), key=mac1_key, digest_size=16)
    msg[116:132] = h.digest()
    # MAC2 = 0
    return bytes(msg)

sk = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
sk.settimeout(2)
sk.sendto(build_init(server_static_pk), (server_ip, server_port))
data, _ = sk.recvfrom(1500)
print(f"Got {len(data)} bytes, first byte 0x{data[0]:02x}")
# Expected: 92 bytes, 0x02 → confirmed WG server
```

**研究問題**：把 MAC1 隨機填，server 應該 silent drop。確認你的 socket 在 timeout 後 raise exception。

### 實驗 6.7.B：架 AmneziaWG，跟 wireguard-go 對比抓包

下載 `amneziawg-go`，跟 `wireguard-go` 各跑一條 tunnel。各抓 50 封 handshake。比對：
- first byte 分布（WG 都 0x01；AWG 隨 H1）
- packet size 分布（WG 148 fixed；AWG 148+S1）

### 實驗 6.7.C：用 Wu 2023 fully-encrypted detection 規則套你抓的 WG 流量

Wu 2023 的核心 rule：**前 6 bytes 中至少 5 bytes 必須在 ASCII printable 範圍才不被視為 fully-encrypted**。對 WG handshake：第一 byte 0x01 → 直接被歸類為 fully-encrypted。**這意味著 GFW 對 SS 的偵測規則對 WG 同樣有效**。

### 實驗 6.7.D（推薦）：把本堂的 fingerprint 矩陣（6 條）逐一寫成 detection Python module，組合成一個 `is_wireguard(udp_flow)` 函數。

## 研究級補遺

### 1. 學界詞彙

- **Protocol mimicry**：偽裝成另一個 protocol。AmneziaWG I1-I5、FTE、Marionette 都屬此。
- **Protocol borrowing / piggybacking**：真的跑另一個 protocol 作為 outer envelope。REALITY、Conjure、Snowflake 屬此。
- **Active probing**：對手主動發 packet 確認 protocol。
- **Fully encrypted detection**：對「看起來像 random bytes」的流量做 entropy + 統計偵測。Wu 2023 是領域標誌。
- **CPS (Chained Protocol Simulation)**：AmneziaWG 2.0 的 I1-I5 機制名稱。

### 2. 對手分類學 / 威脅模型精化

對 WireGuard 在 GFW 場景下的對手等級：

| 等級 | 能力 | 對 WG 影響 |
|---|---|---|
| **passive L4 DPI** | match byte/size | **#1+#2 即破** |
| **passive L4 entropy test** | Wu 2023 規則 | **#1 不過 entropy → 破** |
| **active probe with known server pk** | 偽造 valid MAC1 init | **#5 破** |
| **active probe without server pk** | 隨機 probe | MAC1 silent drop → 安全 |
| **traffic-analysis adversary** | size + timing 序列分析 | **transport size leak inner → 破** |
| **collective enumeration** | IP/port 大量探測 | 配合上述任一即破 |

GFW 在 2023 之後具備上面**全部**能力。**WireGuard 對所有除最後一條外的對手都失效**。

### 3. 形式化定義

**Active probe resistance**（[6.2 §3](6.2-openvpn-anatomy.md) 已給）：

> Server S 對 any probe m（不含 valid pre-shared state）的 response 與 generic non-protocol server 計算上不可區分。

WireGuard 對「**不知 server pk 的 prober**」滿足；對「**已知 server pk 的 prober**」**不滿足**。

G6 的目標：**對任何 prober**（含知道 server identity 的）滿足 probe resistance。實現路徑：REALITY-style 借殼（[7.10](../part-7-proxy-protocols/)）。

### 5. 我們協議的座標 / 設計取捨

G6 在本堂教訓下的具體設計選擇：

```
G6 = WireGuard core + REALITY outer envelope + Wu 2023 anti-detection padding
                                                 + PQ-hybrid KE
                                                 + cover traffic policy
```

- **WireGuard core**：保留 cryptokey routing + Noise IK-style handshake + 16-byte transport overhead 結構。
- **REALITY outer envelope**：握手包進 borrowed real TLS handshake（[7.10](../part-7-proxy-protocols/)）。
- **Wu 2023 anti-detection padding**：first byte / first 6 bytes 滿足 ASCII printable 規則（**或**完全隨機 entropy uniform）。
- **PQ-hybrid KE**：X25519 + ML-KEM-768。
- **Cover traffic policy**：idle 時送統計上不可區分 cover。

### 7. 開放問題（research-level）

1. **WireGuard 能否設計出 backwards-compatible 的 fingerprint mitigation**？AmneziaWG 已示範可以做但要 break compat。能否設計 spec change 讓 legacy WG client 仍能連？答案大概是「不行」——這是 spec 設計死亡的訊號。
2. **AmneziaWG H1-H4 ranged headers** 對 fingerprinting 的具體量化抵抗？目前無 USENIX/NDSS 級實證。能否做 Xue 2022 framework 對 AmneziaWG 的等價分析？
3. **server pk 公開 vs MAC1 設計的根本矛盾**：能否設計 protocol 讓「server identity 公開 + active probe resistance」同時成立？REALITY 給出一條路；還有別的嗎？
4. **WG kernel impl 與 wireguard-go 的 fingerprint 差異**：兩個實作 packet 是否 byte-identical？timing 差異？GFW 是否區分？尚無研究。
5. **AmneziaWG 與 WG 共存 server** 的 fingerprint 隔離：AWG 2.0 拿掉 backwards compat 後，是否有 spec-level 的 multiplexing 設計能讓單一 server 同時服務兩種 client？
