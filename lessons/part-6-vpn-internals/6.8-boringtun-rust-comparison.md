## 動機

兩個關於 BoringTun 的常見誤解：
1. ❌ 「Cloudflare 用 Rust 比 Go 更快才重寫的」——錯。BoringTun 比 wireguard-go **單核略慢**（10~15%）；快的是「**library, 不是 daemon**」的 architectural 選擇。
2. ❌ 「BoringTun 是 production WG 的下一代」——錯。**它停止積極維護自 2023**，Firezone 與其他 fork 是社群繼任者。

那為什麼 BoringTun 重要：
1. **Library-first architecture**：與 wireguard-go 的 daemon-first 對比，提供 mobile app 直接 embed 的乾淨介面。**Cloudflare 1Place app (iOS/Android, WARP) 與 千萬部 iPhone 用的就是它**。
2. **Rust ecosystem 對 crypto 的反思**：BoringTun 同時依賴 `ring`（BoringSSL 包裝）與 `blake2`、`chacha20poly1305` rust-native crate——這個混用是有趣的工程選擇。
3. **不同 platform abstraction**：BoringTun 把「**1 個 protocol handler instance**」與「**OS thread/runtime**」徹底解耦。這對 G6 多平台 reference impl 有強參考價值。
4. **退場的教訓**：為什麼一個 Cloudflare 投入的專案會 stall？這對你考慮 G6 deployment 模型有啟示。

## 核心概念

### 1. 三個 user-space WireGuard 實作的定位差異

| 實作 | 語言 | 行數 | 定位 | 第一順位用戶 |
|---|---|---|---|---|
| **wireguard-go** | Go | ~10000 LoC | reference user-space daemon | self-hosted、CLI 工具、Tailscale |
| **BoringTun** | Rust | ~8000 LoC (excl. tests) | embeddable library + thin daemon | mobile app (iOS/Android/WARP) + millions of devices |
| **Linux kernel WG** | C | ~4000 LoC | kernel module | server / Linux desktop |

**Cloudflare 為什麼選 Rust + library**：
1. **iOS / Android 無法跑 user-space daemon**。必須 link 進 app process。
2. **沒有現成 Go runtime 在 iOS app 中能良好工作**（goroutine + GC pause 對 mobile battery 是災難）。
3. **Rust no-std-able**——容易做 minimal binary 給 embedded / mobile。
4. **memory safety + no GC + zero-cost abstraction**：Rust 對 protocol code 是天然 fit。

### 2. BoringTun 的「Tunn」抽象（noise/mod.rs）

核心 type：

```rust
// noise/mod.rs (簡化)
pub struct Tunn {
    handshake: Handshake,
    sessions: [Option<Session>; N_SESSIONS],   // 8 個 sliding sessions
    current: usize,                            // 當前 session index
    packet_queue: VecDeque<...>,
    timers: Timers,
    rate_limiter: Option<Arc<RateLimiter>>,
    ...
}
```

`Tunn` 是 **per-peer state machine**。它**不持有 socket、不持有 thread、不做 I/O**——只做 protocol state transition。Cloudflare 把它設計成**完全 sans-IO**（async ecosystem 的術語）：caller 餵 input buffer，呼叫 method，拿回 output buffer。

```rust
// noise/mod.rs:
pub enum TunnResult<'a> {
    Done,
    Err(WireGuardError),
    WriteToNetwork(&'a mut [u8]),
    WriteToTunnelV4(&'a mut [u8], Ipv4Addr),
    WriteToTunnelV6(&'a mut [u8], Ipv6Addr),
}

impl Tunn {
    pub fn decapsulate<'a>(&mut self, src_addr: Option<IpAddr>, datagram: &[u8], dst: &'a mut [u8]) -> TunnResult<'a> { ... }
    pub fn encapsulate<'a>(&mut self, src: &[u8], dst: &'a mut [u8]) -> TunnResult<'a> { ... }
    pub fn update_timers<'a>(&mut self, dst: &'a mut [u8]) -> TunnResult<'a> { ... }
}
```

**對比 wireguard-go**：wireguard-go 的 `Device` 自己起 goroutine、自己管 channel、自己跟 socket 互動。BoringTun 的 `Tunn` 是**純 protocol object**，I/O 完全外包。

**這對 G6 是極好的 architecture 範本**：
- core protocol logic 完全 sans-IO，可在任何 runtime（async-std/tokio/sync）跑。
- mobile app 直接 link library，不需要 spawn 額外 OS thread。
- 容易做 unit test（不需 mock socket）。
- 容易做 formal verification（沒有 I/O 副作用）。

### 3. Crypto library 選擇：`ring` + `blake2` + `chacha20poly1305`

`handshake.rs:9-14`：
```rust
use aead::{Aead, Payload};
use blake2::digest::{FixedOutput, KeyInit};
use blake2::{Blake2s256, Blake2sMac, Digest};
use chacha20poly1305::XChaCha20Poly1305;
use rand_core::OsRng;
use ring::aead::{Aad, LessSafeKey, Nonce, UnboundKey, CHACHA20_POLY1305};
```

**混用兩套 ChaCha20-Poly1305**：
- `ring::aead::CHACHA20_POLY1305`（BoringSSL backend）—— transport AEAD。
- `chacha20poly1305::XChaCha20Poly1305`（pure-Rust）—— XChaCha20-Poly1305 for cookie reply（12-byte nonce vs 24-byte extended）。

**為什麼這樣**：
- `ring` 的 ChaCha20-Poly1305 用 BoringSSL 的 AVX2/AVX-512 optimized impl，比 pure-Rust 快 ~2×。
- `ring` 不支援 XChaCha20-Poly1305，所以 cookie reply 用 pure-Rust。
- **trade-off**：依賴兩個 crypto library 是 audit overhead，但效能值得。

對比 wireguard-go 全部用 Go 標準庫 `golang.org/x/crypto`，不混。**Cloudflare 為性能放棄了一致性**——這是 production code 常見取捨。

### 4. Handshake 邏輯逐行對比（handshake.rs vs wireguard-go noise-protocol.go）

**INITIAL_CHAIN_KEY** 與 **INITIAL_CHAIN_HASH** 是 BoringTun **預算硬編碼**的常數（handshake.rs:27-37）：

```rust
const INITIAL_CHAIN_KEY: [u8; 32] = [96, 226, 109, ...];
const INITIAL_CHAIN_HASH: [u8; 32] = [34, 17, 179, ...];
```

對比 wireguard-go：
```go
func init() {
    InitialChainKey = blake2s.Sum256([]byte(NoiseConstruction))
    mixHash(&InitialHash, &InitialChainKey, []byte(WGIdentifier))
}
```

**差異意義**：
- BoringTun **預先算好**避開 runtime init。對 mobile cold start 重要（每 ms 都算）。
- wireguard-go runtime 算。對 daemon 用例無差。
- **trade-off**：BoringTun 寫死，等於 implicitly assert spec 永不變。若 spec 改 NoiseConstruction string（例如 PQ-migrated），這兩個常數要重算 + recompile。**這是 G6 設計的警示**：crypto agility 與 cold-start 速度的取捨。

#### `b2s_hash` / `b2s_hmac` 工具：

```rust
// handshake.rs:39-46
pub(crate) fn b2s_hash(data1: &[u8], data2: &[u8]) -> [u8; 32] {
    let mut hash = Blake2s256::new();
    hash.update(data1);
    hash.update(data2);
    hash.finalize().into()
}
```

對應 wireguard-go 的 `mixHash`。**注意 BoringTun 用兩個 `&[u8]` 參數**避免 caller 自己 concat，省一次 allocation。Rust 的 zero-cost abstraction 在這發揮——`Blake2s256::new().update(a).update(b)` 編譯後是 fused loop。

#### Constant-time 比較：

BoringTun 的 MAC check 用 `subtle` crate（不顯示 in handshake.rs 但隱含於 ring's AEAD verification）：
```rust
use subtle::ConstantTimeEq;
if mac1.ct_eq(&expected_mac1).into() { ... }
```

對應 wireguard-go 的 `hmac.Equal`。Rust 的 `subtle` crate 提供**編譯器 barrier** 保證 LLVM 不優化掉 timing equality——比 Go 的 `hmac.Equal` 更謹慎。**這是 cryptographic engineering 的細節，G6 reference impl 必跟**。

### 5. Session 與 N_SESSIONS = 8 的設計

`mod.rs:42`:
```rust
const N_SESSIONS: usize = 8;
```

**對比 wireguard-go**：wireguard-go 只維護 `current/previous/next` 3 個 `Keypair`，BoringTun 維護 **8 個 session ring**。

為什麼差異？
- BoringTun 預期 mobile 場景**頻繁 roaming**（WiFi/cellular 切換），同 peer 可能 simultaneous 多 session 短期共存。
- 8 是 power-of-two 方便 index masking（`current & 7`）。
- 同時擁有多 session 帶來 anti-replay 與 nonce reuse 風險，BoringTun 額外做 per-session 嚴格驗證。

對 G6 的 implication：**mobile use case 需要 multi-session ring**，desktop daemon 不需要。reference impl 應該**配置可變**。

### 6. 性能：BoringTun vs wireguard-go vs kernel WG

Cloudflare 2019 blog 的 benchmark（Linux, single core, no GSO）：

| Impl | Throughput |
|---|---|
| Linux kernel WG | ~860 Mbps |
| BoringTun (Rust) | ~720 Mbps |
| wireguard-go (Go) | ~660 Mbps |

實測 2024 數據（在 modern Linux 5.15+, GSO enabled）：

| Impl | iperf3 single-stream |
|---|---|
| Linux kernel WG | ~1.5 Gbps |
| wireguard-go (GSO) | ~1.0 Gbps |
| BoringTun | ~0.8 Gbps |

**注意 wireguard-go 在 Linux 上反超 BoringTun**——因為 GSO 整合（[6.6 §6](6.6-wireguard-tun-udp-integration.md)）的 wireguard-go 較好。BoringTun 對 GSO 支援較弱。

**單核 throughput 不是 BoringTun 賣點**。它的賣點：
- **per-instance memory < 1 MB**（vs wireguard-go ~10 MB）。
- **per-tunnel CPU < 5% on mobile**。
- **無 GC pause**——對行動裝置功耗友善。

### 7. Threading model：epoll/kqueue vs goroutine

`device/epoll.rs` 與 `device/kqueue.rs` 是 BoringTun 自己寫的 event loop。**沒有 async runtime**：

```rust
// epoll.rs (簡化)
pub fn epoll_wait_block(&self) -> Vec<Event> {
    let mut events: [libc::epoll_event; MAX_EVENTS] = ...;
    let n = unsafe { libc::epoll_wait(self.epfd, events.as_mut_ptr(), MAX_EVENTS as i32, -1) };
    ...
}
```

**為什麼不用 tokio**：
- 2019 年 Rust async ecosystem 還不穩定。
- mobile 裝置不能容忍 tokio runtime overhead。
- BoringTun 想最大化 portability，自己寫 epoll/kqueue wrap 更乾淨。

**對比 wireguard-go**：goroutine + channel 是 Go runtime 提供的，不需要自己寫 event loop。**Go 的 runtime 是免費的 N:M threading**。

**G6 的選擇**：因為要做 anti-censorship（含 cover traffic、async I/O 對 timing 敏感），建議：
- **reference impl 用 Go**（goroutine 對 prototyping 友善）
- **production embeddable 版本 用 Rust + tokio**（成熟後）

### 8. FFI 介面：給 iOS/Android 用

`boringtun/src/ffi/mod.rs`（397 行）暴露 C ABI：

```rust
#[no_mangle]
pub extern "C" fn new_tunnel(...) -> *mut Tunn { ... }

#[no_mangle]
pub extern "C" fn wireguard_write(tunnel: *mut Tunn, src: *const u8, src_size: u32, dst: *mut u8, dst_size: u32) -> wireguard_result { ... }
```

對應 `wireguard_ffi.h` C header。iOS 用 Swift 直接 import；Android 用 JNI（`jni.rs`）。**Cloudflare WARP 的 iOS app 就 link 這個 .a**。

**這是 BoringTun 最大的工程價值**：把 protocol logic 包成可 link 的 library。對比：
- wireguard-go：要 cgo + Go runtime，iOS link 複雜。
- 自己重寫 Swift/Kotlin impl：reinventing wheel + audit cost。

**對 G6 的啟示**：reference impl day-1 設計 FFI surface（C ABI），讓未來 mobile binding 容易。

### 9. 為什麼 BoringTun 漸漸 stall

從 GitHub 觀察：
- 2024 起 issue 累積，maintainer 回應慢。
- README 顯示「restructuring」狀態超過 1 年未結束。
- 主要 active fork：Firezone 公司的 fork（aims 100% API compat）。
- Cloudflare 內部似乎已 frozen，不接受 community PR。

**推測原因**：
1. Cloudflare WARP 已大規模部署，內部需求飽和，沒動機回 upstream。
2. wireguard-go GSO 提升後性能逼近 BoringTun，社群選擇容易 ship 的 wireguard-go。
3. Rust crypto ecosystem 演進快（ring v0.16 → v0.17 break change），維護成本累積。

**對 G6 的教訓**：
- **不要寫一個沒有 active sponsor 的 library**——OSS 維護是長期承諾。
- **fork-friendly architecture**：sans-IO + clean FFI 讓 fork 容易接手（Firezone 接手 BoringTun 成本 < 重寫）。
- **同時 ship 2~3 個 reference impl**（如 wireguard-go + BoringTun + Linux kernel 三軌）對 protocol longevity 有利。

### 10. BoringTun 給 G6 的具體 take-aways

| 設計選擇 | G6 該不該抄 |
|---|---|
| Sans-IO `Tunn` core | ✅ 必抄 |
| FFI C ABI | ✅ 必抄（mobile 友善） |
| N_SESSIONS ring | ✅ 可選，mobile build 開 |
| 預算 INITIAL_CHAIN_* | ⚠️ 看 spec stability |
| 混用 ring + pure-Rust crypto | ✅ 為性能 |
| 自己寫 epoll/kqueue（不用 tokio） | ❌ 用 tokio（2026 已成熟） |
| Rust no-std 友善 | ✅ |
| 無 GSO 完整整合 | ❌ 不抄此遺憾 |

## 與我們協議設計的關聯

G6 的 reference impl 路線圖：

```
Phase 1 (Part 12.1~12.6): g6-go reference daemon
  ↳ fork wireguard-go 的 tun/conn/device 框架
  ↳ 加 cover-traffic + obfuscation + PQ-hybrid
  ↳ 目標：能在 Linux self-host 跑通 anti-censorship 測試平台

Phase 2 (Part 12.10+): g6-rs library (Rust)
  ↳ 學 BoringTun 的 sans-IO Tunn 設計
  ↳ tokio-based，C ABI FFI
  ↳ 目標：mobile embeddable，WARP-style 部署

Phase 3 (post-thesis): g6-kernel (C / Rust-for-Linux)
  ↳ if spec 穩定，考慮 upstream Linux
  ↳ 目標：production-grade kernel impl
```

BoringTun 的價值是 **Phase 2 的範本**——sans-IO + FFI + mobile-friendly。

use boringtun::noise::{Tunn, TunnResult};
use boringtun::x25519::{StaticSecret, PublicKey};

fn main() {
    let my_static = StaticSecret::random_from_rng(&mut rand_core::OsRng);
    let peer_pk = PublicKey::from(/* ... */);

## 研究級補遺

### 1. 學界詞彙

- **Sans-IO**：一種 protocol library 設計模式——core logic 完全不做 I/O，由 caller 餵 input/拿 output。Python 圈 `h11` (HTTP/1.1)、`h2` (HTTP/2) 為代表；Rust 圈 BoringTun + `quinn`-core 為代表。
- **No-std**：Rust 程式不依賴標準庫，可在 embedded / kernel 跑。BoringTun core 接近 no-std 友善但不完全。
- **FFI ABI surface**：library 對外的 C ABI 介面集合。設計時要平衡彈性與穩定性（ABI 一改全部 binding 重編）。
- **N:M threading**：M 個 OS thread 對應 N 個邏輯任務的調度。Go runtime 是 N:M；BoringTun 偏 1:1 (epoll loop)。

### 2. 對手分類學 / 威脅模型精化

BoringTun 與 wireguard-go 在 attack surface 上**幾乎相同**（同 protocol）。差異點：

| 攻擊面 | BoringTun | wireguard-go |
|---|---|---|
| Memory safety | Rust borrow checker | Go GC + bounds check |
| Timing side channel | `subtle` + `ring` 已 audit | `hmac.Equal` + Go std crypto |
| `unsafe` block | FFI module 有少量 | `cgo` 有少量 |
| Supply chain | crates.io + ring + BoringSSL | Go modules + std crypto |

對 G6：兩者都可接受，**Rust 在 memory safety 略優，Go 在 reference 易讀性略優**。

### 3. 形式化定義

Sans-IO 的形式化好處：可寫成 pure state machine

> `Tunn` is a function `f: (state, input) → (new_state, output)`

純函式 + 確定性 → 容易做 differential testing（同 input 對 wireguard-go 與 BoringTun 應得 byte-identical output）。**這是 protocol implementation 一致性的銀彈**。G6 必抄。

### 7. 開放問題

1. **能否設計 protocol 級的 differential testing framework**？BoringTun vs wireguard-go vs kernel-WG 三方 output 應 byte-identical。這個 framework 是 protocol implementation diversity 的關鍵。
2. **Rust async crypto 的演進**：tokio + `rustls` + `rustls-pki-types` 體系穩定下，是否未來會出現 BoringTun++ ?
3. **G6 reference impl 是否該 day-1 同時 ship Go + Rust？** 工程成本翻倍，但對 protocol longevity 與 audit diversity 有重大好處。
4. **mobile spec subset**：handheld 裝置不適合所有 G6 feature（如 cover traffic 太耗電）。能否 spec 層有 "mobile profile" 子集？
