## 動機

[6.4][6.5] 講了 WireGuard 在 user-space 的 pipeline 與 crypto；本堂講「**這個 pipeline 怎麼接到 OS**」。這比看起來複雜：

1. **TUN device**：Linux `/dev/net/tun`、macOS `utun`、Windows `wintun`、FreeBSD/OpenBSD/iOS/Android 各有 idiosyncrasy。
2. **UDP socket**：要支援 IPv4/IPv6 dual-stack、GSO/GRO、`SO_REUSEPORT`、Linux 上的 fwmark、iOS 上的 NWConnection 限制。
3. **Roaming**：socket 必須能接收任意 source IP/port 並回應正確 destination——這對 NAT 後 server 重要。

對研究 G6 的價值：**OS 邊界的細節決定能不能 ship**。spec / crypto 寫得再漂亮，TUN 整合錯了照樣 throughput 半折。本堂讀完，你能在任何 OS 平台 port wireguard-go 等價邏輯。

## 核心概念

### 1. TUN abstraction（tun/tun.go）

```go
// tun/tun.go 大致介面
type Device interface {
    File() *os.File
    Read(bufs [][]byte, sizes []int, offset int) (int, error)
    Write(bufs [][]byte, offset int) (int, error)
    MTU() (int, error)
    Name() (string, error)
    Events() <-chan Event
    Close() error
    BatchSize() int
}
```

**關鍵設計**：
- `Read` / `Write` 都接受 **批次 `[][]byte`**——對應 [2.x batched syscall]。
- `offset` 參數讓 caller 在 buffer 前留空（給 WireGuard transport header 16 bytes）。
- `BatchSize()` 由平台決定：Linux ~128、macOS 通常 1（無 batched API）。
- `Events()` channel：MTU 變化、interface up/down 由 OS 通報。

### 2. Linux TUN：`tun/tun_linux.go`（720 行，最複雜）

Linux TUN 用 `/dev/net/tun` 加 `TUNSETIFF` ioctl 建立 interface：

```go
// tun_linux.go：CreateTUN (略)
func CreateTUN(name string, mtu int) (Device, error) {
    nfd, err := unix.Open(cloneDevicePath, unix.O_RDWR|unix.O_CLOEXEC, 0)
    var ifr struct {
        name  [16]byte
        flags uint16
        ...
    }
    copy(ifr.name[:], name)
    ifr.flags = unix.IFF_TUN | unix.IFF_NO_PI | unix.IFF_VNET_HDR
    unix.IoctlSetInt(nfd, unix.TUNSETIFF, ...)
    ...
}
```

**3 個 flag 對性能至關重要**：

| Flag | 意義 | 為什麼開 |
|---|---|---|
| `IFF_TUN` | L3 packet（無 ethernet header） | 我們做 L3 VPN |
| `IFF_NO_PI` | 不要 4-byte "Packet Info" 前綴 | 省 4 bytes 對齊計算 |
| `IFF_VNET_HDR` | 啟用 virtio_net_hdr，讓 kernel 透過 TUN 通報 GSO/checksum offload metadata | **這是 user-space GRO/TSO 的關鍵** |

#### `IFF_VNET_HDR` 與 user-space TCP GRO（offload_linux.go）

`tun/offload_linux.go` 有 838 行——比整個 `noise-helpers.go` 大 8 倍。它在做什麼？

**問題**：TUN 從 kernel 讀出的 packet 通常是 single-segment（MTU 大小）。對於 throughput-bound 場景（user 在 VPN 內傳大檔），如果你能**把多個連續 TCP segment 合併成單一 large packet**（GRO），AEAD 一次處理一大塊，比每段都 seal 一次快得多。

Linux kernel 的 GRO 是 NIC driver 層的事，TUN 通常拿不到 GRO 後的大 packet。**但**如果開 `IFF_VNET_HDR`，kernel 會把 GSO metadata（segment size, flags）一起傳過來，user-space 可以**自己做 software GRO**——把多個 single-segment 在 user-space 合併。

`offload_linux.go` 實作完整的 software TCP GRO + UDP GRO：

```go
// offload_linux.go:74-83
func newTCPGROTable() *tcpGROTable {
    t := &tcpGROTable{
        itemsByFlow: make(map[tcpFlowKey][]tcpGROItem, conn.IdealBatchSize),
        itemsPool:   make([][]tcpGROItem, 0, conn.IdealBatchSize),
    }
    return t
}
```

對每個 TCP flow（by src/dst IP+port），維護一個 in-flight GRO entry。每次新 segment 進來，判斷 seq number 是否連續：是 → 合併；否 → flush 舊 entry 並開新。

**這是極好的 systems engineering 例子**：在 user-space 重新實作 kernel 通常做的事，因為 user-space VPN 拿不到 kernel 的 GRO 後 packet。代價是 ~800 行 code，收益是 ~30% throughput 提升。

#### Linux TUN Read：`Read` 批次與 GRO

```go
// tun_linux.go (rough sketch)
func (tun *NativeTun) Read(bufs [][]byte, sizes []int, offset int) (int, error) {
    n, err := unix.Read(tun.tunFile, ...)   // 一次 read 拿到含 virtio_net_hdr 的 raw bytes
    // parse virtio_net_hdr
    // 若 gso_type != none，把 segment 切成多個 buffer
    // 或執行 software GRO 合併
    return n, nil
}
```

實際 Linux read 是 single-shot（`/dev/net/tun` 不支援 `readv` 的批次 GRO），但 user-space `Read` 介面返回 `[][]byte`——`offload_linux.go` 的 GRO 把 raw blob splice 成多個邏輯 packet 還給 caller。

### 3. macOS TUN：`tun/tun_darwin.go`（簡單但慢）

macOS 用 `utun` system socket（不是 `/dev/net/tun`）：

```go
// tun_darwin.go:CreateTUN (簡化)
func CreateTUN(name string, mtu int) (Device, error) {
    fd := unix.Socket(unix.AF_SYSTEM, unix.SOCK_DGRAM, 2)
    var ctlInfo = &unix.CtlInfo{...}
    copy(ctlInfo.Name[:], "com.apple.net.utun_control")
    unix.IoctlCtlInfo(fd, ctlInfo)
    unix.Connect(fd, &unix.SockaddrCtl{...})
    ...
}
```

每封 packet 用一個 syscall（`read` / `write`）——**沒有批次 API**。實測 macOS WireGuard 比 Linux 慢 3~5 倍，主要瓶頸就在這。對 G6 是 reminder：**macOS 開發機方便，但性能評測必須在 Linux**。

`tun_darwin.go:Read`（L204）：
```go
func (tun *NativeTun) Read(bufs [][]byte, sizes []int, offset int) (int, error) {
    // 一次只能 read 一封 packet
    n, err := tun.tunFile.Read(bufs[0][offset-4:])
    // utun 在每封 packet 前面有 4-byte protocol family prefix
    // discard it, return one packet
    sizes[0] = n - 4
    return 1, nil
}
```

**注意 macOS utun 多 4-byte prefix**（protocol family: 0x0000_0002 IPv4 / 0x0000_001E IPv6）。這 4 bytes 與 Linux 的 `IFF_NO_PI` 相對應——macOS 沒辦法關掉。

### 4. Windows TUN：`wintun.dll`

`tun_windows.go` 用 Cloudflare 維護的 `wintun.dll`（基於 NDIS 6.x driver）：
- 用 ring buffer 與 user-space 交換 packet——零 syscall。
- 一次處理 ~256 packet。
- 性能接近 Linux。

[6.8 BoringTun](6.8-boringtun-rust-comparison.md) 會說 Cloudflare 為 Windows 維護 wintun 與 BoringTun 配套。

### 5. UDP Socket：`conn/conn.go` + `conn/bind_std.go`

```go
// conn/conn.go 介面
type Bind interface {
    Open(port uint16) ([]ReceiveFunc, uint16, error)
    Close() error
    SetMark(value uint32) error
    Send(bufs [][]byte, ep Endpoint) error
    ParseEndpoint(s string) (Endpoint, error)
    BatchSize() int
}
```

預設實作 `StdNetBind`（bind_std.go）：

```go
// bind_std.go:140
func (s *StdNetBind) Open(uport uint16) ([]ReceiveFunc, uint16, error) {
    ...
    s.ipv4 = listen IPv4 on uport
    s.ipv6 = listen IPv6 on uport
    return []ReceiveFunc{
        s.makeReceiveIPv4(s.ipv4PC, s.ipv4, rxOffload),
        s.makeReceiveIPv6(s.ipv6PC, s.ipv6, rxOffload),
    }, port, nil
}
```

**兩個 ReceiveFunc** 給 [6.5] 提到的 `RoutineReceiveIncoming` 用：device 起兩個 goroutine（IPv4 + IPv6）各跑一個 ReceiveFunc，並行收 UDP。

`receiveIP`（bind_std.go:226）：

```go
// bind_std.go:226
func (s *StdNetBind) receiveIP(br batchReader, conn *net.UDPConn, ...) (int, error) {
    msgs := s.getMessages()
    for i := range *msgs {
        (*msgs)[i].Buffers[0] = bufs[i]
        (*msgs)[i].OOB = ...   // out-of-band：GSO size, src addr, etc.
    }
    n, err := br.ReadBatch(*msgs, 0)   // recvmmsg-like
    for i := 0; i < n; i++ {
        ep := &StdNetEndpoint{...}
        getSrcFromControl((*msgs)[i].OOB[:], ep)
        sizes[i] = (*msgs)[i].N
        eps[i] = ep
    }
    return n, nil
}
```

**精髓**：
1. `ReadBatch` 是 `golang.org/x/net/ipv4` 的封裝，底層用 Linux 的 `recvmmsg`、其他 OS fallback。
2. `OOB` 拿到 GRO size 與 source IP——這對 dual-stack 與 GRO aggregation 都必要。
3. 一次 read 拿 N 封（`IdealBatchSize` 通常 128）。

#### Linux GRO 整合（conn/gso_linux.go）

```go
// gso_linux.go:22
func getGSOSize(control []byte) (int, error) {
    // 解析 IP_UDP_GRO control message
}

// gso_linux.go:46
func setGSOSize(control *[]byte, gsoSize uint16) {
    // 設定 UDP_SEGMENT 給 sendmmsg
}
```

兩個對稱：**接收時** 從 OOB control buffer 解出 kernel 已 aggregate 的 segment size；**發送時** 在 OOB control buffer 寫 UDP_SEGMENT，叫 kernel 把一個大 datagram 切成多個 MTU-sized segment 送出去。

**這是 Linux 5.0+ 的能力**。沒有它，wireguard-go 性能直接腰斬。`features_linux.go:supportsUDPOffload()` 在 runtime 偵測，落地時若 kernel 不支援自動降回 single-packet syscall。

#### Endpoint sticky（conn/sticky_linux.go）

**問題**：server 是 dual-homed 或 multipath，要保證對同 client 的回包從**同一 source IP** 出，否則 NAT 後 client 收不到。

**解法**：`IP_PKTINFO` ancillary message + `IPV6_PKTINFO`。每次收 UDP，從 control buffer 拿 destination IP（即「我 server 是被哪個 local IP 收到的」），下次回應時設成 source IP。

```go
// sticky_linux.go (rough)
func setSrcFromControl(oob []byte, ep *StdNetEndpoint) {
    // parse IP_PKTINFO -> ep.src
}
```

**這個 100 行的 detail**解掉了「multi-homed server + roaming client」這個傳統 IPsec/OpenVPN 需要 routing-table hacks 才能搞定的問題。對 G6 是必抄。

### 6. fwmark / SO_MARK（conn/mark_unix.go）

Linux 上 `SO_MARK` 給 socket 設 firewall mark，配合 `iptables` / `nftables` / `ip rule` 做 policy routing。

```go
func (s *StdNetBind) SetMark(value uint32) error {
    unix.SetsockoptInt(fd, unix.SOL_SOCKET, unix.SO_MARK, int(value))
}
```

**用途**：避免「VPN 對自己 endpoint 的 UDP 包又被 VPN 路由」的 loop。`wg-quick` 預設用 fwmark + `ip rule` 把 WG 的 outer UDP 走 main routing table（不進 VPN），inner traffic 走 VPN。

**這跟 [1.5] CVE-2024-3661 TunnelVision 關聯**：TunnelVision 利用 DHCP option 121 推 /1 路由壓過 VPN /0——本質是 routing-table layer 的問題，protocol 層解不掉。fwmark + policy routing 是部分緩解。

### 7. netstack：純 user-space TCP/IP（tun/netstack/）

`tun/netstack/` 提供 **不需要 OS TUN device** 的選項——把 gVisor netstack 直接接到 WireGuard。

用途：
- **iOS / 行動裝置** 通常無法或不方便建 TUN，但能跑 user-space TCP/IP stack。
- **單一 process 內 ssh-over-WG** 不想動 OS routing table。
- **測試** 不污染 host network namespace。

```go
// tun/netstack/tun.go (rough)
import "gvisor.dev/gvisor/pkg/tcpip/stack"

func CreateNetTUN(localAddresses []netip.Addr, dnsServers []netip.Addr, mtu int) (tun.Device, *stack.Stack, error) {
    s := stack.New(stack.Options{
        NetworkProtocols:   []stack.NetworkProtocolFactory{...},
        TransportProtocols: []stack.TransportProtocolFactory{tcp.NewProtocol, udp.NewProtocol},
    })
    ...
}
```

**對 G6 的啟示**：reference impl 應同時支援 OS TUN 與 netstack mode——測試與 mobile deployment 都用得上。

### 8. 性能瓶頸分析：把整段 outbound 拆給 perf

對 Linux wireguard-go 跑 `iperf3` 滿載：

| Component | %CPU (typ.) |
|---|---|
| ChaCha20-Poly1305 Seal | 30~40% |
| ChaCha20-Poly1305 Open | 25~35% |
| TUN read / virtio_net_hdr parse | 5~10% |
| Software GRO (offload_linux.go) | 3~8% |
| UDP send (sendmmsg + GSO) | 5~10% |
| Go goroutine scheduling | 3~5% |
| BLAKE2s handshake | < 1%（handshake 是稀疏事件） |
| allowedips trie lookup | < 1% |

**結論**：AEAD 是 CPU bound 的大頭。Cloudflare BoringTun [6.8] 用 Rust + AVX2/AVX-512 手寫 ChaCha20-Poly1305 把這塊壓到 ~50% Go impl。

### 9. 跨平台一致性的「醜陋細節」清單

寫過 multi-OS VPN 才知道有多少細節：

| Issue | Linux | macOS | Windows | iOS |
|---|---|---|---|---|
| Batch syscall | ✅ recvmmsg/sendmmsg | ❌ 單 packet | ✅ wintun ring | ❌ NWConnection |
| UDP GSO/GRO | ✅ 5.0+ | ❌ | partial | ❌ |
| TUN MTU change notification | netlink | KEVENT | wintun event | network extension API |
| IPv6 default routing | accept_ra | scutil | netsh | system-managed |
| fwmark / policy routing | ✅ | ❌（用 PF） | ❌（用 WFP） | ❌ |
| `SO_BINDTODEVICE` | ✅ | ❌（IP_RECVIF） | partial | ❌ |

WireGuard 是少數**在所有上述平台都有 first-class 實作**的 VPN protocol。這是 user-space Go reference + 各平台原生 C/Swift wrapper 的雙軌策略。

## 與我們協議設計的關聯

### G6 reference impl 的 OS 邊界 module

直接抄 wireguard-go 的層級劃分：

```
g6/tun/        ← 抄 wireguard-go tun/
├── tun.go              ← Device interface
├── tun_linux.go        ← Linux /dev/net/tun + IFF_VNET_HDR
├── tun_darwin.go       ← macOS utun
├── tun_windows.go      ← wintun
├── offload_linux.go    ← software GRO（直接 fork wireguard-go）
└── netstack/           ← gVisor netstack option

g6/conn/       ← 抄 wireguard-go conn/
├── conn.go             ← Bind interface
├── bind_std.go         ← cross-platform UDP socket
├── gso_linux.go        ← Linux UDP GSO/GRO
├── sticky_linux.go     ← endpoint sticky
└── mark_unix.go        ← fwmark
```

**不抄但要加**：
- `g6/cover/`：cover traffic generator（與 transport 整合）
- `g6/path/`：path validation（QUIC PATH_CHALLENGE 等價）
- `g6/obfs/`：first-byte 與 size obfuscation 介面

### 性能目標

- Linux：~80% kernel-WG（即 ~800 Mbps single-core，PQ-hybrid 加 ~30% overhead）。
- macOS：~200 Mbps（受 utun batch 限制；無解）。
- 提升空間：用 io_uring / AF_XDP 替代 sendmmsg（[Part 2.6/2.7](../part-2-high-perf-io/)），可期待 1.5 Gbps+。

### 觀察：kernel-space WG 在 PQ 時代的優勢可能縮小

`drivers/net/wireguard/` ([6.9](6.9-wireguard-linux-kernel.md)) 跑 ~1.1× user-space speed。**但**當你加 ML-KEM-768 hybrid（公鑰 1184 bytes、ciphertext 1088 bytes），handshake size 從 148 bytes 飆到 ~1.5KB；雖然 handshake 是稀疏事件，但 kernel impl 對大 handshake message 的緩衝池設計可能要重做。**G6 的 user-space 反而有彈性優勢**——這是支持 user-space-first 的工程論據。

## 研究級補遺

### 1. 學界詞彙

- **TAP vs TUN**：TAP = L2（含 ethernet header）、TUN = L3。VPN 通常用 TUN。
- **virtio_net_hdr**：Linux kernel 提供給 user-space 的 GSO/checksum offload metadata。**對 user-space VPN 至關重要**。
- **GRO/GSO**：Generic Receive/Segmentation Offload。NIC 或軟體把多個 segment 合 / 切。
- **TSO/USO**：TCP/UDP Segmentation Offload（NIC 硬體版的 GSO）。
- **AF_PACKET / AF_XDP**：Linux 上繞 socket stack 的高速封包介面。[2.7](../part-2-high-perf-io/) 已伏筆。

### 2. 對手分類學 / 威脅模型精化

OS-boundary attack surface：

| 攻擊 | 範例 | wireguard-go 防護 |
|---|---|---|
| **TUN ioctl 提權** | tunnel mode 開於 untrusted uid | 多平台 requires root/Admin to create TUN |
| **TUN packet injection** | uid 0 process 寫任意 packet 進 TUN | 信任邊界依賴 OS user model |
| **UDP socket spoof** | 偽 source IP 攻擊 server | endpoint sticky + cryptokey routing 部分緩解 |
| **TunnelVision (CVE-2024-3661)** | rogue DHCP option 121 推路由 | 部分由 fwmark 緩解 |
| **iOS Network Extension limits** | 限制 batch + GSO | 性能損失而非安全 |

### 3. 形式化定義

**Endpoint Sticky** 性質（informal）：

> For all (server S with local IPs L = {l_1, ..., l_n}) and (client packet p arriving at l_i), S's response to p must have source IP l_i.

違反這條 → NAT 後 client 收不到回應 → tunnel 不工作。**這條性質形式化可寫入 [Part 11] TLA+ spec**。

### 5. 我們協議的座標 / 設計取捨

G6 是否要 maintain own TUN/UDP layer？

**Pro**（自己寫）：
- 完全控制 endpoint sticky / fwmark / GSO 行為。
- 容易加 cover traffic（透明插入）。
- 可整合 [Part 2.6 io_uring](../part-2-high-perf-io/) 與 [2.7 XDP](../part-2-high-perf-io/) 等新介面。

**Con**（fork wireguard-go）：
- 多平台支援（macOS / iOS / Android / Windows）已 battle-tested。
- 維護負擔小。

**初步決定**：reference impl fork wireguard-go 的 `tun/` 與 `conn/`，**只在 g6-specific 處（cover traffic 整合、obfuscation 介面）打補丁**。研究期間（Part 12）優先用 fork；ship 時再評估是否完全自寫。

### 6. 必追資源 / 社群入口

- Linux netdev mailing list（UDP_GRO / IFF_VNET_HDR 演進）
- gVisor netstack issue tracker（user-space TCP/IP 演進）
- Tailscale 工程 blog（最大 wireguard-go 商用 fork）
- WireGuard mailing list `tun/conn` 標籤的 patch

### 7. 開放問題

1. **io_uring + zero-copy 對 wireguard-go 的整合**：目前 Go runtime 對 io_uring 支援仍不完整。能否做 hybrid runtime？這是 [Part 2.6](../part-2-high-perf-io/) 後的研究主題。
2. **AF_XDP 給 user-space VPN 的可能性**：BoringTun 已嘗試，但仍非生產級。能否設計穩定的 user-space VPN + XDP kernel bypass？
3. **macOS / iOS 性能天花板**：Apple 不開放 batch UDP API，user-space VPN 永遠慢。能否透過 Network Extension framework 找出更深整合的路徑？
4. **跨平台 stickiness 一致性**：fwmark 在 Linux、PF tag 在 macOS、WFP context 在 Windows——能否設計統一抽象？
