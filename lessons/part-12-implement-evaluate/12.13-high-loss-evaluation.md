## 動機

中國境外連線之 worst-case：3-15% packet loss + 100-300ms RTT。在這條鏈路上，throughput 主導因素 = congestion control 而非 link capacity。所以本堂課等於是 evaluate CC under adversarial loss。

```mermaid
flowchart LR
    PKT[App PKT] --> CC[CC decides rate]
    CC --> SEND[Send]
    SEND --LOSS p%--> NET[Network]
    NET --> RECV
    RECV --feedback--> CC
```

我們把這變 evaluation matrix：

| Loss % | RTT | CC strategy | 預期 metric |
|---|---|---|---|
| 0 | 30 | BBR | full link |
| 1 | 30 | BBR | 80-90% link |
| 5 | 30 | BBR | 40-60% |
| 5 | 30 | Brutal (user 設 100M) | 80-95% (but harm fairness) |
| 10 | 30 | BBR | 10-25% |
| 10 | 150 | BBR | 5-15% |
| 10 | 150 | Brutal (user 設 30M) | 60-80% |

## 核心概念

### 1. 損失類型 (loss model)

```text
1. iid random loss (uniform) — simulator-friendly, 不真實
2. bursty loss (Gilbert-Elliott) — 接近真實 wifi
3. periodic loss (1 of every N) — 接近某些 GFW QoS
4. correlated loss with reordering — multipath / route flap
```

我們對 4 類都跑：

```bash
# (1) iid 5%
tc qdisc add dev eth0 root netem loss 5%

# (2) GE: p=1%, r=10%, h=70%, k=0.1% (低有損率，高 burstiness)
tc qdisc add dev eth0 root netem loss gemodel 1% 10% 70% 0.1%

# (3) 每 100 個 packet 丟 5 個 (近似 5% systematic)
tc qdisc add dev eth0 root netem loss 5% 100%   # corr = 100% 近似 deterministic

# (4) loss + reorder
tc qdisc add dev eth0 root netem loss 3% reorder 5% delay 50ms
```

每 case 跑 5 repeats。

### 2. CC 在 loss 鏈路之 behavior

#### CUBIC (default Linux)
- loss-based：每 loss event 把 cwnd × 0.7
- Mathis equation 適用：$T \le \frac{C \cdot MSS}{RTT \sqrt{p}}$
- 1% loss + 30ms RTT, MSS=1380 → 約 18 Mbps（單流）
- 5% loss → 約 8 Mbps（崩）

#### BBRv2
- BtlBw estimate + min RTT
- 對 random loss 容忍度高（直到 loss > 約 15% 才開始 BtlBw 下調）
- 1% loss + 30ms RTT → 可達 link capacity (假設 link > 1 Gbps)
- 5% loss → 60-80% link
- 10% loss → ramping down，但仍 > CUBIC 10x

#### Hysteria brutal
- 不看 loss；按 user 設 bandwidth 直送
- 1% loss → 設多少送多少
- 但需 receiver 配合 buffer 大；超 link capacity 全部 buffer-bloat
- 道德爭議：不 cooperative，與其他流量搶 bandwidth

#### 我們的選擇：BBRv2 + opt-in Brutal
- 預設 BBRv2 fair
- 允許 user opt-in Brutal mode for 「known-private link」or 「high-loss only」
- Brutal mode UI 顯示 warning

### 3. Loss recovery 細節

#### Selective ACK / NACK
QUIC 用 ACK frame 帶 range；client 端可 ACK out-of-order；server 知道精確哪些 lost。
重傳：每丟 packet 之 retransmission 帶不同 packet number — 避免重傳 packet 被 reorder 識別。

#### Loss detection
- timer-based：RTO 過了沒收 ACK
- count-based：3 個 ACK in a row miss 該 packet
- early retransmit：window 不足以 trigger 3 dupack 時用 RACK

我們 spec § 7 之 loss detection 沿 RFC 9002 + RACK。

### 4. Forward Error Correction (FEC) — 我們的特色

對 5%+ loss 鏈路，retransmission 成本巨大。FEC：sender 每 K 個 source packet 多送 R 個 parity packet。receiver 在 R packet 內任意 K 個就能 reconstruct。

例：K=8, R=2 → 25% overhead，可容忍 2/10 = 20% loss without retx。

實作：Reed-Solomon over GF(2^8) 或 Raptor codes。Hysteria 沒有 FEC；TUIC 沒有；我們加入 = 真實 differentiator。

```rust
pub struct FecEncoder {
    k: usize,
    r: usize,
    block_buffer: Vec<Packet>,
}

impl FecEncoder {
    pub fn input(&mut self, pkt: Packet) -> Vec<Packet> {
        self.block_buffer.push(pkt);
        if self.block_buffer.len() == self.k {
            let parity = compute_rs_parity(&self.block_buffer, self.r);
            let block: Vec<_> = self.block_buffer.drain(..).chain(parity).collect();
            block
        } else {
            vec![]   // wait
        }
    }
}
```

陷阱：FEC 增加 fixed overhead 在 zero-loss link 反而傷 throughput；要 adaptive（loss < 1% 不啟用）。

### 5. Multipath / parallel transmission

對 high-loss link，並行送 2 條 path：
- path A (1% loss)
- path B (5% loss)
- 每 packet 兩條都送 → effective loss = 0.01 × 0.05 = 0.05%

代價：bandwidth × 2。對 control packet（小）合理；對 data packet 太貴。

我們 spec v0.1 不支援 multipath；v1.0 列為 experimental。

### 8. Loss bursts vs uniform: real-world implication

GFW 之 QoS：偶爾全 burst 丟 50-200ms 之 packet (對 UDP 是「斷流幾秒」)。
此情況：
- CC 必 reset (因為看似 RTO)
- 我們協議 send-resume 必快 (< 500ms 內回到 full speed)

實作：BBRv2 保留 cwnd state 1 RTT；超過則 slow-start。
測試：scripted「跑 60s，每 10s 中斷 1s」；量恢復時間。

### 9. Reordering

`tc netem reorder X%` 把 X% 之 packet 與下一個 swap。
對 CC：misinterpret 為 loss → throughput 降。
我們的 packet number 與 SACK 處理應 distinguish reorder vs loss (RACK 之主要 contribution)。

### 10. 結果解讀範例

假設我們得到：

```
Loss% | CUBIC | BBRv2 | our BBR | our +FEC | our Brutal | Hysteria2
0     | 940   | 940   | 945     | 920      | 945        | 940
1     | 180   | 800   | 820     | 850      | 920        | 900
5     | 35    | 480   | 540     | 720      | 880        | 850
10    | 12    | 180   | 280     | 540      | 780        | 720
15    | 5     | 80    | 130     | 350      | 660        | 580
```

解讀：
- CUBIC 是 baseline 慘狀；其他全部勝出（無懸念）
- BBRv2 在 5% loss 是 480；我們 +FEC 720 → FEC 帶來 +50%
- our +FEC 在 10-15% loss 比 Hysteria 略低 (因 Hysteria brutal 直接 push)；但 our Brutal mode 持平
- our +FEC 之 fairness 比 brutal 好 — 對 paper 是賣點

### 11. CPU cost of FEC

Reed-Solomon GF(2^8) k=8 r=2 之 encode: ~1 cycle/byte (with SIMD)。
解碼：需 k 個 packet 完整收後 reconstruct — latency budget 1-2 RTT。對 latency-sensitive (RPC、SSH) 可選 disable。

`cargo bench` 量 RS encode/decode throughput。預期 8 Gbps/core。

### 2. 對手分類學

| 對手 | 攻擊 | 防禦 |
|---|---|---|
| QoS-based shaping | selective UDP drop | TCP fallback, FEC |
| Targeted RTT inflation | hold packet for ms | not really exploitable; just slows |
| Selective packet drop (DPI) | 對「看起來像 proxy」之 packet 丟 | 整形 + AEAD 不可區 |

### 3. 形式化定義

**Goodput** $G = T \cdot (1 - p_{retx}) - O_{FEC}$ where $O_{FEC}$ is FEC overhead, $p_{retx}$ is fraction of bytes retransmitted.
**Recovery time** $T_r$：從 loss event 到回到 BtlBw × 0.8 之時間。

### 7. 開放問題

1. **FEC + congestion control 整合**：當 FEC overhead 與 CC 之 cwnd 互動，formal model 仍 open
2. **Selective loss attack**：對某些 packet（如 control）特定 drop 之 detection 與 防禦

4. **Optimal FEC block size**：given dynamic RTT/loss，optimal (k,r) 是 online RL problem，未有 SOTA
