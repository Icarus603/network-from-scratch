## 動機

被動 DPI 是「看流量」；主動探測是「親自連你 server 看反應」。GFW 之主動探測自 2010+ 起部署成熟，是 most insidious form：你部署 server，馬上被 200+ 中國 IP 連線探測；幾分鐘內被加 block list（IP + port）。

我們協議的 fallback path (12.7) 是 design-level 防禦；本堂課做 evaluation。

## 核心概念

### 1. Probe 類型 taxonomy（Ensafi 2015 + later）

| Type | 對 server 做 |
|---|---|
| **Random byte probe** | 連 :443, 送 random 1-1500 bytes, 看回應 |
| **Replay probe** | 重放看過的 client handshake bytes |
| **TLS-style probe** | 標準 openssl s_client / nmap –sV，看 TLS handshake |
| **HTTP probe** | GET / HEAD, 看 HTTP response |
| **HTTPS-fronted probe** | 真實 TLS handshake + HTTP GET |
| **Timing probe** | 連 + 立即斷, 重複，看 server 之 timing pattern |
| **Slow probe** | 連後 idle 60s, 看 server 是否主動 disconnect |
| **Authenticated-looking probe** (FEP) | 模擬 «有效 auth» 之 ClientHello，但 binder wrong |
| **Long-tail probe** | 拋 magic byte from known proxy （vmess 0x... / shadowsocks 0x... etc.）|

### 2. 評測 harness

實作一個 `prober` binary：

```rust
pub enum Probe {
    RandomBytes { len: usize, n_repeats: usize },
    ReplayCH { captured: Vec<u8> },
    TlsHandshake { sni: Option<String> },
    HttpGet { path: String, headers: Vec<(String, String)> },
    SlowConnect { idle_secs: u32 },
    KnownProxyMagic { bytes: Vec<u8> },
}

pub struct ProbeResult {
    pub probe: Probe,
    pub our_response: Vec<u8>,
    pub caddy_response: Vec<u8>,
    pub our_close_reason: CloseReason,
    pub caddy_close_reason: CloseReason,
    pub our_timing_us: u64,
    pub caddy_timing_us: u64,
}
```

對每 probe：connect 到 our server + caddy reference, compare 兩端 response。**任何 byte-level 或 timing-level 差距 = leakage**。

### 3. Comparison：byte / timing / behavior

```rust
fn evaluate(r: &ProbeResult) -> Verdict {
    if r.our_response != r.caddy_response { return Verdict::ByteDiff; }
    if (r.our_timing_us as i64 - r.caddy_timing_us as i64).abs() > 50_000 {
        return Verdict::TimingDiff;
    }
    if r.our_close_reason != r.caddy_close_reason { return Verdict::CloseDiff; }
    Verdict::Pass
}
```

我們設計目標：對所有 unauthenticated probe，verdict = Pass。

### 4. 對每 probe 之 expected behavior

#### Random byte probe
- Our: fallback path → Caddy → 對 raw bytes 反應 「invalid request」 RST 或 hang
- Caddy: 同上
- ✅ Indistinguishable

#### Replay probe
- 攻擊者重放看過的 ClientHello
- 我們：parse_client_hello 通過 (bytes 是 valid CH)；但 binder 已過期 (epoch 變) → auth fail → fallback
- Caddy 對 valid TLS bytes 之 TLS handshake → server hello
- ⚠️ Our 與 Caddy 看到 valid TLS 之 reaction 不同。Caddy emits TLS server hello，our emits fallback (但 fallback 路徑也送 same bytes → Caddy 處理 → Caddy server hello 一樣)
- 結論：✅ 因為 fallback 走 zero-copy splice 到 Caddy；Caddy 處理同樣 input → 同樣 output

#### TLS handshake probe
- openssl s_client connect to :443
- Our: parse_client_hello fail (real TLS CH not our format) → fallback to Caddy
- Caddy: TLS handshake → cert presented → ok
- ✅

#### HTTP GET probe
- HTTP request 連 :443 raw (no TLS)
- Our: parse_client_hello fail → fallback
- Caddy: response （probably 400 Bad Request or redirect to HTTPS）
- ✅

#### Slow connect / TCP keep-alive probe
- 我們之 server side timeout 必須 match Caddy default (5s for HTTP/2 idle, etc.)
- ✅ 對齊 Caddy timeout config

#### Known proxy magic probe
- prober 送 v2ray vmess magic + replay
- Our: parse_client_hello 不認 → fallback
- Caddy: same
- ✅

#### Authenticated-looking probe (FEP)
- Wu USENIX 2023 之核心 attack：故意把 binder 寫亂或 client_id wrong
- 我們：CT compare → fail → fallback
- ⚠️ 關鍵 invariant：CT compare 之 timing 與 Caddy 對 wrong-cert reject 之 timing 不可以差
- 我們：在 parse 失敗時 **延遲 send fallback bytes 直到 timing 對齊 Caddy**：

```rust
const FALLBACK_FAKE_DELAY_US: u64 = 50; // 與 Caddy parse first packet timing 對齊
fn fallback_with_timing_obscuration(...) {
    let start = Instant::now();
    let need_close = parse_and_authenticate(&bytes); // CT
    let elapsed = start.elapsed();
    if elapsed < Duration::from_micros(FALLBACK_FAKE_DELAY_US) {
        tokio::time::sleep(Duration::from_micros(FALLBACK_FAKE_DELAY_US) - elapsed).await;
    }
    forward_to_cover(stream, bytes).await
}
```

### 5. GFW probe pattern (literature-derived)

對 ss/vmess server (Ensafi 2015 + later):

```text
1. 客戶端剛連完 server：GFW sniff → 觀察 handshake byte pattern
2. 幾秒-幾分鐘內，從 ~100 中國 IP 發起 probe:
   - SS probe: salt + handshake byte sequence
   - vmess probe: random or replay
   - more recently: TLS handshake + check for known panel
3. 若 server response 與 known-circumvention impl 一致 → blacklist IP+port
4. 持續監測：每天 ~10-100 follow-up probe
```

我們的 evaluation：synthetic prober 應模擬此 sequence；若 byte-pass 但仍被 blacklist 證 GFW 用更深 ML — 進入 12.17 ML 評測。

### 6. Probe response 的微觀指標

| Indicator | 我們 vs Caddy |
|---|---|
| First-response byte position | identical (因 fallback 之前無 emit) |
| TCP MSS | identical (Linux kernel 一致) |
| TCP window scaling | identical |
| TCP timestamp | identical |
| Connection close (FIN/RST) | identical (Caddy reaches threshold first) |
| Time to first byte | within 50µs |
| TLS cert chain | identical (Caddy 之 cert) |
| HTTP server header | identical (Caddy default) |
| Robots.txt | identical (Caddy serves real file) |
| TLS session resumption ticket | identical |

每條都要 confirm。

### 7. 經驗：易被忽略的 leak

歷史上 SOTA proxy 之 probe-resistance leak：

| Proxy | Leak | 修法 |
|---|---|---|
| ss before 2014 | 對任何隨機 bytes 都 active reply | ignore-fail-mode |
| obfs3/4 v0 | 對 wrong-password 之 RST 與 normal RST timing 不同 | timing-equalize |
| vmess pre-2020 | recognizable padding pattern | redesigned header |
| trojan early | TLS-but-no-cert-validation 之 timing | match TLS conformance |
| **REALITY** | server 主動 emit SH 對 fallback CH 異常 | spec: SH only after auth |
| **VLESS-vision** | 1-RTT 包 size 比 real TLS 略不同 | not fully fixed (research-level) |

我們 v0.1 必避免上述每一條。

### 8. Probe-resistance test matrix

```text
For each (probe_type, repeat=N):
  Run on:
    A. Our server v0.1
    B. Pure Caddy (control)
    C. Hysteria2 (baseline; known weak to FEP)
    D. VLESS+REALITY (baseline; strong)
    E. TUIC v5 (baseline; medium)
    F. ss-2022 (baseline; weak)
  Compare:
    - response bytes
    - response timing
    - close reason
    - long-run behavior
```

Repeat 1000 次每 type；總 9 types × 6 servers × 1000 = 54k probes。

### 9. 期望 result

| Probe type | Our v0.1 vs Caddy distinguishable? |
|---|---|
| Random bytes | 0% (perfect) |
| Replay | 0% |
| TLS handshake | 0% |
| HTTP GET | 0% |
| Slow connect | 0% |
| Known proxy magic | 0% |
| Authenticated-looking (FEP) | ≤ 5% (timing tolerance) |
| Long-tail (1 week observation) | ≤ 10% (cover site behavior drift) |

對 Hysteria2：FEP-style probe 預期 detectability 70-90%（已知弱）。
對 VLESS+REALITY：FEP-style ~20-30% (Wu 2023 數據)。
我們 v0.1 ≤ 5% 是 SOTA goal。

### 10. 部署測試：實際在 VPS 上跑

不只 lab：我們把 v0.1 server 部署在境內可訪問之 VPS（如 AWS 東京），用境內 IP 訪問 1 個月，觀察：

- GFW 對該 IP+port 之 probe 次數（多少 unique source IP）
- 連線是否被 RST / DROP
- TCP RTT 是否異常變動
- 觀察 baseline (純 Caddy + 同 cert + 同 domain) 對比

如果 GFW 對我們 server 與對純 Caddy 之 probe rate 相同 → 完美。若我們 server 收到 +5-10x probe → 仍有 detectable feature。

### 11. 反向：給 GFW 看的 honeypot

進階 (research-level)：部署「intentional honeypot」用 known-detectable patterns，看 GFW 反應 — 用作 detector behavior 之 reverse-engineering。但這是 high-risk activity，建議僅在 academic ethics approval 下進行。

### 3. 形式化定義

**Probe-indistinguishability**: $\mathsf{Adv}^{\text{probe}}(\mathcal{P}) = |\Pr[\mathcal{P}(\text{S}) = 1] - \Pr[\mathcal{P}(\text{R}) = 1]| \leq \epsilon$, S=our server, R=reference Caddy。
**Timing-indistinguishability sub-property**: $\sup_x |T_S(x) - T_R(x)| \leq \delta$ across all probe input $x$, where $T$ is wall-clock response time.
