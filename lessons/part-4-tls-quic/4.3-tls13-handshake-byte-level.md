## 動機

讀 spec 跟「重現 spec」是兩件事。研究員的標準是：**能不能在沒網路、只有 RFC 8446 PDF 跟一隻筆的情況下，把一段 hex 完全解出來，並且寫一支 parser 通過 wire-format 測試**。這堂課把你逼到這個程度。

更深一層的動機：當我們在 Part 11 設計新協議時，**wire format 一個 bit 的錯就是部署層級的 disaster**。看別人怎麼設計到 byte 級別，是設計自己 wire format 的必要訓練。

## 核心概念

### 1. TLS record layer 的物理現實

任何 handshake message 都被包在 **TLS record** 裡傳。Record 結構（RFC 8446 §5.1）：

```
struct {
    ContentType type;            // 1 byte
    ProtocolVersion legacy_record_version;  // 2 bytes, 永遠 0x0303
    uint16 length;               // 2 bytes (record fragment length)
    opaque fragment[TLSPlaintext.length];   // 可變
} TLSPlaintext;
```

`ContentType` 列舉值：
```
invalid(0)
change_cipher_spec(20)
alert(21)
handshake(22)
application_data(23)
```

握手期間，`ContentType` = 22；handshake key 算出之後，後續 record `ContentType` 變成 23 (`application_data`) 但**內部仍是 handshake message**（因為要藏起來，type 偽裝）。這個「假 type」的真意是：record-level encryption 之後 `ContentType` 變 metadata，**真實 content type 移到 plaintext 結尾的 1 byte `inner_type`**（§5.2）：

```
struct {
    opaque content[TLSPlaintext.length];
    ContentType type;           // <--- 真實 type 在這裡
    uint8 zeros[length_of_padding];
} TLSInnerPlaintext;
```

加密之後變成 `TLSCiphertext`：
```
struct {
    ContentType opaque_type = application_data;  // 永遠 0x17
    ProtocolVersion legacy_record_version = 0x0303;
    uint16 length;
    opaque encrypted_record[TLSCiphertext.length];
} TLSCiphertext;
```

→ 這就是為什麼 Wireshark dissect 1.3 record 時看到 `Application Data`，但其實裡面可能是 `EncryptedExtensions` 或 `Certificate`。

### 2. Handshake message 的通用 header

每個 handshake message 進 record fragment 前還要包一層 `Handshake` struct（§4）：

```
struct {
    HandshakeType msg_type;     // 1 byte
    uint24 length;              // 3 bytes
    select (Handshake.msg_type) {
        case client_hello:          ClientHello;
        case server_hello:          ServerHello;
        case end_of_early_data:     EndOfEarlyData;
        case encrypted_extensions:  EncryptedExtensions;
        case certificate_request:   CertificateRequest;
        case certificate:           Certificate;
        case certificate_verify:    CertificateVerify;
        case finished:              Finished;
        case new_session_ticket:    NewSessionTicket;
        case key_update:            KeyUpdate;
    } body;
} Handshake;
```

`HandshakeType` 列舉值：
```
client_hello(1)
server_hello(2)
new_session_ticket(4)
end_of_early_data(5)
encrypted_extensions(8)
certificate(11)
certificate_request(13)
certificate_verify(15)
finished(20)
key_update(24)
message_hash(254)             // used in transcript hash 構造
```

### 3. ClientHello 結構（RFC 8446 §4.1.2）

```
uint16 ProtocolVersion;
opaque Random[32];

uint8 CipherSuite[2];    // 一個 cipher suite 是 2 bytes

struct {
    ProtocolVersion legacy_version = 0x0303;     // TLS v1.2
    Random random;
    opaque legacy_session_id<0..32>;
    CipherSuite cipher_suites<2..2^16-2>;
    opaque legacy_compression_methods<1..2^8-1>;
    Extension extensions<8..2^16-1>;
} ClientHello;
```

Extension 是 type-length-value：
```
struct {
    ExtensionType extension_type;  // 2 bytes
    opaque extension_data<0..2^16-1>;
} Extension;
```

### 4. 一份真實 ClientHello 的逐 byte 解剖

以下是一份典型 Firefox 122 對 `cloudflare.com` 發出的 ClientHello（**已 redact 隨機欄位**），共 517 bytes。我把 record header + handshake header + ClientHello body 全列出，並逐段解：

```
# Record header
16 03 01 02 00

# Handshake header
01 00 01 fc

# ClientHello body
03 03                                                     # legacy_version = TLS 1.2
de ad be ef 00 00 00 00 00 00 00 00 00 00 00 00
00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00          # client random (32 bytes, redacted)
20                                                        # legacy_session_id length = 32
ca fe ba be 00 00 00 00 00 00 00 00 00 00 00 00
00 00 00 00 00 00 00 00 00 00 00 00 00 00 00 00          # legacy_session_id (32 bytes, redacted)
00 1e                                                     # cipher_suites length = 30 bytes (15 suites)
13 02 13 03 13 01 c0 2c c0 2b cc a9 c0 30 c0 2f
cc a8 c0 24 c0 23 c0 28 c0 27 00 9f 00 9e          # 15 cipher suites
01 00                                                     # legacy_compression_methods (1 method: null)
01 99                                                     # extensions length = 0x0199 = 409 bytes
... (extensions follow)
```

**逐項解**：

**Record header**：
- `16` = ContentType.handshake (22)
- `03 01` = legacy_record_version (TLS 1.0 for ClientHello！注意這裡是 1.0 不是 1.2，RFC 8446 §5.1 規定 the first ClientHello might use 0x0301 to bypass certain middleboxes)
- `02 00` = fragment length = 0x0200 = 512 bytes

**Handshake header**：
- `01` = HandshakeType.client_hello (1)
- `00 01 fc` = uint24 length = 0x0001fc = 508 bytes

**ClientHello body** legacy_version：
- `03 03` = TLS 1.2（注意：**真實 1.3 版本協商在 supported_versions extension 裡**）

**Client random**：32 bytes，前 8 bytes 通常包含 GREASE bytes（FFFx pattern）+ 後面 24 bytes 是 CSRNG 輸出。**redacted 在範例裡**。

**legacy_session_id**：在 TLS 1.3 中不再用於 resumption，但實作仍會填一個 32-byte 隨機值——目的是**讓 server 回應 ServerHello.session_id 跟它一致，這樣 middlebox 看起來像 TLS 1.2 resumption**（middlebox compatibility mode）。

**Cipher suites**：30 bytes = 15 個 cipher suite。其中：
- `13 02` = TLS_AES_256_GCM_SHA384
- `13 03` = TLS_CHACHA20_POLY1305_SHA256
- `13 01` = TLS_AES_128_GCM_SHA256
- 接下來都是 1.2 cipher suites（`c0 2c` = TLS_ECDHE_ECDSA_WITH_AES_256_GCM_SHA384, …）
- 注意：**1.3 cipher suite 必須出現在 1.2 cipher suite 之前**（client 偏好順序）

**legacy_compression_methods**：`01 00` = 1 byte (null compression only)。1.3 強制只允許 null compression。

### 5. ClientHello extensions — 逐個拆

Extensions 是 ClientHello 的真正 payload。一個典型 modern Firefox ClientHello 包含 20+ 個 extensions。下面我列**對 1.3 而言必要 / 對指紋重要**的 extensions，並逐個解 wire format。

#### 5.1 `server_name` (SNI, RFC 6066, type=0x0000)

```
00 00          # extension_type = server_name
00 12          # extension_data length = 18
   00 10                       # ServerNameList length = 16
      00                       # NameType = host_name (0)
      00 0d                    # HostName length = 13
      63 6c 6f 75 64 66 6c 61 72 65 2e 63 6f 6d   # "cloudflare.com"
```

**這 13 bytes 的明文 `cloudflare.com` 就是 GFW 過去 10 年最依賴的探測點**。Part 4.6 的 ECH 主題就是要把這 13 bytes 藏起來。

#### 5.2 `supported_versions` (RFC 8446, type=0x002b)

```
00 2b          # extension_type = supported_versions
00 05          # extension_data length = 5
   04                       # versions list length = 4 bytes
   03 04                    # TLS 1.3 (preferred)
   03 03                    # TLS 1.2 (fallback)
```

**這個 extension 才是真正的版本協商**。RFC 8446 §4.2.1 規定：if this extension is present, the server **must** use it and ignore `legacy_version`.

#### 5.3 `supported_groups` (RFC 7919, type=0x000a)

```
00 0a          # extension_type = supported_groups
00 0c          # extension_data length = 12
   00 0a                    # NamedGroupList length = 10
   00 1d                    # x25519
   00 17                    # secp256r1
   00 18                    # secp384r1
   00 19                    # secp521r1
   01 00                    # ffdhe2048 (FFDHE)
```

NamedGroup 列舉（RFC 8446 §4.2.7）：
- `0x0017` (23) = secp256r1（NIST P-256）
- `0x0018` (24) = secp384r1（NIST P-384）
- `0x0019` (25) = secp521r1（NIST P-521）
- `0x001d` (29) = x25519（Curve25519, RFC 7748）
- `0x001e` (30) = x448（Curve448, RFC 7748）
- `0x0100` ~ `0x0103` = FFDHE 群（2048/3072/4096/6144 bit）
- `0x11ec` (4588) = X25519MLKEM768（draft，post-quantum hybrid，2024+）

#### 5.4 `key_share` (RFC 8446 §4.2.8, type=0x0033)

```
00 33          # extension_type = key_share
00 26          # extension_data length = 38
   00 24                    # KeyShareClientHello length = 36
   00 1d                    # group = x25519
   00 20                    # key_exchange length = 32 bytes
   XX XX XX ... XX          # 32-byte x25519 public key
```

關鍵設計：**ClientHello 直接帶 ECDHE share**。Server 看到後直接回對應的 ECDHE share 在 ServerHello.key_share，雙方 derive shared secret，**不需要 1.2 的額外 ServerKeyExchange 與 ClientKeyExchange round trip**。這就是 1-RTT 的核心。

如果 client 在 `supported_groups` 列了 x25519 + secp256r1，但只在 `key_share` 帶 x25519（最常見的優化）：
- 若 server 也支援 x25519 → 直接用，1-RTT
- 若 server 不支援 x25519 但支援 secp256r1 → server 回 `HelloRetryRequest`（用 ServerHello 的 wire format 但 random 是特殊常數 `CF 21 AD 74 E5 9A 61 11 BE 1D 8C 02 1E 65 B8 91 C2 A2 11 16 7A BB 8C 5E 07 9E 09 E2 C8 A8 33 9C`），告知 client 要哪個 group，client 重發第二個 ClientHello 帶 secp256r1 share → 2-RTT（仍比 1.2 少）

#### 5.5 `signature_algorithms` (RFC 8446 §4.2.3, type=0x000d)

```
00 0d          # extension_type = signature_algorithms
00 18          # extension_data length = 24
   00 16                    # SignatureSchemeList length = 22
   04 03                    # ecdsa_secp256r1_sha256
   05 03                    # ecdsa_secp384r1_sha384
   06 03                    # ecdsa_secp521r1_sha512
   08 04                    # rsa_pss_rsae_sha256
   08 05                    # rsa_pss_rsae_sha384
   08 06                    # rsa_pss_rsae_sha512
   04 01                    # rsa_pkcs1_sha256
   05 01                    # rsa_pkcs1_sha384
   06 01                    # rsa_pkcs1_sha512
   08 07                    # ed25519
   08 08                    # ed448
```

注意：**TLS 1.3 強制禁掉 MD5、SHA-1 in handshake signing**（補 SLOTH 攻擊）。RFC 8446 §4.2.3 明列 deprecated algorithms。`rsa_pkcs1_*` 只在 1.3 客戶端為了 *legacy server cert* 接受，不允許 server 在 CertificateVerify 用。

#### 5.6 `psk_key_exchange_modes` (type=0x002d) + `pre_shared_key` (type=0x0029)

`psk_ke_modes`：
```
00 2d          # extension_type
00 02
   01                       # length = 1
   01                       # psk_dhe_ke (=1)
```

兩種模式：
- `psk_ke` (0)：純 PSK，no DHE → 沒前向保密
- `psk_dhe_ke` (1)：PSK + DHE，有 FS（modern client 預設）

`pre_shared_key`（resumption 才出現）：
```
00 29          # extension_type
00 XX          # length
   XX XX                    # PskIdentities length
      XX XX                 # identity length
      XX XX ... XX          # PSK identity (server-issued ticket)
      XX XX XX XX           # obfuscated_ticket_age (uint32, ticket_age + ticket_age_add)
   XX XX                    # PskBinderEntry list length
      XX                    # binder length
      XX XX ... XX          # HMAC(binder_key, ClientHello prefix)
```

**PSK binder 的角色**：證明 client 確實知道 PSK，且 binder 是對 ClientHello 之前部分的 HMAC——可以防 binder replay。

`pre_shared_key` extension **必須是 ClientHello extensions list 的最後一個**（RFC 8446 §4.2.11.2），因為 binder 是對「整個 ClientHello 直到 binder 開始之前」做 HMAC，必須讓 binder 自己不在 hash 範圍內。

#### 5.7 其他重要 extensions

- `application_layer_protocol_negotiation` (ALPN, RFC 7301, type=0x0010)：客戶端公告支援的 application protocol（`h2`、`http/1.1`、`h3`）
- `extended_master_secret` (RFC 7627, type=0x0017)：1.2 的修補，1.3 不需要但仍要回應（implementation 不會 fail）
- `session_ticket` (RFC 5077, type=0x0023)：1.3 仍接受但用法不同
- `record_size_limit` (RFC 8449, type=0x001c)：限制對方發 record 的最大長度
- `compress_certificate` (RFC 8879, type=0x001b)：協商 Certificate 壓縮
- `delegated_credential` (RFC 9345, type=0x0022)：短 lifetime credential
- `encrypted_client_hello` (ECH, draft-ietf-tls-esni-17, type=0xfe0d)：4.6 主題
- GREASE values (RFC 8701)：`0x?A0A`、`0x?A1A`... 等隨機值，確保 server 不會 hardcode unknown extensions

### 6. ServerHello（RFC 8446 §4.1.3）

結構幾乎跟 ClientHello 鏡像：

```
struct {
    ProtocolVersion legacy_version = 0x0303;     // TLS 1.2
    Random random;
    opaque legacy_session_id_echo<0..32>;        // echo client's session_id
    CipherSuite cipher_suite;                    // chosen suite
    uint8 legacy_compression_method = 0;
    Extension extensions<6..2^16-1>;
} ServerHello;
```

差異：
- 只回**一個** cipher suite
- `legacy_session_id_echo` 必須 echo 客戶端送來的 session_id（不管 server 是否真的 resume）
- `random` 後 8 bytes 是 downgrade sentinel（前一堂講過）

ServerHello extensions 通常只有：
- `supported_versions` (=0x002b, value=0x0304)
- `key_share` (=0x0033, server 對應 group 的 share)
- 若 resumption：`pre_shared_key` (=0x0029, selected_identity)

**所有其他 server 想回的東西都在 EncryptedExtensions 裡**（之後）。

### 7. EncryptedExtensions（§4.3.1）

```
struct {
    Extension extensions<0..2^16-1>;
} EncryptedExtensions;
```

包含 ALPN 選擇、SNI 確認、early_data 接受、ECH retry config 等等。**所有非 "key exchange 必需" 的東西都搬到這裡**，因為這是 ServerHello 之後第一個加密 message。

### 8. Certificate（§4.4.2）

```
struct {
    select (certificate_type) {
        case RawPublicKey:
            opaque ASN1_subjectPublicKeyInfo<1..2^24-1>;
        case X509:
            opaque cert_data<1..2^24-1>;
    };
    Extension extensions<0..2^16-1>;
} CertificateEntry;

struct {
    opaque certificate_request_context<0..2^8-1>;
    CertificateEntry certificate_list<0..2^24-1>;
} Certificate;
```

`certificate_request_context`：在 server-side 第一次 Certificate 是空的（length=0）；post-handshake auth 時 client 對應之前的 CertificateRequest 中的 context value。

**為什麼 server cert 被加密？** Part 4.6 ECH 之外的另一個 SNI 對抗動機：1.2 時 cert 是明文，passive observer 從 SAN/CN 就能識別 server identity。1.3 把 cert 藏進 handshake key 加密之後，**SAN/CN 對 passive observer 隱藏**——但 **SNI 仍明文**，所以 GFW 仍能識別。ECH 是後續修補。

### 9. CertificateVerify（§4.4.3）

```
struct {
    SignatureScheme algorithm;       // 2 bytes
    opaque signature<0..2^16-1>;
} CertificateVerify;
```

簽什麼？**對 transcript 簽**：
```
content = string(64 bytes of 0x20) ||
          context_string ||
          0x00 ||
          Transcript-Hash(Handshake Context, Certificate)
```

- `64 bytes of 0x20` = 64 個空格——padding，讓「TLS 1.3 server CertificateVerify 簽的東西」跟「TLS 1.2 server CertificateVerify 簽的東西」hash 永遠不會碰撞（cross-version anti-collision）
- `context_string` = `"TLS 1.3, server CertificateVerify"`（server）or `"TLS 1.3, client CertificateVerify"`（client）
- `0x00` separator
- `Transcript-Hash(Handshake Context, Certificate)` = 從 ClientHello 到 Certificate（包含）的所有 handshake message 的 hash

**這個簽章的意義**：server 用 cert 對應 private key 證明「我看到的 ClientHello 跟 ServerHello 跟 Certificate 跟你看到的一致」。簽**transcript** 而非簽**簽章特定欄位**是防 KCI（key compromise impersonation）與 transcript collision attack 的關鍵。

### 10. Finished（§4.4.4）

```
struct {
    opaque verify_data[Hash.length];
} Finished;
```

`verify_data`：
```
verify_data =
    HMAC(finished_key,
         Transcript-Hash(Handshake Context,
                         Certificate*, CertificateVerify*))

finished_key = HKDF-Expand-Label(
    BaseKey,
    "finished",
    "",
    Hash.length)

BaseKey = client_handshake_traffic_secret  # for client Finished
       or server_handshake_traffic_secret  # for server Finished
```

**Finished 之後，傳輸切換到 `application_traffic_secret_0`**：
```
client_application_traffic_secret_0 =
    Derive-Secret(Master Secret, "c ap traffic",
                  ClientHello..server Finished)

server_application_traffic_secret_0 =
    Derive-Secret(Master Secret, "s ap traffic",
                  ClientHello..server Finished)
```

Server Finished 結束之後，server 可以送 0.5-RTT application data；Client Finished 結束之後，雙向 1-RTT 完成。

### 11. KeyUpdate（§4.6.3）

```
enum {
    update_not_requested(0),
    update_requested(1),
} KeyUpdateRequest;

struct {
    KeyUpdateRequest request_update;
} KeyUpdate;
```

任一方可以送 `KeyUpdate` 來 rotate traffic key：
```
application_traffic_secret_N+1 =
    HKDF-Expand-Label(application_traffic_secret_N,
                      "traffic upd", "", Hash.length)
```

如果 `request_update = update_requested`，對方收到後也 send 自己的 KeyUpdate。

`KeyUpdate` 的設計用途：long-lived connections rotate AEAD nonce 空間之前 rekey（AEAD 安全 bound 約 2^24.5 record，超過需 rekey）。**這就是 RFC 8446 §5.5 的 record limit 來自 Bhargavan et al. 2017 S&P paper 計算的具體 bound 的緣由**。

## 與我們協議設計的關聯

到此堂為止你應該明白：**TLS 1.3 是一個極其精密的 wire format，每個 byte 的位置都有 anti-attack 或 anti-ossification 的理由**。對我們協議設計的指引：

1. **強制 transcript-hash binding 簽章**：避免 KCI、SLOTH-style collision
2. **handshake header 用 explicit type-length-value**：不要省字節
3. **`HelloRetryRequest` 用 ServerHello 的 wire format + 特殊 random** 是個聰明的 trick——keep wire format flat，state 機制下放
4. **session_id_echo 留 dummy**：教訓是「為了 middlebox 留 ritual bytes 比改 wire format 容易」
5. **任何 message 後面的 padding/extension 必須 length-prefixed**，永遠不靠 sentinel
6. **內層真實 type 在 plaintext 結尾 1 byte** 是個極簡優雅的設計（part 7 protobuf-style trailers 也用類似）

### Anti-fingerprint 的考量

ClientHello 是 *最後一塊明文*。它的每個欄位都被 JA3/JA4 拿來指紋（4.4 詳）。我們協議的 wire format 必須：
- **A 路線（REALITY 路線）**：模仿真實 TLS 1.3 ClientHello，連 extension 順序與 GREASE 值都一致
- **B 路線（MASQUE 路線）**：藏在 QUIC initial packet 裡，先躲開 ClientHello 指紋（但仍有 QUIC long header pattern）
- **C 路線（ECH 路線）**：把 inner ClientHello 整個加密，outer 用「平淡」公開 SNI

Part 6、7、11 反覆比較這三條。

## 動手（必做，60 分鐘）

### 練習 1：用 Wireshark + tls13.xargs.org 對讀一份真實 ClientHello

1. 在 macOS 抓一份本地 curl 到 example.com 的 TLS 1.3 連線：

```bash
sudo tcpdump -i en0 -w /tmp/tls.pcap host example.com and port 443 &
curl -v https://example.com -o /dev/null
sudo killall tcpdump
```

2. 用 Wireshark 開 `tls.pcap`，filter `tls.handshake.type == 1` 找到 ClientHello。
3. **右鍵 → Show Packet Bytes** 看 raw hex。
4. 對著 tls13.xargs.org 把每個 extension 拆出。

> redaction：刪掉 pcap，不要 commit。

### 練習 2：手寫一個極簡 ClientHello parser（Python）

```python
import struct

def parse_client_hello(data: bytes):
    # data starts at record fragment (after record header)
    assert data[0] == 0x01, "must be ClientHello"
    length = int.from_bytes(data[1:4], 'big')
    body = data[4:4 + length]

    # legacy_version
    legacy_version = body[0:2]
    # random
    random = body[2:34]
    # legacy_session_id
    session_id_len = body[34]
    session_id = body[35:35 + session_id_len]

    cur = 35 + session_id_len
    cipher_suites_len = int.from_bytes(body[cur:cur+2], 'big')
    cipher_suites = body[cur+2:cur+2+cipher_suites_len]
    cur += 2 + cipher_suites_len

    compression_methods_len = body[cur]
    compression_methods = body[cur+1:cur+1+compression_methods_len]
    cur += 1 + compression_methods_len

    extensions_len = int.from_bytes(body[cur:cur+2], 'big')
    extensions_data = body[cur+2:cur+2+extensions_len]

    # parse extensions
    extensions = []
    i = 0
    while i < len(extensions_data):
        ext_type = int.from_bytes(extensions_data[i:i+2], 'big')
        ext_len = int.from_bytes(extensions_data[i+2:i+4], 'big')
        ext_data = extensions_data[i+4:i+4+ext_len]
        extensions.append((ext_type, ext_data))
        i += 4 + ext_len

    return {
        'legacy_version': legacy_version.hex(),
        'random': random.hex(),
        'session_id': session_id.hex(),
        'cipher_suites': [cipher_suites[j:j+2].hex() for j in range(0, len(cipher_suites), 2)],
        'extensions': [(hex(t), d.hex()) for t, d in extensions],
    }
```

跑你自己的 ClientHello hex 進去，對著 RFC 確認每個欄位。**這支 100 行 parser 完整覆蓋 RFC 8446 §4.1.2** — 你已經知道 1.3 ClientHello 的全部。

### 練習 3：對著 [tls13.xargs.org](https://tls13.xargs.org/) 走完完整 handshake

xargs 把一份真實的 client + server 完整 1.3 handshake 全部 byte-level annotated。**從頭走到尾**，配合 RFC 8446 §7.1 自己用 HKDF 重算 secrets 試試看（Python `cryptography` 套件 `HKDF` 直接可用）。**這個練習做完才算真的「逐 byte 重現 TLS 1.3」**。

1. **ClientHello 的 `legacy_session_id` 在 1.3 純客戶端視角是廢欄位**。為什麼 implementation 仍會填 32 bytes 而非 0 length？

3. **CertificateVerify 簽 64 個 0x20 + context_string + 0x00 + transcript_hash**。如果省略 64 個 0x20，會有什麼 cross-version attack？提示：1.2 的 CertificateVerify 簽什麼？

### 1. 學界詞彙

| 口語 | 學界用詞 |
|---|---|
| 「逐 byte 拆」 | **Wire format dissection / packet parsing** |
| 「key 換」 | **Traffic key rotation / rekey** |

| 「對 transcript 簽」 | **Transcript-bound signing / handshake commitment** |
| 「Server 提供 ephemeral key share」 | **Static-ephemeral hybrid key agreement** |
| 「session ticket」 | **Resumption PSK / opaque server state token** |

### 2. 對手分類學

ClientHello 是最後一個明文 message。對應的攻擊面：

| 攻擊類型 | 攻擊面 |
|---|---|
| Passive identification (JA3/JA4) | 順序、選項、GREASE 位置 |
| Active probe (ZMap/Censys) | SNI、ALPN |
| Replay attack (0-RTT) | pre_shared_key + early_data ext |
| Transcript collision (SLOTH) | sig_algorithms 中 MD5/SHA-1 |
| Length-based fingerprint | extension order, padding ext |

### 3. 形式化定義

**Transcript hash 的 Merkle-style 線性性**：
$$
T_i = H(T_{i-1} \| m_i), \quad T_0 = \emptyset
$$

TLS 1.3 把 transcript hash 透過 `Derive-Secret(..., Transcript-Hash(...))` 灌入 key schedule，這提供**transcript-binding security**：
> $\forall A$（active attacker），若 $A$ 能讓兩個 honest endpoints commit 到不同的 transcript $T \neq T'$，則 $A$ 能找到 hash collision，與 $H$ 的 collision resistance 矛盾。

這條 lemma 是 1.3 所有安全屬性 proof 的核心。

### 5. 我們協議的座標

ClientHello-equivalent 設計空間：
- ✅ extension TLV 結構（不 footgun）
- ✅ extension 順序由 spec 強制（不留隨意度給 implementation 製造指紋）
- ✅ 強制 GREASE 在某些 slot
- ❓ 是否完全模仿 TLS 1.3 ClientHello（REALITY）vs 自己設計新 inner format（Part 11）

### 7. 開放問題

- ClientHello 的 anti-fingerprinting 與「易部署性」之間的 trade-off：完全模仿真實 ClientHello 增加實作成本與 indistinguishability，但對抗 statistical fingerprint（如 Wu-FEP 2023）能力有限。Part 11 重大設計議題。
- TLS 1.3 + ML-KEM 的 ClientHello size > 1500 bytes，會撞 IP fragmentation。draft-ietf-tls-tlsflags-12 等 proposal 仍在草案。
