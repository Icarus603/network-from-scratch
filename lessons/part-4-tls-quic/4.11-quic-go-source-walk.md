## 動機

讀 RFC 跟讀 **「能跑的 production code」** 是兩件事。RFC 描述「**做什麼**」；implementation 描述「**怎麼做**」。

quic-go 是純 Go 實作，可讀性極佳：
- ~70K LOC，比 quinn (Rust, generics 重) 容易跟
- API 設計乾淨：`Listener` / `Connection` / `Stream` 三 layer
- 廣泛 production-deployed（caddy / sing-box / xray-core 都用過）
- 文件齊全；qlog 整合

讀完應該回答：
- quic-go 的 packet receive loop 在哪、怎麼 dispatch 到 connection
- handshake state machine 在 Go 中如何呈現（vs RFC 8446 Appendix A）
- ACK frame 處理跟 loss recovery 怎麼整合
- HTTP/3 layer 怎麼跟 QUIC layer 整合
- 哪些設計決策跟 RFC 不完全直接 mapping（implementation 補上的細節）

### 2. Packet ingress：UDP datagram → connection

入口：`transport.go` 中的 `Transport.listen()` 啟動 UDP receive loop:

```go
// transport.go
func (t *Transport) listen(conn rawConn) {
    for {
        p, err := t.readNextPacket()  // blocking read
        if err != nil { ... }
        t.handlePacket(p)
    }
}
```

`packet_handler_map.go` 中 `handlePacket(p)` 做兩件事：
1. **Parse short/long header** 取 Destination Connection ID
2. **Lookup** connection by DCID → dispatch

Long header 第一次（Initial）對應的 connection 不存在 → 走 `serverConn.handleInitialPacket()` 建新 connection；短 header / handshake 對應現有 connection → call `connection.handlePacket(p)`。

對應 RFC 9000 §5.2: connection 由 DCID 識別。Implementation 細節：quic-go 維護兩個 maps —
- `handlers map[ConnectionID]packetHandler`
- `resetTokens map[StatelessResetToken]packetHandler`

第二個 map 是為了 stateless reset detection（收到 unknown DCID packet 時查最末 16 bytes 是否 match known token）。

### 3. Connection 主 loop

`connection.go` 是核心。`Connection` 結構含十幾個 goroutine 協同：

```go
// connection.go (simplified)
type connection struct {
    handshakeStream cryptoStream
    streamsMap streamsMap
    sentPacketHandler ackhandler.SentPacketHandler
    receivedPacketHandler ackhandler.ReceivedPacketHandler
    sendConn sendConn
    congestionController congestion.SendAlgorithm
    flowController flowcontrol.ConnectionFlowController
    cryptoSetup handshake.CryptoSetup
    packer *packetPacker
    unpacker unpacker
    framer framer
    closeChan chan struct{}
    sendQueue sendQueue
    ...
}

func (c *connection) run() error {
    go c.cryptoStreamHandler.RunHandshake()
    go c.sendQueue.Run()

    for {
        select {
        case <-c.closeChan:
            return ...
        case p := <-c.receivedPackets:
            c.handlePacketImpl(p)
        case <-c.sendingScheduled:
            c.sendPackets()
        ...
        }
    }
}
```

關鍵點:
- **Single goroutine for state machine**（避 race condition）；I/O 在 sub-goroutine
- **`cryptoStreamHandler` goroutine** 跑 TLS 1.3 handshake——把 received CRYPTO frame 餵給 `handshake.CryptoSetup`，後者用 Go `crypto/tls` 內部 API 跑握手；TLS 算出 secret 後回呼 `installInitialSealer` 等
- **`sendQueue` goroutine** 接收 ready-to-send packets 並 syscall write 到 UDP

### 4. Packet packing — `packet_packer.go`

每次要發 packet，由 `packetPacker.PackPacket()` 組合：
1. 從 `framer.AppendFrames()` 取 outstanding STREAM frames、CRYPTO frames
2. 從 `receivedPacketHandler.GetAckFrame()` 取 ACK frame（如果需要 ack）
3. 從 `flowController` 取 MAX_DATA / MAX_STREAM_DATA frame（如果 window 變）
4. 從 `windowUpdateQueue` 取 frame
5. 計算 packet number、header
6. AEAD seal payload
7. Header protection
8. 回傳 `*coalescablePacket`

Coalescing: 多個 packet（如 Initial + Handshake + 1-RTT）可以同 datagram。`Transport` 把多 packet bytes concat 後一個 syscall send。

### 5. Frame parsing — `internal/wire/`

`internal/wire/` 含每個 frame type 的 parser + serializer。例如 `STREAM` frame:

```go
// internal/wire/stream_frame.go
type StreamFrame struct {
    StreamID       protocol.StreamID
    Offset         protocol.ByteCount
    Data           []byte
    Fin            bool
    DataLenPresent bool
}

func parseStreamFrame(r *bytes.Reader, typeByte byte, ...) (*StreamFrame, error) {
    fin := typeByte&0x01 > 0
    hasOff := typeByte&0x04 > 0
    hasDataLen := typeByte&0x02 > 0
    streamID, _ := quicvarint.Read(r)
    var offset uint64
    if hasOff {
        offset, _ = quicvarint.Read(r)
    }
    var dataLen uint64 = uint64(r.Len())
    if hasDataLen {
        dataLen, _ = quicvarint.Read(r)
    }
    data := make([]byte, dataLen)
    r.Read(data)
    return &StreamFrame{
        StreamID: protocol.StreamID(streamID),
        Offset:   protocol.ByteCount(offset),
        Data:     data,
        Fin:      fin,
    }, nil
}
```

注意 `typeByte` 的 bit semantics 直接對應 RFC 9000 §19.8 — `FIN`(0x01), `LEN`(0x02), `OFF`(0x04)。

Variable-length integer 解碼在 `quicvarint/` package（RFC 9000 §16）：
```go
// quicvarint/varint.go
// 2-bit prefix 決定總長度: 00=1B, 01=2B, 10=4B, 11=8B
func Read(r io.ByteReader) (uint64, error) {
    b1, _ := r.ReadByte()
    prefix := b1 >> 6
    length := 1 << prefix
    val := uint64(b1 & 0x3F)
    for i := 1; i < length; i++ {
        b, _ := r.ReadByte()
        val = (val << 8) | uint64(b)
    }
    return val, nil
}
```

### 6. Handshake — `internal/handshake/`

`crypto_setup.go` 是核心：

```go
type cryptoSetup struct {
    tlsConf *tls.Config
    conn    *qtls.Conn

    runner handshakeRunner
    ...

    initialSealer LongHeaderSealer
    initialOpener LongHeaderOpener
    handshakeSealer LongHeaderSealer
    handshakeOpener LongHeaderOpener
    oneRTTSealer ShortHeaderSealer
    oneRTTOpener ShortHeaderOpener
    zeroRTTSealer LongHeaderSealer
    zeroRTTOpener LongHeaderOpener
}

func (h *cryptoSetup) RunHandshake() {
    err := h.conn.Handshake()  // Go crypto/tls 跑 TLS 1.3 state machine
    ...
}
```

`internal/qtls/` 是 quic-go 對 Go `crypto/tls` 的封裝——Go 1.21+ 提供 `tls.QUICConn` API 讓 QUIC implementer 不必 fork crypto/tls。每次 TLS 算出新 traffic secret → 透過 callback `WriteRecord` / `SetReadSecret` / `SetWriteSecret` 告訴 quic-go 安裝新 sealer/opener。

`Sealer` / `Opener` 是 AEAD + header protection 的 wrapper:

```go
// internal/handshake/aead.go
type LongHeaderSealer interface {
    Seal(dst, src []byte, pn protocol.PacketNumber, associatedData []byte) []byte
    EncryptHeader(sample []byte, firstByte *byte, pnBytes []byte)
    Overhead() int
}
```

`Seal()` = AEAD encrypt + auth tag；`EncryptHeader()` = header protection XOR。對應 RFC 9001 §5.4.1。

### 7. Loss recovery — `internal/ackhandler/`

`sent_packet_handler.go` 對應 RFC 9002:
- 每個 sent packet 記在 `packetHistory`
- 收到 ACK frame → `OnPacketAcked()` 標記 ack 並更新 RTT
- Loss detection trigger:
  - Packet threshold: `kPacketThreshold = 3` (RFC 9002 §6.1.1)
  - Time threshold: `9/8 * max(SRTT, latestRTT)`
- Probe timeout (PTO): `(SRTT + 4*RTTVAR + max_ack_delay) * 2^backoff`
- Detected loss → mark packet's frames as lost → frames 重新進 `framer` queue → 下次發 packet 帶上（**new packet number**）

關鍵：retransmitted frame 不是 retransmitted packet。RFC 9002 §6.2 強調 packet number **不重用**，packet 內含的 frame 才被 retransmit。

### 8. Congestion control — `internal/congestion/`

`congestion.go` 介面：
```go
type SendAlgorithm interface {
    GetCongestionWindow() protocol.ByteCount
    OnPacketSent(...)
    OnPacketAcked(...)
    OnPacketLost(...)
    MaybeExitSlowStart()
    TimeUntilSend(bytesInFlight protocol.ByteCount) time.Time
    ...
}
```

實作 `cubic_sender.go`（CUBIC, RFC 8312）作 default。

```go
// CUBIC: W(t) = C(t - K)^3 + W_max
// where K = (W_max * (1 - beta) / C)^(1/3)
//       beta = 0.7 (typical)
```

`cubic.go` 維護 `wMax`、`epochStart`、`originPointCongestionWindow` 等狀態，按 cubic function update window。

BBR 與 BBRv2 在 quic-go 也有實作（`bbr_sender.go`）。

### 9. Flow control — `internal/flowcontrol/`

`base_flow_controller.go` 是 generic flow controller:
```go
type baseFlowController struct {
    bytesSent uint64
    bytesReceived uint64
    highestReceived uint64

    sendWindow uint64
    receiveWindow uint64
    receiveWindowSize uint64
    maxReceiveWindowSize uint64

    epochStartTime time.Time
    epochStartOffset uint64
    rttStats *utils.RTTStats
}
```

每條 stream 有自己 `streamFlowController`；connection 有 `connectionFlowController`。Window auto-tuning 在 `maybeAdjustWindowSize()`：根據 throughput 動態調大 window。

### 10. Streams — `stream.go` + `framer.go`

`stream.go` 是 user-facing stream API:
```go
type Stream interface {
    StreamID() StreamID
    Read(p []byte) (int, error)
    Write(p []byte) (int, error)
    CancelRead(StreamErrorCode)
    CancelWrite(StreamErrorCode)
    Context() context.Context
    SetReadDeadline(time.Time) error
    SetWriteDeadline(time.Time) error
    SetDeadline(time.Time) error
    Close() error
}
```

Internal: `receiveStream` 持 `frameSorter` 對亂序 STREAM frame 重組；`sendStream` 持寫 buffer。

`framer.go` `AppendFrames()` 在 packing 時拿 stream queue 中 frames 出來：
- 對 `sendStream` 詢問 "有什麼 frame 要送"
- 對 `cryptoStream` 同樣
- 加 ACK frame、flow control frame 等

### 11. HTTP/3 layer — `http3/`

`http3/client.go` + `http3/server.go`:
```go
type RoundTripper struct {
    TLSClientConfig *tls.Config
    QUICConfig *quic.Config
    EnableDatagrams bool
    ...
}

func (r *RoundTripper) RoundTrip(req *http.Request) (*http.Response, error) {
    // dial or reuse QUIC connection
    conn := r.dial(req.URL.Host)
    str := conn.OpenStreamSync(...)
    // write HEADERS frame using QPACK
    encoder.WriteFields(str, headers)
    // write DATA frames
    str.Write(body)
    // read response
    ...
}
```

`frames.go` 含 H3 frame parser/serializer。QPACK 在 `qpack/` subpackage。

對應 RFC 9114 §7 frame types。

### 12. qlog 整合

`qlog/qlog.go` 提供 event recorder。每個 protocol event（packet send/receive, frame, state transition）寫成 qlog event。

對 debug 與 research 極有用 — Part 4.9 已介紹 qvis 可視化。

### 13. 跟 RFC 不直接 mapping 的地方

quic-go implementation choice 對 RFC 的補充：
- **`maxReceiveWindowSize` upper bound**：spec 沒上限，implementation 為防 memory exhaustion 加
- **Frame queueing 順序**：spec 沒指定 ACK vs STREAM vs CRYPTO 順序，implementation 自選
- **Idle timeout granularity**：spec 給 max_idle_timeout，但 implementation 內部 timer 用 monotonic clock + drift compensation
- **0-RTT 接受策略**：spec 給 framework，implementation 決定哪些 application data 接受
- **Path validation timing**：spec 給 framework，implementation 決定何時觸發

這些 implementation-level 決策正是「**RFC 跟 production code 之間的灰色地帶**」——研究員要 carefully calibrate。

## 與我們協議設計的關聯

讀 quic-go 對我們的 implementation 階段（Part 12）有具體 inspiration:

| quic-go pattern | 我們協議的繼承 |
|---|---|
| Sealer / Opener abstraction | ✅ AEAD layer 用同樣 interface |
| Single-goroutine state machine | ✅ Go-based implementation 直接套用 |
| qtls hook into Go crypto/tls | ✅ 用同樣 mechanism for TLS 1.3 |
| qlog event recording | ✅ debug + paper measurement 雙用 |
| Pluggable congestion controller | ✅ 第一版用 CUBIC，後切 BBR / Prague |
| Flow controller two-tier | ✅ 繼承 |
| Connection ID → handler demuxing | ✅ 繼承 + 加 anti-fingerprint mapping |

特別對 Part 12 implementation 章節:
- 把 quic-go fork 或 wrap 是 fastest path
- 加 anti-fingerprint layer (header protection 強化、connection ID rotation policy)
- 加 traffic shaping layer (Part 10/11)
- 加 inner protocol stack (SOCKS5-over-DATAGRAM)

## 自我檢查

1. **quic-go 用 single goroutine 跑 connection state machine + sub-goroutines for I/O**。如果改成 multi-goroutine（每個 stream 一個），會遇到什麼問題？
2. **`Sealer.Seal(dst, src, pn, associatedData)` 第三個參數是 packet number**。Why is packet number used as AEAD nonce input? 對應 RFC 9001 §5.3 怎麼說？
3. **`framer.AppendFrames()` 順序**: ACK 先 vs STREAM 先 vs CRYPTO 先 — 各自影響什麼？quic-go 怎麼決定？
4. **quic-go 用 Go `crypto/tls`** 的 `tls.QUICConn` API (Go 1.21+)。在這個 API 之前 quic-go 怎麼處理（提示：vendored crypto/tls fork）？fork 的維護成本是什麼？
5. **CUBIC implementation 的 `wMax` 變數** 在 fast retransmit 之後怎麼 update? 對應 RFC 8312 公式哪一段?
6. **HTTP/3 RoundTripper 怎麼 reuse connection**? 如果 multiple `RoundTrip` 同時呼叫，會 race 嗎?

## 研究級補遺

### 1. 學界詞彙

| 口語 | 學界用詞 |
|---|---|
| 「實作 vs spec」 | **Reference implementation / compliance test** |
| 「狀態機」 | **Finite state machine / protocol state machine** |
| 「擁塞演算法可插拔」 | **Pluggable congestion control** |
| 「批量打包」 | **Packet coalescing / GSO (Generic Segmentation Offload)** |

### 2. 對手分類學

implementation-level adversary（差別於 protocol-level）:

| 等級 | 能力 |
|---|---|
| I1 | source code review |
| I2 | static analysis (CodeQL, gosec) |
| I3 | dynamic fuzzing (Go-fuzz, syzkaller) |
| I4 | timing side channel (Spectre-style) |
| I5 | implementation-specific bug exploit |

quic-go 有 clusterfuzzlite 持續 fuzzing — 看 `.clusterfuzzlite/` 目錄。

### 3. 形式化定義

implementation correctness:
> 一個 implementation $I$ 正確 implement protocol $P$ iff 對 every spec-compliant trace $T$, $I$ 都能產生 $T$ 且不會產生非 spec 行為。

實際上：證明完整 correctness 對 QUIC 太重，採 **partial verification** (例如 Bhargavan 等用 F* 證 record layer 部分)。

### 4. 領域的關鍵 implementation paper

- Bhargavan et al. *Implementing and Proving the TLS 1.3 Record Layer*. S&P 2017 — verified TLS (precis already done)
- Microsoft msquic team 的 implementation papers
- Cloudflare quiche papers

### 5. 我們協議的座標

到此堂 implementation 路線確定:
- ✅ 基底用 quic-go fork 或 wrap
- ✅ Anti-fingerprint extension layer 在 wire 包裝層
- ✅ qlog 整合作 debug / measurement
- ❓ 是否 rewrite (Rust-based quinn fork) — Part 12 評估

### 6. 必追資源

- quic-go GitHub issues + PRs (active community)
- IETF QUIC interop test runner
- QUIC implementation conformance test suite

### 7. 開放問題

- **Verified QUIC implementation**: 目前無，是 PhD-level research direction
- **Performance gap** quic-go vs C-based (msquic, lsquic) 在 high-throughput scenario
- **Memory safety in non-Rust QUIC**: Go GC pause 對 latency-sensitive 場景
