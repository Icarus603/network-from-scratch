## 動機

TUIC（**T**he **U**ltimate **I**nternet **C**onnector）由 @EAimTY 在 2022 啟動。設計哲學跟 Hysteria 不同：

| 維度 | Hysteria 2 | TUIC v5 |
|---|---|---|
| 賣點 | **速度**（Brutal CC + masquerade）| **協議美學 + 0-RTT + Full Cone NAT** |
| 偽裝 | HTTP/3 masquerade | 純 TLS over QUIC, no app-layer masquerade |
| CC | Brutal（可選 BBR）| user 用 quic-go 預設（BBR / CUBIC） |
| 認證 | HTTP/3 header | TLS Keying Material Exporter (RFC 9000 §6.4) |
| Spec | 演化中（v1, v2） | 嚴格 versioned（v4 → v5 重大改寫） |
| 官方實作 | 自家 Go 程式 | **無官方實作**，only protocol spec |

TUIC 的特色設計：**用 TLS 1.3 Keying Material Exporter 把 user password 跟 TLS session 綁定**——password 不直接出 wire，wire 上的是「password 經 TLS exporter 衍生的 32-byte token」。Replay attack 無效（每個 TLS session token 不同）；MITM 無效（token 由真實 TLS session 衍生）。**這是 TUIC 最漂亮的設計**。

讀完應該回答：

- TUIC v4 出了什麼問題，v5 為什麼幾乎全寫一次？
- TLS Keying Material Exporter 怎麼用？為什麼比 PSK / token 更安全？
- Packet 命令的兩種 UDP relay 模式（quic / native）trade-off 在哪？
- TUIC 為什麼不做 HTTP/3 masquerade？這對抗 GFW 是不是劣勢？

## 核心概念

### 1. v4 → v5 重設計的動機

v4（2022 早期）的問題：

| v4 問題 | 結果 |
|---|---|
| Auth 命令送整個 password | password 在 wire 上明文（被 TLS 加密但可被解密 server log）|
| UDP relay 只有一種 mode | 不能 trade-off lossless vs low-latency |
| ASSOC_ID 4-byte | 過大 |
| 命令型別與 frame 型別混淆 | spec 文件難讀 |
| Heartbeat 用 bidi stream | 浪費 stream ID |

v5（2022 後期）回應：

| v5 改動 | 解決什麼 |
|---|---|
| Auth → TLS Keying Material Exporter | wire 上不含 password；token = `tls_exporter(label="EXPORTER-TUIC", context=UUID, len=32)` |
| Packet 命令支援兩種 mode（quic / native）| user 選 lossless（unidirectional stream）或 low-latency（datagram） |
| ASSOC_ID 縮成 2-byte | 64K UDP session 對個人足夠 |
| 命令統一 `VER(1) | TYPE(1) | OPT(...)` | spec 一目了然 |
| Heartbeat 走 datagram，empty payload | 不占 stream |

### 2. v5 wire 格式（精確 byte-level）

所有命令前綴：

```
+---+---+--------+
|VER|TYP| OPT... |
+---+---+--------+
  1   1   variable
```

- `VER = 0x05`（TUIC v5）
- `TYPE` ∈ {`0x00` Authenticate, `0x01` Connect, `0x02` Packet, `0x03` Dissociate, `0x04` Heartbeat}

#### 2.1 Authenticate (0x00)

```
+---+---+----------+--------+
|VER|TYP|   UUID   | TOKEN  |
|0x5|0x0| 16 bytes | 32 byt |
+---+---+----------+--------+
```

- `UUID`: 標識使用者（多 user 多 UUID）
- `TOKEN`: TLS Keying Material Exporter 衍生，下節詳述

走哪條 stream：**第一條 unidirectional stream** from client to server。server 認證失敗就 close connection；成功不回任何東西，client 直接開後續 stream。

#### 2.2 Connect (0x01)

```
+---+---+---------+
|VER|TYP|  ADDR   |
+---+---+---------+
```

- `ADDR` 格式（共用於多命令）:

```
+------+-----------+------+
| TYPE |   ADDR    | PORT |
+------+-----------+------+
   1     variable     2
```

- `TYPE` ∈ {`0x00` FQDN, `0x01` IPv4, `0x02` IPv6, `0xff` None}
- `0x00 FQDN`: 第 1 byte = length, 之後 length byte 是 host string
- `0x01 IPv4`: 4 byte
- `0x02 IPv6`: 16 byte
- `0xff None`: 0 byte (用於 UDP relay 的 server-to-client 方向，因為 server 不需要告訴 client target)

走哪條 stream：**新的 bidirectional stream**。Connect 送出後 server 立刻 dial target，**之後雙向資料流就走這條 stream**（沒有 ACK frame）。

#### 2.3 Packet (0x02) — UDP relay

```
+---+---+--------+--------+------------+---------+------+------+
|VER|TYP|ASSOC_ID| PKT_ID |FRAG_TOTAL  | FRAG_ID | SIZE | ADDR |
|0x5|0x2|2 bytes | 2 byte |  1 byte    | 1 byte  |2 byt | var  |
+---+---+--------+--------+------------+---------+------+------+
                                                          + PAYLOAD (variable)
```

- `ASSOC_ID`: UDP session ID（client-assigned，類似 SOCKS5 的 UDP ASSOC）
- `PKT_ID`: 單 session 內的 packet 編號
- `FRAG_TOTAL` / `FRAG_ID`: 分片支援
- `SIZE`: payload 長度
- `ADDR`: target 地址（client→server）或 source 地址（server→client）

**兩種傳輸 mode**：

| Mode | 走哪 | 特性 |
|---|---|---|
| `quic` | **Unidirectional stream**（new stream per packet 或 packet group）| Reliable, ordered within stream, 但走 stream → HOLB |
| `native` | **QUIC datagram** (RFC 9221) | Unreliable, no HOLB, low latency, 1200 byte 上限 |

Client 在 config 選一邊，server 對該 session **mirror 同一 mode**。

#### 2.4 Dissociate (0x03)

```
+---+---+--------+
|VER|TYP|ASSOC_ID|
+---+---+--------+
```

走 unidirectional stream。client 主動斷某 ASSOC_ID 的 UDP session（釋放 server 端 NAT mapping）。

#### 2.5 Heartbeat (0x04)

```
+---+---+
|VER|TYP|
+---+---+
```

走 QUIC datagram。client 在 idle 時定時送，防 QUIC connection idle timeout（預設 30s）。

### 3. TLS Keying Material Exporter — 為什麼這設計優雅

**問題**：傳統 proxy 的 auth 方式：

- Shadowsocks-2022 → PSK derive key
- Trojan → SHA-224(password) in plain
- VLESS → UUID in plain

通病：**wire 上 carry 的東西可被 server-side log replay** 或 server 被攻破時洩漏 user credential。

TUIC v5 改用 **TLS 1.3 Exporter (RFC 8446 §7.5)**：

```
TOKEN = HKDF-Expand-Label(
    secret  = TLS Master Secret (Exporter Master Secret),
    label   = "EXPORTER-TUIC",
    context = UUID (16 byte),
    length  = 32
)
```

性質：

| 性質 | 解釋 |
|---|---|
| **session-bound** | TOKEN 跟特定 TLS handshake 的 master secret 綁；session 結束 token 無效 |
| **non-replayable** | 換個 TLS session 就重新衍生 |
| **server-can-compute** | server 知道自己的 master secret，能算出同樣 TOKEN 比對 |
| **MITM-resistant** | 中間人即使竊聽，沒 master secret 算不出 |
| **password 不出 wire** | client 用 password 跑某個 KDF 算 server-saved 預計算結果——TUIC v5 把這步留給實作 |

TUIC v5 spec 沒指定 password → TOKEN 的具體 KDF（這留給實作做），但 wire 上**只有 32-byte exporter output**，這是核心。

**實作細節**（mihomo source `transport/tuic/v5/handshake.go`）：

```go
// Client side
exporter, err := tlsConn.ConnectionState().ExportKeyingMaterial(
    "EXPORTER-TUIC",
    []byte(uuid),  // context = UUID
    32,
)
authCmd := []byte{0x05, 0x00}
authCmd = append(authCmd, uuid[:]...)      // 16-byte UUID
authCmd = append(authCmd, exporter...)     // 32-byte TOKEN
stream.Write(authCmd)
```

Go 的 `crypto/tls.ConnectionState.ExportKeyingMaterial()` 是 RFC 8446 §7.5 的直接實作。

### 4. UDP relay：quic mode vs native mode

這是 TUIC v5 最重要的選擇題。

| 維度 | `quic` mode | `native` mode |
|---|---|---|
| 傳輸 | New unidirectional stream per packet (或 packet group) | QUIC datagram |
| 可靠性 | Reliable (QUIC 重傳) | Unreliable (跟 UDP 一樣) |
| 順序 | Per-stream ordered | No ordering |
| HOLB | Yes（同 stream） | No |
| Latency | 高（要 stream 建立 + 重傳）| 低 |
| Throughput | 中（reliable 浪費頻寬重傳）| 高 |
| Frag 支援 | 可任意大（無 1200 byte 限制）| 限 1200 byte payload |

**何時用 quic mode**：
- 需要可靠 UDP（DNS resolution、TFTP，但這些 protocol 內層有自己 retransmit，邏輯上不該疊）
- 需要大 packet（>1200 byte 的 DNS-over-UDP TXT record）
- 對 latency 不 sensitive

**何時用 native mode**（**主流**）：
- 跑遊戲（CS:GO、Valorant、王者榮耀）
- VoIP、視訊
- WebRTC
- 任何 「丟一個沒事，等下一個」的 traffic

**實務**：sing-box / mihomo 都預設 native。**內層流量本來就 UDP，再加 reliability 是雙重浪費 + 雙重延遲**。

### 5. 0-RTT 設計

TUIC 賣點之一是「0-RTT」。**真相**：QUIC 本身就支援 0-RTT（[Part 4.5](../part-4-tls-quic/4.5-zero-rtt-and-replay.md)）。TUIC 沒做特殊事，只是**允許 client 在 QUIC 0-RTT data 上立刻送 Authenticate + Connect**——若 TLS session ticket 還在，整個 first request 0-RTT。

對比 Trojan / VLESS-TCP：要 TCP handshake 1 RTT + TLS handshake 1 RTT（0-RTT TLS 1.3 可省 1 個）→ 至少 1-RTT 才能送 first request。

QUIC 在這層**結構性優勢**。

但 0-RTT 有 replay 風險（[4.5](../part-4-tls-quic/4.5-zero-rtt-and-replay.md)）：若 client 0-RTT 送 `Connect example.com:443`，attacker 截到 0-RTT data 可 replay 多次讓 server dial 多次。TUIC 對 replay 沒特殊防護，靠 QUIC 本身 anti-replay（基於 PSK binder + ticket age）。

### 6. NAT traversal：Full Cone via persistent session

TUIC 賣點之一是 **Full Cone NAT** 支援 for UDP。意思：proxy 的 source port 對外 stable，允許任何外部 server 主動回送 packet 給 client（peer-to-peer 遊戲、WebRTC ICE 必要）。

**怎麼做**：

- TUIC server 為每個 ASSOC_ID 保留**固定的 outbound UDP socket**（即固定的 source port）。
- 同 ASSOC_ID 的所有 outbound UDP 都從同一 source port 出。
- 對 game server 來說，所有 packet 看起來來自同一 (IP, Port) tuple。
- 任何 game server 對該 tuple 回 packet，server 都 forward 回 client（不檢查 source）。

對比 Shadowsocks 預設 **Symmetric NAT**（每個 (target_ip, target_port) 用不同 source port）：

| NAT 類型 | 用法 |
|---|---|
| Full Cone (TUIC) | 適合 P2P 遊戲、WebRTC、Valorant 等需要 inbound packet 的場景 |
| Symmetric (SS 預設) | 安全性稍高（不接受任意 source 回 packet），但 P2P 不工作 |

實務：**遊戲 user 偏好 TUIC**，正是因為 Full Cone。

### 7. 為什麼 TUIC 沒做 HTTP/3 masquerade

設計哲學差異：

- Hysteria 2: **「假設 attacker probing，所以我要看起來像 web server」**
- TUIC: **「假設 attacker passive，所以我只要 TLS over QUIC 看起來像普通 QUIC」**

實務後果：

- **probing**：對 TUIC server 送 HTTP/3 GET → server 回 connection close 或 reset（因為它不是 HTTP server） → **probing 立刻識別**
- **passive only**：TUIC traffic = QUIC + TLS to a domain → 從外形看跟 Cloudflare 的 QUIC 沒區別

**這在 2024-04-07 之後是個問題**：GFW 已能解 Initial 看 SNI，可以針對「沒有對應網站」的 SNI 做行為觀察（後續 probing）。

TUIC 對 anti-censorship 比 VLESS+REALITY 弱、比 Hysteria 2 也弱（沒 masquerade backend）。它的賣點是**速度 + 設計優雅 + Full Cone NAT**，不是 anti-censorship。

## 與我們協議設計的關聯

| TUIC 設計 | 給我們的啟示 |
|---|---|
| TLS Keying Material Exporter as token | **強烈建議我們協議採用**：password 不出 wire，session-bound, replay-resistant |
| `VER|TYPE|OPT` 統一命令格式 | 採同樣風格；易解析、易擴展 |
| UDP relay 兩種 mode 任選 | **預設 native (datagram)**，提供 `reliable_udp` flag for special cases |
| Full Cone NAT 設計 | **必須支援**（遊戲 user 是 power user） |
| 沒 masquerade → 弱抗探測 | **教訓**：我們協議必須有 masquerade |
| 0-RTT auth in QUIC | 可採；但加 anti-replay nonce |
| Heartbeat via datagram | 採；不要浪費 stream |

## 動手（可選）

### 實驗 1：實作 TLS Keying Material Exporter 計算

Python with `cryptography` lib：

```python
import ssl
from cryptography.hazmat.primitives.kdf.hkdf import HKDFExpand
from cryptography.hazmat.primitives import hashes

# 假設你有 TLS exporter_master_secret（從 wireshark + SSLKEYLOGFILE 抓）
exporter_master_secret = bytes.fromhex("a1b2c3...")

# TLS 1.3 HKDF-Expand-Label per RFC 8446 §7.5
def tls13_exporter(secret, label, context, length):
    full_label = b"tls13 " + label.encode()
    hkdf_label = (
        length.to_bytes(2, 'big') +
        len(full_label).to_bytes(1, 'big') + full_label +
        len(context).to_bytes(1, 'big') + context
    )
    return HKDFExpand(
        algorithm=hashes.SHA256(),
        length=length,
        info=hkdf_label,
    ).derive(secret)

uuid_bytes = bytes.fromhex("00000000-0000-0000-0000-000000000001".replace("-",""))
tuic_token = tls13_exporter(exporter_master_secret, "EXPORTER-TUIC", uuid_bytes, 32)
print(tuic_token.hex())
```

這個 token 應該跟 client / server 各自算出來的完全一致。

## 研究級補遺

### 1. 學界詞彙

| 我們口語 | 學界 |
|---|---|
| TLS Keying Material Exporter | RFC 5705 (TLS 1.2) / RFC 8446 §7.5 (TLS 1.3) exporters |
| Full Cone NAT | "Cone NAT" (RFC 3489 obsoleted, see RFC 5128 for current taxonomy) |
| Session-bound credential | Channel binding (RFC 5056) |
| QUIC datagram | RFC 9221 unreliable datagram extension |
| 0-RTT replay | Anti-replay (RFC 8446 §8) |

### 2. 對手分類學 / 威脅模型精化

對 TUIC 的對手：

- **被動 fingerprint by SNI**: GFW 解 Initial → 看 SNI → 若 SNI 是「自家域名沒對應網站」→ 加 watchlist
- **主動 probing**: 對 TUIC server 用 HTTP/3 客戶端 probe → server 不返回 valid HTTP/3 response → **fingerprint**
- **traffic shape**: UDP/443 流量但**不是 cloudflare/google/etc IP**，且 connection 長存 → 加 suspicion
- **TLS exporter 攻擊**: 在 quantum-secure 世代下 TLS exporter 也會 break；但目前 ECDHE + AES 安全

對 TUIC 的「無 masquerade」是設計缺陷：任何主動 probe 立刻識別。這是它跟 Hysteria 2 比 anti-censorship 弱的根因。

### 3. 領域的關鍵論文 / 規格 / 原始碼

| Source | 為什麼 |
|---|---|
| RFC 8446 §7.5 | TLS exporter 的 normative source |
| RFC 5056 | Channel binding 理論基礎 |
| RFC 5128 | NAT taxonomy（Full Cone, Symmetric） |
| EAimTY/tuic SPEC.md | v5 真實 spec |
| mihomo / sing-box source | 看實作 |
| Bock et al. NDSS 2020 "Detecting Probe-Resistant Proxies" | TUIC 沒採 probe-resistance，這篇是針對 SS-OBFS 等，但類比可借 |

### 5. 我們協議的座標 / 設計取捨

```
Proteus — TUIC 啟示收窄:
- Auth:
  * 必須 TLS-exporter-based 或等價 channel-binding
  * password 不出 wire
- Command format:
  * VER|TYPE|OPT 風格採用
- UDP relay:
  * Native datagram 預設, quic stream opt-in for reliable UDP
- NAT:
  * Full Cone 必須支援 (game user 是 power user)
- Masquerade:
  * 必須有 (TUIC 沒做 → 教訓)
- 0-RTT:
  * 採, 但加 anti-replay nonce (TUIC 沒處理 → 補強)
```

Part 11.5 spec 寫作會回頭引用 `VER|TYPE|OPT` 格式。

### 7. 開放問題

- **OP-1**: TLS exporter 在 QUIC 0-RTT 下的安全性。RFC 8446 §7.5 對 0-RTT exporter 有限制 (early exporter)，但 RFC 9000 怎麼 expose 早期 exporter？實作上常有 bug。
- **OP-2**: Full Cone NAT 在多 user 共享 server 時的隔離問題。是否能讓 user A 的 ASSOC_ID 拿到 user B 的 UDP 回包？實作 bug 風險高，spec 沒明文要求隔離。
- **OP-3**: TUIC 沒做 obfs / masquerade，能否在不破壞 spec 的前提下 layered 上去？社群有 prototype（TUIC + obfs proxy 在前），但無正式 spec。
