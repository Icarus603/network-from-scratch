## 核心概念

### 1. 為什麼有 quic-go fork

upstream `quic-go/quic-go` 是嚴格遵守 IETF QUIC 規格的 reference implementation。它的目標是**正確 + 高效**，不是**anti-censorship**。社群需要的 anti-censorship feature 包括：

- 強制 jumbo Initial（split across 2+ UDP）
- SNI slicing across CRYPTO frames
- 自訂 Initial salt（規避 GFW 的 hardcoded `0x38762cf7…`）
- 自訂 GREASE values / 自訂 transport parameter ID 分布
- 自訂 packet number 編碼策略
- Initial connection ID 長度故意改非 standard
- 自訂 padding 分布在 Initial / Handshake / 1-RTT packet

把這些塞回 upstream 違反「正確實作」原則 → 必須 fork。

### 2. apernet/quic-go：Hysteria 的承載

`apernet/quic-go` 由 Hysteria 主作者 tobyxdd 維護。Xray-core 透過 `github.com/apernet/quic-go` import 它做為 H/3 transport 後端（Xray-core 2024-09 移除自家 legacy QUIC transport 後）。

**主要改動**（從 commit history 與 patches 整理）：

| 改動 | 動機 |
|---|---|
| `SendInitial Datagrams` 可選 | Initial packet 強制跨 2+ UDP datagram |
| `CryptoFrameMaxSize` 可控 | SNI slicing 用 |
| `InitialPacketNumber` 可設 | 規避 GFW 假設 PN=0 起始 |
| `TLSConfig.HelloID` for uTLS | Initial 內 TLS ClientHello fingerprint 模仿 Chrome |
| `Transport.DontSendVersionNegotiation` | server-side 拒絕回 version negotiation packet |
| `Connection.HandshakeIdleTimeout` 細調 | 防 GFW 因 handshake 慢而誤判 |

從 license 角度，apernet/quic-go 是 MIT (跟 upstream 同)，PR 部分會 upstream 回 quic-go，部分留在 fork。

### 3. SNI slicing 實作

SNI slicing 在 quic-go 內的位置：`internal/handshake/crypto_stream.go` 跟 `internal/wire/crypto_frame.go`。

**標準行為**（upstream quic-go 2024 之前）：

```go
// CryptoStream collects TLS bytes, packetizer 把它編 CRYPTO frame
// 預設 frame 盡量大 (~1100 byte payload, 接近 MTU)
maxCryptoFrameSize = computeMaxFrameSize(maxPacketSize)
```

→ TLS ClientHello 通常一個 CRYPTO frame 裝完，剛好一個 Initial packet。

**SNI slicing 改動**（apernet/quic-go 2024-Q4 patch、後上 upstream）：

```go
// 新增 config: 強制 CRYPTO frame 不能 cover ClientHello bytes around SNI extension
type SNISlicingConfig struct {
    Enabled       bool
    SplitBefore   int   // byte offset before which goes in frame 1
    SplitAfter    int   // byte offset starting at which goes in frame 2
    PacketBoundary bool // packet 1 / packet 2 必須在不同 UDP packet
}

// crypto_stream.go 改:
func (s *cryptoStream) PopCryptoFrame(maxSize protocol.ByteCount) *wire.CryptoFrame {
    if s.snicfg.Enabled && needSplitHere(s.offset, s.snicfg) {
        // 強制 emit 此 frame 不跨 SNI extension
        return frameUpToOffset(s.snicfg.SplitBefore)
    }
    return defaultPopFrame(maxSize)
}
```

實際上實作沒寫死「SNI extension」位置，而是讓 caller（client / server）指定 byte offset。

**packet 邊界**：CryptoStream 給 packet builder 多個 frame，packet builder 通常 pack 進同 packet。改動讓 packet builder 在指定 frame 後**強制 packet flush**，下一個 frame 進新 packet：

```go
func (p *packetPacker) packCryptoPacket(crypto *cryptoStream) ... {
    frame := crypto.PopCryptoFrame(...)
    packet.Add(frame)
    if frame.ForceFlush {
        return packet  // 不再裝其他 frame
    }
    // ...
}
```

**Wire 結果**：

```
UDP datagram 1:
  Long Header (Initial)
  CRYPTO frame [offset 0..N, ClientHello bytes 0..N (不含 SNI)]
  PADDING (to MTU)

UDP datagram 2:
  Long Header (Initial, same Packet Number+1 or same with multiple frames)
  CRYPTO frame [offset N..end, ClientHello bytes N..end (含 SNI)]
  PADDING
```

GFW 對 datagram 1 解 → 沒 SNI → pass。Datagram 2 仍然 long header 但 GFW state machine 假設「同 5-tuple 第 N 個 packet 不再檢查」（隨機 sampling）→ pass。或 GFW state 看 packet 1 已 pass → ignore。

### 4. GREASE values：故意送怪 value

**設計目的**：防 middlebox / censor 假設某欄位永遠是某 value。RFC 9287 + RFC 8701（TLS GREASE）的延伸。

**QUIC 內三個 grease point**：

#### 4.1 Version GREASE

RFC 9287 §3.1：client 可送 version field = `?a?a?a?a` 形式（每個 nibble 末位是 `0xA`），server 必須回 version negotiation packet。

Chrome 預設啟用：每隔 N 個 connection 用 grease version → server 必須 negotiate。GFW 若 hardcoded「version=0x00000001 才檢查」就漏抓 grease packet。

**部署現狀**：
- Chrome: 開
- Firefox: 開
- Cloudflare: 支援
- apernet/quic-go: 可選

#### 4.2 Frame Type GREASE

RFC 9000 §22.4 預留某些 frame type id (`0x?a?a?a?a` 形式) 作 grease。Receiver 必須 silently ignore。Sender 故意送 → 強迫 middlebox 別假設 frame type 是固定 enum。

#### 4.3 Transport Parameter GREASE

QUIC transport parameters (RFC 9000 §18) 用 varint ID。RFC 9287 §3.2 預留 grease IDs `0x?a?a?a?a` 給 sender。

apernet/quic-go 在 client side 預設加 1-3 個 random grease transport parameter，每個塞 16-256 random byte → 雙重效果：（1）防 middlebox 拒陌生 ID；（2）讓 Initial packet 更大 → 拉高 GFW 解 cost。

### 5. 自訂 Initial Salt — 規避 hardcoded GFW

RFC 9001 §5.2 規定 QUICv1 用 `salt = 0x38762cf7f55934b34d179ae6a4c80cadccbb7f0a`。GFW 解 Initial 用這個 salt。

若我們 client/server 私下約定**換個 salt**呢？

- Client 用 custom salt 算 Initial key → AES-GCM 加密 ClientHello
- Server 用同 custom salt → 解 → OK
- GFW 用 standard salt → 解出 garbage（plaintext 跟 random 一樣）→ 找不到 TLS ClientHello → 找不到 SNI → **pass**

**問題**：這破壞 RFC 8999 QUIC Invariants（version 0x00000001 必須用標準 salt）。所以做法：

| 做法 | trade-off |
|---|---|
| A. 用 QUICv2 (RFC 9369) | salt = `0x0dede1058a7d066c1e0a8d8e9a8e2e3a31aef6f5` ← 不同；GFW 暫時不支援 |
| B. 自宣 version = grease value `0xfa1afa1a` | server 知道但 GFW 不知道 |
| C. 完全不宣 version，用 short header from start | 違反 RFC 9000；client/server 都要私改 |

apernet/quic-go 支援 QUICv2 + grease version，不支援 (C)（太破壞 spec）。

### 6. Initial Packet Number 起始值

RFC 9000 §17.2.5: Initial Packet Number 起始可任意（沒規定 0）。但實際 implementations 多半從 PN=0 開始。GFW 若假設 PN=0 → 我們從 PN=random 開始 bypass。

apernet/quic-go 提供 `Config.InitialPacketNumber` 設成 random。

實用性：GFW 沒明確基於 PN=0 過濾，所以這 mitigation 是 future-proof（不是當前必要）。

### 7. Initial Connection ID 長度

QUIC long header 內 DCID/SCID 長度自訂（0-20 byte）。但實際多用 8-byte（quic-go upstream 預設）。

GFW 可能對「特定 CID 長度」做 fingerprint。apernet/quic-go 提供 `Config.InitialConnectionIDLength` 0-20 任選 → randomization。

但**長度本身**是 fingerprint。CID length distribution 應該 mimic Chrome / Firefox / Cloudflare 的常見值（8、9、10 byte），不是亂選。

### 8. 量化分析：fork 改動對 GFW 識別率影響

Zohaib 等用 inside-out testbed 量了下列改動對 GFW QUIC 過濾 bypass 率：

| 改動 | bypass 率 |
|---|---|
| 標準 Chrome (single-packet Initial) | 0% (全被擋) |
| SNI slicing (split CRYPTO across 2 packet) | ~100% |
| Jumbo Initial (split single ClientHello across 2 datagram) | ~100% |
| Dummy UDP prepend | ~100% |
| QUICv2 (salt 不同) | ~100% |
| Source port 設 < destination port | ~92% (heuristic miss) |
| Grease version | 視 GFW 是否 hardcoded v1，目前 ~50% |

→ 多種策略合用，bypass 率 100%。

### 9. apernet/quic-go vs upstream：merge 流向

部分 patch upstream 回 quic-go：

- SNI slicing config (mid-2025 merged)
- Grease transport parameter (long-time merged)
- HandshakeIdleTimeout 細調 (merged)

部分留在 fork：

- 強制 Initial split across UDP（upstream 認為太 niche）
- 自訂 InitialPacketNumber（upstream 不認為有用）
- HelloID 整合 uTLS（uTLS license 不 compat，分開 maintain）

**意義**：upstream quic-go 在 anti-censorship 上**逐步開放**（接收 PR），但仍以「正確實作」為主軸。anti-censorship 重度 user 用 apernet fork。

## 研究級補遺

### 1. 學界詞彙

| 我們口語 | 學界 |
|---|---|
| QUIC fork 改 anti-censorship | "Protocol stack modification for circumvention" |
| Initial salt 換 | "Version-derived key obfuscation" |
| Grease | "Protocol unossification via random reserved values" (RFC 9287) |
| SNI slicing | "TLS handshake fragmentation" |
| jumbo Initial | "QUIC Initial datagram spanning" |

### 2. 對手分類學 / 威脅模型精化

對 quic-go fork 啟示後的 censorship resistance：

- **Tier 2 GFW (現)**: bypass 容易（SNI slicing / jumbo）
- **Tier 3 GFW (預測 2026-27)**: 重組 UDP fragments → SNI slicing 失效；但 jumbo across UDP 仍有效因 GFW 不一定重組 CRYPTO across packet
- **Tier 4 GFW**: 重組 CRYPTO frame across UDP → 所有 fragmentation bypass 失效；剩下 QUICv2 / GREASE / 自訂 salt
- **Tier 5 GFW**: 支援所有已知 QUIC version / version negotiation → 自訂 salt 失效；剩下 traffic shape / ML

→ 我們協議**不能只靠 fragmentation**，必須有獨立的 wire-image-mimicry。

### 5. 我們協議的座標 / 設計取捨

```
Proteus — quic-go fork 啟示:
- QUIC stack 選擇:
  * A) 用 upstream quic-go ← 維護輕, anti-censorship 弱
  * B) 用 apernet/quic-go ← anti-censorship 強, 跟 upstream sync 成本
  * C) 自寫 minimal QUIC stack ← 完全 control, 巨大投入
  ← Part 11.4 主架構決策時定
- Anti-censorship features (我們必須有):
  * Initial split across UDP (jumbo / SNI slicing)
  * Grease version / frame type / transport parameter
  * 自訂 Initial salt 選項 (QUICv2 or custom version)
  * Initial PN randomization
  * CID length randomization (with realistic distribution)
  * uTLS ClientHello fingerprint mimicry
- 部署策略:
  * 每 4-6 週 review 一次 upstream quic-go / apernet changes
  * 維持 自己 fork 的 patch set 文件化
```
