## 動機

10.2–10.4 攻擊端講完，現在進入防禦端。本堂是 **defense taxonomy**——把 2009–2024 所有提出過的 WF defense 分類整理，比較其 cost / accuracy / formal property。Part 10.5–10.7 是防禦三部曲：

- **10.5**：classical shaping defenses（本堂）
- **10.6**：PT obfs4 / meek / Snowflake / FTE / Conjure
- **10.7**：regularization disruption（讓「沒規律」本身不成為特徵）

Proteus 的防禦 module 是從這些建造塊組合而來。**Defense engineering 不是發明新東西，而是知道每個塊的 cost / property 並挑出 Pareto-optimal combination。**

### 二、Cost 的多維度

任何 defense 都同時付出多種 cost。**「bandwidth overhead」只是其中一個**：

| Cost 維度 | 度量 | 容忍度（典型 web） |
|---|---|---|
| Bandwidth overhead | (defended bytes − raw bytes) / raw bytes | < 100% (網頁), < 30% (streaming) |
| Latency overhead | 延遲分布變化（p95 / p99） | < 100ms p95 |
| Page-load time overhead | 完整 page render time 增加 | < 50% |
| Energy（mobile） | client CPU/radio 耗能 | 顯著但難量化 |
| Implementation complexity | LoC、protocol changes | qualitative |
| Coordinator cost | 是否需 client-server 配合 | 主要 deployment factor |

### 三、Padding-only 的死路：Dyer 2012 Peek-a-Boo

Dyer et al. 2012 IEEE S&P 是 WF defense 領域的「冷水論文」：他們系統測試了當時所有 padding-based defense（pad-to-MTU、pad-to-uniform、padding to nearest power-of-2、etc.）對 Naïve Bayes / SVM 攻擊的有效性。

#### 結論

1. **單純 padding 都無效**——即使 pad-to-MTU 也只把 attack accuracy 從 91% 降到 80%。原因：direction、count、ordering 沒被處理。
2. **唯一有效的 defense 是 BuFLO**（constant-rate channel），但 overhead > 200%。
3. **「low-overhead WF defense」這個目標可能根本不可能**。

這個論文是 Proteus 的 ground truth：**「我能不能找到 low-overhead 但 effective 的 defense」是 publication-worthy 的問題。** 過去 10 年所有 defense 論文都在挑戰 Dyer 2012 的悲觀結論。

### 四、BuFLO 與 Tamaraw：constant-rate 家族

#### BuFLO (Dyer 12)

- Send packets at **fixed rate** $\rho$，每個 packet 固定 size $s$。
- 連線開始後跑 minimum duration $T$，沒資料就送 dummy。
- 設計目標：把 trace 變成「block of constant-rate」，hide 變動。

**問題**：overhead 巨大（200%+ bandwidth），且資料量大時延遲離譜（>5s 增加）。

#### Tamaraw (Cai 14 CCS)

修 BuFLO：兩個 direction 用不同 rate $\rho_+, \rho_-$，且只 pad 到 next quantum boundary $L$。

- Outgoing rate $\rho_+ = $ user 平均 outgoing rate
- Incoming rate $\rho_- \gg \rho_+$（網頁主要是 download）
- 結束時 pad 到 total packets = $L \cdot \lceil \text{actual} / L \rceil$

**結果**：
- WF acc 從 91% → 11%（強防禦）
- bandwidth overhead ~100%
- latency overhead ~125%

Tamaraw 至今是 **最強 shaping defense baseline**，但 overhead 仍重。

#### CS-BuFLO（Cai 14 同一篇延伸）：congestion-sensitive BuFLO

讓 rate 跟 TCP congestion window 適應，避免 BuFLO 的「狂送 dummy 把連線塞爆」。實務上仍重，但比 BuFLO 友善。

### 五、Traffic Morphing（Wright 09 NDSS）

#### 思路

把 site A 的 trace 「morph」 成 site B 的 trace 分布。用 **packet size distribution-shaping**：

- 給定 source dist $p_A$、target dist $p_B$，求 mapping $T$ 使 $T(p_A) \approx p_B$。
- 用 linear programming optimize $T$。

#### 結果

- vs Liberatore–Levine NB attack：accuracy ↓ 70 → 20%
- overhead 比 BuFLO 低很多（~20%）

#### 為什麼後續失效

只 morph packet size **marginal distribution**，不 morph sequence / timing。**Wang 2014 k-NN 用 ordering 一招破。**

**教訓**：Marginal distribution 一樣不代表 joint distribution 一樣。**Walkie-Talkie 改用 supersequence approach 修這個問題。**

### 六、WTF-PAD（Juarez 16 ESORICS）

#### 動機

Tor 想要 zero-latency defense（不能延遲 user 真實流量）。WTF-PAD 設計：

- **Adaptive padding**：根據實時 IAT 統計，決定何時 inject dummy cell。
- **No delay**：真實 packet 立刻送，dummy 在 idle 期插入。

#### 演算法核心

兩個 states：burst state（真實 packet 集中發） vs idle state（兩 burst 之間）。

- 在 burst state：用 burst histogram 統計 sample dummy IAT，定期 inject。
- 在 idle state：用 gap histogram sample。

#### 結果

- vs k-NN：91% → 60%
- vs k-FP：95% → 65%
- bandwidth overhead ~50%
- **latency overhead = 0**（zero-delay 設計目標）

#### 倒台

DF (Sirinam 18) 把 WTF-PAD accuracy 拉回 90%——WTF-PAD 對 deep learning 沒抵抗力。**WTF-PAD 的核心問題：它對「marginal IAT」做 shaping，但 DL 看的是 sequence-level pattern。** Marginal 統計 一樣 ≠ joint 一樣。**和 Wright 09 同病。**

雖然 WTF-PAD 倒了，**Tor padding spec proposal 254 是基於 WTF-PAD ideas**——所以 Tor real production padding 對 modern DL 也已 known-broken。Tor 社群目前往 Maybenot 框架轉型（後述）。

### 七、Walkie-Talkie（Wang–Goldberg 17 USENIX Sec）

#### 突破點

不 morph distribution，而是**直接把兩個 site 的 trace 變成 identical 的 supersequence**。

#### 思路

1. Client 端把瀏覽器設為 half-duplex mode（一次只送一方向 burst）。
2. 預先計算「site A trace」與「site B trace」的 supersequence——即兩者 cells 的 共同 super-pattern。
3. 訪問 A 時，按 supersequence 結構發送：自己的 cells + dummies 補成 supersequence 形狀。
4. 訪問 B 時也用 same supersequence。
5. attacker 看到 supersequence trace 無法區分是 A 還是 B——理論上 indistinguishable。

#### 結果

- vs k-NN / k-FP：accuracy → ~50%（理論最佳 = random guess between 2）
- bandwidth overhead ~30%
- latency overhead 中等（half-duplex 引入）

#### 倒台

Tik-Tok（Rahman 20）用 timing channel 破。**Walkie-Talkie 在 direction sequence 上 indistinguishable，但 timing 沒處理。**

**教訓**：cross-channel indistinguishability 必要——只覆蓋一個 channel 不夠。

### 八、FRONT 與 GLUE（Gong–Wang 20 USENIX Sec）

#### FRONT (Front-Loaded Random Token)

zero-delay defense，但 padding 集中在 trace 開頭：

- Sample 一個 $n$ from Rayleigh distribution
- 在 trace 前 $T$ 秒內 inject $n$ random dummy cells
- 之後不再 padding

理論基礎：WF feature 對 trace **早期** 的 burst pattern 最敏感。打前面打到 generalization 失效。

#### 結果

- vs DF：98% → 65%
- overhead ~25%
- latency overhead = 0

#### GLUE

把 FRONT 用在 「連續 page visit」 之間——把多個 visit 「glue」 成連續 stream，模糊頁面邊界。對 「parsing fallacy」 (Juarez 14) 那條防線 efective。

#### 倒台 / 評價

對 Tik-Tok / Surakav-aware attacker 仍 vulnerable（acc 80%+）。但作為「便宜 + zero-delay」的 baseline 持續被 cite。

### 九、RegulaTor（Holland–Hopper 22 PoPETs）

#### 設計目標

抽象掉 「padding state machine」，用 **flow-pattern regularization**：

- 把 trace 切成 fixed-duration windows。
- 每個 window 強制 send rate ≤ $R$（threshold）+ inject dummies up to $R$。
- 結果：trace 變成 「regular shape」——所有 site 看起來像同一個 envelope。

#### 結果

- vs DF: 95% → 22% accuracy
- vs Tik-Tok: 95% → 31%
- overhead ~110% bandwidth, ~50% latency

#### 評價

RegulaTor 比 Tamaraw overhead 略低、但 defense 強度相當。**它的 spec 簡單到能直接 deploy**——這在 Tor PT 社群很重要。

### 十、TrafficSliver（De La Cadena 20 CCS）

#### 與其他 defense 正交：traffic splitting

不在 single trace 上 shape，而是**把 user traffic 切分到多個 Tor circuits**。

- Client 端 round-robin / random / chunked 分配 outgoing cells 到 $K$ circuits。
- 每個 circuit 走不同 guard、middle、exit relay。
- Attacker 只觀察單一 relay 時，只看到 trace 的 fraction。

#### 結果

- vs DF（assuming attacker observes 1/K circuits）：accuracy 接近 random
- **若 attacker control multiple guards (e.g. ISP-level)**：保護 degrade

#### 評價

對 ISP-level adversary（如 GFW）保護不充分；對 single-vantage adversary 強。**對 Proteus 的意義**：multipath transport 是天然 split defense。Part 8 的 MASQUE / Multipath QUIC 為 Proteus 提供 transport 基礎。

### 十一、Maybenot（Pulls–Witwer 23 PoPETs）

#### 角色

不是新 defense，而是 **defense framework**——Tor padding v2 的標準化方向。

#### 核心抽象

- Defense = state machine
- State 之間 transition 由 events 觸發（packet sent/received, timer expired, ...）
- 每個 state 定義 action（send dummy, delay packet, modify size）
- Spec 可被 client/server 協商 (negotiation)

#### 為什麼重要

之前每個 defense 都是 ad hoc 實作；Maybenot 提供統一 spec/impl。**對 Proteus 的意義**：可以把 Proteus traffic-shaping 寫成 Maybenot machine，享受其 deployment ecosystem。Part 11.5 詳論。

### 十三、Padding 設計的「不變式」（lessons from 14 年攻防）

1. **Marginal distribution 一樣 ≠ 安全**。必須處理 joint structure。
2. **單 channel 覆蓋 ≠ 安全**。direction + size + timing 必須一起 shape。
3. **Zero-delay defense 一定有 ceiling**——能改的只有「沒在送資料」的空白期。Hard cap 大概是 attacker acc 50–60%。
4. **Trace-end padding 比 trace-begin 重要**：trace 結尾的 burst 通常 idiosyncratic，padding 結尾 cost 較高但收益大。
5. **Decoy trace > Distribution match**：讓 attacker 「無法二選一」比讓「分布看起來像」強。
6. **Network-level split 與 application-level shape 是 orthogonal**——可疊加。

## 與我們協議設計的關聯

1. **Proteus = RegulaTor-style envelope + Surakav-style GAN decoy + Multipath split + Maybenot spec**。四層縱深。
2. **每層的 cost budget**：envelope 30% BW、decoy 30% BW、split 0%（latency 可能 +10%）、Maybenot orchestration 0%。
3. **Cross-channel consistency**：在 protocol level 強制 (direction, size, timing, burst) 都 covered。
4. **形式化 leakage bound**：用 Maybenot state machine 做 quantitative info-flow 分析。**目前 SOTA defense 都缺這個，是 Proteus 的論文賣點。**

## 研究級補遺

### 1. 學界詞彙

- **Padding**: 加 dummy cells 不傳 user data
- **Splitting**: 一個 site 訪問用多個 channel
- **Morphing**: 把 trace 變得「像」另一個 class
- **Decoy / Cover traffic**: 主動生成假流量
- **Constant-rate channel**: 不論 user 是否傳，rate 固定
- **Supersequence**: 兩 site trace 的 共同 super pattern
- **Zero-delay defense**: 不延遲 user 真實 packet
- **Maybenot machine**: defense as state-machine spec
- **Bandwidth overhead**: (defended - raw) / raw bytes
- **Latency overhead**: 延遲（多用 p95 / p99）

### 2. 對手分類學

不再列——10.1 的 5-dim 通用。重點補充：

- **「Defense-aware adversary」**：知道 defense 演算法。Kerckhoffs。
- **「Parameter-aware adversary」**：知道當前 defense 參數。難擋。
- **「Strategic adversary」**：能影響 input distribution（如 publish content forcing user to visit specific page）。
- **本堂大多假設 algorithm-aware**；Surakav / Proteus 走 parameter-aware bar。

### 3. 形式化定義

**$\delta$-indistinguishability defense**

> Defense $\Pi$ 是 $\delta$-indist for site pair $(A, B)$ if total variation distance $TVD(\Pi(A), \Pi(B)) \leq \delta$。

Walkie-Talkie 的理想目標：對 pair $(A, B)$ 達 $\delta = 0$。實際 $\delta \approx 0.5$（timing 沒處理）。

**$(\varepsilon, \delta)$-traffic privacy** (Pulls 17 / Iacovazzi 22 frame)

> Defense $\Pi$ is $(\varepsilon, \delta)$-private if for adjacent inputs (one bit diff in user activity), $\Pi$ output dist satisfies DP form。

實務上 $\varepsilon$ 巨大，未成為主流 metric。**Part 10.12 詳論為何 DP-style 對 traffic 不奏效。**

### 4. 領域的關鍵論文

- Padding-defense canon: Dyer12 → Cai14 (Tamaraw) → Juarez16 (WTF-PAD) → Wang-Goldberg17 (W-T) → Gong-Wang20 (FRONT) → Holland-Hopper22 (RegulaTor)。
- Splitting: TrafficSliver (CCS 20)。
- Adversarial (overlap 10.4): Mockingbird, BLANKET, Surakav。
- Framework: Maybenot 23。
- 反面 / 評估方法：Sirinam-DF (CCS 18), Rahman-TikTok (PoPETs 20), Sheffey-24 PoPETs。

### 7. 開放問題

1. **Pareto-optimal defense**：(BW%, Latency%, attack acc) 的 Pareto front 在哪？2024 仍無完整 characterization。
2. **Adaptive defense**：能否讓 defense parameters 根據 attacker 即時 estimate 動態調整？實作很難（client 沒 attacker model）。
3. **Streaming-aware defense**：streaming（YouTube/Netflix）的 WF defense 與一般網頁完全不同（rate 高、duration 長）。**幾乎沒有專門論文**——是 Proteus 的 niche 機會。
4. **End-to-end defense**：目前 defense 多在 Tor / proxy node 做；application 端也參與才能完整覆蓋（如瀏覽器隨機化 request order）。**Cherubin 17 PoPETs 開始 talked about**，但沒推廣。
