### 2. Design Document（DD）大綱

下方為 Proteus DD 的 RFC-style outline. 用於 academic review 與 NDSS-/PETS-/CCS-/USENIX-Security-class submission preparation.

```
Proteus Design Document — Version 0.1
================================

§A. Abstract
  Proteus is a censorship-resistant high-performance transport protocol designed
  to simultaneously achieve (a) statistical indistinguishability from popular
  cover-protocol traffic, (b) active-probing indistinguishability via REALITY-style
  cover-server fallback, and (c) goodput parity with SOTA QUIC-based proxies.
  Proteus is built on MASQUE CONNECT-UDP over HTTP/3 over QUIC over UDP/443, with
  a hybrid post-quantum key exchange (X25519 + ML-KEM-768) and TLS 1.3-borrowed
  handshake. Forward secrecy and post-compromise security (PCS-weak) of the
  KEYUPDATE ratchet are mechanically verified using TLA+, ProVerif, and Tamarin.

§B. Motivation
  - SOTA anti-censorship protocols (VLESS+REALITY, Hysteria2, TUIC v5) each
    optimize subset of (CAR, PERF, formal-verification).
  - No existing protocol simultaneously achieves all three.
  - Proteus aims to be first SOTA in this combined dimension.

§C. Threat Model
  See lesson 11.1 + spec §11.1.
  - Adversary: on-path active + sub-national GPA + long-term aggregator +
    PQ-capable (SNDL).
  - Capabilities C1–C13 enumerated; C1–C7, C9–C12 in-scope.

§D. Design Goals (quantified)
  - SEC-1..4: confidentiality, integrity, mutual auth + KCI, FS + PCS-weak.
  - CAR-1: ε ≤ 0.20 against 1-day DL classifier; ε ≤ 0.30 stretch for long-term.
  - CAR-2: active-probe indistinguishability via REALITY fallback.
  - PERF-1: goodput ≥ 0.95 BDP.
  - PERF-2: 1-RTT handshake.
  - DEP-1..5: deployable on 1 vCPU/1GB VPS.

§E. Design Space and Choices
  See lesson 11.3 (Proteus-α/β/γ candidates) and 11.4 (selected Proteus-γ MASQUE/H3
  primary with β QUIC fallback and α TLS-TCP last-resort).

§F. Protocol Architecture
  See lesson 11.4 §4–7. Client communicates with Proteus server over UDP/443
  with H3+MASQUE; Proteus server forwards to cover on auth failure; inner
  CONNECT-UDP capsule carries Proteus inner packets.

§G. Specification
  Detailed in `assets/spec/proteus-v0.1.md`. 17 sections including wire format,
  handshake, crypto suite, cover pinning, anti-replay, padding, extensibility,
  conformance, IANA considerations.

§H. Security Analysis
  - Informal: design review (lesson 11.12) tracing each capability to defense.
  - Formal:
    - TLA+: state-machine safety, KeyUniqueness, MutualAuth, PaddingBudget.
    - ProVerif: secrecy, mutual authentication, forward secrecy, KCI,
      partial PQ adversary (phase 2 with custom equation).
    - Tamarin: PCS-weak via KDF ratchet; key uniqueness across epochs.

§I. Evaluation Plan
  Deferred to Part 12.10. Sub-evaluations:
  - SEC-1..4: formal verification (done in §H).
  - CAR-1: train 5 reference attackers (naive, DL CNN, transformer, BiLSTM,
    long-term aggregator); measure Adv_CAR; target ε meets §D.
  - CAR-2: black-box active probing benchmark with 5 known probe tools (zmap,
    masscan, Tor relay-style probes, GFW emulation probes); measure
    response-time distribution diff vs cover server.
  - PERF-1..5: standard QUIC benchmark suite at link bandwidths 100Mbps,
    1Gbps, 10Gbps with packet loss 0.1%, 1%, 5%.
  - DEP-1..5: deployment matrix runs on 1vCPU/1GB, 2vCPU/4GB, 4vCPU/8GB.

§J. Related Work
  See lesson 11.3 (design-space papers) and 11.12 (SOTA comparison).
  - VLESS+REALITY (XTLS team, 2023+)
  - Hysteria2 (apernet, 2023+)
  - TUIC v5 (EAimTY, 2022+)
  - Shadowsocks-2022
  - WireGuard (Donenfeld 2017)
  - MASQUE WG drafts (RFC 9297/9298)
  - Hybrid PQ TLS (draft-ietf-tls-hybrid-design)

§K. Open Problems
  - Long-term ε lower bound (academic problem).
  - PCS-strong without 1-RTT cost.
  - Game-theoretic FPR budget formalization.
  - Spec-to-Tamarin/ProVerif auto-generation.
  - Quantitative residual risk.
  See lesson補遺 across 11.1–11.13 for full open-problem list.

§L. Limitations
  - v0.1 lacks ML-DSA support (Ed25519 only).
  - v0.1 lacks multi-path (single QUIC connection only).
  - v0.1 lacks 0-RTT by default (replay risk).
  - v0.1 lacks IANA codepoint allocation.
  - v0.1 not yet IETF-aligned.

§M. Conclusion
  Proteus v0.1 establishes a research-grade transport spec for high-performance
  censorship resistance. Implementation, evaluation, and possible IETF
  alignment are deferred to Part 12.

§N. Acknowledgments
  - Course author Icarus, by direction of Claude (Anthropic).
  - Building on the SOTA work of XTLS, Hysteria2, TUIC, WireGuard, IETF
    QUIC/MASQUE/TLS WGs.
```

### 6. Lessons learned 寫進 design document

從 Part 11 過程學到的（將寫入 DD §M Conclusion 與 Part 12 retrospective）:

1. **威脅模型最先寫不是 nice-to-have**：先寫 capability matrix, 所有後續設計都自然回頭 trace.
2. **量化 goal 比口號重要**：「同時頂尖 X+Y」沒法 act; ε ≤ 0.20 + 0.95 BDP 可 act.
3. **架構 lock 之前先列 candidate matrix**：避免 premature commitment, 11.3 4 個 candidate 計分卡是關鍵設計動作.
4. **Three formal verification tools 互補**：TLA+ + ProVerif + Tamarin 覆蓋互補 bug 類，缺一不可.
5. **Spec writing 是設計動作**：寫 spec 過程暴露 design ambiguity（例 padding budget 是 per-packet 還是 time-averaged？11.12 才釐清）.
6. **Cover protocol pinning 對 CAR 至關重要**：cover 變則 ε_CAR 重訓.
7. **設計 review 是 audit**：不是 nice ceremony, 是 mechanically catch missed capability defense.
8. **PQ migration 必前驗（confidentiality）後驗（signature）**：對應 Mosca's theorem.
9. **TLS-in-TLS 是 architectural concern**：選 transport substrate 就要避免, 後補很難.
10. **PCS-weak 是 acceptable**：PCS-strong 代價（1-RTT per ratchet）對 anti-censorship 不 worth 多花.
