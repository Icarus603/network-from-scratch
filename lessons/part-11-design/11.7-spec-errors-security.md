## 動機

`Security Considerations` 是 spec 的「壓力測試報告」。

它不是事後補的章節，是把每個 capability 一條一條 walk 過去，回答：「**如果 attacker 做 X 會發生什麼**」。RFC 8446 §11 對 TLS 1.3 寫了 12 大類；Cremers-Horvat-Hoyland-Jackson-van der Merwe 2017 對 TLS 1.3 的 Tamarin 分析又找出更多漏洞——這證明 Security Considerations 即使在公開 review 數年的 spec 也仍漏項。

G6 v0.1 Security Considerations 的最低標：**對 11.1 capability matrix 的每一條 in-scope capability，spec 必須對應一個段落，明說：威脅敘述 + 防禦設計引用 + residual risk**。

本堂的成果直接寫進 spec doc，並作為 11.12 design review 的 input。

## 核心概念

### 1. 錯誤分類學

G6 把 error 分四類：

| 類別 | 例子 | 對外行為 |
|---|---|---|
| **Negative auth path** | wrong auth_tag, stale ts, replay nonce | 整個 connection forward 給 cover（CAR-2） |
| **Authenticated peer protocol violation** | reserved≠0, unknown type，KEYUPDATE transcript mismatch | CLOSE with code，連線終結 |
| **Crypto failure on data path** | AEAD tag mismatch on DATA | silent drop（CAR + DoS resistance） |
| **Operational / resource** | rate-limit、idle timeout、disk full | normal CLOSE 或拒收 |

設計原則：**negative path 不應對 attacker 提供 oracle**。Negative auth → 完全 hide（forward）；data-path AEAD fail → 不 escalate；resource limit → normal close。

#### §11.1 Threat model summary（spec 內）

G6 在 11.1 對手能力矩陣的所有 in-scope capabilities (C1–C7, C9–C12) 下提供以下保證（編號對應 11.1 §5）：

- **SEC-1** Confidentiality (IND-CCA2 reduction to hybrid KEM + AEAD)
- **SEC-2** Integrity (INT-CTXT)
- **SEC-3** Mutual authentication (with KCI resistance)
- **SEC-4** Forward secrecy (per-session ephemeral KEM + KEYUPDATE ratchet)
- **CAR-1** Statistical indistinguishability against cover protocol（target ε_CAR ≤ 0.20 against 1-day classifier, ε_CAR ≤ 0.30 stretch against long-term aggregator）
- **CAR-2** Active-probe indistinguishability (REALITY-style fallback)

Out-of-scope（已在 11.1 §6 列出）：C8 endpoint compromise、C13 cache side-channel、GPA unlinkability、application-level anonymity。

#### §11.2 Cryptographic dependencies

實作 G6 必須採用：

- **AEAD**：ChaCha20-Poly1305（RFC 8439）或 AES-256-GCM（RFC 5116）。MUST NOT use any other.
- **Hash**：BLAKE3 for internal KDF; SHA-256 for TLS 1.3 transcript.
- **DH**：X25519（RFC 7748）.
- **KEM**：ML-KEM-768（NIST FIPS-203）.
- **Signature**：Ed25519（RFC 8032）.
- **KDF**：HKDF-SHA-256（RFC 5869）.

實作者 MUST use constant-time implementations of the above. Reference impl：

- `RustCrypto/x25519-dalek` for X25519
- `RustCrypto/aead` for ChaCha20-Poly1305
- `pq-crystals/kyber` or `RustCrypto/ml-kem` for ML-KEM-768
- `BLAKE3-team/BLAKE3` for BLAKE3

#### §11.3 Forward secrecy

G6 達成 forward secrecy：

- 每 session 採用 fresh ephemeral X25519 + ML-KEM-768 share；
- 連線關閉時 secrets 清零；
- KEYUPDATE 引入 forward-only ratchet。

若 server long-term static SK 洩，已結束 session 的 traffic **不可解**。
若 server long-term ticket-decryption key 洩，**0-RTT data** 可解（warning required for application）.

#### §11.4 KCI resistance

G6 達成 KCI resistance：若 client SK 洩，attacker 不能假冒 honest server 對 client 通信，因 server 認證來自 cover server's TLS cert（attacker 無此 cert chain 對應 SK）。詳見 Krawczyk HMQV 2005 §3 對 KCI 的 formal definition.

#### §11.5 Replay resistance

G6 採三層 anti-replay：

1. **Timestamp window**：±30 分鐘外的 ClientHello reject（→ forward to cover）.
2. **Nonce uniqueness**：Bloom filter 1-hour window of all seen `(client_nonce, timestamp)` pairs.
3. **0-RTT replay**：single-use resumption ticket + Bloom-filter check on early data PSK identity + client_nonce.

實作者 MUST:

- 對 timestamp skew > 30 min 立即 forward, 不可放寬;
- 對 Bloom filter false positive 採 conservative（FPR ≤ 10⁻⁹）；
- 對 0-RTT replay 寧失 0-RTT optimization 也不接受 replay。

#### §11.6 Downgrade protection

G6 對 downgrade 的防護：

1. **Cipher / KEM choice 不可協商**：spec 釘死 ChaCha20-Poly1305 + AES-256-GCM 兩擇一，KEM 釘死 hybrid X25519+ML-KEM-768。實作者 MUST NOT 加入「fallback to classical-only KEM」.
2. **Transcript binding**：handshake 後的 ServerFinished MAC over 整個 transcript (CH..SH..EE..CertVerify..SF), 任何中間欄位被改 → MAC fail.
3. **Version negotiation**：G6 v0.1 沒有 version negotiation——SHA 1 hint 在 ext 0xfe0d 第 0 byte 的 hidden constraint 中（v0.2 將以新 extension type 區分）.
4. **No downgrade signaling like TLS 1.3's `SSL3_RANDOM` last 8 bytes**：因為 G6 不允許 fallback 到舊版。

對比：Bhargavan-Brzuska-Fournet et al. 2016 *Downgrade Resilience in Key-Exchange Protocols* 為 KE downgrade resistance 給 formal definition. G6 滿足該 definition 的「strict no-downgrade」strongest variant.

#### §11.7 Identity hiding

- **Client identity hiding from on-path attacker**：client_id 在 ext 0xfe0d 為 AEAD ciphertext，server's KDF key 加密。對 on-path attacker 看似隨機 16 bytes.
- **Server identity hiding from probe**：active probing 觸發 fallback，attacker 看到 cover server 的 TLS cert + response, 不能區分 G6 server vs cover server.

對比 Noise XK 的 initiator-identity-hiding 性質 (Perrin 2018 §7.7)。G6 達 IK-level（initiator hidden in first message after KEM encap，加 explicit AEAD wrapping）。

#### §11.8 0-RTT considerations

0-RTT data 有兩個 fundamental risks：

1. **Replay**：mitigated by §11.5.
2. **Forward secrecy weakness**：0-RTT key derive from PSK, 若 PSK 洩，所有 past 0-RTT data 可解。Application MUST mark stream as `safe_for_replay` and `non_sensitive_for_FS`.

實作者預設 0-RTT OFF。Server config `enable_0rtt = false`（default）.

#### §11.9 Side-channel resistance

實作要求：

- AEAD primitive 必須 constant-time（reject malleable code paths）.
- ML-KEM-768 Decaps 必須 constant-time（Bellare et al. 2024 對 lattice KEM 的時序攻擊已記載；ML-KEM 的 reject-when-cm-mismatch step 必須 secret-independent timing）.
- HMAC verify 必須 constant-time（避免 Lucky-13 type）.
- TLS cert chain 解析時遠離 PKCS#1 v1.5 解構（避免 Bleichenbacher-derived attacks; 用 RSASSA-PSS 或 Ed25519）.

**Out-of-scope**：cache / power side-channel on shared cloud（C13）。建議部署於 dedicated host or bare metal。

#### §11.10 Anti-active-probing details

REALITY-style fallback 規則：

1. Server 對所有 invalid auth 的 connection MUST forward bytes verbatim to a pre-configured cover URL (specified at G6 server config time).
2. Forward MUST preserve TCP/UDP source port mapping so that cover server's response 可被 routed back to client.
3. Forward MUST NOT inject any G6-specific byte before forwarding.
4. Forward latency budget: < 1ms p99 additional RTT.
5. Cover URL MUST be: (a) globally popular (Alexa top 10k 或 Tranco top 100k); (b) HTTPS-only with valid cert; (c) located in a jurisdiction not blocking G6 server location.

對 Frolov NDSS 2020 列舉的 outside-behaviour leak：

| Leak | G6 對策 |
|---|---|
| Response time variance | < 1ms latency budget, server OS kernel buffer tuning |
| TLS extension order | cover server handles handshake natively |
| TCP option fingerprint | server OS 與 cover server OS 同 family (Linux + standard tuning) |
| Connection reuse semantics | cover server 的 keep-alive 邏輯自然 carry over |

> 對部署者：cover URL 不可選 G6 server 自家 domain（會 self-loop）；不可選 GFW 黑名單上的 domain；不可選 only-HTTP cover.

#### §11.11 Anti-fingerprint padding

實作要求：

- 所有 outgoing UDP datagram 必 round up to 1280 bytes via type-0x07 PADDING inner packets.
- Handshake packets MUST follow `cover_browser_profile`（default: chrome-130）的 ClientHello size distribution（含 cipher suite count、ALPN value, extension order）.
- IAT during active data transfer：採 cover-conditioned shaping; total padding bandwidth overhead ≤ 30% of goodput.
- Idle (> 5s no app data): no keepalive padding sent. Application-level PING 才是流量。

實作必須提供：

- Padding overhead instrumentation (count of padding bytes vs payload bytes).
- Configurable cover profile（chrome / firefox / safari, with profile version pinning to a real-world Chrome major version）.

#### §11.12 Cryptographic agility & post-quantum migration

- Spec v0.1 釘死 hybrid X25519+ML-KEM-768. 任何替代由 spec v0.2 推。
- Signature 仍 classical Ed25519. Migration plan: G6 v0.2 將 add ML-DSA-65 as alternative once `tlsfingerprint.io` 部署率 > 5%.
- Symmetric primitives (ChaCha20, AES-256, SHA-256, BLAKE3) 對 Grover 仍提供 ≥ 128-bit PQ security; no migration planned.

#### §11.13 Denial of Service considerations

| Threat | Defense |
|---|---|
| Handshake DDoS via 1186B junk extension | HMAC auth_tag pre-check (cost ~1μs); fail → forward, no Decap |
| QUIC Initial packet flood | RFC 9000 §8.1 retry token mechanism |
| KEYUPDATE storm | rate-limit: at most one KEYUPDATE per direction per 1 second |
| 0-RTT replay flood | Bloom filter check before Decap |
| AEAD-fail-driven CPU exhaustion | silent drop, no escalation |
| Memory exhaustion via half-open | conn limit per source IP; SYN-cookie-equivalent in QUIC retry token |

對 amplification: G6 inner cell padding 1280B; outer QUIC packet padded similarly. Server response 大小 ≤ 1.2× client request (no amp).

#### §11.14 Compromised endpoints

- **Server side**: if server's long-term private key 洩 ⇒ past sessions 仍 FS-protected (SEC-4); future sessions vulnerable to impersonation.
  Recommended response: rotate server static key out-of-band; revoke old key via client-side update channel.
- **Client side**: if client's user_id-related secret 洩 ⇒ attacker can impersonate user; server should support client-side keypair rotation via admin console.
- **Both endpoints OS compromise (C8)**: out-of-scope; protocol can't help.

#### §11.15 Implementation pitfalls

實作者注意：

1. **Constant-time HMAC compare**：`constant_time_eq()` only; never `==`. Lucky-13 範本.
2. **Bloom filter sizing**：1-hour window × ~1k req/s = ~3.6M entries; 1% false positive rate needs ~10MB. False positive 觸發 forward, 不致命 (CAR-2 safe).
3. **CRT-based RSA, if any**：MUST guard against fault injection (Boneh-DeMillo-Lipton 1997). G6 不用 RSA but cover server's cert可能 has RSA; client side 通常用 standard TLS library.
4. **ML-KEM 解封裝順序**：FIPS-203 §7.3 specifies reject-when-mismatch via implicit secret K_bar; do not branch on H(c) == H(c_recompute).
5. **TCP Forward 的 socket buffer mismatch**：forward 兩端 buffer 應同等，避免變 amplification oracle.
6. **Don't echo error in TLS Alert**: G6 server, even in fallback case, should not generate any TLS Alert by itself. Cover server's Alert reaches the client unmodified.

#### §11.16 Known unattenuated risks

明確記載「我們知道但不解決」：

| Risk | Reason for non-defense |
|---|---|
| Long-term aggregation pushing ε_CAR > 0.3 | No known protocol design has provably bounded long-term ε; we set τ_stretch and acknowledge gap |
| Subnational GPA + N-day flow correlation | structural; no transport-layer fix |
| Cover URL availability collapse (cover censored) | operator must pre-configure multiple cover URLs |
| ML-KEM PQ security review still maturing (FIPS-203 finalized 2024 but field is young) | conservative re-evaluation upon NIST or IACR alerts |
| Adversary's ML/transformer classifier evolving faster than padding budget can absorb | ongoing arms race; v0.2+ may need shaping update |

```
§11.x [Topic]

  [Specific attacker capability, e.g., "attacker mounts active probe via Tor exit
   to G6 server's IP:port"]

Defense:
  [Specific protocol mechanism: "Server forwards bytes to cover URL per §11.10"]

Residual:
  [What's NOT prevented: "Adversary can still observe G6 server's IP exists at
   :443/UDP; mitigated by IP rotation at deployment layer."]

  [Primary literature for both the attack and the defense rationale]
```

## 自我檢查

1. 為什麼 AEAD-fail-in-DATA 是 silent drop 而 KEYUPDATE-transcript-mismatch 是 hard close？答：DATA AEAD fail 可能是 in-path garbage 注入 (no compromise); KEYUPDATE mismatch 表示 authenticated peer 在 protocol violation, must abort.
2. 為什麼 PQ signature 還沒上？答：cert chain size + handshake RTT, 加 deployment readiness 不夠.
3. 為什麼 cover URL pinning 必須在 server 配？答：client-side pin 不可信 (client 可被 attacker 配個 cover-self-controlled URL); server 配，cover 部分 trust anchor in server-side config.
4. 為什麼 0-RTT default OFF？答：(a) replay risk; (b) FS weakness if ticket key 洩; (c) handshake fingerprint distinctness.
5. 為什麼 §11.16 列「不解的 risk」？答：透明披露 > 假裝完美。Review 第一句話就會問 long-term aggregation 怎辦.

### 2. 對手分類學 / 威脅模型精化

**Negative-path attacker subclass**：對手不需破解 crypto，只需 manipulate inputs 引發協議 differentiated responses。Bock 2020 對 GFW 的觀察：error-pattern fingerprint 是 GFW 對 Shadowsocks / Outline 識別的關鍵。

**Probe-resistant proxy attacker**：Frolov NDSS 2020 定義專門對「probe-resistant proxy」的 4 種 outside-behaviour leak。G6 §11.10 對每條給出 specific mitigation.

**Downgrade attacker**：Bhargavan 2016 formal definition. G6 因 spec 釘死 cipher 與 KEM, 達到該文 strongest "no downgrade possible" variant.

### 3. 形式化定義

**Negative-path indistinguishability (formal)**:

```
∀ PPT A,
  Adv^{NPI}(A) = | Pr[A → G6_server with valid auth detects "G6"]
                  - Pr[A → G6_server with invalid auth detects "G6"]
                  - Pr[A → cover_server detects "G6"] |
              ≤ negl(λ)
```

由 forward-bytes-to-cover 機制達成（reduction to cover server's responses being identical regardless of who's relaying）.

**Downgrade-resilience (Bhargavan 2016 §3)**:

```
Game DR:
  A interacts with handshake oracle.
  A wins if oracle accepts (alg_1, alg_2) where alg_1 ≠ alg_2 = (cipher, KEM) supported,
    but neither party intended the downgrade.
  Adv^DR ≤ negl(λ).
```

G6 by construction not negotiable → trivially DR-secure.

### 5. 我們協議的座標 / 設計取捨

本堂 lock：

- Negative-path: forward-to-cover, no oracle.
- Error-codes within authenticated session only, never leak to wire pre-auth.
- AEAD-fail-on-DATA: silent drop.
- Crypto-agility: lock cipher + KEM; no downgrade.
- 0-RTT default OFF.
- Cover URL: server config, multiple allowed.

仍 open（11.8 / 11.12 finalize）：
- Version negotiation 細節（11.8）
- 0-RTT 啟用條件 final review（11.12）

### 7. 開放問題

1. **Forward-to-cover 的 information leak bound**：forward 引入的 < 1ms RTT inflation 是否仍洩漏 high-precision timing fingerprint？沒有 formal proof of "≤ ε leak".
2. **Bloom filter 的 capacity vs cover-URL set 的 trade-off**：cover URL set 越多, replay window 內 nonce space 越分散，是否能換更小 Bloom filter？
3. **Negative-path forward 是否總是 wire-level identical**：linux kernel socket buffer 對 forwarded bytes 是否引入 timing skew？需 packet-level 量測.
4. **Cryptographic agility vs 釘死 ciphers**：spec 釘死 makes downgrade impossible 但 future migration painful. trade-off 不 settled.
5. **Selfie-type attacks on G6**：若 client 與 server 都跑 G6 客戶端 + 服務端 simultaneously（如 mesh deployment），G6 是否仍 mutual auth secure？Drucker-Gueron 2021 對 TLS 1.3 PSK selfie 有 attack，需檢視我們是否引入類似缺陷.
