### 3. Spec 結構選擇的 rationale

**Why §11 Security Considerations 占獨立大章？**

仿 RFC 8446、RFC 9000：security 是 spec 的 first-class citizen, 非 footnote. Review 第一閱讀順序通常是 §1 → §11 → §其他.

**Why §6 把 crypto choice 釘死成 «fixed» 不放彈性？**

避免下游實作 / cipher negotiation 引入 fingerprint surface. v0.2 才開 agility，與 cover_browser_profile 一同對齊.

**Why §7 cover protocol pinning 獨立成章？**

是 G6 anti-censorship 核心機制. 與 standard transport spec 顯著差異, 顯式提示 review 者注意.

**Why §17 changelog 在 spec 內？**

對 IETF submission unusual (changelog 通常在 acknowledgments), 但對「research draft for academic review」必要 — review 者要知道 review 後做了什麼.

### 4. Conformance 框架

§13 conformance 是 v0.1 引入的新章。其他 anti-censorship 協議（VLESS / Hysteria2）通常無 conformance section, 結果 impls 不一. G6 要從一開始有 spec-conformance tests:

- §13.1 wire-format conformance: byte-level test vectors (TBD published with reference impl).
- §13.2 handshake state machine conformance: trace-level test vectors.
- §13.3 crypto: algorithm choice fixed.
- §13.4 test vectors: G6 v0.1 conformance suite released alongside Part 12 implementation.

### 5. 與 Formal Models 的綁定

v0.1 spec §11 引文 explicitly link to:

- TLA+ model: `assets/formal/G6Handshake.tla`
- ProVerif model: `assets/formal/G6Handshake.pv`
- Tamarin theory: `assets/formal/G6Ratchet.spthy`

每個 SEC/CAR claim 引相應 model + verified property name. 這在學術 spec 中是 SOTA practice (Cremers TLS 1.3 Tamarin 2017 開創此 style)。

### 6. Spec 限制：留給 v0.2 的議題

v0.1 explicit defer：

- ML-DSA-65 signature support
- Cipher / KEM negotiation
- Multi-path
- Asymmetric ratchet (PCS-strong)
- 0-RTT default enable
- IANA codepoint registration
- IETF draft alignment（與 MASQUE WG drafts 進度同步）

這些留 v0.2 是**有意的 scope control**, 不是漏寫.

### 7. Spec 與 implementation 的橋接

§16 implementation notes 列具體 lib：

- Rust: rustls, quinn, RustCrypto suite
- Go: sing-box plugin
- CC: BBRv3
- OS: Linux SO_REUSEPORT + eBPF for forward path

這些是 Part 12 實作的 source of truth.

## 自我檢查

1. 為何 §11 Security Considerations 占獨立大章？答：spec first-class citizen，與 RFC 8446 / 9000 對齊.
2. 為何 v0.1 不直接 IETF submit？答：(a) IANA codepoint allocation, (b) MASQUE WG draft alignment, (c) Part 12 實作 feedback.
3. 為何 §13 conformance 必要？答：避免下游 impls 不一致；anti-censorship 領域 historically lacks 此 discipline.
4. 為何 spec changelog 在 spec 內？答：academic draft 風格，review 者需 trace 過去決策.
5. 為何留 ML-DSA 給 v0.2？答：cert chain size + handshake RTT + deployment readiness 不夠.

### 2. 形式化定義

**Spec quality (informal, but operational)**: spec is "ready" iff:

- Byte-exact for every wire field
- Complete state machine
- Every error path defined
- Every security claim has formal proof or empirical evidence
- Every dependency on external spec versioned
- Conformance test vectors available

v0.1 missing the last item (deferred to Part 12). Hence v0.1 is "research-grade ready" not "production-grade ready".
