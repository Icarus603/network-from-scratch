## 動機

TLA+ 證的是 **protocol state-machine safety**（KeyUniqueness, NoDeadlock, PaddingBudget）。ProVerif 證的是 **cryptographic security**（secrecy of session keys, mutual authentication, forward secrecy, KCI resistance）。兩者覆蓋互補的 bug 類別。

ProVerif 用 applied pi-calculus（Abadi-Fournet POPL 2001）model 協議，把 attacker 寫成 Dolev-Yao（Dolev-Yao 1983），然後自動 query secrecy / correspondence。

本堂用 ProVerif 對 Proteus handshake 驗 4 條 query。產出 `assets/formal/ProteusHandshake.pv`.

## 核心概念

### 1. 抽象化選擇

ProVerif 是 symbolic model — perfect cryptography assumption. 具體抽象：

| 真實組件 | ProVerif 抽象 |
|---|---|
| X25519 keygen + DH | `x25519_pub(sk)`, `x25519_dh(sk1, pk2)` with commutativity equation |
| ML-KEM-768 | `kem_encap`, `kem_decap` with decap(sk, encap(pk(sk), n)) = ss equation |
| AEAD | `aead_enc / aead_dec` with reduc rule for correct dec |
| HMAC | `hmac(k, m)` opaque function |
| Signature | `sign / sig_verify` with reduction rule |
| HKDF | `hkdf(salt, ikm)` opaque function (key-deriving) |

> Notable simplification: HKDF 在 ProVerif 預設是 free function（無 algebraic properties）。對 Proteus spec 來說足夠——只需要 HKDF 不能 inverted by attacker.

### 2. Queries（properties to verify）

```proverif
(* Q1: Secrecy of session key *)
query attacker(c_ap_var) ==> false.

(* Q2: Authentication client → server *)
query c_id:host, s_id:host, n:nonce, k:key;
  event(client_completes(c_id, s_id, n, k)) ==>
    event(server_accepts(c_id, s_id, n, k)).

(* Q3: Injective agreement *)
query c_id, s_id, n, k;
  inj-event(client_completes(c_id, s_id, n, k)) ==>
    inj-event(server_accepts(c_id, s_id, n, k)).

(* Q4: Forward secrecy via phase model *)
(* See §6 below *)
```

Q1 確保 traffic key 不可被 attacker 推導。
Q2/Q3 確保 client 不會「完成 session 認為對方是 server S」但 S 未啟動該 session（防 server impersonation）。
Q4 確保 server long-term key 後洩不解 past sessions.

### 3. 完整 model 結構（節錄）

```proverif
type host. type nonce. type sk. type pk. type ct. type key.

free c : channel.
free server_sigsk : sk [private].
free server_kemsk : sk [private].
free client_secret_id : bitstring [private].

fun hkdf(bitstring, bitstring) : key.
fun hmac(key, bitstring) : bitstring.
fun x25519_pub(sk) : pk.
fun x25519_dh(sk, pk) : bitstring.
equation forall a:sk, b:sk; x25519_dh(a, x25519_pub(b)) = x25519_dh(b, x25519_pub(a)).
fun kem_pub(sk) : pk.
fun kem_encap(pk, nonce) : bitstring.
fun kem_decap(sk, ct) : bitstring.
fun kem_ct_of(bitstring) : ct.
fun kem_ss_of(bitstring) : bitstring.
equation forall sk_s:sk, n:nonce;
  kem_decap(sk_s, kem_ct_of(kem_encap(kem_pub(sk_s), n)))
    = kem_ss_of(kem_encap(kem_pub(sk_s), n)).
fun aead_enc(key, nonce, bitstring, bitstring) : bitstring.
reduc forall k:key, n:nonce, p:bitstring, ad:bitstring;
  aead_dec(k, n, aead_enc(k, n, p, ad), ad) = p.
fun sign(sk, bitstring) : bitstring.
fun sig_verify(pk, bitstring, bitstring) : bool
  reduc forall s:sk, m:bitstring; sig_verify(x25519_pub(s), m, sign(s, m)) = true.

event client_starts(host, host, nonce).
event server_accepts(host, host, nonce, key).
event client_completes(host, host, nonce, key).
event server_finished(host, host, nonce, key).

let client(server_idh: host, client_idh: host,
           server_kem_pk: pk, server_sig_pk: pk) =
  new client_eph_sk : sk;
  let client_eph_pk = x25519_pub(client_eph_sk) in
  new ts : nonce;
  new c_nonce : nonce;
  let kem_pair = kem_encap(server_kem_pk, c_nonce) in
  let kem_ct = kem_ct_of(kem_pair) in
  let K_pq = kem_ss_of(kem_pair) in
  let auth_key = hkdf((server_kem_pk, c_nonce, client_eph_pk), <<"g6_auth">>) in
  event client_starts(client_idh, server_idh, c_nonce);
  out(c, (c_nonce, client_eph_pk, kem_ct));
  in(c, (server_eph_pk: pk, sig_t: bitstring, sf: bitstring));
  if sig_verify(server_sig_pk, (server_eph_pk, client_eph_pk), sig_t) = true then
    let K_classic = x25519_dh(client_eph_sk, server_eph_pk) in
    let handshake_secret =
      hkdf((auth_key, (K_classic, K_pq)), <<"hs">>) in
    let c_ap = hkdf((handshake_secret, (client_eph_pk, server_eph_pk)),
                    <<"c ap">>) in
    let s_ap = hkdf((handshake_secret, (client_eph_pk, server_eph_pk)),
                    <<"s ap">>) in
    let expected_sf = hmac(s_ap, (client_eph_pk, server_eph_pk)) in
    if expected_sf = sf then
      event client_completes(client_idh, server_idh, c_nonce, c_ap);
      let cf = hmac(c_ap, (client_eph_pk, server_eph_pk, sf)) in
      out(c, cf).

let server(server_idh: host) =
  in(c, (c_nonce: nonce, client_eph_pk: pk, kem_ct: ct));
  let K_pq = kem_decap(server_kemsk, kem_ct) in
  new server_eph_sk : sk;
  let server_eph_pk = x25519_pub(server_eph_sk) in
  let K_classic = x25519_dh(server_eph_sk, client_eph_pk) in
  let server_kem_pk = kem_pub(server_kemsk) in
  let auth_key = hkdf((server_kem_pk, c_nonce, client_eph_pk), <<"g6_auth">>) in
  let handshake_secret = hkdf((auth_key, (K_classic, K_pq)), <<"hs">>) in
  let c_ap = hkdf((handshake_secret, (client_eph_pk, server_eph_pk)),
                  <<"c ap">>) in
  let s_ap = hkdf((handshake_secret, (client_eph_pk, server_eph_pk)),
                  <<"s ap">>) in
  let sf = hmac(s_ap, (client_eph_pk, server_eph_pk)) in
  let sig_t = sign(server_sigsk, (server_eph_pk, client_eph_pk)) in
  event server_accepts(clientH, server_idh, c_nonce, c_ap);
  out(c, (server_eph_pk, sig_t, sf));
  in(c, cf: bitstring);
  let expected_cf = hmac(c_ap, (client_eph_pk, server_eph_pk, sf)) in
  if expected_cf = cf then
    event server_finished(clientH, server_idh, c_nonce, c_ap).

free clientH : host.
free serverH : host.

process
  let kem_pk = kem_pub(server_kemsk) in
  let sig_pk = x25519_pub(server_sigsk) in
  out(c, (kem_pk, sig_pk));
  ( !client(serverH, clientH, kem_pk, sig_pk)
  | !server(serverH) )
```

> 上面是 simplified spec 用於展示。完整 model 含 client_id ciphertext、HMAC pre-check、auth-tag、replay window 等細節，見 `assets/formal/ProteusHandshake.pv`。

### 4. 跑 ProVerif

```bash
# install ProVerif (via opam or download from prosecco.gforge.inria.fr)
brew install opam
opam install proverif

# run
proverif assets/formal/ProteusHandshake.pv 2>&1 | tee proverif.log
```

期望輸出（節錄）：

```
RESULT not attacker(c_ap_var) is true.            <-- Q1 secrecy holds
RESULT event(client_completes(...)) ==>
       event(server_accepts(...)) is true.        <-- Q2 auth holds
RESULT inj-event(client_completes(...)) ==>
       inj-event(server_accepts(...)) is true.    <-- Q3 injective
```

若任一 query 反例，ProVerif 會給出 attack trace（一系列 message exchange showing how attacker derives secret）.

### 5. 對 Proteus specific 屬性的 query

#### Q1 Secrecy:

```proverif
query attacker(c_ap).
```

預期 `not attacker(c_ap)` = true。

#### Q2 Server-to-client authentication:

```proverif
query c_id:host, s_id:host, n:nonce, k:key;
  event(client_completes(c_id, s_id, n, k)) ==>
    event(server_accepts(c_id, s_id, n, k)).
```

If client_completes happens with key k, then server_accepts must have happened first with the same key.

#### Q3 Injective agreement:

```proverif
query c_id:host, s_id:host, n:nonce, k:key;
  inj-event(client_completes(c_id, s_id, n, k)) ==>
    inj-event(server_accepts(c_id, s_id, n, k)).
```

No two client_completes events can correspond to the same server_accepts (防 replay).

#### Q4 Forward secrecy (phase model):

ProVerif 的 phase 概念 (Blanchet 2002): 把 protocol 分 phase 0 (normal run) + phase 1 (post-compromise)。在 phase 1 attacker 拿 `server_kemsk`：

```proverif
process
  phase 0;
  ( ! client(...) | ! server(...) )
  phase 1;
  out(c, server_kemsk).     <-- compromise
```

然後 query：

```proverif
query attacker(c_ap_var) in phase 0.
```

即「即使在 phase 1 把 server long-term key 給 attacker, phase 0 中的 c_ap 仍 secret」。這是 forward secrecy formal model.

對 Proteus: hybrid KEM 之下，attacker 拿 server_kemsk 後仍需 X25519 ephemeral DH 才能解。但 ephemeral DH key 已 zeroized after session, attacker 不能 retroactively re-derive. ProVerif `phase 1` 模型 verify 此屬性.

#### Q5 KCI resistance:

```proverif
(* attacker compromises client static key *)
phase 1;
out(c, client_secret_id);

(* server still authenticates honest client correctly *)
query c_id, s_id, n, k;
  event(server_accepts(c_id, s_id, n, k)) ==>
    \exists honest_session ...
```

Detailed encoding 需更多 setup，但 Proteus 由 server's TLS cert + signature 對 client 做 server-side auth, attacker 持 client SK 不能偽造 server cert → server 仍 authenticates correctly.

### 6. 對 fallback path 的 modeling

Proteus spec 有「auth fail → forward」路徑。ProVerif 中可 model 為 separate process：

```proverif
let fallback_path =
  in(c, msg: bitstring);
  out(c, msg).   (* relay verbatim to cover, modeled as attacker-controlled cover *)
```

對 attacker 來說，fallback 等同 attacker 自己處理（兩種情況 indistinguishable from attacker's perspective），故 fallback path 不引入新 attack（trivially safe in symbolic model）.

> 注：CAR-2 的「fallback 行為與 cover indistinguishable」是 **computational property**, ProVerif 不直接驗。但 symbolic model 可 verify 「fallback never reveals Proteus secret」這條，是 sub-property of CAR-2.

### 7. 對 PQ adversary 的 model

ProVerif 是 symbolic, default 假設「only-Dolev-Yao 對 perfect cryptography」。對 PQ adversary, 我們 abstract 為 attacker 能 invert classical-only KEM:

```proverif
(* For modeling PQ adversary's attack on X25519 alone *)
phase 2;
fun pq_break_x25519(pk) : sk.
equation forall s:sk; pq_break_x25519(x25519_pub(s)) = s.
```

然後 verify: 即使 `pq_break_x25519` available, secrecy of c_ap **仍** holds because K_pq (ML-KEM-768 SS) 還是 secret. 這對應 hybrid KE 的 security bound max(ε_PQ, ε_classical).

### 8. ProVerif 的局限

| 不能 | 替代工具 |
|---|---|
| stateful 屬性（counters, ratchet 隨時間） | Tamarin (11.11) |
| Probabilistic / computational reduction | CryptoVerif |
| Implementation-level | F* / EasyCrypt |
| Network ordering 屬性 | TLA+ |

對 Proteus v0.1: ProVerif 覆蓋 SEC-1/2/3/4 + (partial) CAR-2; Tamarin (11.11) 覆蓋 ratchet PCS, replay-window invariants; TLA+ (11.9) 覆蓋 state-machine safety.

### 9. 抽象與真實的 gap

ProVerif 是 sound for 「無 attack」（"if it says no attack, then no attack in symbolic model"），但不 complete（可能因 abstraction 過度而漏掉真實 attack）。同時 symbolic model 的 perfect-crypto assumption 可能在 real impl 被 break（side-channel, fault injection）.

對 Proteus: ProVerif 結果是 **necessary** but not sufficient. 要 fully secure 還需:

- CryptoVerif 給 computational bound (optional).
- Implementation-level 驗（Part 12 採 RustCrypto + constant-time review）.
- Side-channel testing (Part 12.10).

## 動手

1. 安裝 ProVerif, 跑 `ProteusHandshake.pv`. 確認 4 條 query 都 true.
2. 故意 break model: 把 `K_classic` 從 handshake_secret derivation 中拿掉 (只剩 K_pq). 跑 ProVerif. 看 secrecy 是否仍 hold (應 hold, 因 K_pq 已足夠 PQ adversary except in classical-only PQ_break phase). 然後加 phase 1 PQ break, 重跑——應 fail.
3. 加 query 「server_finished implies client_completes for any honest client」並 verify.
4. 對 fallback path 寫一個 explicit model, query 確保 fallback 不洩 secret.

## 自我檢查

1. 為何 ProVerif 比 TLA+ 更適合驗 secrecy？答：ProVerif 把 Dolev-Yao attacker 自動 built-in, query attacker(.) is a primitive concept.
2. 為何 inj-event 比 event 強？答：對應「injective agreement」, 防 replay-with-same-trace.
3. 為何 forward secrecy 用 phase 1 model？答：phase 1 區別「session active」與「post-compromise」, 是 Blanchet 2002 的 standard技巧.
4. 若 ProVerif 給 secrecy 但實作仍洩 key 怎辦？答：abstract 與 real impl 的 gap, 需 side-channel test / constant-time review (Part 12).
5. 為什麼 K_classic 與 K_pq 都 mix 進 handshake_secret 而非單一？答：hybrid KE — 攻擊者必須同時 break 兩者才得 secret. ProVerif 在 phase 1 PQ break X25519 下仍 verify, 證明 hybrid 構造 sound.

### 2. 對手分類學 / 威脅模型精化

ProVerif 中對手 capability:

- **Active attacker**：可 listen, inject, modify on `c` channel.
- **Compromise (phase 1)**：可 explicitly leak named secrets via `out(c, secret)`.
- **Computational adversary outside scope**：ProVerif 不直接 model probability — leave to CryptoVerif.

對 Proteus, ProVerif attacker 對應 C1+C2+C4 (Dolev-Yao 標準) + C9 (long-term key compromise in phase 1) + partial C10 (PQ break in phase 2 via custom equation).

### 3. 形式化定義

**Correspondence assertion (Woo-Lam 1993)**:

```
∀ traces τ of protocol:
  ∀ events e_a ∈ τ marked "after",
  ∃ event e_b ∈ τ marked "before" with matching parameters.
```

**Injective correspondence (Lowe 1997)**:

```
... and the mapping e_a -> e_b is injective.
```

對 Proteus:
- `client_completes(c, s, n, k) ==> server_accepts(c, s, n, k)` = non-injective agreement.
- Injective version 加上「無兩 client_completes 對應同 server_accepts」, 防 client-side replay.

### 7. 開放問題

1. **PQ adversary 在 ProVerif 內的精確 model**：custom equation `pq_break_x25519` 是 ad-hoc. 是否有 framework 提供 standardized phase 2 PQ adversary capability set?
2. **ProVerif + CryptoVerif 自動 bridge**: Lipp 2019 對 WireGuard 做了, 但 Proteus 的 hybrid KEM 在 CryptoVerif 中尚無 ready library.
3. **Fallback path 的 computational indistinguishability**：symbolic verifies safety, computational indistinguishability needs real-world timing measurement + game-based proof. 跨工具 gap.
4. **Cover protocol modeling in symbolic**: 能否在 ProVerif 內 model cover protocol 並 verify CAR-2 attacker's behavior is indistinguishable from real H3 fetch?
5. **Spec-to-pv code generation**: 能否從 spec markdown 自動生成 .pv? 目前手工.
