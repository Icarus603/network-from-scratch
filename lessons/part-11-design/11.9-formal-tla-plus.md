## 動機

ProVerif / Tamarin（11.10–11.11）擅長 **crypto-level** symbolic security（secrecy, authenticity, KCI），TLA+ 擅長 **state-machine-level** invariants（mutual exclusion, deadlock freedom, monotonicity）。兩者覆蓋的 bug 類型不同。

G6 有多個 state-machine 屬性需要 TLA+ 驗：

1. **MutualAuth** — CONNECTED 之 client 必對應一個 honest server SH_EE.
2. **NoDeadlock** — 任何 reachable state 至少有一條 enabled action.
3. **KeyUniqueness** — 不同 session 不該 collide on handshake secret.
4. **PaddingBudget** — α 上限不被違反.
5. **ReplayProtection** — server 不在同 window 接受同 nonce 兩次.
6. **FallbackTerminality** — fallback 不能回到正常路徑.

這些都不是 crypto property（不依 reduction to hardness assumption），是**寫程式時容易寫錯的 protocol-state property**。TLC 可以暴力 enumerate model 來驗。

本堂的成果在 `assets/formal/G6Handshake.tla` 與 `G6Handshake.cfg`，可直接餵 TLC tool runner.

## 核心概念

### 1. 為什麼 TLA+ 而不是其他工具

| 工具 | 強 | 弱 | 在 G6 用於 |
|---|---|---|---|
| **TLA+ / TLC** | state-machine invariants, model checking, very expressive temporal logic | crypto symbolic 弱（抽象成 oracle） | 11.9 — protocol-level safety |
| **ProVerif** | 自動證 secrecy / authenticity in Dolev-Yao | state too small / model abstract | 11.10 — crypto layer |
| **Tamarin** | stateful crypto, multi-set rewriting, post-compromise | 慢、需學 multiset rule writing | 11.11 — stateful crypto + ratchet |
| **CryptoVerif** | computational proof, advantage bound | 寫 model 累 | （未在本課程） |
| **F* / EasyCrypt** | code-aligned proof | 工程成本極高 | not used |
| **Alloy** | structural finite model | 不適合 temporal | not used |

G6 採三件套：**TLA+（11.9）+ ProVerif（11.10）+ Tamarin（11.11）**，覆蓋互補面。

### 2. TLA+ 建模 G6 handshake：抽象選擇

抽象決定哪些細節留、哪些 mock：

| 細節 | 處理 |
|---|---|
| Crypto primitives (X25519, ML-KEM-768, AEAD) | 抽象為 oracle: `Verify`, `Decaps` 是 boolean choice |
| Bytes / wire format | discrete msg type (`"CH"`, `"SH_EE"`, etc.) |
| Network ordering | atomic actions (1 step = 1 packet processed) |
| Packet loss / retransmit | not modeled (orthogonal to spec) |
| Multiple sessions | finite set `1..MaxSessions` |
| Clock | discrete minutes |

> 這是經典 "model the spec, not the world" — Lamport, *Specifying Systems* §1.3.

### 3. State variables

```tla
VARIABLES
    state,             \* sid -> state name
    sessions_at,       \* party -> set of sid held
    handshake_secret,  \* sid -> secret token (or 0)
    transcript,        \* sid -> sequence of messages
    seen_nonces,       \* server -> set of (nonce, timestamp)
    padding_bytes,     \* sid -> nat
    data_bytes,        \* sid -> nat
    clock              \* global logical minute
```

`state` 對應 11.6 state machine 的 12 個 state name。每個 SessionId 走自己 state.

### 4. Actions（state transitions）

每個 11.6 state diagram 的 edge 對應一個 TLA+ action：

| 11.6 edge | TLA+ action |
|---|---|
| INIT → AUTH_PENDING (recv CH) | `ClientStartHandshake` |
| AUTH_PENDING → VERIFY (HMAC OK) | `ServerVerifyOK` |
| AUTH_PENDING → FALLBACK | `ServerVerifyFail` |
| VERIFY → DECAPS (Decap OK) | `ServerDecapsOK` |
| DECAPS → SECRET_DERIVED → HANDSHAKE_DONE (send SH) | `ServerSendSH`（合併 SECRET_DERIVED, 因 atomic） |
| HANDSHAKE_DONE → CONNECTED (recv CF) | `ClientFinish` |
| CONNECTED → KEYUPDATE_PENDING | `RotateKey` |
| KEYUPDATE_PENDING → CONNECTED | `ConfirmKey` |
| CONNECTED → CLOSED | `CloseSession` |
| (clock advance) | `ClockTick` |

完整定義見 `assets/formal/G6Handshake.tla`.

### 5. Invariants

```tla
KeyUniqueness ==
    \A s1, s2 \in SessionId :
        (state[s1] = "CONNECTED" /\ state[s2] = "CONNECTED" /\ s1 # s2)
        => handshake_secret[s1] # handshake_secret[s2]
```

這條 invariant 保證 `ServerDecapsOK` 在 pick 新 secret 時 explicitly avoid collision（已在 action precondition 寫入）。TLC 會 verify 該 precondition 對所有 reachable state 確實生效。

```tla
PaddingBudget ==
    \A s \in SessionId : state[s] = "CONNECTED" => PaddingOk(s)
```

`PaddingOk(s)` 定義為 `padding_bytes[s] * 10 ≤ Alpha * data_bytes[s] + 1280`. `Alpha = 3` 表 30% 預算。額外 +1280 是 handshake startup 容差。

```tla
MutualAuth ==
    \A s \in SessionId :
        state[s] = "CONNECTED"
        => \E i \in 1..Len(transcript[s]) :
                Head(SubSeq(transcript[s], i, i))[1] = "SH_EE"
```

CONNECTED state 必有 transcript 包含 SH_EE 訊息（即 server 確實送過 SH）。

### 6. Temporal property: Liveness

```tla
Liveness ==
    \A s \in SessionId : (state[s] = "AUTH_PENDING") ~> (state[s] \in {"CLOSED", "FALLBACK", "FALLBACK_FORWARDING"})
```

`~>` 是 leads-to operator: 從 AUTH_PENDING 出發，**eventually** 達到 terminal state。這檢查無 deadlock。

> 注：用 leads-to 不用 enabled-once-must-fire（強 fairness），是因為 TLA+ 預設 weak fairness 對 enabled action. 若沒人 trigger CloseSession，CONNECTED 可永久停留，但這不算 deadlock（spec 允許 long-lived connection）。對「stuck in AUTH_PENDING」這條才用 leads-to.

### 7. 模型 size 與配置

`G6Handshake.cfg`:

```
Clients = {c1}
Servers = {s1}
MaxSessions = 3
Alpha = 3
TimestampWindow = 4
```

State space 估計（TLC 報告）：~10⁴–10⁶ states，秒級檢查。

### 8. 用 TLC 找到的潛在 bug 範例

走過下面三個假想 bug，看 TLC 怎麼 catch:

#### Bug A: 移除 KeyUniqueness 的 precondition

如果在 `ServerDecapsOK` 移除 collision check：

```tla
ServerDecapsOK(sid, secret) ==
    /\ state[sid] = "VERIFY"
    /\ secret \in Secrets
    \* /\ \A s2 \in SessionId : state[s2] # "CONNECTED" \/ handshake_secret[s2] # secret  ← 移除
    /\ ...
```

TLC 會 explore 到「兩個 session 都選同 secret」這個 state，violation of `KeyUniqueness`。Counter-example：

```
1. ClientStartHandshake(c1, s1, sid=1, nonce=1, ts=1)
2. ServerVerifyOK(s1, sid=1)
3. ServerDecapsOK(sid=1, secret=5)
4. ServerSendSH(sid=1, s1)
5. ClientFinish(sid=1)
6. ClientStartHandshake(c1, s1, sid=2, nonce=2, ts=1)
7. ServerVerifyOK(s1, sid=2)
8. ServerDecapsOK(sid=2, secret=5)  ← collision
9. ServerSendSH(sid=2, s1)
10. ClientFinish(sid=2)
   KeyUniqueness violated
```

#### Bug B: padding budget 不 enforce

如果 `SendData` 不 limit:

```tla
SendData(sid, n_data, n_pad) ==
    /\ state[sid] = "CONNECTED"
    \* /\ n_pad ≤ MaxAlpha[sid]  ← 設想正確版本
    /\ ...
```

TLC enumerate `n_pad = 2, n_data = 0`，violation of PaddingBudget。

#### Bug C: replay-after-window

如果 `ServerVerifyOK` 漏 clock - ts ≤ TimestampWindow:

TLC 找到一條 trace：clock = 5, ts = 0, TimestampWindow = 4 → 應拒絕但 accepted → ReplayProtection might still hold (因 nonce 唯一)，但若再加一條 client 用同 nonce 重 send → violation.

### 9. Liveness check 限制

TLC 對 liveness 是 explicit weak-fairness check。對 G6 model，沒有強 fairness assumption（每個 action 不一定無限發生），但 `ClockTick` 的存在保證 stay-in-AUTH_PENDING 不會永久（clock 到 max → 不能 ServerVerify → 必 timeout → 由 Server 觸發 fallback 或 close）.

要 verify liveness，cfg 加 `CHECK_DEADLOCK TRUE`（TLC 預設）.

### 10. 對 ProVerif/Tamarin 的 interface

TLA+ 證的是 state-level safety, **不證 crypto secrecy**. 例如：

- TLA+ 認為 `handshake_secret` 是個 token，不問它能不能被 attacker 推出.
- TLA+ 認為 `Verify` 是 boolean choice，不問 attacker 能不能 forge.

那 secrecy / authenticity 怎辦？交給 ProVerif (11.10):

```
ProVerif goal:
  query attacker(server_long_term_sk) ==> false
  query injective_event_correspondence(client_accept(s) => server_running(s))
```

兩者**互不重複**, 一起構成 G6 v0.1 的 formal verification certificate.

### 11. 模型擴展（v0.2 預備）

當 G6 v0.2 加新 cipher、新 ext, TLA+ model 改：

- 新增 state（如 `CIPHER_NEGOTIATING`）.
- 新增 invariant（如 `NoDowngradeFromV02ToV01`）.
- 重跑 TLC.

抓 v0.2 → v0.1 downgrade 是 TLA+ 強項——只需多寫一條：

```
NoDowngrade ==
    \A s \in SessionId : state[s] = "CONNECTED" /\ v0_2_advertised[s] => version_used[s] = "v0.2"
```

## 自我檢查

1. 為什麼把 crypto 抽象成 oracle 而不在 TLA+ 內模 X25519 算術？答：state space 爆炸 + 不是 TLA+ 強項. 留給 ProVerif.
2. 為什麼 KeyUniqueness 寫 precondition 而非僅 invariant？答：spec 必須**強制**新 secret 不 collide; precondition 表示「spec 義務」, invariant 是 "expected truth".
3. 為什麼 Liveness 用 leads-to 不用 G(eventually)？答：leads-to 是 "if A then eventually B", G(eventually) 是 "always eventually", semantics 不同.
4. 為什麼模型 `SendData(sid, n_data, n_pad)` 用具體 1..2, 0..2 而非 Nat？答：TLC enumerate finite, 限 small bound to keep state space manageable; spec layer 仍 Nat.
5. 為什麼 fallback 路徑也被 model？答：spec 的一個重要 invariant 是 fallback 不再回正常 (FallbackTerminal)，TLA+ verify 此 invariant.

### 2. 對手分類學 / 威脅模型精化

TLA+ 對 adversary 的處理：

- **G6Handshake.tla 預設**：actions 由 environment / scheduler 任意 trigger, 等同 adversary 可控所有 timing.
- **未來擴展**：可加 `AdvForge` action 模擬 attacker 注 invalid msg, verify spec 對該 msg 的 handling 正確（fallback or drop）.

### 3. 形式化定義

**Temporal Logic of Actions (TLA, Lamport 1994)**:

```
A behavior is an infinite sequence of states.
A specification is a temporal formula on behaviors.
A safety property: Init /\ [][Next]_vars
A liveness property: <>F, []<>F, F ~> G
```

`[][Next]_vars` means "always (next state via Next, or vars unchanged)" — square brackets are "always" in temporal logic.

**Model-checking decidability**: TLA+ 在 finite state model 是 decidable (TLC complete on finite); on infinite state 是 semi-decidable.

### 7. 開放問題

1. **TLA+ + ProVerif 自動 cross-check**: 兩工具的 attacker model 不同, 是否有 framework 自動證明 「TLA+ result + ProVerif result → end-to-end security」？
2. **PCS 是否能 TLA+ 表達**: TLA+ 的 secrecy 是 trivial（沒有 attacker）, 能否藉 explicit attacker model 表達 PCS？目前不便.
3. **State-space 爆炸對複雜協議**: G6 加 0-RTT + multiple cover URL + multi-version 後 state space 是否需要 abstraction？需研究 partial-order reduction.
4. **Spec evolution 與 model evolution 同步**: 從 spec 文字自動生成 TLA+ model 是否可能？目前手動.
5. **Quantitative TLA+**: 對 padding budget 這種 quantitative invariant, TLC enumerate 是 OK; 但對更複雜的 quantitative property (如 expected latency under fairness), 是否需要 probabilistic model checker（PRISM, Storm）？
