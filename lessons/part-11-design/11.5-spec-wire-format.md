## 動機

協議的 wire format 是它的「物理身體」。

Spec 寫得不夠精確，會出現兩種惡果：

1. **互通失敗**：不同實作對同一 byte 解讀不一致（IPv6 部署過程的反面教材）。
2. **形式驗證崩潰**：ProVerif/Tamarin 的 input 是 abstract message structure，spec 含糊則 model 含糊，verification 失去意義。

本堂用 **RFC 風格** byte-by-byte 定義 G6 v0.1 的全部 wire types。每個欄位都有：

- 位元位置
- 編碼方式（varint / fixed-width / length-prefixed）
- 取值範圍
- 解析失敗時的處理
- 對應的 SEC/CAR/PERF 目標 ID

最終匯成 `assets/spec/g6-v0.0-wire.md`（spec draft，11.13 升為 v0.1）。

> **RFC 2119/8174 強制詞彙**：MUST / MUST NOT / SHOULD / SHOULD NOT / MAY / OPTIONAL，全部按 BCP 14 用法。中文版註解可寫「必須 / 必不 / 應 / 不應 / 可」。

## 核心概念

### 1. G6 packet hierarchy

```
UDP datagram (to UDP/443)
  └─ QUIC long/short header packet (RFC 9000)
       └─ QUIC CRYPTO / STREAM / DATAGRAM frames
            └─ TLS 1.3 handshake (in CRYPTO frames)
            │    └─ G6 extension 0xfe0d (handshake auth)
            └─ HTTP/3 frames (in STREAM frames, RFC 9114)
                 └─ CONNECT-UDP capsule
                      └─ G6 inner packet (wire types below)
```

G6 自家 wire types 都在 **innermost** 層——即 MASQUE CONNECT-UDP 之上的 application data。外層三層完全沿用 RFC 9000/9001/9114/9297/9298 spec，G6 不改它們。

> 設計哲學：盡量少自己定義 wire format。外層用 IETF 規格，G6 自家只占最 inner 一層。Spec 工作量最小化，CAR 屬性也借 RFC 規格之 popularity 加固。

### 2. G6 ClientHello extension 0xfe0d (handshake auth)

QUIC Initial 包內含 TLS 1.3 ClientHello。Hello 內 extensions 是 G6 認證資料的載體。

#### 2.1 Extension type assignment

```
extension_type = 0xfe0d  (private use range, per RFC 8446 §4.2; reserved by us, no IANA registration)
```

> 註：IETF private-use range `0xff00–0xffff` 是 official RFC 5705 allocation；`0xfe00–0xfeff` 是 "reserved for private use" by convention. G6 取 `0xfe0d` 是 deliberate（"fe0d" = "feed", easy to grep）。Final IANA-track 申請推到 v1.0 (Part 12 之後)。

#### 2.2 Extension data layout

```
struct G6AuthExtension {
    opaque   client_nonce[16];        // CSPRNG, fresh per Hello
    opaque   client_x25519_pub[32];   // ephemeral ECDH share (separate from TLS 1.3 key_share)
    opaque   client_mlkem768_ct[1088];// ML-KEM-768 ciphertext encapsulating to server PK
    opaque   client_id[16];           // pseudonym (encrypted-under-server-PK identity)
    uint16   timestamp_unix_minutes;  // current minute mod 65536 (anti-replay)
    opaque   auth_tag[32];            // HMAC-SHA256 over the above, key derived from server PK
} G6AuthExtension;
```

Total size: 16 + 32 + 1088 + 16 + 2 + 32 = **1186 bytes**。

Encoded as TLS extension value (length-prefixed, 2-byte length per RFC 8446):
```
| 0xfe 0x0d | uint16 ext_length = 1186 | <1186 bytes payload> |
```

Total extension on the wire: **1190 bytes**。

#### 2.3 Field-by-field semantics

| Field | Bytes | Encoding | Purpose / target |
|---|---|---|---|
| client_nonce | 16 | random | replay protection, mixed into KDF (SEC-3) |
| client_x25519_pub | 32 | RFC 7748 little-endian Montgomery | classical ECDH share (SEC-1 classical part) |
| client_mlkem768_ct | 1088 | FIPS-203 ML-KEM Encaps output | PQ share (SEC-1 PQ part, C10) |
| client_id | 16 | AEAD(server_PK_kdf, key=server_id_key, n=zero, ad=nonce, "G6_CID") | pseudonym; resists ID linkage (SEC-3) |
| timestamp_unix_minutes | 2 | big-endian | reject if abs delta > 30 minutes (SEC-2 anti-replay) |
| auth_tag | 32 | HMAC-SHA256 | server quick reject without ML-KEM decap (CAR-2 perf) |

#### 2.4 auth_tag 計算

```
auth_key = HKDF-Extract(salt=server_pq_fingerprint, IKM=client_x25519_pub || client_nonce)
auth_tag = HMAC-SHA256(auth_key,
                      client_nonce || client_x25519_pub || client_mlkem768_ct ||
                      client_id || timestamp)
```

Server 只需要 `auth_key`（從 client_x25519_pub + nonce 推得），不需要做 ML-KEM Decaps 就能驗。失敗 → 直接 fallback 給 cover，不浪費 PQ Decaps（~50us）。

#### 2.5 Encoding rules

- **Endianness**：所有 multi-byte 數值欄位用 **big-endian (network byte order)**，符合 RFC 8446 + RFC 9000。
- **Booleans**：避免用單獨 bool 欄位；改用 flag bit 在 reserved 高位。
- **Strings**：no strings on wire. ASCII names 僅出現在 ALPN（H3 alpn = "h3"）這類 IETF-defined 處。
- **Padding inside extension**：MUST NOT pad inside auth extension. Padding for fingerprint normalization 由 cover handshake natural shape 提供。

### 3. G6 inner packet (capsule payload) types

當 H3 / MASQUE CONNECT-UDP tunnel 建立後，inner payload 用 G6 自家 wire types。每個 inner packet wrapped in Capsule (RFC 9297):

```
Capsule {
    Capsule-Type     = 0x00 (HTTP_DATAGRAM, RFC 9297)
    Capsule-Length   = varint
    Context-ID       = varint (per RFC 9298, identifies CONNECT-UDP context)
    UDP-Proxying-Payload = <G6 inner packet>
}
```

#### 3.1 Inner packet header

```
struct G6InnerHeader {
    uint8   type;            // packet type, see table below
    uint8   flags;            // bit-field
    uint16  reserved;         // MUST be zero; reject if not
    uint64  seqnum;           // monotonic, per-direction
}
```

8 bytes fixed header。Reserved 欄位將來啟用為 version-bump field 時用（11.8 細說）。

#### 3.2 Inner packet types

| Type | Name | Direction | Payload |
|---|---|---|---|
| 0x01 | DATA | both | application stream data |
| 0x02 | ACK | both | seqnum range ack（內部 G6 ACK，獨立於 QUIC ACK） |
| 0x03 | NEW_STREAM | client→server | open new logical stream |
| 0x04 | RESET_STREAM | both | abort stream |
| 0x05 | PING | both | keep-alive probe |
| 0x06 | KEYUPDATE | both | rotate AEAD key (SEC-4 forward-secrecy ratchet) |
| 0x07 | PADDING | both | pure padding, MUST be ignored |
| 0x08 | CLOSE | both | graceful close with error code |
| 0xf0–0xff | private/extension | both | application extensions |

> 為什麼有「inner ACK」？MASQUE Capsule 上 inner 是 unreliable datagram-style；我們在 inner 跑一個 lightweight reliability layer，避免外層 QUIC stream 的 HoL。Hysteria2 / TUIC 思路類似。

#### 3.3 DATA packet payload

```
struct G6DataPayload {
    uint16   stream_id;
    uint16   offset_high;     // upper 16 bits of 32-bit offset
    uint16   offset_low;
    uint16   data_length;
    opaque   data[data_length];
    opaque   aead_tag[16];    // Poly1305 tag over (header || payload), AAD = inner header
}
```

Stream-level multiplexing 在 inner layer 完成。Stream_id 0 reserved for control.

#### 3.4 ACK packet payload

```
struct G6AckPayload {
    uint16   range_count;
    G6AckRange ranges[range_count];
}

struct G6AckRange {
    uint64   smallest;
    uint16   length;        // (largest - smallest + 1)
}
```

#### 3.5 KEYUPDATE packet payload

```
struct G6KeyUpdate {
    uint8    next_epoch;     // current_epoch + 1
    opaque   handshake_transcript_hash[32];   // confirm party agreement on ratchet point
}
```

對應 SEC-4 forward-secrecy ratchet（11.6 詳述 key schedule）。

### 4. Cell padding（CAR-1）

每個 outgoing UDP datagram MUST round up to **1280 bytes** by appending PADDING-type inner packets (type=0x07). 1280B 是 IPv4 MTU - QUIC long-header overhead 的 safe-bound。

```
struct G6CellPadding {
    G6InnerHeader (type=0x07);
    uint16  padding_length;
    opaque  zeros[padding_length];   // MUST be zero (CAR doesn't require random, AEAD obfuscates)
}
```

例外：

- handshake packets (Initial/0-RTT) — 取 cover protocol natural shape。
- KEYUPDATE / CLOSE — 立即發送，可 pad 或不 pad。

### 5. Varint encoding

G6 採用 QUIC varint encoding (RFC 9000 §16):

```
| 2-bit length tag (MSB) | 6/14/30/62 bits value |

tag 00 → 1-byte total,  6-bit value range [0,63]
tag 01 → 2-byte total, 14-bit value range [0, 16383]
tag 10 → 4-byte total, 30-bit value range [0, 2^30-1]
tag 11 → 8-byte total, 62-bit value range [0, 2^62-1]
```

所有 varint 欄位（如 Capsule-Length, Context-ID, 內部 seqnum）採此編碼。

### 6. ALPN

G6 server 在 TLS 1.3 ALPN extension 中宣稱 `"h3"`（RFC 9114）。**MUST NOT** 宣稱 G6 自家 ALPN——CAR-1 要求對外 indistinguishable from H3。

### 7. 失敗處理（解析 error）

| 錯誤 | 處理 | 對應 SEC |
|---|---|---|
| `extension 0xfe0d` 缺失 | server MUST forward to cover; do not respond as G6 | CAR-2 |
| `extension 0xfe0d` length 不對 | server MUST forward to cover | CAR-2 |
| `auth_tag` 校驗失敗 | server MUST forward to cover | CAR-2 |
| `timestamp` skew > 30 minutes | server MUST forward to cover (looks like stale replay) | CAR-2 + SEC-2 |
| inner packet `reserved != 0` | client/server MUST close connection with code 0x10 | SEC-2 |
| `aead_tag` mismatch | silently drop packet; MUST NOT emit error | SEC-2 + CAR-1 (no error timing) |
| `seqnum` replay (already received) | silently drop | SEC-2 |
| invalid `type` (unknown but in 0x09–0xef range) | silently drop | SEC-2 (forward-compat) |
| extension type (0xf0–0xff) unknown | silently ignore frame | DEP (forward-compat) |

**設計原則**：所有 negative path 走「silent drop or fallback」，不發 explicit error。Error-message-driven 是 oracle，會洩漏 internal state（Bock 2020 §5 對 Outline 的 error-pattern attack 證明）。

### 8. Wire types 對 capability matrix 的 trace

| Capability | Wire-level 反制 |
|---|---|
| C1 inline read | AEAD over inner DATA; outer QUIC long-header 已加密 (RFC 9001) |
| C2 modify/drop | inner ACK + retransmit |
| C4 active probing | 0xfe0d extension validation → forward 給 cover |
| C5 flow fingerprint | 1280B cell padding |
| C9 long-term key compromise | per-session ephemeral KEM + KEYUPDATE ratchet |
| C10 PQ adversary | ML-KEM-768 in extension |
| C11 TLS-in-TLS | inner DATA wrapped in Capsule + cell padding obscures segment boundary |

### 9. 完整 byte 範例：G6 ClientHello 第一個 packet 的 hex dump（節錄）

模擬 client 連 G6 server，借 SNI = "www.microsoft.com"：

```
QUIC Long Header (Initial):
  c0                      # flags: long header, Initial type
  00 00 00 01             # QUIC version 1
  08                      # DCID length
  XX XX XX XX XX XX XX XX # DCID (8 bytes, random by client)
  00                      # SCID length = 0
  XX XX                   # Token length (varint, 0 here)
  XX XX                   # Packet number length

TLS 1.3 ClientHello (inside CRYPTO frame):
  03 03                   # legacy_version
  XX..XX                  # random (32 bytes)
  20                      # session_id length
  XX..XX                  # session_id (32 random bytes)
  00 04                   # cipher_suites length
  13 01 13 03             # TLS_AES_128_GCM_SHA256, TLS_CHACHA20_POLY1305_SHA256
  01 00                   # compression methods (null)
  YY YY                   # extensions length

Extensions:
  00 00                   # server_name extension
  XX XX                   # ext length
  ... "www.microsoft.com"
  00 0a                   # supported_groups (X25519 + ML-KEM-768 hybrid)
  XX XX
  00 1d 11 ec             # X25519 + private code for x25519+mlkem
  00 33                   # key_share
  XX XX
  ... share data ...
  00 2b                   # supported_versions = TLS 1.3
  ...
  00 10                   # ALPN
  XX XX
  02 68 33                # "h3"
  ...
  fe 0d                   # G6 auth extension!
  04 a6                   # length = 1190 - 4 = 1186 bytes
  ... (G6AuthExtension 1186 bytes as defined above) ...
  ...
  00 15                   # padding extension (to round ClientHello shape)
  XX XX
```

> 注：把 `0xfe0d` 放在 ALPN 之後、padding 之前的位置，是因為 popular browser ClientHello（Chrome、Firefox）的 extension order 通常在 padding 之前還有 private-use extensions（如 `0x4469` GREASE）。G6 客戶端需 follow chrome-latest-stable 的 extension order（CAR-1）。具體 order 由 client config option `cover_browser_profile` 決定，預設 = `chrome-130`（自動跟版本，避免 ossification）。

### 10. Server response

Server 在 TLS 1.3 ServerHello 內**不**回應 0xfe0d 擴展（spec MUST NOT echo）。回應通道：

- 若 auth OK：server 開 H3 stream, 用 CONNECT-UDP capsule 對 client 開放 inner channel。
- 若 auth fail：server forwards 整個 Initial packet 給 cover server，由 cover 自然回應。

> 這意味著「authenticated」與「failed」兩個 server-side state 在 wire 上的 distinguishability 完全靠 H3 應用層之後 inner 流量的存在 vs 不存在判斷。Attacker 在 fail case 看到的是 cover server 的真實 H3 response。

### 11. Inner packet padding / ACK frequency 細節

- ACK frequency：每 `min(64 packets, 50ms)` 觸發 inner ACK。
- 不採用 immediate-ACK（高 freq → 易識別）。
- ACK 內 range 數 ≤ 16，超過則發兩個 ACK。
- ACK seqnum 自己也是普通 seqnum（無 separate ack-only space），keepalive-friendly。

### 12. Stream / connection IDs

inner stream_id：

- 0：control（用於 KEYUPDATE / CLOSE / PING）
- 1..N：data streams
- 偶數 client-initiated, 奇數 server-initiated（沿 QUIC convention）

### 13. CLOSE packet codes

| Code | Meaning |
|---|---|
| 0x00 | NO_ERROR |
| 0x01 | PROTOCOL_VIOLATION |
| 0x02 | INTERNAL_ERROR |
| 0x10 | RESERVED_FIELD_VIOLATION |
| 0x20 | AUTH_REPLAY |
| 0x21 | AUTH_EXPIRED |
| 0x30 | FLOW_CONTROL |
| 0xff | UNKNOWN |

> 從 client 視角，server-side AUTH_* error codes 永不見——因為 fail 即 forward 給 cover。CLOSE 只在 authenticated session 中發。

## 自我檢查

1. 為什麼 inner packet `reserved` 強制 0 而不是 ignore？答：給 v0.2 升版時可重用，現在 reject 是 strict-then-loose 原則（RFC 7322 鼓勵）。
2. 為什麼 inner ACK 而非依賴外層 QUIC ACK？答：QUIC stream HoL 對 padding 友善但對 inner reliability 浪費。
3. 為什麼 `auth_tag` 採 HMAC-SHA256 而非 ML-KEM Decaps 直接驗？答：性能——HMAC ~1μs vs ML-KEM Decaps ~50μs，server-side anti-DDoS。
4. 為什麼 CLOSE 有 error codes 但對外 wire 不分？答：認證後 channel 加密，error code 只在 authenticated 雙方知。pre-auth fail 走 forward，無 close packet 對 attacker 可見。
5. 為什麼 cell 大小是 1280B？答：IPv4 minimum MTU = 576B 但實際路徑常 ≥ 1280B；1280B 是 IPv6 minimum MTU 強制保證；扣 QUIC long-header overhead 約 40B + UDP 8B + IP 40B = 88B → MSS ≈ 1280 - 88 = 1192B inner payload。

### 2. 對手分類學 / 威脅模型精化

對 wire format leak 的 attacker：

- **C1' Wire-level fingerprint**：extension length、order、specific byte patterns 都是 fingerprint surface。G6 對策是 `cover_browser_profile` 鎖。
- **C4' Decoy oracle**：attacker 對 `extension 0xfe0d` 缺失但其他都對的 ClientHello 試探，看 server 行為。G6 forward to cover 切斷此 oracle。

### 3. 形式化定義

**Wire format precision**：

```
A wire format spec is precise if and only if:
  ∀ byte sequence b ∈ {0,1}*:
    Decode(b) ∈ {valid_packet | reject}
  and the decision is deterministic and depends only on b.
```

G6 spec 對每個欄位給出 explicit range + reject rule，滿足此定義。

**Anti-replay invariant**：

```
∀ session S, ∀ packet P sent in S:
  P.seqnum_in_S 不重複; receiver maintains a sliding window W of size 1024;
  packet with seqnum < min(W) MUST be silently dropped.
```

### 7. 開放問題

1. **Extension order entropy**：fixed extension order 是 fingerprint。能否在 G6 spec 內定義 deterministic-but-pseudo-random extension order 基於 session secret，使得跨 session 看起來「隨機」但同 session 內 deterministic？
2. **Varint vs fixed-width 對 fingerprint 的影響**：varint length-prefix 對 small payload 是 1-byte，對 large 是 4-byte。是否引入 size-leak fingerprint？
3. **Reserved field 的長期 cost**：reserve 16-bit 給 v0.2，是否值得？對比直接 version bump 的 trade-off？
4. **Cell padding 對 wire-level entropy 的影響**：1280B fixed 對 LSTM-based timing analysis 是否仍洩漏 burstiness pattern？
