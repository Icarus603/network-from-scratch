## 動機

ProVerif 無法直接表達 **stateful** protocol properties — 例如：

- 「ratchet 一次後，舊 key 永遠不應再產生」
- 「每個 epoch 對應唯一 key」
- 「post-compromise security (PCS)」：在 key reveal 後，下一個 ratchet 之後的 key 仍 secret

這些屬性需要 **stateful** 表達 — protocol 的 state 隨時間演化，property 對「state of the world at time t」做斷言。Tamarin 用 multiset rewriting + first-order temporal logic 處理。

本堂用 Tamarin 對 Proteus 的 KEYUPDATE ratchet 驗 PCS 與 key uniqueness。產出 `assets/formal/ProteusRatchet.spthy`.

## 核心概念

### 1. Tamarin vs ProVerif

| | ProVerif | Tamarin |
|---|---|---|
| 基礎 | applied pi-calculus | multiset rewriting (MSR) |
| State | 隱含於 process | 顯式於 facts |
| Adversary | 自動 Dolev-Yao | 自動 + customizable |
| 屬性語法 | event correspondence / phase | first-order linear temporal logic |
| 自動化 | 完全自動（往往） | 半自動（需 helper lemma） |
| 強項 | secrecy / auth | stateful protocols, ratchets, PCS, DH-related advanced properties |
| 弱項 | stateful | 速度慢，需 user guidance |

對 Proteus: **ProVerif 處理 handshake (11.10)**, **Tamarin 處理 ratchet + PCS (本堂)**.

### 2. Proteus ratchet 模型抽象

```
Setup_Server   : create long-term server key
HandshakeDone  : new session, K_0 derived (abstracted from handshake)
Ratchet_Step   : K_{n+1} = hkdf(K_n, label_ratchet)
Use_Key_To_Send: send some msg encrypted under K_n
Reveal_Key     : attacker compromises K_n at specific epoch
Reveal_LongTerm: attacker compromises server long-term key
```

每條 rule 標記 events，properties 對 events 寫 lemma.

### 3. Tamarin rules（節錄）

```
rule Setup_Server:
    [ Fr(~ltk) ]
  --[ ServerKey($S, ~ltk) ]->
    [ !ServerLTK($S, ~ltk) ]

rule Handshake_Establish:
    [ !ServerLTK($S, ~ltk), Fr(~master) ]
  --[ HandshakeDone($C, $S, h(<~master, 'epoch_0'>)),
      SessionStart($C, $S),
      OutKey(h(<~master, 'epoch_0'>)) ]->
    [ !Session($C, $S, h(<~master, 'epoch_0'>), '0') ]

rule Ratchet_Step:
    [ !Session($C, $S, K_old, idx) ]
  --[ Ratchet($C, $S, K_old, hkdf(K_old, label_ratchet), idx),
      OldKeyDeleted(K_old) ]->
    [ !Session($C, $S, hkdf(K_old, label_ratchet), h(idx)) ]

rule Reveal_Key:
    [ !Session($C, $S, K, idx) ]
  --[ KeyReveal($C, $S, K, idx) ]->
    [ Out(K) ]
```

- `Fr(~x)` = fresh value
- `!Session(...)` = persistent fact
- `[ A ] --[ EventX ]-> [ B ]` = rewrite A to B, generating event EventX

### 4. Properties (lemmas)

```
lemma forward_secrecy:
  "All C S K m #i.
       SentWithKey(C, S, K, m) @ i &
       (not Ex #j. KeyReveal(C, S, K, j) @ j & j < i + 1)
     ==> not (Ex #k. K(m) @ k)"
```

`K(m) @ k` = attacker knows `m` at time k.

```
lemma post_compromise_security:
  "All C S K_old K_new idx #i #j.
       Ratchet(C, S, K_old, K_new, idx) @ i &
       KeyReveal(C, S, K_old, idx) @ j &
       j < i
       & (not Ex K' idx' #k. KeyReveal(C, S, K', idx') @ k & k > i)
     ==> not (Ex #l. K(K_new) @ l)"
```

意思：若 K_old 在時間 j 被 reveal，j < i (ratchet at i)，且 i 之後沒再 reveal — 那麼 K_new 不被 attacker 知道。這正是 PCS 的形式定義（Cohn-Gordon 2016）.

### 7. 為何 Proteus 的 ratchet 達 PCS-weak/medium 而非 PCS-strong

Cohn-Gordon 2016 區分：

- **PCS-weak**：reveal K_n 後，K_{n+1} 仍 secret if attacker stops其 active role.
- **PCS-medium**：reveal K_n 後，K_{n+1} secret if attacker has not seen any of K_{n+1}'s direct inputs.
- **PCS-strong**：reveal K_n + any pre-ratchet leak 後仍 secret if **fresh randomness** is mixed into ratchet.

Proteus ratchet 是 **KDF-only** (`K_{n+1} = hkdf(K_n, label)`)，沒有 fresh randomness 注入。若 attacker 持 K_n 並 derive 後續 keys, attacker 知道所有未來 keys.

但**若 K_n 是 forgotten by attacker** (compromise was momentary), and attacker doesn't see K_n directly — 則 K_{n+1} = hkdf(K_n, label) 對 attacker 是 secret. 這是 PCS-weak.

要達 PCS-strong, Proteus v0.2 必須讓 ratchet 內注入 fresh DH share (Signal-style asymmetric ratchet)，每 ratchet 引入 1 RTT cost. Proteus v0.1 不開. 已在 11.6 §5.4 提.

Tamarin lemma `post_compromise_security` 上對 Proteus verify 的是 **PCS-weak**: attacker 在 j 拿 K_old, attacker stop reveals — 在 ratchet 之後 K_new 仍 secret. 結果 = verified.

### 8. 對 replay-protection 的 stateful 驗證

ProVerif 用 inj-event 處理 replay (count); Tamarin 用 facts 可 explicitly model replay window. 範例：

```
rule Server_ReceiveCH:
  let
    new_nonce = ~nonce
  in
    [ !ServerLTK($S, ltk),
      In(<n, c_x_pub, kem_ct, c_id, ts, tag>) ]
  --[ Process($S, n, ts) ]->
    [ Process_marker($S, n, ts) ]

rule Server_Reject_Replay:
    [ Process_marker($S, n, ts),
      Process_marker($S, n, ts) ]  // duplicate
  --[ Replayed($S, n) ]->
    [ ]   // empty, but Replayed event is generated → can lemma against it

lemma no_replay:
  "All S n #i #j.
     Process($S, n, _) @ i & Process($S, n, _) @ j
     ==> #i = #j"
```

該 lemma 驗 spec 中「server 不 process 同 nonce 兩次」.

### 9. Tamarin 限制

對 Proteus v0.1 v0.2 ratchet model: 收斂. 對大型 protocol (e.g., 完整 TLS 1.3 0-RTT + Cookie + Resumption): Tamarin 需要 hours 與 user-provided helper lemmas. Cremers 2017 對 TLS 1.3 用 hundreds of helper lemmas.

對 Proteus: 預期 PCS lemma 可在 minutes 內 verify. 若超時, 加 helper lemma like:

```
lemma helper_key_chain [reuse]:
  "All C S K_n K_m idx_n idx_m #i #j.
     Ratchet(C, S, K_n, K_{n+1}, idx_n) @ i &
     Ratchet(C, S, K_m, K_{m+1}, idx_m) @ j &
     K_m = K_{n+1}
     ==> #j > #i & K_m = hkdf(K_n, label_ratchet)"
```

### 10. 完整 Tamarin model 與 ProVerif 模型的對應 mapping

| ProVerif (ProteusHandshake.pv) | Tamarin (ProteusRatchet.spthy) |
|---|---|
| client / server process | rules Setup, Handshake_Establish |
| event client_completes | event SessionStart / HandshakeDone |
| event server_accepts | event HandshakeDone (single rule) |
| query attacker(c_ap) | lemma forward_secrecy + post_compromise_security |
| phase 1 KEM compromise | rule Reveal_LongTerm |
| inj-event | implicit via fact uniqueness |

### 11. 對 Tamarin model 的 spec hook

Spec §11.3 forward secrecy 段引：

```
Forward secrecy and post-compromise security of the KEYUPDATE ratchet
have been verified in Tamarin (model file `assets/formal/ProteusRatchet.spthy`),
specifically:

  - lemma forward_secrecy: VERIFIED
  - lemma post_compromise_security: VERIFIED (PCS-weak, see Cohn-Gordon 2016)
  - lemma key_uniqueness: VERIFIED
  - lemma fs_after_ltk_reveal: VERIFIED

The ratchet provides PCS-weak: after a single key reveal at epoch n,
the attacker cannot derive K_{n+1} as long as it does not interact with
the session between the reveal and the ratchet event. To achieve PCS-strong
(resilient to active attacker even during ratchet), an asymmetric ratchet
with fresh DH/KEM mix is required; this is deferred to Proteus v0.2 per §13.4.
```

## 動手

1. 安裝 Tamarin (`brew install tamarin-prover`), 跑 ProteusRatchet.spthy. 確認 4 lemmas verified.
2. 修改 ratchet rule: 把 `K_new = hkdf(K_old, label_ratchet)` 改成 `K_new = K_old`. 重跑 — PCS lemma 應 fail (counter-example: K_old revealed → K_new = K_old → attacker has K_new).
3. 加 helper lemma 對 key_uniqueness 加速.
4. 嘗試 PCS-strong model: 加 fresh DH 進 ratchet rule, verify lemma `post_compromise_security_strong` (in active attacker scenario).

## 自我檢查

1. 為什麼 Tamarin 比 ProVerif 更適合驗 ratchet？答：ratchet 是 stateful — state evolution over time. ProVerif process abstraction 隱含 state, Tamarin 顯式 facts 表達.
2. 為何 Proteus v0.1 只達 PCS-weak？答：unidirectional KDF ratchet, no fresh randomness; PCS-strong 需 asymmetric ratchet.
3. Tamarin 為何需要 helper lemma？答：因 free first-order logic search-space 爆炸; helper lemma 提供 induction hint.
4. 為何 fs_after_ltk_reveal 重要？答：long-term key 與 session key 是分開的 — 確保 spec 結構正確.
5. 對 Tamarin proof script 是否有 false-positive 風險？答：Tamarin sound — 證的就是真. 但 spec model 不對則證錯 spec; spec-to-model 一致性是另一回事.

### 2. 對手分類學 / 威脅模型精化

Tamarin attacker:

- Standard Dolev-Yao (自動)
- + Reveal rules: explicit compromise of named values (PCS modeling)
- + Adaptive: attacker decides when to compromise

對 Proteus: attacker can reveal session key at any epoch + can reveal long-term key. Tamarin verify PCS even under this strongest adaptive compromise model (for the ratchet portion).

### 3. 形式化定義

**Multi-set rewriting rule (Tamarin)**:

```
[ premise_facts ]   // multiset of facts that must be present
  --[ action_events ]->  // observed in the trace
[ conclusion_facts ]   // multiset replacing the premise
```

**Linear temporal logic (Tamarin)**:

```
∀ var. φ(var, #i, #j, ...) ==> ψ(...)
```

`#i, #j` 為 time variables (節點 in trace DAG).

**PCS-weak formal (Cohn-Gordon 2016 §4)**:

```
∀ session, ∀ epoch n.
  CompromiseAt(session, n) ∧ ¬ FurtherInteraction(session, ≥ n)
  ⇒ Secret(K_{n+1}, ..., K_∞)
```

### 7. 開放問題

1. **PCS-strong 對 anti-censorship transport 的成本**：每 ratchet 加 1-RTT DH 對 Proteus 1-RTT handshake 預算的影響? 必 measure.
2. **Tamarin 與 ProVerif 結果 cross-check 自動化**：可否寫 framework, 給定 protocol spec, 自動生成 Tamarin + ProVerif models 並 cross-check 一致?
3. **Quantitative PCS**：PCS-weak/medium/strong 是定性. 能否定義 ε-PCS (attacker advantage in compromise scenario)?
4. **Stateful CAR property in Tamarin**：能否 model padding budget 為 stateful counter 並 verify "α never exceeded"? Tamarin 對 counter 有 native support.
5. **Multi-session interaction**：Proteus deployment 多 client 同時跑. Tamarin 對 concurrency 是 native, 但 proof search 加劇 expensive.
