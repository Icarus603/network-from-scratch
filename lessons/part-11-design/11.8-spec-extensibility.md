## 動機

協議若不能演進就是死的。

擴展性難在三點：

1. **新功能不能破舊客戶端**（forward / backward compatibility）.
2. **中間盒不能逐漸把選項「凝固」**（ossification, e.g. TCP options 被廣泛 drop）.
3. **新版本的 wire-level 變化不能破壞 CAR fingerprint**.

第三點是 Proteus 獨有的——一般協議只考慮 1+2，Proteus 還要保證新版本仍能 mimic cover protocol。

本堂定義 Proteus 的 extension framework、version negotiation、與 anti-ossification（GREASE 風格）機制。

## 核心概念

### 1. 擴展機制：兩種選擇

| 風格 | 範例 | 優 | 劣 |
|---|---|---|---|
| **TLS-style typed extensions** | TLS 1.3 ClientHello / EncryptedExtensions | 已 IETF 證型；wire format 可預期 | 加 ext 需 IANA 配 code point |
| **QUIC-style transport parameters** | QUIC v1 transport params | 雙端自然協商，varint 編碼 | 對 Proteus inner protocol 重複 wheel |
| **Proteus-style: hybrid** | inner packet type 0xf0-0xff + extension type sub-field | 內外分明 | 多一層 spec |

Proteus 採 **hybrid**：

- **Handshake-layer extensions**：TLS-style，跟在 ClientHello/ServerHello 內。Sub-types 用 Proteus reserved range `0xfe00–0xfeff`. `0xfe0d` 是 auth, `0xfe0e` reserved future, etc.
- **Inner-data-layer extensions**：使用 inner packet type 0xf0–0xff. 第一個 byte 是 extension type, 後續是 ext-specific payload.

### 2. Version negotiation

#### 2.1 Proteus v0.1 不協商版本

設計選擇：v0.1 spec 釘死所有 cipher、KEM、core packet types。**没有版本協商欄位**.

理由：

- 版本協商欄位本身會成為 fingerprint（GFW 已對 QUIC version number 訓 classifier）.
- v0.1 為 "initial deployment"，鎖死 simplifies threat model.
- 將來升 v0.2 由 client side initiate—不在 wire 出現。

#### 2.2 v0.2+ 的協商策略：parallel-version handshake

當 Proteus v0.2 出世，client 與 server 都支援多版本時：

- Client 在 ClientHello 的 Proteus extension type 上 carry version hint:
  - `0xfe0d` = v0.1 auth (1186 bytes payload)
  - `0xfe1d` = v0.2 auth (different payload)
  - Both can be sent in same CH if client wants to support both.
- Server picks the **highest version supported** and responds via SH `EncryptedExtensions`.
- Client 收到回應自然知是 v0.2 → 用 v0.2 後續流程.
- 若 server 不支援 v0.2 → fallback to v0.1.

> 這比顯式 version field 隱蔽（version 隱於 ext type 編號），有助 CAR-1.

#### 2.3 不支援 downgrade

Server 若同時支援 v0.1 與 v0.2，且 client 同時 carry 兩者 ext, 仍**首選 v0.2**（永不 prefer 舊版）.

對 Bhargavan 2016 downgrade resilience: Proteus 由結構達 "no downgrade unless client explicit only sends v0.1".

### 3. 擴展機制：handshake-layer

Proteus reserved sub-ext range：

| Code | Status | Purpose |
|---|---|---|
| 0xfe00–0xfe0c | reserved | future low-IDs |
| **0xfe0d** | spec v0.1 | auth payload |
| 0xfe0e | reserved future | v0.2 auth |
| 0xfe0f | reserved future | identity proof for mesh deploy |
| 0xfe10–0xfe1f | reserved | core extensions |
| 0xfe20–0xfeff | private use | implementer experiments |

新增 ext rules：

- Server MUST silently ignore unknown ext types (forward-compat).
- Client MUST be prepared to receive zero EncryptedExtension response for ext sent.
- New core extensions MUST be specified via dedicated spec amendments (Part 12 onwards).

### 4. 擴展機制：inner-data-layer

inner packet types 0xf0–0xff：

```
struct ProteusExtension {
    ProteusInnerHeader header { type = 0xfN };
    uint8        sub_type;       // extension-specific
    uint16       sub_length;
    opaque       payload[sub_length];
    opaque       aead_tag[16];
}
```

雙方未協商的 ext: silent ignore by receiver.

### 5. GREASE: anti-ossification

借 RFC 8701（TLS GREASE）思路。Client 在 ClientHello 內**故意插入 Proteus spec 未定義** code points：

- 每連線隨機選 1-2 個 GREASE values from `0xfeXa, 0xfeXb` (a single-byte hi-nibble GREASE block)，or arbitrary unknowns.
- Server MUST silently ignore them.

目的：

- 防止 middlebox / future censor 訓 classifier 對「Proteus ext set 一定是某幾個」做 fingerprint.
- 維持 wire-level 多樣性—與 Chrome/Firefox 的 GREASE 行為對齊（CAR-1 borrow）.

GREASE rules：

- GREASE codepoint MUST be ignored by spec-compliant impl.
- GREASE payload MUST be valid TLS extension shape (length-prefixed).
- GREASE position random in extension list.

### 6. Cipher / KEM agility

Proteus v0.1 lock。但 spec leaves room for v0.2 agility：

#### 6.1 Cipher 協商（v0.2 design sketch）

```
Proteus ext 0xfe0e (v0.2) {
    uint16  supported_ciphers[];   // 0x1303 ChaCha20Poly1305, 0x1302 AES256GCM
    ...
}
```

但 cipher set MUST stay short, predictable, and ordered to match cover_browser_profile.

#### 6.2 KEM 協商

當 ML-KEM-1024 在 v0.2 加入，client 可 carry 一個或多個 KEM ciphertexts。Server 選最強. 但 wire size 增加會引入 CAR risk—v0.2 必須重新跑 CAR-1 evaluation.

### 7. 對應 ossification 的防護

| Ossification source | Proteus 對策 |
|---|---|
| Middlebox 對 QUIC version field 凝固 | Proteus 用 QUIC v1, 不協商; 全靠 QUIC WG 努力（unicorn version 21d1c1c1 等） |
| Middlebox 對 TLS extension order 凝固 | Always follow `cover_browser_profile` |
| Middlebox 對 cipher suite list 凝固 | Always 提 ChaCha20 + AES-256 順序與 browser 一致 |
| Censor 對 ext code point 凝固 | GREASE values inserted |
| 上層協議 ossification (H3 / HTTP datagram) | 跟 IETF MASQUE WG 演進 |

### 8. Forward / backward compatibility 規則

| 場景 | 行為 |
|---|---|
| 舊 client + 新 server | server uses old protocol (v0.1) for this session; advertises v0.2 capability via NewSessionTicket extension (optional) |
| 新 client + 舊 server | client sends both v0.1 (0xfe0d) and v0.2 (0xfe1d) exts; server picks v0.1 |
| 兩端皆新 | both support v0.2; server uses v0.2 |
| 客戶端 ext 未知 | server 必 silent ignore, 不錯誤 |
| 服務端 ext 未知 | client 必 silent ignore, 不錯誤 |

### 9. Wire-format invariants（不可變）

無論 v0.x 怎變，下列 invariants MUST 保持，否則 CAR-1 fingerprint 整套重訓：

1. Outer transport 仍 H3-over-QUIC over UDP 443.
2. ClientHello total size 維持 chrome-distribution range (1.4KB–2.5KB).
3. Cell padding 1280B.
4. ALPN advertised as `h3`.
5. SNI 借 popular cover.

任何破壞 invariant 的變更須 → v1.0（major version bump，重新 CAR-1 evaluation cycle）.

### 10. Extension review process（過程，非 spec 內容）

對未來 contributor 加 ext 的流程：

```
1. Propose: spec amendment doc 寫進 Proteus repo /spec/proposals/
2. Threat impact: 對 §11 capability matrix 逐條評估
3. CAR impact: 訓 reference attackers, 跑 ε_CAR measurement
4. PERF impact: bench
5. Formal model update: ProVerif / Tamarin model patch
6. Acceptance: 3 reviewer approval + 30-day comment window
7. Allocate code point: 從 0xfe10–0xfe1f reserve 取
```

### 11. Spec versioning 規則

仿 SemVer：

- **v0.x → v0.y**: feature add / new ext, fully backward-compatible.
- **v1.0**: first major; wire-format invariants 可能不變但 anti-fingerprint 重新校.
- **v2.0+**: 任何破壞 wire-format invariant 的變更.

每次 minor bump：
- spec doc 版本記錄 (RFC-style versioning).
- ProVerif/Tamarin model 同步更新.
- CAR-1 benchmark suite re-run.

## 自我檢查

1. 為什麼 Proteus v0.1 不直接帶 version field 而靠 ext type 編號隱含？答：CAR-1 — version field 自身是 fingerprint.
2. 為什麼 GREASE 必要？答：anti-ossification + CAR multiplicity. Chrome / Firefox 都 GREASE, 我們不 GREASE 反而 stick out.
3. 為什麼 v0.2 cipher 協商不直接 enumerate 所有 cipher？答：cipher list 自身是 fingerprint surface; 必須對齊 cover browser profile.
4. 為什麼 ossification 防護寫進 spec 而不是「implementer's note」？答：因為 implementer 沒 spec 強制會 cut corner; ossification 是 collective action problem.
5. 為什麼 wire-format invariants 列為 "不可變"？答：CAR-1 是 calibrated against specific cover distribution; invariant 變則 calibration 失效，需 v1.0.

### 2. 對手分類學

middlebox 與 censor 都是 ossifiers：

- **Passive middlebox**: 對未知 wire shape 採 fail-close. GREASE 抗.
- **Active middlebox**: 對 ext code 點過濾或重寫. cover_browser_profile 鎖.
- **Censor adaptive classifier**: 對 ext set entropy 學. GREASE multiplicity.

### 3. 形式化定義

**Extension forward-compat (Thomson RFC 9170)**:

```
A receiver R is forward-compatible iff for any byte sequence b that is a valid
encoding of some future-spec extension, R does not throw error; R either ignores
or accepts.
```

Proteus 由 "silent ignore unknown" 達到此屬性.

**Ossification-resistance metric**: 

```
Let M(t) = set of mainstream impls at time t.
Let V(t) = set of allowed wire-shapes accepted by all impl in M(t).
Ossification at t = | V(t) | / | V(spec) |
```

GREASE 不直接消除 ossification 但維持 |V(t)| 動態變化。

### 5. 我們協議的座標 / 設計取捨

本堂 lock：

- v0.1 no version field; v0.2+ encoded in ext type
- GREASE mandatory for client
- 1-2 GREASE codepoints per CH
- Cipher / KEM agility deferred to v0.2
- Forward-compat: silent-ignore-unknown is spec norm

仍 open：
- GREASE codepoint pool 具體值（之後與 cover_browser_profile 一起 release）
- Cipher agility 在 v0.2 怎做（待 PERF/CAR 量測再定）

### 7. 開放問題

1. **GREASE 在 anti-censorship 場景下的 limit**：GREASE 假設 honest middlebox; 對 adversarial censor 是否仍提升 multiplicity？或反而 censor 訓 classifier 學 GREASE pattern？
2. **Ext-type 編碼隱含版本的 fingerprint cost**：若 Proteus v0.2 用 `0xfe1d`, attacker 是否能對「同時 carry 0xfe0d + 0xfe1d」做版本 detection？需 量測.
3. **Spec evolution 的 game**：CAR-1 與 protocol evolution 是 fundamentally opposing forces（前者要 stability, 後者要 change）. 是否能 formalize 為 multi-armed bandit?
4. **Extension review process 自動化**: 能否寫一個工具自動 evaluate ext proposal 對 §11 capability matrix 的影響？
5. **Cipher agility 與 cover browser profile 一致性 的 closed-form formula**：給定 cover = chrome-N, 怎麼自動選哪些 cipher 該開哪些不該開？
