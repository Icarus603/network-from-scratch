## 動機

Active probing 是 GFW 對 fully-encrypted-traffic 的 ultimate 確認手段。從 2011 年第一次對 obfs2/obfs3 探測，到 2020 對 SS 的 7-family、2023 對所有 FET 流量的通用探測，**這條路徑十年間從 ad-hoc 工程升級為標準化 attack pipeline**。本堂橫跨時間軸把這條路徑徹底拆解。

研究級目標：
- 理解 active probing 的 **decision pipeline**：誰觸發、誰執行、誰確認、誰執行 block。
- 學會 [[frolov-probe-resistant-ndss20]] 級的「**TCP-layer side channel**」設計與識別。
- 把以上整合成 lesson 9.12 testbed 的設計藍圖。

> **Failure framing**：active probing 沒有「不被 detect 的服務器」這個 absolute——只有「**fingerprint 相對小於 N**」的服務器。N 隨對手能力升級而下降。10 年來 N 從「容易」變「需 ML」變「需多 modality fusion」。設計協議要假設 N 還會繼續下降。

### 2. Trigger 的演化：從 Tor → SS → FET

| 時期 | 對象 | Trigger heuristic | 文獻 |
|---|---|---|---|
| 2011–2014 | obfs2 / obfs3 / Tor bridges | 已知 bridge IP 表 + Tor handshake replay | Winter, Lindskog FOCI 2012; [[ensafi-gfw-probing]] |
| 2015 | Tor + obfs3 改進 | + 觀察 client → server 的 entropy + TLS-like 但無正確 cert | [[ensafi-gfw-probing]] |
| 2019–2020 | Shadowsocks (libev + Outline) | first-packet length mod 16 + entropy | [[alice-bob-carol-ss-imc20]] |
| 2022–2023 | 全 FET（SS, VMess, …） | 6 條 byte-class exemption 規則 | [[wu-fep-detection]] |
| 2024+ | + QUIC traffic | + QUIC Initial decryption + SNI | [[zohaib-quic-sni-usenix25]] |

**演化方向**：從「**已知工具特定 fingerprint**」到「**任何沒有 plaintext header 的流量**」。Trigger 越通用，**probe 階段越要精確**——因為 false positive 上升。

### 3. Probe family 的演化

**[[ensafi-gfw-probing]] (IMC 2015)** 觀察到的 Tor bridge probe families：
- Plain TLS ClientHello（嘗試啟動 TLS）
- obfs2 handshake replay（從之前的 client → bridge 流量裡 copy）
- obfs3 handshake replay
- Vanilla Tor handshake (TLS-like)

**[[alice-bob-carol-ss-imc20]] (IMC 2020)** 對 SS 觀察到的 7 family（lesson 9.2 已詳）：
1. Verbatim replay
2. Single-byte mutation
3. Multi-byte mutation
4. Random 221 bytes
5. Random length-matched
6. Truncated replay
7. Concatenated replays

**[[frolov-probe-resistant-ndss20]] (NDSS 2020)** 對 6 種 probe-resistant proxy 觀察到的 attack-side probe：
- 不同長度的 random payload (51, 100, 150, 200 ...)
- 不同 known protocol prefix (HTTP, TLS, SSH banner)
- 不同 TLS handshake (with/without SNI, with/without ALPN)
- ZMap-style single-packet scan

**通用 pattern**：
1. **替代 fingerprint**（送 HTTP / TLS / SSH / SS prefix）：看 server 是否誤回應 → 識別協議家族。
2. **格式化 garbage**（特定長度 random）：看 server 行為與真實 TCP server 的差異。
3. **時序分析**（probe 後 X 秒 connect 同 port 再探）：identify residual blocking pattern。
4. **連線層級觀察**（TCP timestamp, ISN entropy）：抓 server kernel 指紋。

### 4. Prober pool: scale 與 centralisation

[[alice-bob-carol-ss-imc20]] 對 2019-20 SS probing 觀察的 prober pool：

- **12 000+ distinct IP addresses** 在 4 個月內出現過。
- **TCP TS clustering** → 推測背後只有 **十多台** central machine。
- **Source IP 並非 spoofed**——response packet 確實能 reach prober (因為 prober 從這個 IP 收到 TCP handshake)。**這些是真正的中國境內 IP**，可能由：
  - GFW 操作 multi-NIC 主機，多個 IP 對外。
  - 與 ISP 合作借用 IP 池。
  - 部分被合法用戶共用（住宅 IP）—— 這個假設未被排除，OONI 觀察 some prober IPs 也來自 residential ISP。

**對比 [[ensafi-gfw-probing]] (2015)** 觀察的 prober pool：
- **數百個** distinct IP，相對小。
- 同樣有中央化 evidence（時序同步、port 分布）。

**演化**：5 年 prober 池擴張 50x。可能是 source-IP rotation 系統的工程改進。

### 5. Decision logic：從 single-shot 到 multi-stage

早期 (2011–2015)：
- Single-shot probe → server responds with TLS-like + correct Tor cert → "Tor bridge confirmed" → block。

2020 之後：
- Multi-probe sequence：先 verbatim replay → silent server → 標記為「可能 SS」→ 後續 probe 變異 → 觀察 timing → 確認。
- **Decision tree** 形態（per [[frolov-probe-resistant-ndss20]]）：
  ```
  if server responds with TLS_handshake_alert: NOT proxy
  elif server responds with HTTP_400: NOT proxy
  elif server times_out_30s with FIN: maybe obfs4
  elif server FIN_at_byte_50_RST_at_byte_51: probably obfs4
  elif server times_out_forever: maybe MTProto / Outline
  ...
  ```

### 6. Probe-resistance: 防禦演化

#### (a) obfs4 (2013)
- 用 **Elligator-encoded** ephemeral pk（看起來 uniform random）。
- Server 收到後計算驗證；驗證失敗則 silent-hold + 30s timeout。
- **Probe-resistance L1**：probe replay 無法觸發 server response。

[[frolov-probe-resistant-ndss20]] 攻擊：probe size 49 byte FIN-timeout 30s、50 byte FIN-immediate、51 byte RST-immediate — **threshold 即指紋**。

#### (b) Shadowsocks AEAD silent-hold (2019)
- 失敗 AEAD 解密後保持 socket open 30 秒，不發任何 byte。
- 對 [[frolov-probe-resistant-ndss20]] threshold 攻擊有幫助（沒有 byte 邊界差異）。
- 但 [[wu-fep-detection]] 顯示 passive entropy heuristic 已不需要 probing 即可識別。

#### (c) MTProto Telegram (perpetual read)
- Server 永遠 read socket，從不 close、從不 timeout。
- **[[frolov-probe-resistant-ndss20]] 評為「唯一倖存」協議**。
- 弱點：DoS（client 不主動 close，server resource 被佔）。實作上 MTProto 用 `keep_alive` 與 OS-level idle timeout 緩解。

#### (d) Snowflake + meek (browser WebRTC fronting)
- 用 WebRTC SDP signaling，bridge 是「rendezvous」服務+ WebRTC peer。
- Probe 在 SDP rendezvous 階段 → 看到 WebSocket / Fastly CDN，**不是直接 probe bridge IP**。
- 弱點：CDN 仍可被 SNI block；2022 後部分被打。

#### (e) REALITY 的 fallback-to-real-backend (lesson 9.4)
- 失敗認證 → server byte-for-byte forward 給真 cover site。
- Probe 看到的是真 cover site。
- 弱點：timing 精確度、IP-SNI 對應。

### 7. [[ensafi-gfw-probing]] 的歷史貢獻

IMC 2015 是第一篇系統化記錄 GFW active probing 的學術論文。關鍵發現（lesson 9.6 角度）：

1. **GFW 從 2011 開始 active probing Tor bridges**，至 2015 已有規模化的 prober pool。
2. **Probe 行為與 client 行為惟妙惟肖**：probe 用 TLS handshake、Tor 協議特徵，幾乎像真實 client。
3. **Probe 來源 IP 數量隨時間增長**——已預警今後 prober pool 會擴張。
4. **首次量化 probe 時間延遲**：被動觸發後 0.5–60 秒內 probe 到達。

對研究方法論的貢獻：建立了「**部署 honeypot bridge → 記錄 probe → 分析 prober pool**」的範本，後續所有 active probing 研究都沿用。

### 8. Wu 2023 的 active probing post-FET

[[wu-fep-detection]] 觀察到 **GFW 對 FET 的處理**：
- Passive 階段已 sufficient 偵測（6 條 exempt）。
- Active probing 仍存在，但更像 **confirmation** 而非 primary detector。
- 對 SS-style replay probe 依然發送。
- 偵測判定後幾秒到幾分鐘內加入 IP blocklist。

**演化意義**：active probing 從 primary detector 退化為 secondary confirmer，但仍是 pipeline 不可或缺的一環。

## 與我們協議設計的關聯

從 active probing 10 年史抽取的設計原則：

1. **假設對手 probe budget 是 O(每個目標 100 probes)**。任何「需要 100 次 probe 才能識別」的設計不夠。
2. **Probe-resistance L1（silent-hold）不夠**——[[frolov-probe-resistant-ndss20]] 的 threshold 攻擊揭穿。
3. **Probe-resistance L2（perpetual read）強但成本**——對 honest client 也持久持 socket。
4. **Probe-resistance L3（REALITY-style real-backend forward）目前最強**——但 timing 與 IP 漏洞仍 open。
5. **Probe-resistance L4（cryptographic indistinguishability of server identity）是 Part 11 目標**——即使對手能 probe，response 必須 cryptographically indistinguishable from a real cover server response。

Part 11.5「服務器 fallback 設計」會詳細展開 L3 → L4 的 design space。

## 動手

**任務**：建立你自己的 honeypot bridge 並記錄 GFW probes。

> ⚠️ 警告：這會讓你的 VPS IP 進入 GFW 觀察名單，**至少數週**內這個 IP 上的其他服務可能也被影響。**用 disposable VPS**。

### 設置
1. 境外 VPS A：跑 `obfs4proxy` 作為 Tor bridge，使用獨立 IP。
2. 從境內 client（或用一個 tor browser + obfs4 連線）連 A 一次，發起 obfs4 handshake → 觸發 GFW 觀察。
3. 切斷 client。在 A 上跑 tcpdump 24h：
   ```bash
   tcpdump -i any -w obfs4-probes.pcap 'port 25555 and src not 你的client_IP'
   ```

### 預期觀察
- 24h 內 100–500 probes from various IPs。
- 大部分 probe 是 TLS ClientHello、obfs4 handshake replay。
- Source IP 分散，TCP TS 集中。

### 分析
1. 提取每個 probe 的 (src_IP, src_port, TS_val, payload_first_bytes, total_bytes, response, close_pattern)。
2. 統計：
   - Source IP 數量。
   - TS val cluster 數量。
   - Payload 類別分布。
3. 把 source IP 按 ASN 解析，看是中國電信、聯通還是教育網。

### 進階
- 對部分 probe **故意回應錯誤的東西**（例如 fake Tor consensus、HTTP 503），看 GFW 後續是否改變 probing 行為。
- 對部分 probe **不回應**，看 GFW 是否會 retry。

輸出：`assets/measurements/2026-active-probe-honeypot/`，redact bridge IP 與 client IP。

## 研究級補遺

### 1. 學界詞彙

| 中文 | 學界標準 | 說明 |
|---|---|---|
| 主動探測 | **active probing** | 由 censor 主動發起的 TCP/UDP connect 與 payload 發送 |
| 探測族 | **probe family** | 同類但變異的探測集合 |
| 重放探測 | **replay probe** | 用之前 capture 的 client traffic 作為 probe |
| 探測排程器 | **probe scheduler** | 控制 timing、family 選擇、target 選擇 |
| 抗探測協議 | **probe-resistant protocol** | 對 active probe 不洩漏身份 |
| 殘留封鎖 | **residual blocking** | 觸發後一段時間繼續封鎖同一 dst |
| 蜜罐橋 | **honeypot bridge** | 故意暴露的 bridge，用來收集 probes |

### 2. 對手分類學精化

GFW probing 對手：
- **Capability axes**:
  - probe budget: 高（per target 多達百次）
  - probe diversity: 高（多 family、多 source IP）
  - timing precision: 中（延遲 0.5–60s）
  - cross-target correlation: 高（可從 prober pool 對所有可疑 IP 同時 probe）
- **Limitations**:
  - 必須有 trigger（不對全 internet 隨機 probe）
  - probe scheduler 有 latency
  - prober 來源 IP 真實存在（可被 honeypot 記錄）

對比 academic adversary models：[[frolov-probe-resistant-ndss20]] 假設 ZMap-scale active probe，但只 single-shot。實際 GFW 是 multi-shot、stateful。

### 3. 形式化定義

**Probe-resistance** 的 game-based definition：

對 PPT 對手 $\mathcal{A}$，給定 (server $S$, oracle access to probe S with up to $K$ messages)，$\mathcal{A}$ 輸出二元判斷 "$S$ 是 proxy" or "$S$ is non-proxy"。

協議 $\Pi$ 是 $(K, \epsilon)$-probe-resistant iff：

$$
\left| \Pr[\mathcal{A}^{S_\Pi}(1^\kappa) = 1] - \Pr[\mathcal{A}^{S_{\text{cover}}}(1^\kappa) = 1] \right| \leq \epsilon
$$

對任意 $\mathcal{A}$ runtime $\leq T(\kappa)$，probe 次數 $\leq K$。

**目前已知 K-bounds**:
- obfs4 silent-hold: $K \geq 3$ probes (49/50/51) gives $\epsilon \geq 0.9$ ([[frolov-probe-resistant-ndss20]])
- MTProto perpetual-read: $K = \infty$ 仍 $\epsilon \approx 0$ ([[frolov-probe-resistant-ndss20]])
- REALITY: empirically $\epsilon \approx 0$ for $K \leq $ 任意 known probe family，但 timing precision 未被測量

### 6. 開放問題（research-level）

1. **L4 probe-resistance 是否存在**？即「對任意 PPT prober 都 cryptographically indistinguishable from cover server」的協議是否可構造？
   - 必要條件：server 對 garbage 必須產生 cover server bit-exact response（包含 timing）。timing precision 是難點。
2. **Prober identification by honeypot consortium**：能否組織多個 honeypot 共享 prober IP/TS database，做 community-level prober identification?
3. **Multi-stage probing analysis**：對 GFW 的 multi-stage probing decision 做 reverse engineering 仍未完成。設計 series of probes-and-responses 來逆向 decision tree。
4. **Cross-jurisdiction probing comparison**：China vs Iran vs Russia 的 probing 行為定量比較。
