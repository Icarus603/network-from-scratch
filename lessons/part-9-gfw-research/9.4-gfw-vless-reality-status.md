## 動機

VLESS+REALITY 是 2022 年 RPRX（XTLS 維護者）提出的代理協議，**到 2026 年仍是中國地區事實上最被廣用且未被大規模封鎖的「家用」方案**。研究問題：

1. REALITY 的核心 trick 是什麼，為何比 Trojan 強？
2. 哪些 attack surface 仍 open（GFW 哪天決定要打就能打）？
3. 我們設計新 SOTA 協議，能從 REALITY 學到什麼、避免什麼？

REALITY 不是學術論文產物，沒有 peer-reviewed paper。**最權威來源是 RPRX 的 GitHub spec + Xray-core 源碼**。本堂不假設你看過源碼——我們**會把 reality.go 的關鍵段落直接拆給你看**。

> **Failure framing**：REALITY 在 2026 還活著，但 **它的存活很可能是「GFW 還沒有經濟動機去打」而非「GFW 打不了」**。Iran 已觀察到 REALITY 被間歇性 block；GFW 一旦部署類似機制，REALITY 預計同樣被打。Don't fall in love with REALITY.

## 核心概念

### 1. REALITY 的設計目標：解決什麼問題

回顧 Trojan 的問題（lesson 9.3）：
- 服務器用自己的真實 cert + LE 申請的 domain → **IP-SNI 對應不對**（OVH IP 配自己 domain）。
- TLS 完成後流量形態是 TLS-in-TLS，與真實 browsing 不同 → **traffic-shape leak**。
- Fallback 邏輯在 application layer → **TLS-layer probe 揭穿**。

REALITY 的設計目標：
- **借用真實大站的 cert**（沒有自己的 domain；client 用 `apple.com` 之類連線）→ 修「IP-SNI 對應」（**仍不完美**，因為服務器 IP 通常不是 Apple 的 ASN）。
- **server 完整模擬 TLS handshake**，handshake 過程中 byte-for-byte 像真實 apple.com → 修 ClientHello/ServerHello 一致性。
- **REALITY 完成後立即進入 application data，不再 TLS-in-TLS**：直接用 TLS session key 加密 VLESS 訊息 → **減少 TLS-in-TLS shape leak**（但不消除——後續 9.8、10.2 詳）。
- **Active probe / wrong client** → **TLS hand-off 給真實 backend**（apple.com）→ 探測者看到真 Apple 回應 → **完美 fallback**。

### 2. 「Handshake stealing」: 核心 trick 拆解

```mermaid
sequenceDiagram
    participant C as Client
    participant S as REALITY Server (IP_S)
    participant A as Real apple.com

    Note over C,S: Client 想連 apple.com 但實際 dst_IP = IP_S
    C->>S: TCP SYN → IP_S:443
    S->>C: SYN-ACK
    C->>S: TLS ClientHello (SNI=apple.com, key_share=ephemeral_pk + auth_pk)
    Note over S: Server 解析 ClientHello.key_share<br/>提取 client 的 auth_pk（hidden 在 key_share 中）<br/>用 X25519 算 shared_secret = X25519(server_sk, client_auth_pk)<br/>auth_key = HKDF(shared_secret, "REALITY")
    alt 認證通過（client 是真 REALITY client）
        S->>S: 自己生成 ServerHello + Cert (借用真 apple.com cert)<br/>用 auth_key 簽 ServerHello 的 random 部分
        S->>C: ServerHello + Certificate + CertVerify + Finished
        Note over C: Client 驗證 auth_key signature<br/>確認對方是真 REALITY server
        C->>S: TLS Finished
        Note over C,S: TLS 完成；隨後流量是 VLESS over TLS 1.3
    else 認證失敗（任意其他 client 或 prober）
        S->>A: TCP connect apple.com:443
        S->>A: Forward 該 ClientHello 原封不動
        A->>S: ServerHello + ...
        S->>C: Forward ServerHello + ...
        Note over C,S,A: 此後 server 是純 byte 轉發器
        Note over A: Probe 看到真 Apple 回應
    end
```

**關鍵點**：
1. **Server 端「無 cert 私鑰」**：server 不擁有 apple.com 的 cert 私鑰；ServerHello/Certificate 內容是從 apple.com 真實抓來的（部分由 server 啟動時去探測 apple.com 並 cache）。但 **server 無法產生符合 apple.com 私鑰簽名的 CertificateVerify**。**這個矛盾怎麼解？** 見下節。
2. **Auth key 隱藏在 ClientHello.key_share**：client 把 auth pk 嵌在 key_share extension 內（具體：mutates 一個 ECDH key share 的部分 bytes），server 提取後算 X25519。
3. **Fallback 通過 in-band TCP forward**：認證失敗（auth_key 不對）→ 直接把該 TCP 流轉給真 apple.com。**TLS session 從 client 視角看是直接跟 apple.com 完成**。

### 3. CertificateVerify 的處理：REALITY 的密碼學技巧

這是 REALITY 最 subtle 的部分。

**問題**：TLS 1.3 中 server 必須產生 `CertificateVerify`，是用 cert 私鑰簽 transcript hash。Server 沒有私鑰怎麼簽？

**答案**：REALITY **不簽**真實 cert 私鑰，而是 **用 ephemeral X25519 derived key 簽**，並 mutate 對應的 cert 來 match。

具體（從 `reality.go:108-180` 推導）：
1. Server 啟動時去 apple.com 抓真實 cert，記住 `subject`, `SAN`, `issuer`, `validity`。
2. Handshake 時 server 生成一個 **mutated cert**：保留 apple.com 的 subject 與 SAN，但 **替換 public key field** 為自己的 ephemeral pk（由 `auth_key` 派生）。
3. Server 用對應的 ephemeral sk 簽 CertificateVerify。
4. Client（合法 REALITY client）用相同 `auth_key` 派生相同 ephemeral pk，驗證 CertificateVerify pass。Client 知道 cert 是 mutated 的，但不在 cert chain 上做 CA 驗證——它信任 `auth_key`。

**對 active probe 的意義**：
- 普通瀏覽器（任意 probe）連線時，看到 ServerHello + mutated cert → **cert chain 驗證失敗**（issuer 簽名對不上 mutated pk）→ TLS alert / disconnect。
- **這就是 fallback 觸發的時機**：server 偵測到 client 認證失敗（auth_key 不對），**從 TLS hello phase 開始**改用 transparent forward，client 看到的是真 apple.com 的 cert chain。

**等等——這怎麼可能無縫切換？** Server 必須在送出 ServerHello 前就決定是否要 fallback。從 ClientHello 中 server 已能從 `key_share` 提取 auth pk 並驗證。所以 fallback 決策在 **server 送出第一個 byte 之前** 做。

> **設計優雅之處**：fallback 在 TLS 第一個 server-side byte 之前發生。從 probe 角度看，**沒有任何「REALITY 路徑」的痕跡——它從未進入 REALITY 模式**。

### 4. SNI 選擇：為何 apple.com / microsoft.com / cloudflare.com 而非 example.com

REALITY 對 SNI 有強烈建議（XTLS docs）：

| 條件 | 為什麼 |
|---|---|
| TLS 1.3 only | REALITY 設計需要 TLS 1.3 record 結構 |
| 不在 GFW 黑名單 | 否則 SNI-based block |
| 大量使用、有 H2/H3 | 隱藏在熱門 SNI 池中 |
| Cert 是 publicly 可探測的 | server 啟動時要去抓 |
| 不要求 client cert | mTLS site 不行 |

具體推薦：`www.microsoft.com`、`www.cloudflare.com`、`addons.mozilla.org`、`yahoo.com`。**不推薦** `apple.com`（會 301 redirect）、`google.com`（IP 池太多變）、自己的 domain。

### 5. REALITY 仍有的漏點

#### (a) IP-SNI ASN 對應

如 lesson 9.3 所言，client 連 IP_S（你的 VPS）但 SNI=`microsoft.com`。GFW passive 觀察：
- 是否 `microsoft.com` 真的 host 在 IP_S 對應的 ASN？

對策（部分）：
- **使用 CDN-fronted 域名**：CloudFlare/Fastly 之類，因為這些 CDN 服務海量 site，**任何 IP/SNI mismatch 都是常見的**——例如 Apple、Microsoft 部分服務也走 Akamai。
- 但即便如此，VPS 的 ASN 多半不是 CDN 的 ASN。

#### (b) TLS-in-TLS 形態 leak

REALITY 之後的 application data 是「**直接用 TLS session key 加密 VLESS frame**」，**不是真正 TLS-in-TLS**——只有一層 TLS。所以 record 結構是真實 TLS 1.3 record。

但 **application data 的 byte 分布** 仍是 VLESS frame 不是 HTTP/2 frame。一個訓練在「真實 microsoft.com 瀏覽流量」上的 traffic-shape classifier 仍可能識別。

對策（部分）：
- **XTLS-Vision**：RPRX 的後續工作，添加流量 padding 與 splice 機制，讓 application data 在 byte-pattern 上更接近 HTTP/2。但這是 heuristic，無 guarantee。

#### (c) 中央化 cert cache 的 stale 問題

Server 啟動時抓 apple.com cert 做 mutation。若 apple.com cert 更新（每 30-90 天），server cache 過期，mutated cert 中的 subject/SAN 與當前 apple.com 不一致。Client 如何驗證 mutated cert 仍 match 真實 apple.com？

實作做法：client 也 **不檢查 cert chain**，而是用 `auth_key` 簽 ServerHello.random 來認證。**Cert 內容對 client 而言只是 cover——它不驗證真 issuer**。

但這意味著 **任何 cert 都可以**——server 完全可以送一個 fake cert 給 client，client 不會發現。**這暴露對 client 的 anti-MITM 弱點**：若 GFW 做 MITM，REALITY client 不會知道。但 GFW 通常不做 MITM（會破壞 TLS 信任），所以這個攻擊路徑罕見。

#### (d) Server-side IP 暴露

Client 必須知道 server IP（不能 DNS lookup `microsoft.com`，那會去真 Microsoft）。所以 client 配置中包含 server IP。**Client IP 一旦曝光，server 立即可被掃描**：
- ZMap 全網 port 443 掃描。
- 對每個 IP 嘗試 REALITY handshake with random auth_pk。
- 若 server 反應與真 microsoft.com 不同（時序、ServerHello 差異）→ 標記為 REALITY server。

**這就是 [[frolov-probe-resistant-ndss20]] 級威脅在 REALITY 上的版本**。對策：server 對 wrong-auth_pk 必須 transparent-forward microsoft.com，**byte-for-byte**，包括 timing。Xray 實作有沒有完美做到？社群尚無公開驗證。

#### (e) `key_share` mutation 的可檢測性

REALITY client 把 auth_pk 嵌在 ClientHello.key_share 中。具體做法：用一個 X25519 key share，**前 32 bytes 是 client 的真 ephemeral pk，但某些 byte 位置被 auth_pk 的部分覆蓋**——這是 RPRX 的設計選擇。

**問題**：如果 GFW 收集大量 ClientHello 並做統計分析，REALITY 的 key_share entropy / 結構是否與正常 ClientHello 有微小但可區分的差異？目前無公開 measurement，但這是一個 open research question。

> **2025-08 一個 The Register 報導**（基於 GFW.report 觀察）指出 GFW 在某段時間對 QUIC 與 TLS 都嘗試「異常 ClientHello」識別。REALITY 是否在這個 surface 內被影響，未知。

### 6. REALITY 與 ECH 的對比

ECH (Encrypted ClientHello, RFC 9460 in progress) 的目標：把 inner ClientHello（含真 SNI）加密在 outer ClientHello 內。

REALITY vs ECH：

| 維度 | REALITY | ECH |
|---|---|---|
| 標準化 | 無，社群 | IETF 標準化中 |
| 部署 | 自己控 server | 需要 CDN 支援 |
| 對 GFW 的位置 | 已部署，被容忍 | 部分被 GFW 干擾（2024 IETF mailing list） |
| Cert 假設 | 借用真 cert，無私鑰 | 真實 cert + ECH key 公佈在 DNS HTTPS RR |
| 對 IP-SNI mismatch | 仍有 | CDN 自然有 |
| 對 active probing | 完美 fallback | 視 CDN 配置 |
| 對 ML traffic shape | 有 leak | 有 leak |

REALITY 的勝出在 **「不需要 CDN 合作」** 與 **「fallback 完美」**。ECH 在 **「標準化、普遍部署、有正當性」**。

我們協議設計（Part 11）應該 **同時利用 ECH（當可用時）與 REALITY-style fallback**。

### 7. 為何 2026 中國尚未大規模封 REALITY

公開觀察（OONI、GFW.report、社群）的綜合判斷：

1. **能識別的方法存在**（IP-SNI 對應、active probe 對 corner case TLS handshake、ML 流量形態），但 **目前未大規模部署**。GFW.report 與 v2fly 社群均未報告 systematic block。
2. **可能原因**：
   - **collateral damage 太大**：REALITY 借用的 SNI（microsoft、apple、cloudflare）都是熱門站。Block 任何一個會炸掉大量真實流量。
   - **誤封代價**：對 microsoft.com 做 SNI 黑名單會影響 Windows update。GFW 不會這麼做。
   - **政策優先序**：GFW 對 SS、VMess、obfs4 都有歷史「優先打擊」。REALITY 是後來者，且使用者基數雖大但「**個人翻牆」性質**，不是商業大規模 service。
3. **Iran 的情況不同**：Iran 對 collateral damage 容忍度更高，已觀察到 REALITY 被間歇性影響。**這是「對手不同」的明證**——我們設計協議要考慮**對手譜系**（China conservative vs Iran aggressive vs Russia variable）。

## 與我們協議設計的關聯

REALITY 給我們的 design lessons：

1. **「Handshake stealing」是有效 fallback pattern**。但要徹底——必須 byte-for-byte forward，包括 timing。
2. **Auth 隱藏在 ClientHello 內** 是 viable，但 mutation 必須 entropy-correct。Part 11.4 詳。
3. **Cert chain 在 client 端的處理要設計清楚**。REALITY 的「不檢查 cert chain，只認 auth_key」雖然簡單，但對 MITM 弱。我們可以加 `auth_key` 簽 cert hash 之類，增強強度。
4. **不要依賴特定 cover SNI**——SNI 列表應該可動態更新、client 自己 detect 哪些 SNI 在當地未被 block。
5. **traffic-shape mitigation 是 REALITY 沒解決的問題**，必須補。Part 10 整段處理。

## 動手

**任務**：在自己的 VPS 上部署 REALITY，並對它做幾種 active probe，觀察 fallback 行為。

> ⚠️ 同樣用臨時 instance。不要 probe 你的生產 REALITY。

1. Xray-core 部署一個 REALITY server，cover SNI = `www.cloudflare.com`，dest = `www.cloudflare.com:443`。
2. 從境外 host（不是中國，避開 GFW 影響）：
   - **Probe A**：用 `openssl s_client -connect IP_S:443 -servername www.cloudflare.com`，觀察 ServerHello、cert、handshake 結果。對比直接 `openssl s_client -connect www.cloudflare.com:443`。
   - **Probe B**：用 `curl https://IP_S` 連線（無 SNI 對應）。
   - **Probe C**：用 nmap `--script ssl-enum-ciphers` 對 IP_S。
   - **Probe D**：用 `tlsfuzzer` 做一些 corner-case probes（TLS 1.2 fallback、bad SNI、empty extensions）。
3. 記錄每個 probe 的 (response time, response bytes, observed cert, alert)。
4. 對比 server-side log（Xray debug-level）：哪些觸發了 fallback？哪些被當作 valid client？哪些 timing 是異常的？

**分析重點**：
- ServerHello 的 random 32 bytes 是否真隨機？entropy test。
- 從建立 TCP 連線到 server 第一個 byte 的 latency 是否 與 直接連 `www.cloudflare.com` 相近？任何延遲都可能是 fingerprint。
- 對 `Probe D` 的 corner case，REALITY 是否 forward 給 cloudflare.com 並且 forward 完整？

## 研究級補遺

### 1. 學界詞彙

| 中文 | 學界標準 | 說明 |
|---|---|---|
| 握手盜用 | **handshake stealing / borrowing** | server 模擬 cover server 的 TLS handshake |
| 派生認證 | **derived authentication key** | 從 ECDH 共享 secret 派生 auth key |
| 透明轉發 | **transparent forwarding / SNI-based reverse proxy** | 失敗時 forward 給真 backend |
| 突變憑證 | **mutated certificate** | 修改 cert 公鑰 field 配合 ephemeral key |
| 共驅憑證 | **cert chain bypassing** | client 不驗 cert chain，只用 auth key |

### 2. 對手分類學精化

針對 REALITY 的對手：
- Capability: passive ClientHello stats + 主動 TLS probe + IP/SNI 對應 + traffic shape classifier + ZMap-style scan。
- Limitation: 由於 fallback 是真實 Cloudflare/Microsoft 流量，**任何打擊 REALITY 的方法都伴隨對 cover site 的 collateral damage**。這是 REALITY 的核心防禦。
- 中國目前 cost-benefit 不支持大規模 block；Iran 容忍 collateral damage。

### 3. 形式化定義

REALITY 的安全宣稱（informal）：

對 PPT 對手 $\mathcal{A}$：
- $\Pr[\mathcal{A} \text{ 識別 REALITY 流量 } | \text{ 只有 ClientHello}] - \Pr[\mathcal{A} = 1 | \text{ 真實 cover site 流量}] \leq \mathsf{negl}(\kappa)$
  （前提：client 的 uTLS profile 與真 Chrome 一致）。
- $\Pr[\mathcal{A} \text{ 識別 REALITY server } | \text{ 任意 active probe}] - \Pr[\mathcal{A} = 1 | \text{ 真實 cover site server}] \leq \mathsf{negl}(\kappa)$
  （前提：fallback 完美，server 是 byte-for-byte forward 包括 timing）。

**注意**：流量形態（application data 階段）**不** 在 REALITY 的安全宣稱範圍。它只承諾 handshake unobservability。

### 5. 我們協議的座標 / 取捨

| 設計選擇 | 學 REALITY | 改進 |
|---|---|---|
| Auth 載體 | ClientHello.key_share | 同；考慮 ECH inner SNI 雙重隱藏 |
| Cert 處理 | 借用真 cert 不簽 | 簽 cert hash 增強 MITM 阻力 |
| Fallback | TLS-layer forward | 同；加 timing-precise mirror |
| SNI 池 | 靜態 | 動態 + client side selection |
| Traffic shape | 不處理 | 加 padding / pacing (Part 10) |

### 6. 開放問題

1. **ClientHello key_share mutation 的 entropy footprint**：嚴格的統計學能否區分 REALITY 與真 Chrome 的 key_share 分布？
2. **Active probe budget**：對手用多少 probe 才能以 95 % 置信度識別 REALITY server？目前無公開估算。
3. **2024+ GFW 對 ECH 的態度** 變化是否會 carry 到 REALITY 上？
4. **服務器 timing fingerprint**：REALITY server 在 fallback 路徑上的 latency 與真 backend 的差異，能否 ZMap 級別測得？
