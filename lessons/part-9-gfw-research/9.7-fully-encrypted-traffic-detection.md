## 動機

[[wu-fep-detection]] 是 **2023 年最重要的審查研究論文**（之一）。它證明：

> 2021 年 11 月起，GFW 開始 **純被動** 識別「全加密協議」（FEP）流量，不需 active probing。識別只用 **5 條** byte-level heuristic，且 **誤封率 0.6%**（被 IP 範圍 + 機率限制降到 ~0.04 % 實際誤封）。

這對協議設計界是 **paradigm-shift**：
- 過去 10 年 SS / VMess / obfs4 賭「看起來像隨機 = 不可辨識」這個假設。
- Wu 2023 用 6 個月實證打爆此假設。**任何純加密、無 plaintext header 的協議，都會被打**。

研究級目標：把 5 條規則拆到「**寫程式時能直接 check**」的精度。每條規則拆三段：
1. 規則確切是什麼（byte-level）。
2. 為什麼這條規則 work。
3. 怎麼 evade（已在 shadowsocks-rust 等部署）。

> **Failure framing**：論文最後一節（§8）給出 5 個 circumvention strategies，到 2023-02 仍 work。**2026-05 是否仍 work？** GFW 在 2023-2024 間數次升級規則，社群觀察到部分 evasion 已失效。我們會在「2026 更新」段落補充當前狀態。

## 核心概念

### 1. 為什麼 GFW 從 active 改為 passive

回顧 lesson 9.2/9.6 的 active probing pipeline：
- 計算成本：低（per probe O(1)）。
- 延遲：高（被動觸發 → probe 延遲秒至分鐘）。
- 精確度：高（probe 確認後 false positive 接近 0）。

但 active probing 不能 scale 到 **所有 internet 流量**：
- 每秒新建 TCP 流數量 ~10^9 級。即使只對 1% 流量 probe，也是 10^7 probe/s。
- Probe pool 有 IP 預算上限。

純被動 detection 的成本/收益不同：
- 每包成本：O(L)（讀首 N byte）。
- 延遲：~0（line-rate）。
- 精確度：依靠 heuristic 質量。

**Wu 2023 的核心發現**：GFW 願意用更低精度（0.6 % 誤封率）換 line-rate detection，並用 **IP-範圍限制 + probabilistic blocking** 把 collateral damage 控制在 ~0.04 %。

### 2. 5 條 exemption rules（精確版）

GFW 用 **OR-of-exemptions** 邏輯：**任何一條** 命中即 **放行**；**全部不命中** → 標記為 FEP → blocking。

#### Ex1: Popcount/byte（entropy proxy）

```
popcount(byte) = byte 中 1 bit 的數量 ∈ [0, 8]
mean_popcount = sum(popcount(b) for b in first_payload) / len(first_payload)

if mean_popcount ≤ 3.4 OR mean_popcount ≥ 4.6:
    PASS（放行）
```

**為何 work**：
- 真隨機 byte 的 popcount 期望 = 4（平均每 byte 有 4 bit 是 1）。
- 普通協議的 plaintext 含 zero bytes、ASCII bytes 等 popcount ≠ 4 的 byte → 偏離 4。
- SS-AEAD 等 FEP 的首包接近真隨機 → popcount ≈ 4。

**Evade**：
- 在加密 byte 中插入 padding bytes 讓 mean popcount 偏離。
- 具體：把首 N byte 中 25% 改為 `0x00`（popcount=0）或 `0xff`（popcount=8）→ mean drops below 3.4 或 above 4.6。
- **shadowsocks-rust 已部署** `prefix` 模式，在首段前 prepend 一個 user-defined byte sequence，內容可以是 ASCII string 或 zero-padded。

#### Ex2: First 6 bytes printable

```
all(0x20 <= b <= 0x7e for b in first_payload[:6])  → PASS
```

**為何 work**：
- HTTP 首 6 byte 是 `GET / `、`POST /`、`HEAD /` 等 → 全 printable。
- SSH 首 N byte 是 `SSH-2.0-...` → printable。
- IMAP/SMTP/POP banner 都 printable。
- FEP 首 6 byte 隨機 → expected 機率全 printable ≈ (95/256)^6 ≈ 0.018 → 通常不滿足。

**Evade**：
- 首 6 byte 強制為合法 ASCII（如 `HTTP/1` 或自選字串）。
- 後續 byte 不受此規則限制。
- **trojan-rust 與 shadowsocks-rust 的 `prefix: "HTTP/1.1"` 配置即此目的**。

#### Ex3: > 50% printable

```
printable_count = sum(1 for b in first_payload if 0x20 <= b <= 0x7e)
if printable_count / len(first_payload) > 0.5: PASS
```

**為何 work**：類似 Ex2 但比例化。HTTP request line + headers 通常 100% printable。

**Evade**：增加 printable byte 比例至 > 50%。**注意與 Ex2 互補**：Ex2 要求前 6 byte 連續 printable，Ex3 要求 50% printable 但可不連續。**滿足 Ex2 也常滿足 Ex3**。

#### Ex4: 連續 ≥ 20 printable byte

```
max_run = max(連續 printable byte 的長度)
if max_run > 20: PASS
```

**為何 work**：HTTP header value（如 `User-Agent: Mozilla/5.0 ...`）通常有 > 20 連續 printable。FEP 隨機 byte 連續 printable run 期望 < 5。

**Evade**：插入 20+ byte 的 printable string，例如 `Host: example.com` 之類。

#### Ex5: 已知協議 prefix 匹配

```
if first_payload[:N] matches:
    TLS (e.g. \x16 \x03 [\x00-\x09] \x__ \x__ \x01 ... ClientHello)
    HTTP (GET / POST / HEAD / OPTIONS / PUT / DELETE / TRACE / CONNECT / PATCH)
    or other recognised protocol → PASS
```

**為何 work**：明顯 plaintext header → 是已知協議。

**Evade**：偽裝成 TLS ClientHello 或 HTTP request。**Trojan、VLESS+REALITY 走這條**。

### 3. Blocking behaviour 細節

論文 §3-4：

- **觸發後 IP 封鎖**：標記為 FEP 後，封鎖 `(client_IP, server_IP, server_port)` 三元組 **180 秒**（residual）。180 秒後嘗試再連可能再次觸發。
- **只看 client→server 第一個 data packet**：TCP handshake 完成後的第一個 payload。**SYN 與 SYN-ACK 不算**。
- **TCP 重組無**：GFW 純粹看「第一個 segment 內容」，**不重組**——若 client 把首段分成多 segment，GFW 只看第一個。這就是 `brdgrd` 對 SS 仍有效的原因延伸。
- **僅 inspect TCP**：UDP 不在範圍（這個系統不打 UDP）。**QUIC 不受此影響**——但 lesson 9.5 的 QUIC SNI filter 是另一套系統。
- **Probabilistic blocking**：對符合 trigger 的流，**只以 ~26.3% 機率封鎖**。理由（推測）：減少誤封 collateral damage + 降低系統負載。
- **IP 範圍限制**：**只對 ~26% IP** 啟用 detection。具體：DigitalOcean、Linode、Alibaba、Constant 等熱門 VPS 機房 IP 段。Cloudflare、Akamai 等 CDN IP 段 **不啟用**。

### 4. 規則的設計合理性分析

從攻防角度看 5 規則：

| 規則 | TP（真 FEP 被偵測） | FP（誤偵測正常流量） | 對手成本 |
|---|---|---|---|
| Ex1 popcount | high（SS first segment 全隨機） | mid（部分二進位協議如 BitTorrent） | low（加 padding） |
| Ex2 前 6 byte | high | low | low（加 prefix） |
| Ex3 50% printable | mid | mid | low |
| Ex4 連 20 byte | mid | high（很多協議有長 ASCII） | low |
| Ex5 prefix match | high | very low（明確協議特徵） | high（必須完整模仿） |

**OR 邏輯**：對手只需滿足 **任一條** 即放行。所以 evade 最便宜的是 Ex2 + Ex4（加個 HTTP-like prefix）。**但 Ex5 是最 robust**——一旦 GFW 升級用 ML 分類器看 entropy 分布，Ex2/3/4 可能仍被打，但 Ex5 因為**整個流被當作 HTTP/TLS 處理**，最難打。

### 5. 第一個 packet 之後的「自由」

論文 §3-4 反覆強調：**只看 first packet**。這給 evasion 留了大空間：

- First packet 偽裝為 TLS ClientHello（任意內容，可以是 hand-crafted）→ 過 Ex5。
- Server 對偽 ClientHello 回應任意 ServerHello（不必合法 TLS）。
- 之後 application data 完全自由——隨機 byte、任何 cipher 都可以。

**這是當前主流 evasion path**。Shadowsocks-2022 規範、shadowsocks-rust prefix 模式都基於此。

**但 prepare for the upgrade**：如果 GFW 升級到 **看前 N 個 packet** 或 **重組 TLS handshake** 並 verify completion，此 evasion 就失效。**那一天 REALITY 風格的「真實完整 TLS handshake」變成必須**。

### 6. 與 SS detection 系統的關係

論文證明：FEP detection 與 lesson 9.2 講的 SS-specific detection 是 **兩個獨立但用相同 traffic-analysis algorithm 的系統**：

- SS-specific detection（2019, [[alice-bob-carol-ss-imc20]]）：用 length mod 16 + entropy + active probing。
- FEP detection（2021-11, [[wu-fep-detection]]）：用 5 條 byte-level heuristic + IP 範圍 + 機率 + 純被動。

兩個系統的關係：
- FEP detection **不依賴 active probing**。
- 但 **active probing 系統仍存在**——對通過 FEP detection 但仍可疑的流可再 active probe。
- 兩系統的 underlying algorithm 似乎共用——都是 byte-level heuristic。

**對抗論意義**：evade FEP detection 不等於 evade everything。可能仍要 evade active probing。

### 7. 已部署的 evasion 策略（2022-2026）

社群在 2022 後部署的 evasion：

**(a) Shadowsocks-2022（規範）**：
- 標準化 prefix 機制。
- 預設 prefix 為 `HTTP/1.1 200 OK\r\n` 之類 → 過 Ex2/Ex4。
- 後續 byte 全 AEAD 隨機。

**(b) shadowsocks-rust 的 `prefix: ...`**：
- 配置中可指定 prefix bytes。
- 用戶常用 `[0xff] * 100` 或自選 ASCII。

**(c) VMess AEAD header masking**：
- VMess 2022+ 把 AEAD header 改為「**looks like TLS ClientHello bytes**」。
- 開頭 `\x16 \x03 \x01 ... \x01` → 過 Ex5。

**(d) Outline / SSL prefix mode**：
- Outline 1.6+ 加 SSL handshake prefix。

**(e) Hysteria2 / TUIC（UDP-based）**：
- 不受此系統影響（規則只 inspect TCP）。

### 8. 2026 更新（基於 GFW Report 與社群觀察）

到 2026-05 的可觀察狀況：

1. **5 條規則仍有效**——Wu 2023 列出的 evasion 策略多數仍 work。
2. **IP 範圍可能擴大**——OONI 2024 觀察到部分 CDN IP 段也被 inspect（具體哪些未公開）。
3. **PROBABILISTIC blocking 持續**——但 probability 可能調整（2024 部分觀察到 ~40% 而非 26.3%）。
4. **新規則 candidates**——GFW.report 2024 blog post 暗示 GFW 在實驗 **「前 2 packet 統計」** detection（未確認）。

## 與我們協議設計的關聯

從這篇論文抽取的 **強制性** 約束：

1. **首包必須過 Ex5**——也就是首包必須是 plausible TLS ClientHello 或 HTTP request。**不能只滿足 Ex1/2/3/4，因為這些是 weak evasion**。
2. **首包不應依賴後續 packet** 才能被識別為合法。即「首包 stand-alone」必須通過 detection。
3. **針對 GFW 規則升級的 forward-compatibility**：
   - 如果未來 GFW 重組首 N packet，**前 N 個 packet 加總** 必須仍像 TLS handshake。
   - 如果未來 GFW verify TLS handshake completion，**必須完成真實合法 TLS handshake**（REALITY 路徑）。
4. **IP 部署考量**：CDN IP 段可能更安全，但 IP 來源不能 hard-code，必須運維可變。
5. **UDP 路徑作為輔助**：當 TCP 路徑被打時 UDP 仍是 escape hatch，但 lesson 9.5 揭示 UDP 也在被加強掃描。

Part 11.4「首包設計」會以這 5 條為硬約束。

## 動手

**任務一**：實作 5 條規則 detector，並用真實 traffic 測。

1. Python 程式：讀取 `.pcap`，對每個 TCP flow 提取首包 payload，套用 5 條規則，輸出 (flow_id, exempt_rules_matched, would_be_blocked)。
2. 測試資料：
   - 你的 SS / VMess / Trojan traffic capture。
   - HTTPS browsing traffic capture。
   - 各種其他協議 (SSH、IMAP、DNS-over-TCP) capture。
3. 計算 TP rate（FEP correctly identified）與 FP rate（normal traffic mis-identified）。

骨架（放到 `assets/code/fep-detector/detector.py`）：

```python
from scapy.all import rdpcap, TCP, IP

def popcount_mean(payload: bytes) -> float:
    if not payload: return 0
    return sum(bin(b).count("1") for b in payload) / len(payload)

def all_printable(b: bytes) -> bool:
    return all(0x20 <= x <= 0x7e for x in b)

def is_printable(x: int) -> bool:
    return 0x20 <= x <= 0x7e

def max_printable_run(payload: bytes) -> int:
    best, cur = 0, 0
    for x in payload:
        if is_printable(x):
            cur += 1
            best = max(best, cur)
        else:
            cur = 0
    return best

def matches_tls(payload: bytes) -> bool:
    return (
        len(payload) >= 6
        and payload[0] == 0x16          # Handshake
        and payload[1] == 0x03          # SSL/TLS major
        and 0x00 <= payload[2] <= 0x09  # minor
        and payload[5] == 0x01          # ClientHello
    )

def matches_http(payload: bytes) -> bool:
    for verb in (b"GET ", b"POST ", b"HEAD ", b"OPTIONS ", b"PUT ", b"DELETE ", b"TRACE ", b"CONNECT", b"PATCH "):
        if payload.startswith(verb):
            return True
    return False

def is_exempt(payload: bytes) -> list[str]:
    matched = []
    p = popcount_mean(payload)
    if p <= 3.4 or p >= 4.6: matched.append("Ex1")
    if len(payload) >= 6 and all_printable(payload[:6]): matched.append("Ex2")
    if sum(is_printable(x) for x in payload) / max(1, len(payload)) > 0.5: matched.append("Ex3")
    if max_printable_run(payload) > 20: matched.append("Ex4")
    if matches_tls(payload) or matches_http(payload): matched.append("Ex5")
    return matched
```

**任務二**：用 detector 跑你 Clash / sing-box 的 outbound 流量，**哪些 outbound 會被你模擬的 FEP detector 攔截**？

預期：
- VLESS+REALITY → Ex5 命中（過）。
- 純 SS-AEAD（無 prefix）→ 全不命中（被攔）。
- SS-2022 with prefix → Ex2/Ex4 命中（過）。
- Hysteria2 over QUIC → 不適用（UDP）。

## 研究級補遺

### 1. 學界詞彙

| 中文 | 學界標準 | 說明 |
|---|---|---|
| 全加密協議 | **fully-encrypted protocol (FEP)** | 每 byte 加密、無 plaintext header |
| 排除規則 | **exemption rule** | 命中即放行 |
| 線速偵測 | **line-rate detection** | 在 Tbps 級流量上即時處理 |
| 機率封鎖 | **probabilistic blocking** | 以一定機率執行 block |
| 殘留封鎖 | **residual censorship** | 觸發後一段時間繼續封 |
| 客製化 prefix | **customizable payload prefix** | 由 client 加在 ciphertext 前的 plaintext bytes |

### 2. 對手分類學精化

GFW FEP 偵測對手：
- **Capability**: pure on-path passive, line-rate, stateless per-flow.
- **Detection algorithm**: 5 byte-level heuristic, OR-of-exemptions.
- **Coverage**: ~26% of source IP ranges (popular VPS), TCP only.
- **Blocking**: 26.3% probabilistic, 180 s residual.
- **Active probing system**: 並行，用相同 traffic-analysis algorithm。
- **Adversary upgrade trajectory** (預測): multi-packet inspection → ML on byte sequence → in-flight TLS handshake verification → eventually L4 cryptographic indistinguishability requirement。

### 3. 形式化定義

**FEP indistinguishability problem**：

定義一個 PPT 對手 $\mathcal{A}$，給定 TCP flow first packet $P \in \{0,1\}^L$，輸出 bit $b$。

$$
\mathsf{Adv}^{\mathsf{FEP}}_{\mathcal{A}}(\kappa) = \Pr[\mathcal{A}(P) = 1 | P \sim \text{FEP}] - \Pr[\mathcal{A}(P) = 1 | P \sim \text{NormalTCP}]
$$

GFW 的 5-rule classifier 達到 $\mathsf{Adv} \approx 1.0$ 對 vanilla SS-AEAD。

協議 $\Pi$ 滿足 $\epsilon$-FEP-indistinguishability iff 對 GFW 等價 classifier：$\mathsf{Adv} \leq \epsilon$。

vanilla SS：$\epsilon = 1.0$。
SS-2022 with HTTP-prefix：$\epsilon \approx 0$ on Ex2/Ex4。
VLESS+REALITY：$\epsilon \approx 0$ on Ex5。
