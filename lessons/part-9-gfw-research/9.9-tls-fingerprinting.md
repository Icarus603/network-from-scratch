## 動機

TLS ClientHello 是 censor 可看到的**唯一 plaintext** TLS message（在 ECH 部署前）。它的內容（cipher list、extensions order、supported groups、ALPN）構成一個 **per-implementation fingerprint**——同一個 Chrome 版本 + 同一作業系統 + 同一 build = 同一 fingerprint。

研究級目標：
1. 把 JA3 / JA4 系列徹底拆——byte-by-byte 看 ClientHello 怎麼被 hash 成 fingerprint。
2. 理解 [[frolov-utls-ndss19]] 為何 uTLS 是現代 circumvention 必備。
3. 把這套知識 transfer 到 server-side（JA3S、JA4S、JARM）與 QUIC（JA4 over QUIC）。
4. 為 lesson 9.11 testbed 中部署 nDPI/Zeek 的 TLS detector 鋪路。

> **Failure framing**：TLS fingerprint「絕對 unique」是個誤解。它是 **「per-implementation」** 的——同樣 Chrome 92 大致同一 fingerprint，但 corner case 與 GREASE 隨機性讓 hash 有 small variance。一個「mimic Chrome 92」的 circumvention 工具要做到 **byte-identical** 才能完美藏身——這是 uTLS 的核心承諾，也是其複雜度的來源。

## 核心概念

### 1. ClientHello 結構回顧

RFC 8446 §4.1.2:

```
ClientHello {
    ProtocolVersion legacy_version = 0x0303;     // TLS 1.2 for compatibility
    Random random;                                 // 32 bytes
    opaque legacy_session_id<0..32>;              // for TLS 1.3 session resumption indicator
    CipherSuite cipher_suites<2..2^16-2>;         // list of supported ciphers
    opaque legacy_compression_methods<1..2^8-1>;  // always [0x00] in TLS 1.3
    Extension extensions<8..2^16-1>;              // structured extensions
}
```

關鍵：**extension 是 list，順序也是 fingerprint 的一部分**。

GREASE (RFC 8701)：Google 為防止 ossification，要求 ClientHello 在 cipher_suites、extensions、supported_groups、signature_algorithms 等位置 insert **random unused values**（如 `0x0A0A`、`0x1A1A`），這些是「保留位」。GFW 不能用 GREASE bytes 做 fingerprint（會破 RFC），所以 JA3/JA4 都 **strip GREASE before hashing**。

### 2. JA3 規格（2017）

`https://github.com/salesforce/ja3` — Althouse, Atkinson, Atkins (Salesforce, 2017)。

JA3 string format:
```
<TLSVersion>,<Ciphers>,<Extensions>,<EllipticCurves>,<EllipticCurvePointFormats>
```

每個 field：
- `TLSVersion`: legacy_version field decimal (e.g. `771` for `0x0303`).
- `Ciphers`: cipher suite IDs decimal, dash-separated, GREASE stripped.
- `Extensions`: extension types decimal, dash-separated, GREASE stripped.
- `EllipticCurves`: supported_groups extension content.
- `EllipticCurvePointFormats`: ec_point_formats extension content.

JA3 = MD5(JA3 string).

範例 (Chrome 89):
```
771,4865-4866-4867-49195-49199-49196-49200-52393-52392-49171-49172-156-157-47-53,0-23-65281-10-11-35-16-5-13-18-51-45-43-27-17513,29-23-24,0
```

→ `MD5(...) = 6734f37431670b3ab4292b8f60f29984`

**JA3S** 是 server-side 對應，hash ServerHello。

### 3. JA3 的限制

- **MD5**：不 collision-resistant，但 GFW 用 hash equality 不在乎。
- **Hash space 小（128 bit）**：理論上夠，實務上沒問題。
- **Extension order 完全進 hash**：好處是精細，壞處是任何 Chrome minor update 就改 hash。
- **不處理 ECH / Encrypted Extensions / HelloRetryRequest**：2017 設計沒考慮。
- **不可逆**：給定 MD5 你不知道哪些 cipher 在裡面。Debug 困難。

### 4. JA4/JA4+ 規格（2023+）

FoxIO 由 John Althouse 主導，2023 年起發布 JA4 系列。設計修正 JA3 的限制：

**JA4 fingerprint 結構**（example: `t13d1517h2_8daaf6152771_b6f405a00624`）：

```
JA4 = a_b_c

a = 1 char (t/q/d) + 2 char TLS version + 1 char SNI presence + 2 char cipher count + 2 char ext count + 2 char ALPN
b = first 12 chars of SHA256(sorted ciphers, hex, comma-sep)
c = first 12 chars of SHA256(sorted exts ∖ {SNI, ALPN}, comma-sep, + sig algorithms)
```

具體解讀 `t13d1517h2_8daaf6152771_b6f405a00624`:
- `t`: TLS over TCP（`q` = QUIC，`d` = DTLS）
- `13`: TLS 1.3
- `d`: SNI exists
- `15`: 15 cipher suites
- `17`: 17 extensions
- `h2`: ALPN is `h2`（first + last char of first ALPN value）
- `_8daaf6152771`: SHA256(sorted ciphers, hex, comma-sep)[:12]
- `_b6f405a00624`: SHA256(sorted exts excluding SNI and ALPN, then sig algos)[:12]

**設計優勢**：
- **a-b-c 結構** 允許部分 match（只看 `a` 識別 protocol、看 `a_b` 識別 cipher class、全 `a_b_c` 完整 fingerprint）。
- **Sorted hashing** 對 extension order 變化 robust（Chrome update 改順序不影響 hash）。
- **SHA256** 而非 MD5。
- **每 field 人類可讀**：debug 容易。
- **GREASE stripped**：同 JA3。

### 5. JA4 完整家族

| Fingerprint | 對象 | 用途 |
|---|---|---|
| **JA4** | TLS Client | Standard client mimicry detection |
| JA4S | TLS Server | Server-side fingerprint |
| JA4H | HTTP Client | HTTP layer (User-Agent + headers) |
| JA4L / JA4LS | Light Distance | Geographic/network distance proxy |
| JA4X | X.509 Certificate | Cert structure fingerprint |
| JA4SSH | SSH | SSH banner + KEX |
| JA4T / JA4TS / JA4TScan | TCP | SYN/SYN-ACK fingerprint |

### 6. uTLS：偽造 ClientHello 的標準工具

[[frolov-utls-ndss19]] 出生背景：他們在 CU Boulder 量到 Lantern / Psiphon / meek 等工具的 TLS fingerprint **在 11 billion 連線中 prevalence < 0.0003 %**——意即「肉眼可見的 minority」，GFW block 它們 zero collateral damage。

uTLS 解法：**Go's crypto/tls 的 fork，允許 caller assemble 任意 ClientHello**。

具體 API（uTLS）：

```go
import tls "github.com/refraction-networking/utls"

config := &tls.Config{InsecureSkipVerify: true, ServerName: "www.cloudflare.com"}
conn := tls.UClient(tcpConn, config, tls.HelloChrome_102)
err := conn.Handshake()
```

`tls.HelloChrome_102` 是預設 preset，含 cipher list、extension order、padding、key share。caller 也可用 `tls.ClientHelloSpec` 完全自訂。

**uTLS 對齊精度**：到 byte 級。包含 GREASE 位置、extension padding length、key share 是否含 X25519+P256+Kyber 等細節。

**已採用 uTLS 的工具**：Psiphon、Lantern、TapDance、meek、Tor (Snowflake)、Xray-core (REALITY)、sing-box、HTTP/3 客戶端（部分）。

### 7. uTLS 的維護壓力

Chrome 每 6 週 stable release。每次 release 可能改：
- 新 cipher suite（PQ key share）
- Extension order
- ALPN 列表
- Padding length

**uTLS 必須跟上**——否則 mimicry 是 stale 版本，仍有 fingerprint leak。

實際情況（2026-05）：uTLS 有 active maintainer，但 lag 通常 1-2 個 Chrome stable cycle。**社群工具如 v2fly / Xray 通常 pin 到 specific uTLS commit**，每幾個月 bump 一次。

**對協議設計**：必須選擇是否 vendoring uTLS、是否 auto-update、是否提供 admin override（讓 user 在工具更新前手動指定 Chrome version）。

### 8. ServerHello / JA3S / JA4S 對 REALITY 的意義

REALITY server 自己生成 ServerHello（lesson 9.4）。**ServerHello 是 censor 可看到的另一個 fingerprint**：

- 真實 Cloudflare server 的 JA3S：固定 hash。
- REALITY server 模仿的 ServerHello：必須產生**相同 JA3S/JA4S**。

REALITY 實作怎麼做？
1. Server 啟動時去探測真 Cloudflare ServerHello，記錄 cipher、selected_extension、server_random 格式。
2. Handshake 時用相同 cipher、extension list、相同 server_random 結構（雖然 random bytes 不同）。
3. 結果 JA3S / JA4S hash 與真 Cloudflare 相同。

**驗證方法**：用 JA3S/JA4S 工具同時對真 Cloudflare 與 REALITY server 計算 hash → 應該相同。**社群應該定期跑這個 sanity check**。

### 9. JA4 over QUIC

JA4 design 包含 `t` (TCP) / `q` (QUIC) prefix。對 QUIC ClientHello（在 Initial packet 內）：
- 同樣解密 Initial（[[zohaib-quic-sni-usenix25]]）。
- 提取內部 TLS ClientHello。
- 計算 JA4 with `q` prefix。

GFW 完全有能力對 QUIC 流做 JA4 fingerprint。**Hysteria2 / TUIC 用自訂 ALPN 但 ClientHello 其他部分若不 uTLS 化會洩漏**。

### 10. 對抗 fingerprint 的工程實踐

**Best practice for our protocol**:
1. **Client side**: 用 uTLS（或 rustls equivalent），preset = current Chrome stable。
2. **Server side**: 模仿 cover server 的 JA3S/JA4S（REALITY pattern）。
3. **Auto-update**: 把 uTLS preset 編譯到二進位，每月 release 更新一次。
4. **Random selection**: 不要永遠用 Chrome——也用 Firefox、Safari preset 輪換，匿名在 user-population 中。
5. **Per-destination preset**: 連 Microsoft 用 Edge/Chrome、連 Apple 用 Safari、連 GitHub 用 Chrome——配合 destination ASN。

## 與我們協議設計的關聯

從 TLS fingerprint 全景對協議設計的硬約束：

1. **Client must use uTLS**（或同類庫）—— 不可使用 standard library TLS。Part 11.4 寫入硬要求。
2. **Client ClientHello 必須 byte-match 當前 Chrome stable**。維護人員必須 monthly bump uTLS preset。
3. **Server ServerHello 必須 byte-match cover server**（如果是 REALITY-style handshake stealing）。
4. **不要 hard-code 一個 fixed JA3/JA4**——隨 Chrome 更新而 rotate。
5. **JA4SSH、JA4T 也要 考慮**——TCP SYN fingerprint、SSH（若 SSH-cover 路徑）也是 attack surface。
6. **PQ key share rollout**——當 Chrome 預設 enable Kyber768Hybrid 時，uTLS 必須跟上。

### 2. 對手分類學精化

- **Tier 1 (passive)**: hash ClientHello, lookup blocklist of "rare fingerprints"
- **Tier 2 (passive)**: 統計每個 fingerprint 在 dst IP 上的 prevalence → 異常分佈標記
- **Tier 3 (active)**: 看到罕見 fingerprint 就 trigger active probe
- **Tier 4**: 多 layer fusion（JA3 + JA4H + JA4T + IP/ASN cross-reference）

### 3. 形式化定義

定義 **TLS-fingerprint indistinguishability**：

協議 $\Pi$ 是 $\epsilon$-TLS-fingerprint-indistinguishable from cover impl $C$ iff：

$$
\Pr[\mathsf{JA4}(\mathsf{ClientHello}_\Pi) \neq \mathsf{JA4}(\mathsf{ClientHello}_C)] \leq \epsilon
$$

對 Chrome 102 preset uTLS：$\epsilon \approx 0$（exact match）。

但**隨機性影響**：每次連線 ClientHello.random / GREASE position 不同，造成 raw byte sequence 不同，但 JA4 設計上不受 GREASE / random 影響——所以 $\epsilon$ 真的可以 = 0。
