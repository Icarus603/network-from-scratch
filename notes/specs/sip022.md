**Status**: 主流 sing-box / shadowsocks-rust / shadowsocks-libev 全面實作。

## One-line
把 SIP004 的歷史包袱（HKDF-SHA1、0x3FFF chunk cap、無 replay 防護、UDP 無 session）一次清光，並在 EIH 補上單 port 多用戶。Wire format 與 SIP004 **不相容**。

## Cipher Suites
| Method | Key | Salt | Nonce | 必選 |
|---|---|---|---|---|
| `2022-blake3-aes-128-gcm` | 16 B | 16 B | 12 B | required |
| `2022-blake3-aes-256-gcm` | 32 B | 32 B | 12 B | required |
| `2022-blake3-chacha20-poly1305` | 32 B | 32 B（TCP）／24 B XChaCha nonce（UDP） | — | optional，給 ARM / 無 AES-NI |

PSK 必須是 base64-encoded raw bytes，**不再經 EVP_BytesToKey**——使用者的弱密碼漏洞被一刀切除。

## Session Subkey Derivation
```
session_subkey := blake3::derive_key(
    context:      "shadowsocks 2022 session subkey",
    key_material: psk || salt
)
```
EIH 多 PSK 場景另用：
```
identity_subkey := blake3::derive_key(
    context:      "shadowsocks 2022 identity subkey",
    key_material: iPSK_n || salt
)
```
BLAKE3 derive_key 模式取代 HKDF-SHA1；context string 是領域分隔。

## TCP Wire Format
**Request stream**:
```
[salt(16/32 B)]
[enc fixed_request_header(11 B) || tag(16 B)]
[enc variable_request_header(var) || tag(16 B)]
[enc length(2 B) || tag(16 B)]
[enc payload(≤ 0xFFFF) || tag(16 B)]
... (length, payload) repeated
```

**Fixed request header (11 B, big-endian)**:
| 欄位 | 大小 | 值 |
|---|---|---|
| Type | 1 B | `0x00` = client → server |
| Timestamp | 8 B | Unix epoch seconds, u64be |
| Length | 2 B | 接下來 variable header 的長度，u16be |

**Variable request header**:
| 欄位 | 大小 | 說明 |
|---|---|---|
| ATYP | 1 B | SOCKS5 address type |
| Address | var | domain（含 1 B len 前綴）／IPv4 4 B／IPv6 16 B |
| Port | 2 B | u16be |
| Padding length | 2 B | u16be |
| Padding | var | 隨機 bytes |
| Initial payload | var | 第一段業務資料（可選） |

**Response stream**:
```
[salt(16/32 B)]
[enc fixed_response_header(27/43 B) || tag(16 B)]
[enc payload || tag] [enc length || tag] ...
```
**Fixed response header**:
| 欄位 | 大小 |
|---|---|
| Type | 1 B（`0x01` = server → client） |
| Timestamp | 8 B u64be |
| Request salt | 16 / 32 B（綁定到 request 防 cross-session replay）|
| Length | 2 B u16be |

注意 response 的 length 欄位**前置在 header 裡**而非每個 chunk 前——因此第一段 payload 不需要獨立 length chunk。後續 chunk 才回到 `(payload, length)` 對。

**Length cap 提升至 0xFFFF (65535)**：相比 SIP004 的 0x3FFF 提升 4 倍，大幅減少 syscall 次數。

## UDP Wire Format（AES-GCM 變體）
```
[enc separate_header(16 B)]   ← 用 session_id 衍生的 AES-ECB 加密，無 tag
[enc body || tag(16 B)]       ← 用 session subkey + nonce
```

**Separate header (16 B)**:
| 欄位 | 大小 |
|---|---|
| Session ID | 8 B |
| Packet ID | 8 B u64be |

`(session_id, packet_id)` 同時當作 AEAD nonce 的後 12 B（前 4 B 為 0）。Separate header 用 AES block cipher（**ECB，無 tag**）加密以隱藏 session ID 給被動觀察者，body 才走完整 AEAD。

**Client → server message header**:
| Type=0x00 (1 B) | Timestamp (8 B) | Padding length (2 B) | Padding (var) | ATYP (1 B) | Address | Port (2 B) | Payload |

**Server → client message header**（多帶 client session ID）:
| Type=0x01 (1 B) | Timestamp (8 B) | Client session ID (8 B) | Padding length (2 B) | Padding (var) | ATYP | Address | Port | Payload |

**XChaCha20-Poly1305 UDP 變體**：
```
[nonce(24 B)] [enc body || tag(16 B)]
```
XChaCha 的 24 B nonce 大到可以隨機，不需要 separate header trick。

## Replay Protection
- **TCP salt**：server 必須記錄所有收到的 salt **至少 60 秒**；重複即拒絕。Strict mode（SIP022-3）要求記錄到 PSK 完整生命週期。
- **Timestamp window**：±30 秒。`|server_now - client_timestamp| > 30 s` 即拒絕。同時防 clock skew 太大的 active replay。
- **UDP packet ID sliding window**：每個 session 維護 sliding window（建議參考 WireGuard `replay.c` 的 2048-bit window 實作），重複或超出窗口即丟棄。
- Session 至少保留 60 秒，避免合法 client 短暫離線後被當成新 session。

## Padding Rules
- Request variable header **必須**含「payload OR 非零 padding」二者之一。Server 收到「payload 為空 AND padding length = 0」直接拒絕——這設計是為了打破 Frolov 2020 的 entropy fingerprint（純隨機 stream 起手 → server reply → 被識別為 SS）。
- Response header 永遠帶 payload，故不需 padding。
- UDP 允許 optional padding，但要尊重 MTU。

## Session Semantics
- **TCP**：1:1 socket，client 發 request stream，server 發 response stream。每連線新 salt + 新 subkey。
- **UDP**：client 自己生 random session ID，server 以 session ID 而非 source addr/port 路由 → **NAT rebinding / mobile network 切換不會掉 session**。這是相對 SIP004 UDP 的最大改進，也是和 QUIC connection ID 同樣的設計哲學。

## EIH（Extensible Identity Headers，SIP022-2）
- 多用戶情境：每 client 拿到 `iPSK_0 : iPSK_1 : ... : uPSK_n` 的多層 PSK。
- Wire format 在 salt 之後**插入一或多個 identity header**：
  ```
  identity_header_n = AES-ECB(
      key = identity_subkey_n,
      plaintext = blake3::hash(iPSK_{n+1})[0..16]
  )
  ```
  `identity_subkey_n = blake3::derive_key("shadowsocks 2022 identity subkey", iPSK_n || salt)`
- Server 收到後：用自己的 iPSK 衍生 subkey、解開最外層 → 拿到下一層 PSK 的 BLAKE3 hash → 在預先建好的 `hash → user` 表裡查找 → 確認該 user 的 uPSK → 用 uPSK 走 SIP022 主流程。
- UDP 版本還會把 plaintext 與 session_id ⊕ packet_id XOR，避免 ECB 在多包之間出現重複密文 block。
- Response 不帶 identity header（server 已知 user）。

## Key differences vs SIP004（記在腦裡的對照表）
| | SIP004 | SIP022 |
|---|---|---|
| KDF | HKDF-SHA1 | BLAKE3 derive_key |
| Password derivation | EVP_BytesToKey (MD5) | 直接 base64 PSK |
| Chunk cap | 0x3FFF (16 KB) | 0xFFFF (64 KB) |
| Replay protection | 無 | salt window 60 s + ts ±30 s + UDP sliding window |
| UDP session | 無，每包獨立 salt | session ID + packet ID（QUIC-like） |
| Multi-user single port | 無 | EIH 層疊 PSK |
| Header padding | 無 | 強制 payload-or-padding |

## How it informs our protocol design
- **Session ID 與 source addr 解耦**——QUIC / SIP022 / Hysteria2 都這樣做。我們的協議必須沿用，否則行動網路掉線體驗會輸。
- **Padding-or-payload 拒空封包**是廉價但有效的 anti-fingerprint trick，值得抄。
- **EIH 的 identity header 層疊**是一個漂亮的設計：被動觀察者只看到「一坨密文」，server 卻能在 O(1) hash lookup 找到 user。我們的多用戶設計可參考，但要注意 AES-ECB 是 known-plaintext-friendly，多包之間必須 XOR 變化（SIP022 也踩過這坑）。
- **BLAKE3 derive_key context 字串**是領域分隔的好範例——同樣 PSK 用在 session vs identity 衍生不會撞 key。

## Quirks / spec foot-guns
1. **UDP separate header 用 AES-ECB 而非 AEAD** — 故意的：因為要把 nonce（session_id, packet_id）藏起來，又不想為前 16 B 多花 16 B tag。代價：每個 session 的「first block」key stream 是固定的，需 XOR mitigation。
2. **Response header 把 length 塞進 fixed header 末段** — 與 request stream 的 `(length, payload)` chunk 對不一致，初次讀規格容易踩。
3. **`shadowsocks.org/doc/sip022.html` 與 `Shadowsocks-NET/shadowsocks-specs` 兩處的措辭略有差異**——前者較簡，後者是 normative source。實作時以後者為準。
4. **EIH spec 不在 sip022.html**——必須去 Shadowsocks-NET repo 抓 `2022-2-...md`。`shadowsocks.org` 主站長期未補上 EIH 文件，是常見混淆點。
5. **Strict mode（SIP022-3）** 要求 salt 永久記憶到 PSK 換完為止——server 端記憶體成本可觀，多數部署沒開。

## Open questions
- SIP022 仍假設 wire 上看起來是「pure entropy」可被接受。在 GFW 開始普查 entropy stream（Wu et al., USENIX Security 2023 *FEP*）後，純 SS-2022 在 censored network 已不安全，必須再包一層 obfuscation（v2ray-plugin / shadow-tls / REALITY）。我們的協議是否該把 obfuscation 直接做進 wire format，而非外掛？這是 Part 11 設計章節要決的。
- EIH 的 AES-ECB 抉擇雖然 XOR 補丁修了，但仍非首選。是否該直接改用 AES-GCM-SIV 或 deterministic AEAD？trade-off：每包多 16 B tag。
