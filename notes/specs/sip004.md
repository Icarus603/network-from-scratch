# SIP004 — Shadowsocks AEAD Cipher Specification

**Source**: <https://shadowsocks.org/doc/aead.html> (canonical) ／ `shadowsocks/shadowsocks-org/docs/doc/aead.md` ／ `shadowsocks/shadowsocks-org#30`
**Authors**: proposed by @Mygod，design input from @wongsyrone, @Noisyfox, @breakwa11；amended by SIP007（@riobard）。
**Status**: ratified 2017，被 2017–2021 期間所有 mainline / libev / sing-box / xray 採用。**已被 SIP022 (Shadowsocks 2022) 取代**，但仍然是現行 deployed base 的多數。
**Read on**: 2026-05-16（Part 7 預備材料）

## One-line
舊版 Shadowsocks 把 stream cipher 換成 AEAD（AES-GCM / ChaCha20-Poly1305）以阻擋主動探測攻擊（GFW 2015 的 redirect probe），規格只有兩頁但留下兩個歷史包袱：HKDF-SHA1 與 14-bit chunk 上限。

## Key Derivation（"Key Derivation" 節）
- Master key 由密碼經 OpenSSL `EVP_BytesToKey(3)` 產生（MD5 based，**已過時**）。
- Per-session subkey: `HKDF_SHA1(master_key, salt, info="ss-subkey") → subkey`。`info` 字串硬編碼為 `ss-subkey`（無引號）。
- Salt 大小依 cipher：
  - `AES-128-GCM`: 16 B
  - `AES-256-GCM`: 32 B
  - `CHACHA20-POLY1305`: 32 B
- "Salt must be unique through the entire life of the pre-shared master key." → 但規格本身**沒**強制 server 維護 replay 表，這是 SIP022 才補上的洞。

## TCP Chunked Record Framing（"TCP" 節）
連線開頭：`[salt(16/32 B)] || [encrypted chunk] || [encrypted chunk] || ...`

每個 encrypted chunk 結構（密文形式）：

| 欄位 | 大小 | 說明 |
|---|---|---|
| `encrypted DATA_LEN` | 2 B | payload 長度，big-endian u16 |
| `DATA_LEN_TAG` | 16 B | 上面 2 B 的 Poly1305/GCM tag |
| `encrypted DATA` | ≤ 0x3FFF B | payload |
| `DATA_TAG` | 16 B | payload 的 tag |

**0x3FFF (16383) 上限**：規格原文「the higher two bits are reserved and must be set to zero」。也就是 16-bit 長度欄只用低 14 bit。當年的考量是讓 sender 能夠在不增加 framing overhead 的前提下保留 2 bit 給未來擴充——結果這 2 bit 從未被用到，反而變成 throughput 瓶頸（每 16 KB 就要一次 GCM auth + 兩次 syscall）。SIP022 直接拉到 65535。

## Nonce 構造（"Authenticated Encryption/Decryption" 節）
- 12-byte nonce，**初始全零**。
- 每次 AEAD encrypt/decrypt 後 nonce 視為 **little-endian u96** 加 1。
- 每個 chunk 消耗 **2 個 nonce**（一個給 length、一個給 payload）。
- 上行 / 下行各自獨立計數（規格未明寫但實作通行做法；底層保證是 subkey 由不同 salt 衍生）。
- **無 explicit nonce in wire format**——sender / receiver 必須同步狀態，任一封包丟失即整條連線報廢，故只能跑在 TCP 上。

## UDP Format（"UDP" 節）
```
[salt(16/32 B)] || [encrypted payload] || [tag(16 B)]
```
每個 UDP packet **獨立加密**，nonce 全零。代價：salt 占滿包頭，且不能複用 → 小包效率極差。SIP022 把 UDP relay 整個重寫。

## SIP007 amendment
SIP007 補上「sub-key salt 必須來自 CSPRNG」與部分相容性細節，未改變 wire format。

## Limitations / 為什麼需要 SIP022
1. **HKDF-SHA1 + EVP_BytesToKey** — 兩個皆已過時。SHA-1 collision (SHAttered, 2017) 雖未直接破壞 HKDF，但符號上不該再出現在新規格。
2. **無 replay protection** — server 不檢查 salt 重複。Frolov et al., USENIX Security 2020 *Detecting Probe-resistant Proxies* 直接把它列為攻擊面。
3. **0x3FFF chunk cap** — 高速連線下每 16 KB 一次 syscall + AEAD 切換，CPU bound。
4. **UDP 效能差 + 無 session 概念** — 每個 packet 獨立 salt + 32 B 開銷。
5. **無 multi-user / EIH** — 一個 port 只能對應一個 PSK。

## How it informs our protocol design
- SIP004 是「協議設計時把 framing 做太死」的反例：14-bit length 與固定 nonce schedule 都讓未來擴充變難。我們的協議要保留 versioning byte + length 欄位足夠寬。
- HKDF + per-session subkey 的層級是對的，但底層 hash 該用 BLAKE3（吞吐 ~3 GB/s，無 length-extension issue）或 HKDF-SHA-256。
- TCP-only 的 stateful nonce 證明 in-band sequence number 不可省——QUIC / Hysteria2 都帶 explicit packet number 不是浪費。

## Open questions
- 為何 2017 設計時不直接 sequence number 帶 in-band？答案多半是「為了不讓被動觀察者看到任何明文」——但這個目標在 TLS-in-TLS 時代已失去意義，反而招來 0x3FFF 與 stateful 的代價。值得在 Part 11 設計章節重新評估。
