# BLAKE3: One Function, Fast Everywhere
**Venue / Year**: 2020 whitepaper (https://blake3.io)
**Authors**: Jack O'Connor, Jean-Philippe Aumasson, Samuel Neves, Zooko Wilcox-O'Hearn
**Read on**: 2026-05-14 (in lesson 3.3)
**Status**: full PDF (`assets/papers/blake3-2020.pdf`)
**One-line**: 把 BLAKE2 大幅簡化 + 加 Merkle tree 結構 + SIMD 平行 → 達 0.5 cycles/byte AVX-512，比 SHA-3 快 25×；統一 hash / KDF / MAC / XOF 四 mode；2020 年代純軟體 hash 的事實王者。

## Problem
BLAKE2 已經夠快 (~3 c/b) 但仍 strictly sequential——大檔案無法平行 hash。AVX-512 / NEON 在 2018 年代普及但 hash 軟體沒充分用。BLAKE2 有兩個變體 (b/s) 與多個 mode (keyed, salt, tree)，API 複雜。需要：**單一 function、單 API、SIMD 充分、Merkle tree 平行、KDF/MAC/XOF 統一**。

## Contribution
1. **Merkle tree 結構**：input 切 1024-byte chunks，每 chunk 獨立 hash 為 chaining value (CV)；CVs 兩兩 hash 上溯為 binary Merkle tree。
   - **平行**：8 chunks 可同時 hash on AVX-512（每 chunk 用獨立 SIMD lane）。
   - **增量驗證**：給定 root + chunk + proof path，可驗證單個 chunk 不需 full hash。
   - **streaming**：可邊讀邊 hash，保持 partial tree state。
2. **單一 API 四 mode**：
   - hash(m): plain hash
   - keyed_hash(K, m): MAC
   - derive_key(context, KM): KDF (similar to HKDF without Extract step)
   - XOF: arbitrary-length output via squeeze-like
3. **減 round 數 + 簡化 G function**：BLAKE2b 的 G function 略簡，round 從 12 降到 7（BLAKE3 7-round 對比 BLAKE2b 12-round）。
4. **效能**：
   - 1 thread, AVX-512: ~0.5 cycles/byte (~6 GB/s on 3 GHz)
   - 32 threads, AVX-512: ~17 GB/s
   - vs SHA-256 ~5 c/b (no SHA-NI), vs SHA3-256 ~12 c/b

## Method
**Tree structure**：
```text
Input: byte string (any length)
Chunk size: 1024 bytes (16 blocks of 64 bytes each)

For each chunk i:
    chunk_state = compress chunk_i with chunk_chaining
    chunk_CV[i] = chunk_state output

If only 1 chunk: root_CV = chunk_CV[0] with ROOT flag set
Else:
    Build Merkle tree of CVs:
        parent(L, R) = compress_pair(L, R)
    root_CV = top of tree with ROOT flag

Output:
    hash: first 32 bytes of compress(root_CV) with output counter 0
    XOF: concatenate compress(root_CV) with output counter 0, 1, 2, ...
```

**Compression function** 是 BLAKE2 的 G function 變體（7 rounds, 32-bit lanes 但內部 64-bit-friendly arithmetic）。

**Mode separation via flags**：
- CHUNK_START / CHUNK_END / PARENT / ROOT / KEYED_HASH / DERIVE_KEY_CONTEXT / DERIVE_KEY_MATERIAL
- 不同 flag 給不同 domain separation，避免 cross-mode collisions。

## Results
- 廣泛採用於 modern Rust ecosystem：cargo / rustc 內部用 BLAKE3 hash artifacts。
- libsodium、Python (`blake3-py`)、Node.js (`@noble/hashes`) 等都有實作。
- WireGuard 部分 fork 評估換 BLAKE3 (但目前主線仍 BLAKE2s 因 spec stability)。

## Limitations / what they don't solve
- **較新 → cryptanalysis 不如 SHA-2/3 充分**：BLAKE3 2020 公開，至 2026 約 6 年密碼分析；最佳已知 attack 對 round-reduced 5/7-round。Margin 充足但不如 SHA-2/3 那麼 battle-tested。
- **不在 NIST 標準**：FIPS 環境必須用 SHA-2/3。
- **Quantum security**：256-bit output → ~128-bit classical, ~85-bit quantum (BHT)；對 Proteus 我們認為 sufficient。

## How it informs our protocol design
- **Proteus hash agility**：spec 預留升級到 BLAKE3 路徑（hash_id field）。
- **Proteus cover-traffic padding 可選 BLAKE3 XOF**：替代 SHAKE128，效能更好。
- **Proteus v1 default 仍 SHA-256**：TLS 1.3 互通優先；BLAKE3 為 future v2 候選。

## Open questions
- BLAKE3 的 round-reduced cryptanalysis 至 8/10-round 仍 open active。
- Quantum cryptanalysis of BLAKE3 round function 仍少有研究。
- Tree-hash 的 incremental verification 在 protocol design 中的 utility (e.g., Merkle proofs in light clients) 仍 underexplored。
